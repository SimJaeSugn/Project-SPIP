⚠ 전 항목 공통 보안 불변식 (위반 시 게이트 실패)

  • L-1 렌더링은 textContent만. 스캔/메일 데이터를 innerHTML에 절대 넣지 않는다. 모든 노드는 el() 헬퍼로 생성.
  • CSP(index.html:13-14 + main의 onHeadersReceived 이중 주입): default-src 'none'; script-src 'self'; style-src 'self'; connect-src 'none'. 외부 CDN·인라인 스크립트·네트워크 fetch 전부 금지. 인라인 <script> 0, 인라인 이벤트 핸들러 0.
  • style-src 'self' → 인라인 style 속성은 허용되나(현재 코드가 대량 사용), 인라인 <style> 블록·동적 스타일시트 주입은 불가. 색은 JS에서 el.style.color= CSSOM으로 설정(app.js:3279 참조).
  • H-1/H-2 경로는 pathGuard 검증, 외부 프로세스는 safeExec(절대경로·shell:false). 프런트엔드 항목엔 직접 영향 적으나 새 IPC 추가 시 준수.
  • 런타임 의존성 최소: 유일 런타임 의존성 electron-updater. 단 SortableJS는 이미 vendor 로컬 번들로 도입되어 있음(index.html:29, ./vendor/Sortable.min.js) — npm 런타임 의존성이 아닌 self-host 자산.

3. 각 섹션 드래그로 위치 이동 (프리뷰·워터폴 재배치)

현재 상태홈 섹션에는 드래그 없음. 홈 레이아웃은 고정 2-컬럼 flex(좌 1.65 / 우 1, app.js:1572-1584)에 섹션을 하드코딩 순서로 appendChild: 좌=주의필요·생산성·타임라인, 우=할일·메일·디스크·기능추가. 순서 저장 메커니즘 없음.
드래그 코드는 존재하나 "프로젝트 카드"에만 적용: SortableJS가 .cards 컨테이너에 부착(initCardSortable app.js:4894), placeholder 라이브 프리뷰(ghostClass) + onEnd에서 commitReorder→setOrder 영속(app.js:4914-4924).

제약·리스크① 포커스 트랩(getTabbables(dialog) 2127)은 좌측 카테고리 버튼도 트랩에 포함됨 — 좌우 분할 후에도 Tab 순환 정상 동작 확인 필요. ② 스크롤 복원(render의 SCROLL_SEL에 .modal__body 포함, 1155) — 우측 패널 셀렉터가 바뀌면 복원 셀렉터도 갱신. ③ 전체 재렌더라 카테고리 선택 상태는 store에 보관해야 유지(예 store.settingsTab). ④ CSP: 레이아웃은 인라인 style 속성/CSS 클래스로 — flex 2분할은 무난.
건드릴 곳renderSettings(2142)를 좌(카테고리 리스트) + 우(선택 섹션) 2-pane로 재구성. store에 활성 카테고리 상태 추가, 카테고리 클릭 시 해당 render*Settings()만 우측에 그림. buildModal에 옵션 추가 또는 별도 레이아웃. styles.css의 .modal 계열 확장.

8. 프로젝트 대시보드 검색창 한글 자음·모음 분리 입력 버그 (원인 확정)

현재 상태대시보드 검색 input(renderHeader:3528-3535)이 input 이벤트마다 debounce(render, 120)를 호출(app.js:3533-3534). render()는 app.replaceChildren()로 input 노드를 통째 교체한다(1171). IME 조합 중(자음·모음 입력 중)에 노드가 교체되면 composition이 깨져 자모가 분리되어 입력됨.
원인 가설(거의 확정)현재 코드에 IME 방어가 전혀 없음 — 코드베이스 전체에서 isComposing / compositionstart / compositionend 검색 결과 0건. 부분 완화책으로 render()가 "포커스/캐럿 복원"을 함(FOCUS_SEL=.topbar__search-input·.orbit__search, app.js:1160-1168, 1196-1207). 그러나 이는 노드 교체 후 포커스·selectionRange만 복원할 뿐, 진행 중이던 IME composition 세션은 복원 불가 → 조합 도중 노드가 사라지면 브라우저가 조합을 강제 종료하고 자모가 깨진다. 120ms debounce가 빠른 타이핑에서 조합 중 발화할 수 있음.
관련 위치버그 input 핸들러 app.js:3534 (searchInput.addEventListener('input', (e) => { store.state.search = ...; debounced(); })) · debounce 정의 5218 · render 노드교체 1171 · 불완전 포커스 복원 1160-1207. 동일 위험의 다른 input: 궤도 검색 .orbit__search(3355), 할 일 입력(1822), 각종 설정 input(2313/2516/2619/2782/4129) — 이들은 대부분 render를 즉시 부르지 않거나 복원 셀렉터에 없어 증상이 다를 수 있음.
건드릴 곳 / 해결 방향① composition 가드: input 핸들러에서 e.isComposing(또는 compositionstart/end 플래그)이면 render 보류, compositionend 후 1회 반영. ② 또는 검색 재렌더 시 input 노드를 교체하지 않도록 — 검색은 결과 영역만 갱신하고 헤더 input은 보존(전체 replaceChildren 구조와 상충 → 부분 갱신 도입 필요). ③ 최소수정안: input 핸들러에서 조합 중이면 store만 갱신하고 debounced() 호출 스킵. 전체 재렌더 아키텍처가 근본 원인이라 항목 2(실시간 재렌더 강화)와 직접 충돌.

PM 주의 — 교차 이슈(항목 간 충돌·동반 설계 필요)

  • 전체 재렌더 vs 입력/드래그/차트 — 근본 구조가 render()=replaceChildren()(전체 교체)다. 이것이 8번(IME 깨짐)·3번(드래그 파괴)·4번(차트 인스턴스 파괴)의 공통 원인. 항목 2(실시간 갱신 강화)는 재렌더 빈도를 높여 이 세 문제를 모두 악화. 설계 우선순위: 부분 갱신(또는 갱신 가드)을 먼저 정립해야 나머지가 안정적으로 얹힌다. 이미 가드 인프라(store._dragging, 오버레이 체크, 포커스/스크롤 복원)가 있으니 확장하는 방향 권장.
  • 실시간 갱신(2) × 한글 IME(8) — 라이브 push로 재렌더가 발생하면 사용자가 검색어를 한글 조합 중일 때도 input 노드가 교체돼 조합이 깨진다. 8번 수정 없이 2번을 강화하면 입력 중 깨짐이 더 잦아짐. composition 중에는 라이브 렌더도 보류하는 가드를 8번 수정에 포함시킬 것.
  • 차트 라이브러리(4) × CSP — script-src 'self' + connect-src 'none' 때문에 CDN·런타임 다운로드 전부 불가. 반드시 self-host(vendor/). canvas 기반(Chart.js)이 안전, SVG 라이브러리는 동적 <style> 주입으로 style-src 위반 위험 → 도입 전 CSP 적합성 검증. "런타임 의존성 최소" 규약은 SortableJS 선례로 self-host 자산은 예외 취급 가능하나 근거 문서화 필요.
  • 드래그(3) × uiState 스키마 — 현재 order는 프로젝트 id 해시 전용(ID_RE). 홈 섹션 순서 저장은 uiStateStore에 새 키·정규화·IPC·preload 노출 신설이 필요(백엔드 작업 포함). SortableJS·드래그 가드 인프라는 재사용 가능.
  • 메뉴 제거(5) × 단축키(6) × 설정(7) — 한 묶음으로 설계하면 일관됨: 메뉴 제거 → 잃는 기능(폴더추가·재스캔·새로고침·정보)을 헤더 버튼/단축키로 이관(F5=새로고침 신설), "정보"를 설정 모달로, 단축키 안내를 설정 카테고리로. 단축키 매핑은 단일 상수로 두고 메뉴/안내표/keydown 핸들러가 같은 출처를 참조해야 동기화 깨짐을 막음. 메뉴 제거 시 preload onMenu 화이트리스트·app subscribeMenu 죽은 채널 정리(SEC-L1).
  • 헤더 통일(1) × 메뉴 제거(5) — 메뉴 기능을 헤더로 옮기면 홈/프로젝트 헤더 구성이 자연히 영향받음. 1번과 5번을 함께 설계해 두 헤더의 액션 영역을 한 번에 정리할 것.
