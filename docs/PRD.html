1. 개요

Project-SPIP는 한 대의 PC에 여러 폴더로 흩어져 있는 VS Code 프로젝트들을 자동으로 탐지하고, 각 프로젝트의 언어/스택, 활동 신선도, Git 상태, 규모/의존성 같은 인사이트와 함께 로컬 웹 대시보드로 보여준다. 사용자는 더 이상 탐색기를 뒤지지 않고도 어떤 프로젝트가 어디에 있고 어떤 상태인지 한눈에 파악하며, 카드에서 곧바로 VS Code로 열 수 있다.

핵심 가치 제안: "내 모든 프로젝트를 한 화면에서 — 어디 있는지, 어떤 상태인지, 바로 열 수 있게."

2. 배경 / 문제 정의

사용자 문제

  • 프로젝트가 여러 드라이브·폴더(~/dev, D:\work, 바탕화면, 임시 폴더 등)에 흩어져 있어 전체 목록을 한눈에 볼 방법이 없다.
  • 어떤 프로젝트가 최근에 손댄 것이고 어떤 것이 방치(stale)됐는지, 어디에 미커밋/미푸시 변경이 남아 있는지 일일이 폴더를 열어봐야만 안다.
  • 오래된 프로젝트의 node_modules 등이 디스크를 잠식하지만, 어디가 무거운지 파악이 어렵다.
  • 특정 프로젝트를 열려면 경로를 기억해 탐색기로 찾아 들어가 VS Code를 여는 반복 작업이 필요하다.

비즈니스 / 제품 목표

  • 개발자 본인의 프로젝트 관리 비용을 줄이고, 흩어진 자산에 대한 가시성과 통제감을 제공한다.
  • 설치·실행이 가벼운 로컬 전용 도구로, 외부 의존성과 프라이버시 리스크를 최소화한다.

4. 대상 사용자 / 페르소나

항목내용
페르소나"멀티 프로젝트 개발자" — 본인 1인
상황여러 클라이언트/사이드/실험 프로젝트를 한 PC의 여러 위치에 보관. Git 사용. VS Code가 주 에디터.
니즈전체 프로젝트를 한눈에 보고, 상태(신선도·git·용량)를 빠르게 파악하고, 원하는 것을 즉시 열기.
기술 수준높음. CLI/Node 실행에 익숙. 단, 도구 자체는 가볍고 즉시 작동하길 원함.
실행 환경로컬 PC, 단일 사용자, 인증 없음. 브라우저로 localhost 대시보드 접속.

5. 사용자 스토리 (요약)

전체 백로그와 상세 수용 기준은 docs/UserStories.html, 단일 기능 명세는 docs/FeatureSpec.html 참조.

  • 개발자로서, 내 모든 프로젝트의 위치를 파악하기 위해 지정 폴더(또는 전체 드라이브)를 스캔하고 싶다.
  • 개발자로서, 어떤 프로젝트가 방치됐는지 판단하기 위해 각 프로젝트의 최종 수정일·최근 커밋 시각을 보고 싶다.
  • 개발자로서, 커밋·푸시를 깜빡한 프로젝트를 찾기 위해 git 미커밋 변경·미푸시 여부를 한눈에 확인하고 싶다.
  • 개발자로서, 디스크를 정리하기 위해 어떤 프로젝트의 용량·node_modules 크기가 큰지 알고 싶다.
  • 개발자로서, 작업을 빠르게 재개하기 위해 카드에서 바로 VS Code로 열고 싶다.
  • 개발자로서, 원하는 것을 빨리 찾기 위해 언어·신선도·git 상태로 필터·정렬·검색하고 싶다.

6. 기능 요구사항

우선순위는 MoSCoW. Must = MVP 필수, Should = MVP 직후, Could = 여유 시, Won't = 이번 버전 제외. 각 요구는 추적성 매트릭스 입력용 R-ID를 가진다.

6.1 스캔 / 탐지

ID요구사항우선수용 기준 (Acceptance Criteria)
R-01 지정 경로 하위에서 VS Code 프로젝트 폴더를 탐지한다. Must Given 루트 경로 하위에 .git / package.json / .vscode 중 하나 이상을 가진 폴더가 있을 때, When 스캔을 실행하면, Then 그 폴더가 프로젝트로 1회만 탐지된다. 같은 트리에서 중첩 탐지 시 최상위 1건으로 집계한다.
R-02 거대·불필요 폴더를 제외 규칙으로 건너뛴다. Must Given node_modules, .git 내부, dist/build/.cache, OS 시스템 폴더가 트리에 있을 때, When 스캔하면, Then 해당 폴더 하위로 재귀 진입하지 않는다. 제외 목록은 설정으로 추가 가능하다.
R-03 탐색 깊이/심볼릭 링크 안전 처리. Should When 순환 심볼릭 링크나 과도한 깊이를 만나면, Then 무한 루프 없이 상한(기본 깊이 한도)에서 중단하고 경고를 로그에 남긴다.

6.2 스캔 범위 설정

ID요구사항우선수용 기준
R-04 사용자 지정 루트 폴더 목록을 설정한다 (기본 방식). Must Given 설정 파일(예: spip.config.json) 또는 CLI 인자로 1개 이상 루트 경로를 줄 때, When 스캔하면, Then 지정한 경로들만 대상으로 한다. 기본 동작(확정): 첫 실행 시 스캔 루트가 미지정이면 자동 스캔하지 않고 "스캔할 루트 폴더를 지정하세요" 안내 후 종료한다(홈 전체 자동 스캔은 기본 아님). 전체 드라이브 스캔은 R-05(후속 M4) 옵션으로만 제공한다.
R-05 전체 드라이브 자동 탐색 옵션을 제공한다. Should When spip scan --all-drives(또는 설정 플래그) 실행 시, Then 사용 가능한 모든 드라이브 루트를 대상으로 R-02 제외 규칙을 적용해 탐색하고, 시작 전 사용자에게 시간이 오래 걸릴 수 있음을 고지한다.

6.3 인사이트 수집 (4종)

ID요구사항우선수용 기준
R-06 언어/프레임워크 분포 감지. Must Then 각 프로젝트의 주 언어/스택을 package.json의 deps와 파일 확장자 비율로 추정해 표시하고, 전체 집계(예: JS 12개, Python 5개)를 통계 뷰에 보여준다. 감지 불가 시 "Unknown"으로 표기한다.
R-07 활동/신선도 정보. Must Then 각 프로젝트의 최종 파일 수정일과(가능 시) 최근 커밋 시각을 보여주고, 기준일(기본 90일) 이상 활동 없는 프로젝트를 stale로 표시한다. 기준일은 설정 가능.
R-08 Git 상태 수집. Must Then 저장소 여부, 현재 브랜치, 미커밋 변경 유무, 원격 대비 ahead/behind(미푸시 여부)를 표시한다. git 미설치/비저장소면 R-12에 따라 안전 처리하고 "N/A"로 표기한다.
R-09 규모/의존성 정보. Should Then 프로젝트 총 용량, node_modules 크기(있으면), 의존성 개수(deps/devDeps)를 보여준다. 용량 계산이 무거운 경우 비동기/지연 측정 또는 근사치 허용을 명시한다.

6.4 대시보드 표시

ID요구사항우선수용 기준
R-10 프로젝트를 카드/리스트로 표시하고 전체 통계 요약 뷰를 제공한다. Must Then 각 항목은 이름·경로·설명·주 언어·신선도·git 요약 배지를 보여준다. 상단에 전체 개수, 언어 분포, stale 개수, 총 용량 요약이 표시된다. 데이터가 0건이면 빈 상태 안내를 보여준다.
R-11 필터·정렬·검색을 제공한다. Must When 사용자가 언어/신선도/git 상태로 필터하거나 이름·최종수정·용량으로 정렬하거나 이름·경로로 검색하면, Then 목록이 클라이언트 측에서 갱신된다. 측정 기준: 200개 프로젝트 로드 상태에서 입력 이벤트→목록 DOM 재렌더 완료까지 performance.now() 계측값 중앙값 ≤ 200ms(§3 측정 정의). 필터 조합은 AND로 동작한다.

6.5 액션 / 연동

ID요구사항우선수용 기준
R-12 'VS Code로 열기' 액션 (code . 실행). Must When 카드의 '열기'를 클릭하면, Then 서버가 해당 프로젝트 경로에서 code <path>를 실행하고 성공/실패 피드백을 UI에 반환한다. 응답 기준: 클릭 후 2초 이내에 UI에 "실행 중/성공/실패" 피드백이 표시된다(이 시간 기준은 본 요구에서 단일 정의하며 하위 명세는 이를 참조). code CLI 미설치 시 안내 메시지를 표시하고 크래시하지 않는다.
R-13 git/외부 도구 미존재 시 graceful 처리. Must Given git 또는 code CLI가 없거나 권한이 없을 때, When 관련 동작을 시도하면, Then 프로세스가 죽지 않고 해당 항목만 "N/A"/안내로 처리하며 나머지 기능은 정상 동작한다.

6.6 CLI

ID요구사항우선수용 기준
R-14 spip 명령으로 스캔을 실행한다. Must When 전역 설치/링크 후 spip(= node scan.js)를 실행하면, Then 설정/인자 기준으로 스캔이 수행되고 결과 데이터(예: projects.json)가 생성되며 요약이 콘솔에 출력된다.
R-15 스캔 진행 표시(progress) 제공. Should When 스캔이 1초 이상 걸리면, Then 처리 중인 경로/진행 카운트를 콘솔(또는 대시보드)에 주기적으로 표시한다.
R-16 대시보드에서 재스캔 트리거. Could When 대시보드의 '새로고침/재스캔'을 누르면, Then 서버가 스캔을 재실행하고 갱신된 데이터를 화면에 반영한다.

7. 비기능 요구사항

ID분류요구사항 / 수용 기준
N-01성능지정 루트(프로젝트 ~200개) 스캔은 30초 이내. 전체 드라이브 스캔은 제외 규칙을 항상 적용하고 진행 표시(R-15)를 동반한다.
N-02보안/프라이버시서버는 127.0.0.1(localhost)에만 바인딩한다. 외부 네트워크로 어떤 데이터도 전송하지 않는다.
N-03경로 안전성 / 프로세스 인젝션대시보드/API는 스캔 루트 밖 경로 접근·디렉터리 순회(../)를 차단한다. '열기'·파일 읽기는 스캔으로 등록된 경로 화이트리스트 내에서만 허용한다. 외부 프로세스 호출 규칙(명시): 모든 외부 프로세스 호출(code, git 등)은 shell:false + 인자 배열 전달을 강제하며, 사용자/스캔 유래 입력(경로·브랜치명 등)은 셸 문자열로 연결하지 않는다. (상세 보안 설계·판정은 security-reviewer 영역)
N-04의존성Node 16+, CommonJS. 외부 npm 의존성 최소화(가능하면 Node 표준 모듈 위주). 대시보드 UI는 빌드/프레임워크 없는 순수 HTML/CSS/JS.
N-05견고성읽기 권한 없는 폴더, 손상된 package.json, 비-UTF8 파일을 만나도 해당 항목만 건너뛰고 전체 스캔은 완료한다.
N-06데이터스캔 결과(JSON)는 로컬에만 보관하며, OS 사용자 캐시 폴더(예: Windows %LOCALAPPDATA%, macOS ~/Library/Caches, Linux $XDG_CACHE_HOME)에 영속화하여 프로젝트 폴더를 오염시키지 않는다. 단일 사용자, 인증 없음.
N-07접근성/UX Should · M4MVP 비포함(후속 M4). 단일 사용자 로컬 도구 특성상 MVP 범위에서 제외한다. 다만 MVP 단계에서도 권고 수준으로 기본 키보드 포커스 이동과 색상 외 상태 라벨(텍스트 병기), 적정 대비를 지향한다. 정식 접근성 검증·완성은 M4에서 수행한다.

8. 사용자 플로우 / 주요 화면

핵심 경로(Happy path): CLI로 스캔 → 대시보드 접속 → 목록·통계 확인 → 필터/검색으로 대상 찾기 → 상세 확인 → 'VS Code로 열기'. 화면별 상세 흐름은 docs/FeatureSpec.html 참조.

9. 범위 밖 (Out of scope)

9.1 이번 제품에서 제외 (Won't — 진짜 범위 밖)

제외 항목분류사유
다중 사용자 / 계정 / 인증Won't로컬 단일 사용자 도구. 인증 없음.
원격/클라우드 동기화·공유Won't프라이버시·로컬 전용 원칙(N-02).
git 커밋/푸시 등 쓰기 동작Won't상태 읽기만. 쓰기는 VS Code/터미널의 몫.
VS Code 외 에디터 연동Won't이번 버전은 VS Code 전용. 추후 검토.

9.2 MVP 범위 밖 · 후속 후보 (Could — 이번 버전 미포함, 향후 가능)

아래 항목은 "제외(Won't)"가 아니라 MVP 이후 후속 후보다. 마일스톤 M4 또는 그 이후 검토 대상이며, MVP 기준 집합에는 포함되지 않는다.

후속 후보 항목분류비고
실시간 파일 감시(watch)·자동 재스캔CouldMVP는 수동 스캔. R-16(대시보드 재스캔)도 Could/후속.
프로젝트 태그/메모 등 사용자 메타 편집Could핵심 가치 검증 이후 확장.

10. 가정

  • 사용자는 Node.js 16+가 설치된 PC에서 본인 권한으로 실행한다.
  • 대부분의 프로젝트는 Git을 사용하거나 package.json/.vscode를 가진다(탐지 신호로 사용).
  • VS Code의 code CLI가 PATH에 등록돼 있다(없으면 R-12에서 안내).
  • 스캔 결과는 단일 PC 기준이며, 결과 데이터 파일명/세부 형식은 구현팀이 결정한다(예: projects.json). 저장 위치는 N-06로 확정(OS 캐시 폴더).

확정된 결정 (이전 미해결 질문 → PM 확정)

항목확정 내용
기본 스캔 루트사용자 지정 강제. 첫 실행 시 루트 미지정이면 자동 스캔하지 않고 안내 후 종료. 홈 전체 자동 스캔은 기본 아님. 전체 드라이브 스캔은 R-05(M4) 옵션. (R-04 반영)
stale 기준최종 수정 후 90일 경과 시 방치(stale)로 표시. 설정에서 변경 가능. (R-07 반영)
서버 동작고정 포트 사용, 브라우저 자동 오픈 없음. 시작 시 콘솔에 접속 URL 출력. (구체 포트 번호는 구현팀 결정, 기본값 문서화)
데이터 저장 위치OS 사용자 캐시 폴더에 스캔 결과(JSON) 영속화. 프로젝트 폴더 비오염. (N-06 반영)

11. 리스크 및 미해결 질문

R1 — 전체 드라이브 스캔 성능. 드라이브 전체 순회는 수십만 폴더에 달할 수 있다. 제외 규칙·깊이 한도·진행 표시가 없으면 체감 성능이 크게 나빠진다. (완화: R-02/R-03/R-15)
R2 — git/code CLI 환경 편차. OS·설치 상태에 따라 CLI 부재 가능. graceful 처리(R-13)로 크래시는 막되, 사용자 안내 문구 품질이 중요.
R3 — 용량 측정 비용. node_modules 용량 계산은 무겁다. 동기 측정 시 스캔이 느려질 수 있어 지연/근사 전략 필요(R-09).
R4 — 경로 보안. 로컬이라도 디렉터리 순회로 임의 파일 노출 위험. 화이트리스트 강제(N-03)가 필수.

미해결 질문

이전 1회차 미해결 질문 Q1~Q4(기본 루트·stale 90일·포트/오픈·영속 위치)는 모두 PM 확정 완료(§10 확정 결정 표 참조). 현재 설계 착수를 막는 미해결 기획 질문은 없음. 구체 포트 번호·결과 JSON 스키마 등 세부는 구현팀 결정 사항이다.
