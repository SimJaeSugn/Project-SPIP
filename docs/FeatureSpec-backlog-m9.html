1. 배치 개요 및 목표

이 8개 백로그(→9개 요구)는 모두 홈/프로젝트 대시보드의 사용 경험을 다듬는 작업으로, 네 갈래의 문제를 함께 해결한다.

  • UX 일관성 — 홈·프로젝트 헤더 구성 통일(R-27), 네이티브 메뉴 제거 후 헤더/단축키로 기능 이관(R-28), 설정 좌우 분할 정리(R-30).
  • 실시간성 — 모든 홈 섹션이 라이브 데이터를 반영(R-31). 단, 이를 켜면 재렌더 빈도가 올라가 입력·드래그·차트가 깨진다.
  • 커스터마이즈 — 홈 섹션을 드래그로 재배치(R-32), 주간 생산성 차트를 인터랙티브 라이브러리로(R-33).
  • 버그 수정 — 검색창 한글 자음·모음 분리 입력(R-26).
왜 이 8개를 함께 묶는가 — 공통 근원이 하나다. 감사 결과, 현재 render()가 app.replaceChildren()로 전체 DOM을 매번 통째 교체하는 구조라는 점이 R-26(한글 IME 자모분리)·R-32(드래그 파괴)·R-33(차트 인스턴스 파괴)의 공통 원인이며, R-31(실시간 갱신)은 재렌더 빈도를 높여 이 셋을 모두 악화시킨다. 따라서 "무엇을 만들 것인가"보다 먼저 부분 갱신/재렌더 가드 기반(R-25)을 정립해야 나머지가 안정적으로 얹힌다. R-25를 기반(foundational) 요구로 별도 명시하고, R-26·R-31·R-32·R-33이 여기에 의존하도록 수용 기준·의존성에 박았다.

이 배치에서 반드시 지키는 불변식 (감사 §보안 불변식)

  • L-1: 모든 렌더는 textContent(el() 헬퍼)만. 스캔/메일/차트 라벨 등 신뢰불가 문자열을 innerHTML에 넣지 않는다.
  • CSP(default-src 'none'; script-src 'self'; connect-src 'none'): 외부 CDN·인라인 <script>·런타임 fetch 전부 금지. (만약 신규 라이브러리가 필요하면 반드시 self-host — 단, 이 배치에서는 차트를 SVG 자작으로 확정해 신규 라이브러리 도입 자체가 없다.)
  • 런타임 의존성 최소: npm 런타임 의존성은 electron-updater 유지. 이 배치는 차트를 SVG 자작해 새 라이브러리·self-host 차트 자산을 추가하지 않는다(R-33 확정). 기존 SortableJS(self-host)는 R-32에서 재사용만 한다.
  • 새 IPC는 register.js guard(senderFrame 정확 일치)·고정 에러코드(L-3) 경유. uiState 영속은 0600 원자적 쓰기.

4. 요구 간 의존성

실선 초록 = 강한 선행 의존(기반 R-25 먼저) 실선 회색 = 상호 참조/동반 설계 점선 = 악화/주의 관계
R-25 ↔ R-26 ↔ R-31 단일 진술(검증 F-3): R-26(IME 버그)은 R-25 기반 위에서 먼저 해소하고, R-31(실시간 push)은 R-25의 composition 보류 훅을 반드시 경유해 한글 조합 중 input 교체를 막는다. 아래 구현 순서·R-26 수용 기준·다이어그램이 모두 이 진술과 일치한다.
구현 순서 권고: ① R-25(기반) → ② R-26 / R-27 / R-28 / R-29(Must, 병렬 가능하나 R-26은 R-25 의존, R-28↔R-29는 단일출처 공유) → ③ R-30 / R-31 / R-32(Should) → ④ R-33(Could). R-31은 R-25의 composition 보류 훅을 경유하며, R-26 해소 이후 활성화한다.

5. 요구 상세

R-25 부분 갱신 / 재렌더 가드 기반 정립 (foundational) 기반
구현 개발자로서, 후속 요구(R-26·R-31·R-32·R-33)가 의존할 수 있는 안정적인 부분 갱신·재렌더 가드 계약을 원한다. 왜냐하면 현재 render()가 전체 DOM을 통째 교체해 한글 조합·드래그 세션·차트 인스턴스가 매 렌더마다 파괴되며, 이 공통 근원을 먼저 잡지 않으면 후속 요구들이 각자 임시방편을 만들어 충돌하기 때문이다. (이 요구의 직접 수혜자는 최종 사용자가 아니라 후속 요구를 구현하는 개발자다 — 사용자 가치는 R-26·R-31·R-32·R-33을 통해 간접 실현된다.)
API surface 최종 확정은 chief-architect의 첫 설계 과제로 위임(검증 F-2). 함수명·시그니처·모듈 배치 등 구체 API 표면은 본 명세에서 못 박지 않는다. 아래 코드 식별자(shouldDeferRender 등)는 예시·의도 전달용이며 확정 계약이 아니다. 다만 후속 요구가 의존할 계약의 의도 4가지는 확정이며, 아키텍트는 이 4가지를 충족하는 구체 API를 첫 설계 산출물에서 정의한다.
상세 기능 명세 — 확정된 계약 의도 (4가지, 아키텍트가 구체 API로 구현)
  • (1) composition(IME) 진행 중 재렌더 보류 훅 — compositionstart→compositionend 사이 진행 상태를 단일 진입점에서 추적하고, 그동안 라이브 갱신/재렌더를 보류한다. (R-26·R-31이 이 훅을 경유.)
  • (2) 노드 보존 부분 갱신 — 특정 DOM(헤더 검색 input 등)을 재렌더 시 파괴하지 않고 보존하여 입력값·포커스·캐럿(selectionStart/End)을 유지한다.
  • (3) stateful 위젯의 destroy/recreate 라이프사이클 훅 — 차트(R-33)·SortableJS 컨테이너(R-32)처럼 인스턴스를 가진 위젯이 재렌더 시 일관되게 정리·재생성되도록 표준 훅을 제공(중복 인스턴스·메모리 누수 방지).
  • (4) 실시간 push 갱신 coalesce — 보류 중 누적된 라이브 갱신을 상태 해제 시 병합·디바운스하여 1회만 반영한다(드래그/조합/오버레이 중 갱신 묶음 처리).

전면 가상DOM/프레임워크 도입은 비범위 — 기존 가드 인프라(store._dragging·오버레이 체크·FOCUS_SEL/SCROLL_SEL 복원)를 확장하는 방향. 가드 코드는 마커(예 RG-1·RG-2)로 추적.

수용 기준 — 완료 정의(DoD, 검증 가능 체크리스트)
  • 의도(1): composition 진행 상태가 true인 동안 라이브 갱신 트리거 시 재렌더가 호출되지 않고 보류 플래그가 설정된다 — 보류 판정 순수함수가 false 반환함을 헤드리스 단위테스트로 검증.
  • 의도(1): 드래그 중·오버레이(설정/도움말/드로어) 열림 상태에서도 동일하게 라이브 갱신이 보류된다(단위테스트).
  • 의도(2): 보존 대상 노드(헤더 검색 input)가 재렌더를 거쳐도 입력값·포커스·캐럿 위치(selectionStart/End)·주요 스크롤 위치가 유지된다(동작 확인 + 기존 복원 회귀 0).
  • 의도(3): stateful 위젯이 재렌더 시 이전 인스턴스가 destroy되고 재생성되어 중복 인스턴스·이벤트 핸들러 누적·메모리 누수가 없다(훅 호출 단위테스트 또는 동작 확인).
  • 의도(4): 보류 상태 해제 시 누적된 갱신이 정확히 1회 병합 반영된다(중복 렌더 0) — coalesce 단위테스트.
  • 가드/coalesce 판정 로직은 app.js DOM과 분리된 순수 함수로 단위테스트 가능하다.
  • 아키텍트가 4가지 의도를 충족하는 구체 API surface(함수명·시그니처·모듈)를 첫 설계 산출물에 문서화했고, R-26·R-31·R-32·R-33 설계가 그 API를 참조한다.
  • 기존 dashboard/orbit/home/scan 뷰 회귀 0, 전체 테스트 스위트 그린 유지. 가드 코드에 추적 마커 병기.

기반 / Must 우선 처리   선행 의존: 없음 (모든 후속의 토대)   피의존: R-26 R-31 R-32 R-33

R-26 프로젝트 대시보드 검색창 한글 자음·모음 분리 입력 버그 수정 Must
사용자로서 프로젝트 검색창에 한글을 끊김 없이 입력하고 싶다. 왜냐하면 현재 "프로젝트"를 치면 "ㅍㅡㄹㅗㅈㅔㅋㅡㅌㅡ"처럼 자음·모음이 분리돼 검색이 사실상 불가능하기 때문이다.
상세 기능 명세 (감사: 원인 거의 확정)
  • 원인: 검색 input 핸들러가 매 입력마다 debounce(render,120)를 호출 → render()가 input 노드를 replaceChildren로 통째 교체 → IME 조합 세션이 강제 종료되어 자모가 분리됨. 코드베이스에 isComposing/compositionstart/compositionend 방어가 0건.
  • 해결 방향(R-25 가드 활용): ① 검색 input 핸들러에서 e.isComposing 또는 store._composing이면 store.state.search만 갱신하고 render() 호출(debounced)을 스킵한다. ② compositionend 발생 시 1회 반영한다. ③ 가능하면 R-25의 "보존 영역"으로 검색 결과 영역만 갱신하고 헤더 input 노드는 보존(상위 해법).
  • 동일 위험의 다른 input(궤도 검색 .orbit__search, 할 일 입력 등)도 동일 가드를 적용해 회귀를 막는다.
수용 기준 (검증 가능 체크리스트)
  • 검색창에 "프로젝트", "스카이림", "한글테스트" 등 받침 포함 단어를 빠르게 입력해도 자모가 분리되지 않고 완성형으로 입력된다(사용자 PC 1회 수동 확인 — 헤드리스 한계).
  • 조합 중(isComposing===true 또는 _composing) 입력 핸들러는 render()/debounced 재렌더를 호출하지 않는다 — 핸들러 분기 순수함수 단위테스트.
  • compositionend 후 검색 결과가 1회 반영되어 최종 검색어로 필터링된다.
  • 영문/숫자 입력 시 기존 debounce 검색 동작은 변함없이 유지된다(회귀 없음).
  • R-31(실시간 push)은 R-25의 composition 보류 훅을 경유하므로, 한글 조합 중 라이브 push가 발생해도 input 교체가 일어나지 않아 입력이 깨지지 않는다(R-26은 R-25 기반 위에서 먼저 해소·검증 F-3 단일 진술).
우선순위 · 의존성

Must (사용자 영향 큰 버그)   depends: R-25 (composition 추적·갱신 보류 가드 사용)

R-27 홈 ↔ 프로젝트 대시보드 헤더 구성 통일 Must

  • 워터폴 재배치: "적절한 워터폴(masonry) 재배치"를 위해 2-컬럼 flex를 단일 그리드/컬럼 모델로 레이아웃 재설계(섹션 높이가 달라도 빈틈 없이 채움). (구현 방식은 아키텍트 결정 — masonry CSS vs JS 재배치.)
  • 순서 영속(백엔드 신설): 현 uiState.order는 프로젝트 id 해시 전용(ID_RE). 홈 섹션 순서는 새 스키마 키(예 homeLayout) + 정규화/검증 + IPC(spip:setHomeLayout) + preload 노출을 신설한다. 0600 원자적 영속.
  • R-25 가드: 드래그 중(_dragging) 라이브 재렌더 보류. 전체 재렌더 시 SortableJS init/destroy 표준 훅 사용.
수용 기준 (검증 가능 체크리스트)
  • 홈 섹션을 드래그로 이동할 수 있고, 드래그 중 놓일 위치 프리뷰(placeholder)가 표시된다(사용자 PC 확인).
  • 드래그 종료 후 섹션들이 워터폴/그리드 방식으로 재배치되며, 정량 기준을 만족한다: (a) 섹션 간 시각적 겹침 0(서로 다른 섹션 박스의 경계 영역 중첩 없음), (b) 규정된 간격(gap) 외 의도치 않은 빈 칸 없음(컬럼 내 섹션은 위에서부터 gap만큼만 띄워 연속 배치), (c) 워터폴 재배치 애니메이션이 완료될 때까지 중간 프레임에서도 레이아웃 깨짐(요소 사라짐·튐·중첩) 없음(사용자 PC 시각 확인).
  • 재배치한 섹션 순서가 앱 재시작 후에도 유지된다(homeLayout 영속) — 정규화/검증 순수함수 단위테스트 + 사용자 PC 재시작 확인.
  • homeLayout 스키마가 알 수 없는 섹션 id·중복·길이 초과를 정규화로 거부/정리한다(단위테스트).
  • 섹션 순서 IPC는 register.js guard·고정 에러코드 경유, 실행/경로 의미 없는 식별자만 다룬다(보안).
  • 드래그 중 라이브 갱신이 발생해도 드래그 세션이 파괴되지 않는다(R-25 _dragging 가드).
우선순위 · 의존성

Should   depends: R-25 (드래그 중 재렌더 보류)   백엔드 신설 동반(uiStateStore 스키마·IPC·preload).

R-33 주간 생산성 차트를 인터랙티브하게 구현 (SVG 자작 · 외부 라이브러리 없음) Could
사용자로서 주간 생산성(일 커밋 빈도)을 툴팁·호버 등 인터랙션이 있는 차트로 보고 싶다. 왜냐하면 현재는 div 막대를 인라인 높이로 수작업 렌더해 막대에 마우스를 올려도 정확한 수치를 알 수 없기 때문이다.
상세 기능 명세 (사용자 확정: SVG 자작)
  • 현황: renderHomeProductivity()가 순수 div의 인라인 height/background로 7일 막대를 수작업 생성. 호버 툴팁·강조 없음.
  • 구현 수단(확정): 외부 차트 라이브러리 없이 svg() 헬퍼로 인터랙티브 SVG 자작 — 새 런타임 의존성 0, self-host 자산도 불필요(자작이므로). Chart.js 등 라이브러리 도입은 비범위(디자인 문서 §7 "차트도 SVG/canvas 자작" 원칙 일치).
  • 인터랙션: 막대(또는 스파크라인) 호버 시 정확한 수치 툴팁 + 해당 막대 강조(hover 하이라이트). 툴팁은 DOM/SVG로 구성, 텍스트는 textContent.
  • L-1·CSP: 차트에 숫자만 전달(안전). 라벨에 프로젝트명 등 신뢰불가 문자열은 textContent 경로로만. 외부 네트워크 요청 0(자작이므로 vendor 자산·CDN 모두 불필요).
  • R-25 라이프사이클 훅: 전체 재렌더 시 SVG 차트의 이벤트 핸들러·DOM 노드를 R-25의 destroy/recreate 훅으로 일관 정리해 핸들러 누적·중복 노드를 방지한다.
수용 기준 (검증 가능 체크리스트)
  • 주간 생산성 차트의 막대(또는 동등 시각화)에 호버 시 정확한 수치 툴팁이 뜨고 해당 막대가 강조된다(사용자 PC 확인).
  • 외부 네트워크 요청 0 — 차트는 SVG 자작이라 외부 라이브러리·CDN·vendor 자산을 일절 추가하지 않으며, 앱 실행 중 외부 도메인 fetch가 발생하지 않는다(개발자도구 네트워크 탭 확인).
  • 새 런타임 의존성·차트 라이브러리 도입 0 — package.json 의존성 변경 없음, public/vendor/에 차트 자산 추가 없음(코드 리뷰).
  • CSP 위반(콘솔 CSP 에러) 0건 — 인라인 <style>·인라인 script 없음, 색·스타일은 CSSOM/속성으로 설정.
  • 차트 텍스트(라벨·툴팁)는 전부 textContent/안전 DOM 경로로 렌더(L-1).
  • 전체 재렌더 시 이전 차트의 이벤트 핸들러·노드가 정리되어 핸들러 누적·중복 노드·메모리 누수가 없다(R-25 라이프사이클 훅 — 동작 확인).
우선순위 · 의존성

Could   depends: R-25 (차트 노드/핸들러 destroy/recreate 훅)   구현 수단은 SVG 자작으로 확정(라이브러리 비범위).

6. 비범위 (Out of scope)

  • 전면 가상 DOM/프레임워크 도입 — R-25는 기존 render() 구조 위에 갱신 가드·보존 영역을 얹는 것까지다. React/Vue·diff 엔진 전환은 비범위(순수 HTML/CSS/JS·번들러 없음 규약 유지).
  • 단축키 사용자 편집(리매핑) UI — R-29는 단축키 표시 전용. 사용자가 단축키를 바꾸는 기능은 비범위.
  • 설정 카테고리의 검색/필터 — R-30은 2-pane 분할까지. 설정 항목 검색창은 비범위.
  • 홈 섹션 추가/삭제·표시 토글 — R-32는 기존 섹션의 순서 재배치까지. 섹션 자체를 끄거나 새 섹션을 추가하는 커스터마이즈는 비범위.
  • 커밋 차트 외 신규 데이터 수집 — R-31/R-33은 보유/기존 수집 데이터(commitActivity 등) 활용까지. 새 git 메트릭 수집기 신설은 비범위(필요 시 별도 요구).
  • 프로젝트 카드(대시보드)의 워터폴/그리드 재배치 — R-32는 홈 섹션 한정. 기존 카드 reorder(R-19/R-23)는 그대로 유지.
  • 차트 라이브러리 도입(Chart.js 등) — R-33은 SVG 자작으로 확정되어 외부 차트 라이브러리·self-host 차트 자산은 비범위다. 새 npm 런타임 의존성 추가 금지(전 요구 공통).

7.2 아키텍트 위임 (검증 F-2)

R-25의 구체 API surface 최종 확정은 chief-architect의 첫 설계 과제로 위임한다. 본 명세는 후속 요구가 의존할 계약의 의도 4가지만 확정한다 — (1) composition(IME) 진행 중 재렌더 보류 훅, (2) 노드 보존 부분 갱신(입력/포커스/캐럿 유지), (3) stateful 위젯의 destroy/recreate 라이프사이클 훅, (4) 실시간 push 갱신 coalesce. 함수명·시그니처·모듈 배치는 아키텍트가 첫 설계 산출물에서 정의하고, R-26·R-31·R-32·R-33 설계가 그 API를 참조한다.
