Project-SPIP — 기능 명세서

PRD(docs/PRD.html)의 MVP Must 기능을 정상/예외 흐름과 엣지 케이스까지 상세화한다. 대상: 구현 개발자·QA. 본 문서는 F-1 스캔(R-01 등)·F-2 VS Code로 열기(R-12)·F-3 대시보드 표시(R-10)·F-4 필터/정렬/검색(R-11) 4개 MVP 기능을 명세한다. MVP 경계는 PRD §12.1 단일 MVP 요구 집합을 따른다.

마일스톤 델타 안내 (MVP 이후). 본 명세는 MVP 기준이며, 이후 마일스톤(M4·M6·M7·M8)의 기능 델타는 마일스톤별 설계 정본과 추적성 매트릭스에서 관리한다 — docs/architecture/{m4,m6,m7,m8}-design.html · docs/traceability.html · docs/pm/progress.html. 최신: M8 — R-24 상태 주시(라이브 갱신): 한 번 스캔한 뒤 재스캔 전까지 git·freshness를 주기 폴링(기본 15초)으로 재수집해 대시보드에 라이브 반영(StateWatcher · 단방향 push · 보안 0 약화 · ADR-M8-1). 정본: docs/architecture/m8-design.html.

F-1. 프로젝트 스캔 (탐지 + 인사이트 수집)

기능명프로젝트 스캔
관련 요구 IDR-01, R-02, R-03, R-04, R-05, R-06~R-09, R-14, R-15
목적지정 루트(또는 전체 드라이브) 하위의 VS Code 프로젝트를 탐지하고 4종 인사이트를 수집해 결과 데이터로 저장한다.
대상 사용자멀티 프로젝트 개발자 본인 (CLI 실행)
트리거(MVP) spip / npm run scan. (후속) 대시보드 재스캔 — R-16, Could/M4. MVP 필수 트리거 아님.

사용자 스토리 + 수용 기준

  • 개발자로서 내 프로젝트 위치를 한 번에 파악하기 위해 스캔을 실행하고 싶다.
    수용 기준: 스캔 종료 시 탐지된 프로젝트 수, 소요 시간, 결과 파일 경로가 콘솔에 요약 출력된다. 탐지 신호(.git/package.json/.vscode) 중 하나라도 가진 폴더는 1건으로 집계된다.

정상 흐름 (Happy path)

  1. 사용자가 spip 실행. 설정/인자에서 스캔 루트 목록을 로드(R-04). 없으면 기본값 사용 + 안내.
  2. 각 루트를 너비/깊이 우선으로 순회하며, R-02 제외 규칙(node_modules, .git 내부, dist/build/.cache 등) 폴더는 재귀 진입하지 않음.
  3. 탐지 신호를 가진 폴더를 프로젝트로 등록(중첩 시 최상위 1건).
  4. 각 프로젝트에 대해 인사이트 수집: 언어/스택(R-06), 신선도(R-07), git 상태(R-08), 규모/의존성(R-09).
  5. 1초 이상 소요 시 진행 카운트/현재 경로 표시(R-15).
  6. 결과를 로컬 데이터(예: projects.json)로 저장하고 요약 출력.

예외 / 에러 흐름

상황처리
루트 경로가 존재하지 않음해당 루트만 건너뛰고 경고 출력, 나머지 루트는 계속(R-13/N-05).
읽기 권한 없는 폴더해당 폴더 스킵 + 경고 로그. 스캔 전체는 완료(N-05).
git 미설치 / 비저장소git 인사이트를 "N/A"로 채우고 계속(R-08/R-13).
손상된 / 비-UTF8 package.json언어/의존성 정보 일부 누락 처리, 프로젝트 자체는 유지(N-05).
결과 파일 쓰기 실패(권한/디스크)명확한 에러 메시지 출력 후 비정상 종료 코드 반환.

엣지 케이스

  • 순환 심볼릭 링크 / 과도한 깊이 → 깊이 한도에서 중단(R-03).
  • 한 폴더가 여러 탐지 신호를 동시 보유 → 중복 등록하지 않음(1건).
  • 중첩 프로젝트(monorepo 내부 패키지) → 최상위만 등록(설정으로 하위 포함 옵션은 Could).
  • 매우 거대한 node_modules → 용량은 지연/근사 측정 허용(R-09), 재귀 진입은 항상 차단(R-02).
  • 탐지 0건 → 빈 결과 + "프로젝트를 찾지 못함, 루트 설정 확인" 안내.

의존성 / 제약

  • git 상태 수집은 git CLI에 의존(없으면 graceful).
  • 전체 드라이브 스캔(R-05)은 시간이 길 수 있어 진행 표시·제외 규칙 필수.
  • Node 16+, CommonJS, 외부 의존성 최소(N-04).

F-2. VS Code로 열기

기능명VS Code로 열기
관련 요구 IDR-12, R-13, N-03
목적대시보드 카드/상세에서 클릭 한 번으로 해당 프로젝트를 VS Code로 연다.
트리거카드/상세의 'VS Code로 열기' 버튼 클릭

사용자 스토리 + 수용 기준

  • 개발자로서 작업을 빠르게 재개하기 위해 카드에서 바로 VS Code로 열고 싶다.
    수용 기준: 버튼 클릭 후 PRD R-12 응답 기준(2초 이내 UI 피드백)을 준수하며 code <path>가 실행되고 성공/실패 피드백이 표시된다. (시간 수치는 R-12에서 단일 정의하며 본 명세는 이를 참조한다.) 경로는 스캔으로 등록된 화이트리스트 내에서만 허용된다(N-03).

정상 흐름

  1. 사용자가 카드의 '열기' 클릭 → 프런트가 프로젝트 식별자(또는 경로)를 서버 API로 전송.
  2. 서버는 해당 식별자를 등록된 프로젝트 목록과 대조해 실제 경로 확정(N-03 화이트리스트 검증).
  3. 서버가 code <path> 실행.
  4. 성공 시 UI에 "VS Code에서 여는 중" 피드백 표시.

예외 / 에러 흐름

상황처리
code CLI 미설치 / PATH 부재"VS Code CLI(code)를 찾을 수 없음" 안내 + 설치/PATH 등록 가이드 표시. 크래시 없음(R-13).
경로가 화이트리스트 밖 / 조작된 요청요청 거부(403 류), 실행하지 않음(N-03).
경로가 더 이상 존재하지 않음(삭제됨)"경로가 존재하지 않음, 재스캔 권장" 안내.
프로세스 실행 실패에러 메시지를 UI에 반환, 다른 기능은 정상 유지.

엣지 케이스

  • 동일 프로젝트 '열기' 연타 → 중복 실행 방지(버튼 일시 비활성/디바운스).
  • 경로에 공백/한글/특수문자 → 안전하게 인자 전달. 모든 외부 프로세스 호출(code·git)은 shell:false + 인자 배열로 실행하고 사용자/스캔 유래 입력을 셸 문자열로 연결하지 않는다(N-03 규칙). 상세 보안 판정은 security-reviewer 영역.

화면 흐름: 목록 → 상세 → 액션

F-3. 대시보드 표시 (목록 · 통계 요약)

기능명대시보드 표시
관련 요구 IDR-10, N-02, N-07(권고)
목적스캔 결과 데이터를 localhost 대시보드에서 카드/리스트 + 상단 통계 요약으로 보여준다.
대상 사용자멀티 프로젝트 개발자 본인 (브라우저)
트리거서버 실행(npm run start) 후 콘솔에 출력된 고정 포트 URL로 접속(자동 오픈 없음)

사용자 스토리 + 수용 기준

  • 개발자로서 전체 프로젝트를 한눈에 보기 위해 카드/리스트와 통계 요약을 보고 싶다.
    수용 기준: 각 항목은 이름·경로·설명·주 언어·신선도(stale 90일 배지)·git 요약 배지를 표시한다. 상단에 전체 개수·언어 분포·stale 개수·총 용량 요약이 표시된다. 데이터 0건이면 빈 상태 안내를 표시한다.

정상 흐름 (Happy path)

  1. 서버(server.js)가 OS 캐시의 결과 JSON(N-06)을 읽어 프로젝트 목록·집계를 계산.
  2. 브라우저가 대시보드 페이지를 로드 → 상단 통계 요약(개수·언어 분포·stale 수·총 용량) 렌더.
  3. 본문에 프로젝트 카드/리스트를 렌더(이름·경로·설명·언어·신선도·git 배지).
  4. 각 카드에 'VS Code로 열기' 액션 버튼 노출(F-2로 연결).

예외 / 에러 흐름

상황처리
결과 JSON이 아직 없음(스캔 미실행)"먼저 스캔을 실행하세요(spip/npm run scan)" 안내 + 빈 상태 표시. 크래시 없음.
결과 JSON 손상/파싱 실패"결과 데이터를 읽을 수 없음, 재스캔 권장" 안내. 서버는 계속 동작.
일부 항목 필드 누락(인사이트 N/A)해당 배지만 "N/A"로 표기하고 카드 자체는 정상 렌더(R-13).
외부 접속 시도127.0.0.1에만 바인딩되어 외부에서 접근 불가(N-02).

엣지 케이스

  • 탐지 0건 → 카드 영역에 빈 상태 + "프로젝트 없음, 루트 설정 확인" 안내, 통계는 0으로 표시.
  • 프로젝트 수 대량(예: 200+) → 가독성 유지(목록 가상화/페이지네이션은 구현 선택, 성능 기준 N-01 준수).
  • 설명(description) 없음 → "(설명 없음)" placeholder.
  • 매우 긴 경로/이름 → 말줄임 처리, 전체값은 title/tooltip로 확인 가능.
  • 상태 배지는 색상만이 아니라 텍스트 라벨 병기(N-07 권고 — 정식 검증은 M4).

의존성 / 제약

  • F-1 스캔 산출 데이터에 의존(없으면 빈 상태).
  • UI는 빌드/프레임워크 없는 순수 HTML/CSS/JS(N-04). 서버는 localhost 전용(N-02).

F-4. 필터 · 정렬 · 검색

기능명필터 / 정렬 / 검색
관련 요구 IDR-11
목적목록에서 원하는 프로젝트를 언어·신선도·git 상태 필터, 정렬, 검색으로 빠르게 찾는다.
트리거대시보드의 필터 컨트롤 변경 / 정렬 키 선택 / 검색어 입력

사용자 스토리 + 수용 기준

  • 개발자로서 원하는 프로젝트를 빨리 찾기 위해 필터·정렬·검색하고 싶다.
    수용 기준: 언어/신선도/git 상태 필터는 AND 조합으로 동작한다. 이름·최종수정·용량으로 정렬할 수 있다. 이름·경로로 검색(부분 일치)한다. 성능: 200개 프로젝트 로드 상태에서 입력 이벤트→목록 DOM 재렌더 완료까지 performance.now() 계측 중앙값 ≤ 200ms(PRD §3 측정 정의·R-11).

정상 흐름 (Happy path)

  1. 사용자가 필터(언어/신선도/git)를 선택 → 클라이언트가 메모리상 목록에 AND 필터 적용.
  2. 정렬 키(이름/최종수정/용량) 선택 → 오름/내림차순 정렬.
  3. 검색어 입력 → 이름·경로 부분 일치로 추가 필터(필터·정렬과 결합).
  4. 결과 목록과 상단 카운트(필터 결과 N건)를 즉시 재렌더.

예외 / 에러 흐름

상황처리
필터/검색 결과 0건"조건에 맞는 프로젝트 없음" 안내 + 조건 초기화 버튼 노출. 전체 데이터는 보존.
정렬 키 값 결측(용량 N/A 등)결측 항목을 일관 규칙으로 정렬 말단 배치(예: 항상 뒤), 크래시 없음.
검색어에 정규식/특수문자리터럴 문자열로 처리(정규식 해석/주입 방지).

엣지 케이스

  • 필터·정렬·검색 동시 적용 → 필터(AND) → 검색 → 정렬 순으로 결합, 일관된 결과.
  • 모든 필터 해제 → 전체 목록 복원.
  • 대량(200+) 항목 빠른 연속 입력 → 디바운스/효율 렌더로 200ms 기준 유지.
  • 대소문자·공백 → 검색은 대소문자 무시·trim 적용.

의존성 / 제약

  • 클라이언트 측 처리(서버 왕복 없이 메모리 데이터에 적용) — 응답성 확보 및 N-01 충족.
  • F-3 목록 렌더에 의존.
