Project-SPIP — M8 설계 델타 (상태 주시 · 라이브 갱신)

기존 Electron 데스크톱 앱(electron-migration.html v1.1, m7-design.html rev.2 기준)에 사용자 요청 1건을 점진 확장으로 반영하는 설계 델타다. 한 번 스캔으로 발견된 프로젝트의 가변 신호(git·freshness)를 재스캔(파일시스템 워크) 없이 주기적으로 재수집해 대시보드에 라이브 반영한다(재스캔 전까지 계속 주시). 기존 스캔 파이프라인·스냅샷 계약·IPC·보안 베이스라인(EC·EM·M4·M6·M7 통제)을 0 약화로 보존하고, 새 채널 1개(단방향 push)와 워처 모듈 1개만 추가한다. 본 문서는 설계이며 실행 코드를 포함하지 않는다.

문서 유형: 설계 델타 · 작성: 수석 아키텍트 · 작성일: 2026-06-21 · rev.1 · 입력: scanner.js(buildProject §8.1 shape)·collectors/{git,freshness}.js·snapshotStore.js·scanController.js(status phase·store.load)·register.js(senderFrame guard)·preload.js(구독 채널 패턴)·app.js(subscribe*·toViewModel·Sortable)
설계 원칙: ① 단순성 — 무거운 walker/detector/size를 돌리지 않고, 이미 확정된 known project 경로에 git·freshness 수집기만 재적용(저비용·N-01 보존). ② 기존 보안 불변 — 경로는 매 tick pathGuard.canonicalize로 재해석(H-1), git은 resolveBin('git') 절대경로 + git -C(H-2), 페이로드는 스냅샷이 이미 노출하는 §8.1 shape만(L-3). ③ 추적성 — 요청을 R-24(상태 주시)로 부여하고 R-14(스캔)·R-15(진행 push)·R-08(git)·R-07(freshness)과 연결. ④ 결정 — 폴링 vs fs.watch 트레이드오프를 ADR-M8-1에 명시(폴링 채택).

1. 개요 · 동인 · 변경 범위

사용자 요청을 신규 요구 ID로 정리한다. 기존 코드를 갈아엎지 않는 추가로 설계한다.

요구사용자 요청아키텍처 동인 / 결정 영향
R-24한 번 스캔한 뒤로는 재스캔 전까지 상태를 계속 주시(라이브 갱신).M8 핵심. 스냅샷은 스캔 시점의 동결값이라 git/수정시각이 바뀌어도 수동 재스캔 전까지 stale. StateWatcher(주기 폴링)가 known project 경로에 git·freshness 수집기만 재적용해 변경분을 메인 창에 단방향 push. 무거운 워크/탐지/size 미수행(저비용). 스캔과 경합하지 않도록 스캔 중 tick 건너뜀. 영속은 재스캔만 담당(메모리 오버레이).

2. 품질 속성 요구

  • 성능(N-01) — 매 tick은 known 경로에 git(status+log -1)·freshness(얕은 mtime 표본)만. walker/detector/size 미수행. 기본 주기 15초(하한 3초 clamp). 직전 tick 미완 시 다음 tick 건너뜀(_busy).
  • 가용성 — 워처 타이머는 unref(프로세스 종료 비차단). tick 예외는 항목/패스 단위 격리(스캔·앱 무영향). 스캔 중(phase∈{scanning,finalizing}) tick 무동작.
  • 정확성 — 변경된 항목만 push(diff). git/freshness 9개 표시 필드 비교. 재스캔이 store.load()로 목록을 교체하면 다음 tick이 새 목록을 자동 주시.
  • 보안 — H-1 경로 재해석·H-2 git 절대경로·L-3 최소 노출(아래 §4).
  • UX — 드래그(SortableJS) 중에는 데이터만 병합하고 재렌더 보류(DnD 파손 방지). reduced-motion 무관(데이터 갱신만).

3. 주시 메커니즘 (폴링 · 재수집 · diff)

StateWatcher는 setInterval(tick, intervalMs)로 동작한다. 한 tick의 의사 흐름:

// lib/server/stateWatcher.js (의사코드)
async tick():
  if _busy return []                 // 재진입 금지
  if isScanning() return []           // 스캔과 경합 회피
  projects = store.getProjects()
  if projects.length == 0 return []
  _busy = true
  updates = []
  for p in projects:
    if isScanning() break             // 도중 스캔 시작 시 중단
    real = pathGuard.canonicalize(p.path)  // H-1: 소멸/이탈이면 null → skip
    if real == null continue
    git  = normalizeGit( await gitCollector.collect(real, ctx) )       // H-2: git -C real
    fresh= normalizeFreshness( freshCollector.collect(real, {git}) )
    if stateChanged(p, {git,fresh}):
      updates.push( store.applyLiveState(p.id, git, fresh) )            // 메모리 in-place
  _busy = false
  if updates.length: onUpdate({ projects: updates })                   // main → renderer push
  return updates

normalizeGit/normalizeFreshness는 scanner.buildProject와 동형 정규화(§8.1 shape)로, 라이브 값이 스캔 결과와 1:1 비교·표시 정합을 갖게 한다. stateChanged는 표시에 영향을 주는 git 6필드(status·isRepo·branch·dirty·ahead·behind) + freshness 3필드(lastModified·lastCommit·isStale)만 비교한다.

4. 보안 표면 · 통제

통제내용
H-1매 tick 각 프로젝트 경로를 pathGuard.canonicalize로 재해석한 뒤에만 수집한다. 소멸·심링크 이탈(null)은 건너뛴다. 워처는 스냅샷의 known 경로만 대상으로 하며 renderer 입력으로 경로가 주입되지 않는다(자율 폴링).
H-2git 수집기는 resolveBin('git') 절대경로 + git -C <path>(cwd 비의존)로 호출 — 부분신뢰 폴더의 위장 git.exe 미선택. safeExec 타임아웃·버퍼·in-flight 상한(M-4) 그대로.
L-3push 페이로드는 이미 getProjects가 노출하는 §8.1 project shape(갱신분)만 담는다 — 추가 경로·내부 상태·절대경로 비노출. 채널은 단방향 송신(main→renderer)이며 renderer가 호출하는 invoke 채널을 신설하지 않는다(공격 표면 0 증가).
격리tick·수집기 예외는 패스/항목 단위 try로 격리(스캔·앱 무영향). 구독자(onUpdate) 예외도 격리.

6. 데이터 흐름 · 렌더러 병합

파일시스템 변화 ──(15s tick)──▶ StateWatcher.tick
  └▶ git/freshness 재수집 ─▶ stateChanged? ─▶ SnapshotStore.applyLiveState (메모리 in-place)
                                          └▶ onUpdate({projects}) ─▶ win.webContents.send('spip:projectsUpdated')
renderer: spip.onProjectsUpdated(cb) ─▶ applyProjectsUpdate(payload)
  ├ store.raw / viewModels 를 id 로 교체 (toViewModel 재매핑)
  ├ stale KPI 로컬 재계산 (freshness 변동 반영)
  └ view=='dashboard' && !_dragging 면 render()  // 드래그 중엔 보류
  • 단일 진실 원천 — 워처가 applyLiveState로 메모리 스냅샷을 갱신하므로, 이후 getProjects(렌더러 새로고침/재로드)도 라이브 값을 본다. push는 즉시 반영용, 스토어는 정합 보존용.
  • 드래그 보류 — SortableJS onStart/onEnd가 store._dragging을 토글. 드래그 중 도착한 갱신은 데이터만 병합하고 재렌더를 미뤄, 드롭 후 자연 렌더에 반영(DnD 그리드 파손 방지).
  • 수동순서/필터 보존 — applyProjectsUpdate는 id 단위 치환만 하므로 현재 sortMode·필터·manual order 컨텍스트가 유지된다.

7. 생명주기 · 재스캔 경합

  • 생성 — context.js.buildContext가 stateWatcher를 단일 인스턴스로 생성(scanController와 동급 컴포지션).
  • 시작 — main.js가 창·IPC 배선 직후 start({store, config, isScanning, onUpdate}). 빈 store(0개)면 tick은 무동작이라 항상 시작해도 안전 — 첫 스캔으로 store.load()되면 그때부터 실제 주시("한 번 스캔한 이후" 충족).
  • 재스캔 경합 — isScanning = () => phase∈{scanning,finalizing}. 스캔 중 tick은 무동작. 스캔 완료 시 scanController가 store.load()로 목록을 교체(scanController.js:203) → 다음 tick이 새 목록을 자동 주시. 별도 재시작 불필요.
  • 정리 — disposeResources()(앱 종료 1지점)에서 stateWatcher.stop()(멱등). 타이머 unref로 누수 0.

10. ADR

ADR-M8-1 — 상태 주시 메커니즘: 주기 폴링 채택(vs fs.watch)

맥락. 스냅샷은 스캔 시점 동결값이라 git/수정시각 변화를 재스캔 전까지 못 본다. "재스캔 전까지 계속 주시"를 충족하려면 known 프로젝트의 가변 신호를 지속 갱신해야 한다.

결정. 주기 폴링(기본 15초)으로 known 경로에 git·freshness 수집기만 재적용한다. fs.watch(이벤트 기반)는 채택하지 않는다.

근거. ① 단순·예측가능 — 단일 타이머, 외부 의존 0, 자원 사용 상한이 명확(주기·_busy·스캔 게이트). ② 플랫폼 안정성 — 다수 디렉터리 재귀 watch는 Windows에서 핸들/이벤트 폭주·중복 이벤트·디바운스 복잡도가 크고 네트워크/이동식 드라이브에서 불안정. ③ 비용 적정 — git status/log·얕은 mtime 표본은 프로젝트당 저비용이고, 변경분만 push하므로 렌더 부하도 작다. ④ 경합 단순화 — 스캔과는 phase 게이트로 간단히 분리.

대안. fs.watch — 더 즉각적이나 자원·플랫폼 리스크·구현/검증 복잡도가 높아 보류. 하이브리드(watch+폴링) — 정확도 최고지만 과설계. 폴링으로 충분하며, 필요 시 후속 마일스톤에서 watch를 폴링 위에 얹는 점진 확장이 가능(인터페이스 동일: onUpdate).

영향. 최대 갱신 지연 = 폴링 주기(기본 15초). 즉시성이 필요하면 주기를 낮추거나(하한 3초) 향후 watch 보강. 영속은 재스캔만 — 라이브 값은 메모리 오버레이(재시작 시 마지막 스냅샷부터 다시 주시).

11. 리스크 · 가정 · 미해결

  • 가정 — 프로젝트 수가 수십~수백 규모. 매우 큰 모집단에서는 주기를 늘리거나 배치 분할(후속) 고려. 현재는 tick 내 순차 수집 + _busy 가드로 폭주를 막는다.
  • 지연 — 변화 후 최대 1주기(기본 15초) 후 반영. ADR-M8-1 영향 참조.
  • 영속 미수행 — 라이브 값은 디스크에 쓰지 않는다(잦은 쓰기 회피). 재시작 시 마지막 스냅샷 기준으로 다시 주시 시작. 의도된 단순화.
  • 미해결 — size(용량) 라이브 주시는 범위 외(측정 비용). 필요 시 별도 요구로 분리.
