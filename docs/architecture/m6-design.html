1. 개요 · 아키텍처 동인

백로그(docs/ref.md) 5개 항목을 아키텍처 동인으로 정리하고 신규 요구 ID를 부여한다. 모든 항목은 기존 코드를 갈아엎지 않는 추가(addition)로 설계한다 — 신규 IPC 채널은 기존 register.js guard에 등록만 하고, 영속화는 기존 lib/common/paths·config + folders.js의 0600 원자적 쓰기 패턴을 재사용한다.

2. 품질 속성 요구

품질 속성목표M6에서의 구조적 충족
보안 (최우선)기존 베이스라인 0 약화. 사용자 지정 실행 경로가 임의 명령 실행 표면이 되지 않음신규 채널 전부 register.js guard 경유(senderFrame·try/catch·FORBIDDEN/INTERNAL). resolveBin의 절대경로·.exe 한정·shell:false·인자 배열 강제를 사용자 지정 경로에도 동일 적용. 실행 직전 캐시 우회 강제 재검증(M6-H-1)·실행 인자 [projectPath] 고정·사용자 args 없음(M6-H-2)·toolId 화이트리스트(M6-M-1). 클립보드는 main clipboard.writeText(텍스트 한정).
유지보수성기존 모듈 경계 보존, 신규 책임 격리R-18 → lib/common/toolRegistry.js(신규, 도메인) + electron/ipc/tools.js(어댑터). UI 상태 → electron/ipc/uiState.js + lib/common/uiStateStore.js. 트레이 → electron/tray.js. 기존 파일 수정은 최소 지점(아래 §10).
확장성향후 에디터/툴 추가 시 코드 분기 없이 데이터로 확장R-18 툴 레지스트리는 tools: { <toolId>: { path, label } } 맵 구조 — 새 툴 추가는 맵 항목 추가뿐(args 필드 없음 — M6-H-2 시정). resolveBin은 toolId 무관 일반 함수. 단 toolId는 레지스트리 등록 known id 화이트리스트로만 실행(M6-M-1).
성능드래그/즐겨찾기 토글 즉시 반응(<50ms 체감). 영속 쓰기가 UI를 막지 않음UI 상태는 renderer 메모리에서 즉시 반영(낙관적)하고 IPC 영속은 비동기 fire-and-forget. ui-state.json은 작은 파일(수 KB)이라 0600 원자적 쓰기 비용 무시 가능.
가용성/견고성ui-state.json 손상·부재여도 graceful(빈 상태). 트레이만 남은 상태에서 안정 동작uiStateStore는 부재/손상 시 기본값(빈 favorites·빈 order)으로 폴백(config.js의 graceful 패턴 동일). 트레이는 창 파괴와 독립 생명주기.
접근성 (N-07 정합)드래그/즐겨찾기/슬라이더 키보드 도달 가능드래그는 키보드 대체(상/하 이동 버튼) 병행. 즐겨찾기 토글은 <button aria-pressed>. 슬라이더는 좌/우 버튼 + 화살표 키.

3. 영속화 데이터 모델 (통합 설계)

3종 신규 영속 데이터의 저장 위치를 성격에 따라 분리한다 — 설정류(tools)는 기존 spip.config.json(CLI와 공유되는 동작 설정), UI 상태류(favorites·card order)는 신규 ui-state.json(앱 GUI 전용, CLI 무관)에 둔다. 둘 다 동일 앱 데이터 폴더(appDir() = %APPDATA%\spip) 아래 두고, folders.js의 0600 원자적 쓰기(임시파일→fsync→rename, M-2)를 재사용한다.

3.4 uiStateStore 읽기 가드 (M6-M-4 — DoS / 손상 방어)

ui-state.json은 로컬 파일이라 디스크 손상·외부 조작으로 거대 JSON·깊은 중첩이 들어오면 JSON.parse 단계에서 메모리/시간 DoS가 가능하다. id가 실행·경로에 미사용이라 오염 영향은 작지만, 읽기 단계 가드를 명시한다(기존 H-3 _safeParse 패턴 재사용).

// lib/common/uiStateStore.js read() 가드(설계 — 구현 아님)
const MAX_UISTATE_BYTES = 1 * 1024 * 1024;   // 1MB 상한(M6-M-4)
const DEFAULT_STATE = { schemaVersion: 1, favorites: [], order: [], sortMode: 'auto' };

function read(deps) {
  try {
    const st = fs.statSync(uiStatePath());
    if (st.size > MAX_UISTATE_BYTES) return clone(DEFAULT_STATE);   // 크기 상한 초과 → 빈 상태 폴백
    const raw = fs.readFileSync(uiStatePath(), 'utf8');
    if (raw.length > MAX_UISTATE_BYTES) return clone(DEFAULT_STATE);  // 길이 재확인(symlink·경합 대비)
    const obj = _safeParse(raw);                                  // H-3 가드: 깊이 제한·예외 흡수, 실패 시 null
    if (!isPlainObject(obj)) return clone(DEFAULT_STATE);
    return normalizeState(obj);   // schemaVersion 불일치·배열 길이 상한(favorites≤512/order)·id 형식·중복 제거를 파싱 직후 재적용
  } catch (_) { return clone(DEFAULT_STATE); }   // 부재/손상/권한 → graceful 빈 상태
}
요지: ① 파싱 전 바이트 크기 상한(1MB) → 초과 시 즉시 기본값. ② _safeParse(JSON 깊이/예외 가드, 기존 H-3 패턴) 경유. ③ 파싱 직후 normalizeState가 배열 길이 상한·id 형식·중복 제거·sortMode 화이트리스트·schemaVersion 불일치 폴백을 재적용. 어떤 실패든 빈 상태로 graceful, 0600 유지.

5. R-17 — 프로젝트 경로 복사

카드/드로어에서 프로젝트 절대경로를 클립보드로 복사한다. 클립보드 접근은 main의 Electron clipboard.writeText로 한다(renderer navigator.clipboard 아님).

항목설계
클립보드 위치main clipboard.writeText. spip:copyText(text) 채널 신규. ADR-M6-1 참조.
preload 노출copyText(text) 1개만 최소 노출. 채널명 하드코딩.
모듈electron/ipc/clipboard.js(신규, copyText(args,{clipboard}) 순수 함수 — clipboard 주입으로 헤드리스 테스트).

데이터영속 없음(순간 동작).

// lib/common/toolRegistry.js (신규 도메인 — 외부 의존성 0, Electron 미import)
// config.tools(맵)를 정규화하고, toolId의 실행 경로를 "사용자 지정 우선 → PATH 폴백"으로 해석.

// 알려진 toolId 집합(M6-M-1/SQ-3). MVP는 'code'. 새 툴 추가 시 여기에 등록해야 실행 대상이 됨.
const KNOWN_TOOL_IDS = new Set(['code']);   // 향후: 'cursor','idea' 등 명시 추가

// 1) 정규화(loadConfig가 호출): 형식 검증·폴백. args 필드는 읽지 않고 무시(M6-H-2)
function normalizeTools(input, logger) {
  const out = {};
  if (!isPlainObject(input)) return out;
  for (const [id, v] of Object.entries(input).slice(0, LIMITS.maxTools)) {
    if (!/^[a-z0-9_-]{1,32}$/.test(id)) continue;       // 형식 화이트리스트
    if (!KNOWN_TOOL_IDS.has(id)) continue;               // 등록 known id만 채택(M6-M-1) — 임의 id 거부
    if (!isPlainObject(v)) continue;
    out[id] = {
      path: (typeof v.path === 'string' && v.path.length <= 4096) ? v.path : null,
      label: sanitizeLabel(v.label, id),                 // 제어/방향제어문자 제거·인쇄가능만·≤64(L-2). v.args는 읽지 않음(M6-H-2)
    };
  }
  return out;
}

// 2) 실행 경로 해석(actions.open이 호출): 사용자 지정 우선 → PATH. force 재검증으로 캐시 우회(M6-H-1)
function resolveTool(toolId, config, deps) {                // deps.resolveBin 주입 가능
  if (!KNOWN_TOOL_IDS.has(toolId)) return { bin: null, source: 'none' };  // 화이트리스트 외 toolId 즉시 거부(M6-M-1)
  const rb = deps.resolveBin;
  const tool = (config.tools && config.tools[toolId]) || null;
  // 사용자 지정 절대경로는 캐시를 믿지 않고 매 실행 force 재검증(절대·존재·.exe·실행파일·realpath)
  if (tool && tool.path) {
    const abs = rb(tool.path, { force: true });           // ★M6-H-1: 캐시 우회 강제 재검증(safeExec.js:77 force 분기)
    if (abs) return { bin: abs, source: 'config' };
  }
  // 폴백: 화이트리스트 멤버인 toolId만 PATH에서 탐색(임의 PATH 바이너리 호출 차단)
  const viaPath = rb(toolId, { force: true });
  if (viaPath) return { bin: viaPath, source: 'path' };
  return { bin: null, source: 'none' };
}
resolveBin 확장 최소화 + 캐시 우회(M6-H-1): 실측 safeExec.resolveBin(safeExec.js:82~91)은 이미 "name이 절대경로면 .exe·실행파일 검증만 수행"하며 opts.force=true 시 _binCache를 우회하고 fs를 재확인한다(line 77). 따라서 R-18은 resolveBin 자체를 바꾸지 않고, resolveTool이 매 실행 {force:true}로 호출하여 등록 후 파일이 악성으로 교체돼도 spawn 직전 최신 fs 상태로 재검증한다(TOCTOU·캐시 신뢰 제거). 기존 H-2 정책(.exe만·.bat/.cmd 거부·shell:false)이 사용자 지정 경로에도 그대로 적용된다 — 보안 핵심.
캐시 무효화 정책(P3-2): setToolPath 성공 시에도 해당 path/toolId에 대해 resolveBin(.,{force:true}) 재해석(또는 _clearBinCache)을 수행해 이전 캐시값을 폐기한다. getTools의 resolved 산출도 동일 force 경로를 사용해 항상 최신 상태를 반영한다.

6.2 open 흐름 일반화

// electron/ipc/actions.js openInVsCode 델타(설계) — resolveBin 직접 호출을 resolveTool로 교체
async function openInVsCode(args, ctx) {
  const id = sanitizeOpenId(args);
  if (id === null) return { ok:false, code:'ID_NOT_FOUND' };
  const toolId = sanitizeToolId(args && args.toolId) || 'code';   // 미지정 하위호환='code'
  if (!toolRegistry.isKnownToolId(toolId)) return { ok:false, code:'TOOL_NOT_FOUND' }; // ★M6-M-1: 화이트리스트 외 거부
  const project = ctx.store.getById(id);
  if (!project) return { ok:false, code:'ID_NOT_FOUND' };
  // H-1: pathGuard canonicalize/isAllowed (불변)
  const real = pg.canonicalize(project.path);
  if (real === null) return { ok:false, code:'PATH_GONE' };
  if (!pg.isAllowed(project.path, ctx.store.getAllowKeySet())) return { ok:false, code:'PATH_NOT_ALLOWED' };
  // H-2 + ★M6-H-1: resolveTool이 {force:true}로 spawn 직전 캐시 우회 강제 재검증(절대·존재·.exe·realpath)
  const r = toolRegistry.resolveTool(toolId, ctx.config, { resolveBin: rb });
  if (!r.bin) return { ok:false, code: toolId === 'code' ? 'CODE_CLI_NOT_FOUND' : 'TOOL_NOT_FOUND' };
  // H-2/M-4: shell:false·인자 배열·detached(불변). ★실행 인자는 [real] 고정 — 사용자 args 없음(M6-H-2)
  await exec(r.bin, [real], { shell:false, detached:true, inflightKey:'open:'+toolId+':'+id, maxInflight: MAX_INFLIGHT_PER_ID });
  return { ok:true, code:'OPENING' };
}
inflightKey 형식 변경(P3-1): 실측은 'open:'+id(actions.js:92)이나 본 일반화는 툴별 분리를 위해 'open:'+toolId+':'+id로 변경한다. 동작 영향은 없으나(내부 키) api-contract/electron-migration 갱신 시 이 형식 변경을 명시해 구현자 혼동을 막는다. 실행 인자는 항상 [real] 단일 고정(검증된 프로젝트 경로) — 사용자 입력 인자가 spawn에 도달하는 경로는 존재하지 않는다(M6-H-2 원천 제거).

9.1 트레이 + 생명주기(close-to-tray)

Electron Tray + 컨텍스트 메뉴(대시보드 열기 / 즐겨찾기 / 종료)를 추가하고, 대시보드 창을 닫아도 앱은 트레이에 상주한다. 트레이 '종료'만 완전 종료다.

9.2 생명주기 변경 상세

// main.js 생명주기 델타(설계) — 기존 close 핸들러·window-all-closed를 close-to-tray로
let isQuitting = false;   // 트레이 '종료'·앱 quit 시에만 true

// 트레이 생성(onReady 내, 창 생성 후)
// ★P2-2: onQuit가 app.quit()을 직접 부르면 before-quit이 먼저 발화→disposeResources()가 즉시 실행되어
//   이후 Q4 '취소' 시 scanController가 이미 dispose된 불일치 발생. → onQuit는 win.close() 트리거로 통일하고
//   실제 종료(destroy+quit)는 'close' 핸들러가 Q4 통과 후에만 수행하는 단일 경로로 만든다.
tray = createTray({
  onShowDashboard: () => { if (win) { win.show(); win.focus(); } },
  onShowFavorites: () => { if (win) { win.show(); win.webContents.send('spip:tray:favorites'); } },
  onQuit: () => { isQuitting = true; if (win) win.close(); else doFinalQuit(); },  // app.quit() 직접 호출 안 함
});

// 실제 종료 1지점: 자원 dispose는 여기서만(=Q4 통과/창 없음 확정 후).
function doFinalQuit() { disposeResources(); tray && tray.destroy(); app.exit(0); }

win.on('close', (e) => {
  if (!isQuitting) {                // ★ 평소 X 클릭: 종료 아닌 트레이로 숨김
    e.preventDefault();
    win.hide();
    return;
  }
  // 완전 종료 경로에서만 Q4(스캔 진행 중 확인) 적용 — 기존 로직 보존. ★dispose는 아직 안 함(P2-2)
  const phase = ctx.scanController.status().phase;
  if (phase === 'scanning' || phase === 'finalizing') {
    e.preventDefault();
    const choice = dialog.showMessageBoxSync(win, { type:'question', buttons:['종료','취소'], defaultId:1, cancelId:1, message:'스캔이 진행 중입니다. 종료하시겠습니까?' });
    if (choice === 0) { doFinalQuit(); win.destroy(); }   // 확정 시에만 dispose
    else { isQuitting = false; }    // ★취소: 종료 의도 해제·자원 미dispose로 정상 상태 복귀(스캔 지속)
  } else { doFinalQuit(); }
});

// ★ window-all-closed에서 quit 제거 — 트레이 상주(비 macOS도). 완전 종료는 onQuit→close→doFinalQuit가 담당.
app.on('window-all-closed', () => { /* no-op: 트레이 상주. 창 닫혀도 quit 안 함 */ });
// ★before-quit: dispose를 여기서 하지 않는다(중복·조기 dispose 방지). isQuitting 가드만.
//   외부 OS 종료(로그오프 등)로 before-quit이 직접 와도 doFinalQuit이 멱등하면 안전.
app.on('before-quit', () => { isQuitting = true; });
// ★M6-M-3: second-instance의 argv/cwd payload는 비신뢰 — 절대 파싱·경로 사용·실행 안 함. win.show 트리거로만.
app.on('second-instance', (_e, _argv, _cwd) => { if (win) { win.show(); if (win.isMinimized()) win.restore(); win.focus(); } });
상호작용설계
단일 인스턴스 락불변 유지. 둘째 실행 → second-instance에서 win.show()+focus(트레이로 숨어 있어도 복원). 트레이 상주 중 재실행해도 새 프로세스 안 뜸.
second-instance argv 신뢰 경계 (M6-M-3)second-instance가 전달하는 argv/cwd payload는 비신뢰 — 절대 파싱·경로 해석·실행에 쓰지 않고 win.show()/restore()/focus() 트리거로만 사용. 향후 "argv로 프로젝트 열기" 확장 시에도 이 payload는 신뢰 경계 밖임을 불변식으로 유지.
트레이 push action 화이트리스트 (M6-M-3)트레이 메뉴 클릭 push spip:tray:<action>의 action은 dashboard|favorites 고정 화이트리스트만. 임의 action push 금지. push는 메인 win.webContents로만(단방향).
Q4 종료 다이얼로그 + dispose 순서 (P2-2)완전 종료 경로(isQuitting=true)에서만 Q4 적용. 평소 X(close-to-tray)는 스캔 진행과 무관하게 hide만(스캔은 백그라운드 지속). 자원 dispose는 Q4 '종료' 확정 후 doFinalQuit()에서만 — Q4 '취소' 시 isQuitting=false로 복귀하고 scanController는 dispose되지 않은 정상 상태로 스캔 지속(조기 dispose 불일치 제거).
onQuit / before-quit (P2-2 단일 경로)트레이 '종료'는 app.quit()을 직접 호출하지 않고 win.close() 트리거→close 핸들러가 Q4 통과 후 doFinalQuit()(=disposeResources+tray.destroy+app.exit) 1지점에서 종료. before-quit은 isQuitting=true 가드만(dispose 중복·조기 실행 방지). doFinalQuit은 멱등 설계.
macOS기존 activate/Dock 동작 유지. close-to-tray는 전 플랫폼 공통(1차 타깃 Windows).

13. ADR

ADR-M6-1 — 클립보드 접근 위치: main clipboard (renderer navigator.clipboard 아님)

상태: 제안(승인 권고)

맥락: R-17 경로 복사를 (a) renderer navigator.clipboard.writeText vs (b) main clipboard.writeText + IPC 중 선택.

고려한 대안:

  • a. navigator.clipboard: IPC 불필요(코드 적음). 그러나 CSP default-src 'none'·sandbox·포커스/권한 프롬프트 의존으로 동작 불안정, "renderer는 OS 자원 직접 접근 0" 원칙 위배.
  • b. main clipboard + spip:copyText: 채널 1개·preload 함수 1개 추가. main이 OS 클립보드 단독 소유. 일관·예측 가능.

결정: b. spip:copyText(text) 신규, main clipboard.writeText. preload엔 copyText만 최소 노출.

근거: 기존 아키텍처(모든 OS 접근은 main, renderer는 트리거만)와 정합. 권한·CSP 변수 제거. 텍스트 한정이라 표면 극소.

결과: (+) 일관·안전·CSP 무영향. (−) IPC 왕복 1회(체감 무시). 클립보드 읽기는 미노출(필요 없음).

ADR-M6-2 — 외부 툴: 단일 code 경로가 아닌 툴 레지스트리(map)

상태: 제안

맥락: 백로그가 "향후 다른 툴도 추가할 수 있도록 확장성 고려"를 명시. 단일 codePath 설정으로 끝낼지, 일반 레지스트리로 갈지.

고려한 대안:

  • A. config.codePath(단일): 가장 단순하나 새 툴(cursor·neovim 등) 추가 시 매번 설정 키·코드 분기 필요 — 확장성 요구 미충족.
  • B. tools 맵 레지스트리: tools:{ <id>:{path,label} }(rev.2: args 필드 삭제 — M6-H-2). 새 툴 = 맵 항목 + KNOWN_TOOL_IDS 등록뿐. open은 toolId 일반화. resolveBin은 그대로({force:true}로만 호출).
  • C. 플러그인/스크립트 등록: 과설계 — 임의 실행 스크립트는 보안 표면 폭증, 단일 사용자 도구에 불필요.

결정: B(툴 레지스트리 맵). resolveTool(toolId)가 "사용자 지정 경로 우선 → PATH 폴백"하되 ① toolId는 KNOWN_TOOL_IDS 화이트리스트 멤버만(M6-M-1) ② 매 실행 resolveBin(.,{force:true})로 캐시 우회 강제 재검증(M6-H-1) ③ 실행 인자는 [real] 고정·사용자 args 없음(M6-H-2). resolveBin·safeExec 무수정 재사용.

근거: 확장성 요구를 데이터(맵)로 충족하면서 코드 분기 0. C의 임의 스크립트 표면은 거부. resolveBin이 이미 절대경로 검증·force 캐시 우회를 지원하므로 추가 비용 작음. args 제거·toolId 화이트리스트·force 재검증으로 보안 게이트(M6-H-1/H-2/M-1)를 설계 단계에서 해소.

결과: (+) 새 툴 데이터로 추가·보안 정책(.exe·shell:false·force 재검증) 일괄 적용·LOLBin/TOCTOU 표면 제거. (−) open 시그니처에 toolId 추가(미지정 시 'code' 하위호환). config 스키마 1키 증가. 새 툴은 코드의 KNOWN_TOOL_IDS에도 등록 필요(데이터만으로는 임의 추가 불가 — 의도된 보안 트레이드오프). 사용자가 실행 플래그를 커스터마이즈하는 기능은 미지원(보안 우선).

ADR-M6-3 — 영속화 위치: tools→config.json, favorites/order→신규 ui-state.json

상태: 제안

맥락: 3종 신규 영속 데이터를 (a) 전부 config.json (b) 전부 신규 ui-state.json (c) 성격별 분리 중 선택.

고려한 대안:

  • a. 전부 config.json: 파일 1개로 단순. 그러나 GUI 상태(빈번 갱신·폐기 가능)가 CLI 공유 설정과 섞여 오염·충돌·CLI 불필요 키 노출.
  • b. 전부 ui-state.json: tools는 동작 설정이라 CLI가 못 봄(git 경로 등 향후 공유 불가). 설정/상태 경계 모호.
  • c. 성격별 분리(2파일): tools=설정류→config.json, favorites/order=UI 상태→ui-state.json. 각자 검증·graceful 정책 적용.

결정: c. config.json에 tools 추가(loadConfig 편입), 신규 ui-state.json에 favorites·order·sortMode. 둘 다 0600 원자적 쓰기.

근거: 데이터 성격(공유성·검증·손상 허용도)이 다르면 분리가 더 안전·단순. 파일은 2개로만 한정(과분할 안 함). config 손상 위험(scanRoots 유실)과 UI 상태 폐기 가능성을 격리.

결과: (+) 책임 분리·CLI 호환·graceful 격리. (−) 파일·로더 1개 추가(uiStateStore, persistScanRoots 패턴 재사용으로 비용 작음).

ADR-M6-4 — close-to-tray: window-all-closed→quit 제거, 트레이 '종료'만 완전 종료

상태: 제안

맥락: 백로그가 "대시보드는 닫아도 트레이는 남아 있어야 한다 / 종료는 전체 종료"를 명시. 현재 main은 window-all-closed → app.quit().

고려한 대안:

  • A. 최소화로 트레이: minimize 가로채기. 그러나 "닫아도 상주" 요구(X 클릭)와 불일치.
  • B. close 가로채 hide + isQuitting 플래그: X→hide, 트레이 '종료'만 isQuitting=true로 실제 quit. window-all-closed no-op.

결정: B. win.on('close')에서 !isQuitting이면 preventDefault+hide. 트레이 '종료'는 app.quit()을 직접 부르지 않고 isQuitting=true; win.close() 트리거 → close 핸들러가 Q4 통과 후 doFinalQuit()(dispose+tray.destroy+exit) 단일 종료 경로로 종료(rev.2 P2-2). Q4(스캔 중 확인)는 완전 종료 경로에서만이며, 자원 dispose는 Q4 '종료' 확정 후에만·'취소' 시 isQuitting=false로 정상 복귀(미dispose).

근거: 백로그 요구 정확 충족. isQuitting 플래그는 Electron close-to-tray의 표준 패턴. 단일 인스턴스 락·Q4와 자연 정합. app.quit()→before-quit이 먼저 disposeResources를 실행해 Q4 취소 시 scanController 조기 dispose 불일치를 일으키므로, 종료 결정·dispose를 close 핸들러 한 곳으로 통일(P2-2 해소).

결과: (+) 트레이 상주·요구 충족·dispose 시점 일관(취소 시 정상 상태). (−) 사용자가 "닫았는데 안 꺼짐" 혼동 가능 → 최초 hide 시 트레이 풍선 안내 1회(MQ-4 확정). 스캔이 트레이 숨김 상태로 백그라운드 지속(의도된 동작). before-quit은 isQuitting 가드만 두고 dispose를 옮겨 외부 OS 종료 시 doFinalQuit 멱등성에 의존.

ADR-M6-5 — 즐겨찾기 슬라이더: 별도 창 아닌 메인창 내 오버레이

상태: 제안

맥락: 트레이 '즐겨찾기'의 우측하단 슬라이딩 UI를 (a) 별도 작은 BrowserWindow vs (b) 메인창 내 오버레이로.

고려한 대안:

  • A. 별도 BrowserWindow: 데스크톱 위젯처럼 항상 위 표시 가능. 그러나 webPreferences·preload·CSP·senderFrame·하드닝을 한 벌 더 구성·검증(보안 표면·복잡도 2배), favorites/projects 데이터 재적재 IPC 필요.
  • B. 메인창 내 오버레이(position:fixed 우측하단): 기존 보안 컨텍스트·데이터 그대로 재사용. 추가 창·preload 0. "우측하단 표시"는 CSS로 충족.

결정: B(오버레이). 트레이 '즐겨찾기'→main이 win.show()+spip:tray:favorites push→renderer가 오버레이 토글.
