Project-SPIP — 즐겨찾기 셸프 위젯 아키텍처

홈 대시보드("홈·브리핑")에 임의의 URL·폴더·파일을 한 "셸프"에 꽂아 두는 새 위젯을 추가한다. 스파인(접힘)↔배너(펼침) 전환, 활성 카드 자동 스크롤, 드래그/휠 가로 스크롤은 초안 favorites-shelf-widget.dc.html의 동작을 그대로 옮긴다. URL 메타데이터(og:image·title· description) 크롤링은 메인 프로세스에서 실제로 수행한다 — 이 위젯은 렌더러 connect-src 'none'를 유지한 채 외부 네트워크를 main에서만 허용하는, 본 앱 최초의 사용자 승인 egress 표면이다. 코드가 아니라 청사진이다 — 시그니처·스키마·의사코드·다이어그램까지만 정의하고 구현은 개발팀에 위임한다.

1. 개요 · 범위 · 아키텍처 동인

홈(store.state.view==='home')의 카드 그리드(renderHomeSection)에 합류하는 새 위젯 "즐겨찾기 셸프"를 설계한다. 책 셸프에 책등(스파인)을 꽂듯 즐겨찾기를 가로로 늘어놓고, 활성 항목만 배너 카드로 펼쳐 보여준다. 항목은 프로젝트 id가 아니라 임의의 URL/폴더/파일 경로를 직접 가리킨다.

기존 "프로젝트 즐겨찾기 위젯"과 완전 분리. electron/favoritesWidget.js· public/favorites.*·uiState.setFavorite·spip:favorites-changed 는 프로젝트 id 기반 별도 기능이며 일절 건드리지 않는다. 본 위젯은 신규 네임스페이스 spip:shelf:*· 신규 영속 키 shelfBookmarks로 독립한다.

2. 품질 속성 요구

보안(최우선)
main egress는 http/https + SSRF 차단 + 크기/타임아웃/리다이렉트 상한. 렌더러는 textContent만(L-1). 임의 경로는 canonicalize + 민감경로 deny.
성능
홈 진입 시 셸프는 캐시된 메타로 즉시 렌더(네트워크 0). 크롤은 add/refresh/스케줄러에서만. 크롤 1건 타임박스 ≤ 8s, 이미지 ≤ 100KB.
유지보수성
순수 함수(detectType·ogParse·ssrfGuard·pathPolicy·shelfWidgetVM)를 fs/net과 분리 → 헤드리스 단위테스트.
견고성
크롤/스캔 실패는 graceful — 항목은 폴백 메타로 꽂히고 error 상태 토스트. 손상 영속은 빈 셸프로 폴백.
확장성(데이터)
셸프 항목 상한 MAX_SHELF=64. 텍스트 필드 길이 상한. og:image 바이트는 ui-state 밖 별도 캐시(1MB 상한 보호).
호환성
CommonJS·바닐라·번들러 없음. electron-builder files(electron/·lib/·public/)에 자동 포함. 새 의존성 0.

3. 시스템 컨텍스트 · 신뢰 경계

본 위젯은 앱에 새로운 신뢰 경계를 도입한다: 메인 프로세스 → 임의 인터넷 호스트(사용자가 붙여넣은 URL). 이 경계가 SSRF·악성 응답의 입구이므로 다이어그램에 명시한다. 이후 security-reviewer의 위협 모델링 입력이다.

신뢰 IPC(검증된 origin) 비신뢰 egress(SSRF 표면 — 방어 집중) 로컬 FS I/O
핵심 위협(security-reviewer 입력). ① SSRF — 내부망/메타데이터 IP로의 크롤(§9.2). ② 악성 응답 — 거대 HTML/이미지·무한 리다이렉트·decompression bomb. ③ 경로 — 임의 경로 북마크의 민감 경로 열람/실행(§10). ④ 저장형 XSS — 크롤로 받은 title/desc를 innerHTML 렌더(L-1로 차단). ⑤ 정보노출 — og:image/data: URI 크기로 ui-state 비대(§11로 분리).

6. 홈 마운트 · 렌더러 통합

홈은 이미 구현된 마소너리 카드 그리드다(renderHome → applyHomeLayout → renderHomeSection(id)). 셸프는 새 섹션 id 'shelf'로 합류한다 — 별도 창/뷰가 아니라 홈의 한 카드다(SH-A1).

  1. HOME_SECTION_IDS에 'shelf' 추가(양쪽 enum). 위치는 기본순서상 activity 뒤 권장(상호작용 위젯 군집). 토글 가능(=featureAdd 아님 → 위젯 갤러리로 숨김/표시).
  2. renderHomeSection의 switch에 case 'shelf': return renderHomeShelf(); 추가.
  3. renderHomeShelf()는 shelfWidgetVM(store.shelf, active, composer)(순수)가 만든 plain object를 받아 dc.html 마크업을 el({text})(textContent)로 빌드. 배너 이미지는 el('img',{attrs:{src:dataUri}})(img-src data: 허용).
  4. 가로 스크롤 행에 wireSwipe(el)(드래그/휠)·focusActive(id)(활성 자동 스크롤)를 초안 로직 그대로 부착. ref 콜백 대신 마운트 후 querySelector로 행 노드 획득.
  5. store 확장: store.shelf(목록 캐시), store.shelfActive(펼친 id), store.shelfComposer({type,input,state:'idle'|'loading'|'error'}).
  6. 라이브 갱신: 스케줄러/refresh가 spip:shelf:changed push → onShelfChanged 구독이 store.shelf 갱신 후 홈이면 재렌더(메일 onMailUpdated 패턴과 동일).
레이아웃 주의. 마소너리는 CSS columns라 카드 폭이 컬럼 폭으로 제한된다. 셸프는 카드 안에서 overflow-x:auto로 가로 스크롤하므로 좁은 컬럼에서도 동작하지만, dc의 펼친 배너(290px)가 컬럼보다 넓을 수 있다. → 셸프 카드는 컬럼 스팬(전체폭) 옵션을 검토(또는 펼침폭을 컬럼에 맞춰 클램프). 미해결 §16-Q3.

7. 데이터 모델 · 영속

북마크 목록은 ui-state.json(0600)의 신규 키 shelfBookmarks에 저장한다(ADR-SH-1). og:image 바이트는 저장하지 않고 캐시 키(해시)만 둔다 — 1MB 상한·정규화 비용 보호(§11).

7.1 엔트리 스키마(정규화 후)

ShelfBookmark = {
  id:        string,   // 'b' + 6~32 hex. main 생성(crypto.randomBytes). 렌더러 시각 비신뢰
  type:      'url' | 'folder' | 'file',
  ref:       string,   // url=원본 URL(검증된 http/https) / folder·file=canonicalize된 실경로
  // ── 크롤/스캔 캐시 메타(표시용 · 전부 sanitize·길이상한 · L-1 textContent 렌더) ──
  name:      string,   // 스파인 라벨(브랜드/폴더명/파일명)  ≤ 120
  title:     string,   // 배너 제목  ≤ 200
  sub:       string,   // host 또는 tidy 경로  ≤ 200
  desc:      string,   // 설명  ≤ 500
  color:     string,   // '#RRGGBB' 형식만(정규식). 배너 그라데이션·스파인 배경
  mono:      string,   // 1~4자 모노 라벨
  cat:       string,   // 카테고리 칩  ≤ 32
  status:    string,   // '200 · 120ms' / '128개 파일 · 12MB' 등  ≤ 80
  bannerKey: string|null, // og:image 캐시 키(sha256 32hex) 또는 null. 바이트는 별도 파일
  lastChecked: number|null, // ms epoch. main 스탬프
  addedAt:   number,   // ms epoch. main 스탬프
  order:     number    // 정렬 키(reorder가 갱신). 또는 배열 순서 자체로 표현
}
order 표현. 배열 순서 = 표시 순서로 두고 reorder(ids)가 배열을 재배열하는 방식을 권장(별도 order 필드 불요, 프로젝트 setOrder와 동형). 위 order 필드는 선택.

7.2 정규화 규칙 — normalizeShelfBookmarks(input)

  • 비배열 → []. 항목 MAX_SHELF = 64 상한, 초과분 절단.
  • id: /^b[0-9a-f]{6,32}$/ 외 폐기. 중복 id 제거.
  • type: 화이트리스트 외 폐기.
  • ref: url은 validateHttpUrl 통과만(http/https·자격거부·≤2048). folder/file은 비어있지 않은 문자열(열기 시 재-canonicalize·재게이트 — §10).
  • 텍스트 필드: sanitizeText(C0/DEL·방향성 제어 제거 + trim + 필드별 길이상한). 빈 문자열 허용(폴백 표시).
  • color: /^#[0-9a-fA-F]{6}$/ 외 → 기본색. bannerKey: 32hex 외 → null.
  • 수치(addedAt/lastChecked): 유한수만, 아니면 null.
  • schemaVersion 폴백: 키 부재/구버전 → defaultState()가 shelfBookmarks:[] 제공(브리핑 키와 동일 graceful).

7.3 영속 흐름

read: uiStateStore.read() → 정규화된 shelfBookmarks 포함. write: uiStateStore.write(state)가 0600 원자적 쓰기·상승세션 write 보류·1MB 상한·깊이 가드를 그대로 적용. IPC list는 read 결과에 imageCache.toDataUri(bannerKey)를 붙여 반환(저장본엔 키만).

9. URL 크롤링 서브시스템(main) · SSRF 방어

렌더러 connect-src 'none'는 유지된다. 크롤은 main의 lib/shelf/urlMeta.crawl(url)가 node:https/node:http로 수행한다. SSRF가 본 기능 최대 위험이므로 다층 방어한다.

9.1 요청 정책

  • 메서드 GET만. 헤더 최소화(고정 User-Agent, Accept: text/html, no-referrer 의미).
  • 타임아웃 ≤ 8s(연결+응답). 응답 본문 크기 상한 512KB(초과 시 소켓 destroy). content-type이 text/html 계열 아니면 본문 미파싱(폴백 메타).
  • 리다이렉트 최대 3 hop, 각 hop마다 §9.2 재검증(Location URL 파싱→스킴·자격·DNS·IP 게이트). 무한 루프·credential leak 방지.
  • 압축 응답은 크기 상한을 해제 후 누적 바이트 기준으로도 재확인(decompression bomb 방지) — 또는 Accept-Encoding: identity로 압축 비요청(단순·안전, 권장).
  • 쿠키 저장·전송 없음. 인증 헤더 없음.

9.2 SSRF IP 게이트 — ssrfGuard

validateBriefingUrl(http/https·자격거부·길이)을 일반화한 validateHttpUrl로 1차 검증 후, 호스트명을 DNS resolve해 결과 IP를 검사한다. 다음에 해당하는 IP가 하나라도 있으면 BLOCKED_HOST로 거부:

분류IPv4IPv6
루프백127.0.0.0/8::1
사설10/8, 172.16/12, 192.168/16fc00::/7 (ULA)
링크로컬169.254.0.0/16fe80::/10
클라우드 메타데이터169.254.169.254, 100.100.100.200fd00:ec2::254

멀티캐스트/예약224/4, 240/4, 255.255.255.255ff00::/8
  • DNS rebinding/TOCTOU 완화: resolve로 얻어 게이트를 통과한 그 IP로 직접 연결(https.request({ lookup })의 lookup을 검증된 IP만 반환하도록 핀, 또는 options.host=검증IP + servername=원래host + Host 헤더). 연결 시점에 도메인을 다시 해석하지 않는다.
  • 리다이렉트 hop의 새 호스트도 동일 절차 반복.
  • 숫자 IP 직접 입력(http://10.0.0.1)도 동일 게이트에 걸린다(파싱→분류).

9.3 ogParse(순수)

정규식 기반 경량 추출(cheerio 미도입, ADR-SH-5). 추출 우선순위: og:title → <title>, og:description → <meta name=description>, og:image(절대 URL로 resolve), og:site_name → host. 기본 HTML 엔티티만 디코드(&amp; &lt; &gt; &quot; &#39; &#NNN;). 결과는 전부 sanitize·길이상한 후 반환 — 신뢰불가 문자열이므로 렌더러에서 textContent로만 표시(L-1). 파싱 실패/빈 메타 시 fallback(host)(초안 로직)로 폴백.

정규식 파서 한계. 비정상/거대 HTML에 대해 ReDoS·과매칭 위험 → 입력 바이트는 이미 512KB로 상한, 정규식은 비탐욕·앵커·캡처 최소화로 작성하고 매치 횟수 상한(예: meta 태그 첫 N개만 스캔)을 둔다.

10. 폴더/파일 메타 수집 · 경로 정책

임의 경로 북마크는 프로젝트가 아니므로 scanRoots 화이트리스트 강제는 부적절하다(사용자가 직접 추가). 대신 canonicalize(H-1) + 민감경로 deny 게이트 + 자기참조 화이트리스트의 3중 정책을 쓴다(ADR-SH-4).

10.1 add/refresh 시 수집(localMeta)

  • pathGuard.canonicalize(ref) → 실경로(심링크/junction 해소·NFC). null이면 PATH_GONE.
  • pathPolicy.isDenied(real) 통과 검사 — 통과(=거부) 시 PATH_DENIED.
  • fs.stat로 유형 확인: folder=isDirectory, file=isFile. 불일치 시 type 자동 보정 또는 거부.
  • folder 메타: 얕은 순회(maxDepth·maxEntries·budgetMs 예산, 기존 size 수집기 규약 차용)로 파일 수·용량 근사. 예산 초과 시 partial 표기.
  • file 메타: 확장자→언어/모노/색(초안 fileMeta 맵 포팅), 크기·mtime.
  • ref에는 canonicalize된 실경로를 저장(표시는 tidyPath로 ~ 치환).

10.2 민감경로 deny 게이트 — pathPolicy.isDenied(real)

거부 대상예
드라이브/파일시스템 루트C:\, D:\, /
OS 시스템 디렉토리%WINDIR%(C:\Windows\**), /etc, /sys, /proc, /dev, /usr, /bin, /System(macOS)
자격/비밀 디렉토리~/.ssh, ~/.aws, ~/.gnupg, ~/.config/gh 등(접두 매칭, 폴드)
해소 실패canonicalize=null

비교는 pathGuard.foldForCompare로 폴드(대소문자 비민감 FS 일관). 시스템 디렉토리는 접두(하위 포함) 매칭으로 차단. 사용자 홈 일반 하위(프로젝트·문서 등)는 허용된다.

10.3 열기(open) 정책

  • 저장된 ref(실경로)를 열기 시점에 재-canonicalize·재-게이트(TOCTOU 축소). 저장 후 경로가 시스템 디렉토리로 바뀌었으면 거부.
  • folder → safeExec(resolveTool('code'), [real], {shell:false, detached:true})(VS Code, H-2, 기존 openInVsCode와 동일 규약). 도구 미설정/실패 시 shell.openPath(real) 폴백(OS 기본앱).
  • file → shell.openPath(real)(OS 기본앱, PM #2 확정 — file은 항상 기본앱, VS Code 강제 안 함).
  • url → shell.openExternal(ref). 단 http/https 재검증 필수(javascript:·file:·커스텀 스킴 차단 — openExternal는 임의 스킴을 OS 핸들러로 넘길 수 있어 위험).
  • 실행 인자는 [real] 고정(사용자 args 없음). id별 in-flight 상한.

11. og:image 캐시 · 전달

렌더러는 외부 이미지를 직접 받을 수 없다(img-src 'self' data:; connect-src 'none'). 현재 CSP가 data:를 허용하므로(실측 public/index.html CSP meta — img-src 'self' data:), main이 다운로드한 og:image를 data: URI로 렌더러에 전달한다(ADR-SH-2). 바이트는 ui-state에 넣지 않고 appData 캐시 파일에 둔다.

  • fetch: §9.2 SSRF 게이트 통과한 og:image URL만. content-type이 image/(png|jpeg|gif|webp|svg+xml?)이어야 함. SVG 주의 SVG는 스크립트 내장 가능 → data: URI로 <img>에 넣으면 Chromium이 스크립트 실행 안 함(이미지 컨텍스트)이나 보수적으로 SVG 제외 권장.
  • 크기 상한 100KB(PM #4 확정, 초과 시 거부 → bannerKey=null, 그라데이션 배너 폴백). 이미지 리사이즈는 의존성 0 제약상 미수행(§16-Q1 해소).
  • 저장: imageCache가 appData/shelf-images/<sha256(refURL)>.<ext>로 0600 write. 키=해시.
  • 전달: list/add/refresh 응답에서 toDataUri(bannerKey)(파일 읽어 base64 + 검증된 mime)로 bannerImage 필드 생성. 저장본엔 키만(payload 비대 방지: 64항목×100KB×1.34 ≈ 8.6MB 최악 — list는 일괄이므로 지연 로드 검토: 펼친 항목만 data: URI, 스파인은 색만).
  • GC: 북마크에서 참조되지 않는 캐시 파일은 주기/remove 시 정리. 총 캐시 용량 상한.
대안(미채택, ADR-SH-2 참조). app:// 예약 라우트(app://shelf-img/<key>)로 캐시 디렉토리 서빙 → 지연/브라우저 캐시 유리하나 보안 민감한 프로토콜 핸들러를 수정해야 한다. MVP는 data: URI(프로토콜 무수정). 목록이 커져 payload가 부담되면 app:// 라우트로 전환(2차).

12. 자동 재크롤 스케줄러(6시간)

초안 푸터 "6시간마다 자동으로 다시 스캔·크롤링" 문구를 충족. lib/shelf/scheduler.js가 main에서 setInterval(6h)로 북마크를 순회 재수집한다.

  • 저빈도·타임박스: 항목당 크롤 ≤ 8s, 동시 1건(직렬) 또는 소수 병렬 + in-flight 상한. 전체 배치 watchdog.
  • 변경분만 영속(메타 diff). 갱신 후 spip:shelf:changed push → 홈 재렌더.
  • 상승세션(elevated)이면 write 보류(uiStateStore가 자동 처리) — 크롤 자체는 스킵 권장(불필요 egress 회피).
  • 기본값: ON + 끄기 토글(설정 영속) — PM #1 확정(Q2 해소). 백그라운드 자동 egress가 사용자 모르게 발생하므로 on/off 설정을 제공하되, 기본은 켜짐이며 사용자가 끌 수 있다. 토글 off면 스케줄러 tick은 무동작(수동 refresh만). 앱 실행 중에만 동작(데몬/백그라운드 상주 아님).
  • 앱 시작 직후 즉시 실행 금지(부팅 지연 회피) — 첫 tick은 지연 후.

13. 보안 불변식 체크리스트

  • L-1 크롤/스캔 산출 문자열(title·desc·og·name·status)은 전부 textContent로만 렌더. innerHTML 금지. 배너는 <img src=data:>(속성 주입, 마크업 주입 아님).
  • H-1 folder/file는 add·open 양쪽에서 pathGuard.canonicalize + 폴드 비교. ref는 실경로 저장.
  • H-2 folder 열기는 safeExec 절대경로·shell:false·인자 [real] 고정(VS Code), .bat/.cmd 차단 그대로. file은 shell.openPath(OS 기본앱, PM #2) — 외부 프로세스 spawn 아님. 양쪽 모두 열기 시점 canonicalize+pathPolicy 재게이트.
  • 경로 deny pathPolicy 민감/시스템 경로 차단(드라이브 루트·시스템·자격 디렉토리).
  • SSRF URL·og:image 모두 scheme/자격/길이 + DNS IP 분류 게이트 + IP 핀 + hop 재검증.
  • CSP 렌더러 connect-src 'none' 유지(외부 fetch 0). 모든 egress는 main IPC. img-src data: 활용(현 CSP 무변경).
  • IPC 전 채널 guard(senderFrame origin) 경유. preload 채널 하드코딩. main 인자 재검증.
  • 영속 0600 원자적 쓰기·상승세션 write 보류·1MB/깊이 상한(uiStateStore 재사용). og:image 바이트는 ui-state 밖 캐시(0600).
  • L-3 고정 에러코드만 반환. 실경로·해소 IP·스택은 로그 전용.
  • 응답 한도 크롤 본문 512KB·이미지 100KB·타임아웃 8s·리다이렉트 3·압축 미요청(bomb 방지).
  • 의존성 새 런타임 의존성 0(node 내장만). 셸프 항목 64·텍스트 길이 상한.
  • openExternal url은 http/https 재검증 후에만 외부 열기(임의 스킴 차단).

14. ADR — 핵심 결정

본 위젯의 결정은 자족성을 위해 이 문서에 둔다(프로젝트 표준 adr.html에는 요약 1줄 추가 권장).

ADR-SH-1 승인 — 북마크 영속 위치

맥락. 임의 URL/폴더/파일 북마크 목록을 0600·정규화·상한·상승세션 보류로 안전 영속해야 한다.

대안. ① ui-state.json 신규 키 shelfBookmarks(기존 인프라 재사용) ② 별도 shelf.json 신규 store ③ config.json에 저장.

결정. ①. uiStateStore에 normalizeShelfBookmarks 추가.

근거. 원자적 쓰기·1MB/깊이 가드·상승세션 보류·schemaVersion 폴백이 이미 검증됨. todos/briefing 신규 키 추가 선례 동일. ②는 동일 인프라를 재구현하는 중복. ③ config는 CLI/스캔 설정 영역이라 부적합. og:image 바이트는 제외(별도 캐시)해 1MB 상한 보호.

결과. +구현·테스트 최소, 일관성. −ui-state 단일 파일에 책임 증가(항목 상한 64로 완화). 추적: SH-A2·SH-A7

ADR-SH-2 승인 — og:image 렌더러 전달 방식

맥락. 렌더러는 외부 이미지를 직접 못 받는다(connect-src 'none'). 현 CSP는 img-src 'self' data:.

대안. ① main 다운로드→data: URI 전달(캐시는 appData 파일) ② app:// 예약 라우트로 캐시 서빙 ③ og:image 미렌더(그라데이션 배너만).

결정. ① data: URI(MVP). 목록 비대 시 펼친 항목만 data: URI(지연), 추후 ②로 확장 가능.

근거. 현 CSP 무변경(data: 이미 허용), 보안 민감한 프로토콜 핸들러 수정 회피. ②는 깔끔하나 프로토콜 표면 확대. ③은 승인된 og:image 기능 상실.

결과. +프로토콜 무수정·빠른 구현. −일괄 list payload 증가(지연 로드·64·100KB 상한으로 완화). 추적: SH-A3·SH-A4

ADR-SH-3 승인 — 크롤을 main에서 수행 + SSRF 방어

맥락. og 메타를 실제로 가져오려면 외부 네트워크가 필요. 렌더러는 차단됨.

대안. ① main node:https 크롤(+SSRF 게이트) ② 렌더러 fetch(CSP 완화) ③ 크롤 없이 host 기반 더미 메타.

결정. ①. egress는 main 단독, 다층 SSRF 방어.

근거. ②는 connect-src 'none' 불변식 파괴(앱 전체 보안 약화) — 불가. ③은 사용자 승인 기능(실제 og) 미충족. main 한정 egress는 LLM/IMAP/updater 선례와 일관.

결과. +불변식 유지·실제 메타. −SSRF/응답 방어 구현·테스트 부담(신뢰 경계 신설). 추적: SH-A3·SH-A4·SH-A5

ADR-SH-4 승인 — 임의 경로 북마크 보안 정책

맥락. 사용자가 직접 추가하는 폴더/파일은 프로젝트가 아니라 scanRoots 화이트리스트가 없다.

대안. ① scanRoots 강제(추가 불가능해짐) ② 무검증 canonicalize 후 그대로 열기 ③ canonicalize + 민감경로 deny 게이트 + 자기참조 화이트리스트(열기 시 재검증).

결정. ③.

근거. ①은 기능 자체를 막음(부적절). ②는 시스템/자격 경로 열람·실행 위험. ③은 H-1 정신(canonicalize 단일 진실)을 유지하면서 임의 경로를 허용하되 위험 경로만 차단. 열기 시 재-canonicalize·재게이트로 TOCTOU 축소.

결과. +유연성+안전 균형. −deny 목록 유지보수(플랫폼별). 추적: SH-A2·SH-A6

ADR-SH-5 승인 — HTML 메타 파싱 방식

맥락. og 태그 추출에 HTML 파싱 필요. 새 런타임 의존성 0 원칙.

대안. ① 자작 경량 정규식 파서 ② cheerio/parse5 등 의존성 추가 ③ 렌더러 DOMParser(불가 — main에 DOM 없음).

결정. ① 자작 ogParse(meta/title만 대상, 비탐욕·매치상한·엔티티 일부 디코드).

근거. ②는 CLAUDE.md 의존성 0 위배. ③ main은 DOM 없음. og/meta 추출은 정규식으로 충분(완전 파서 불요). 입력 512KB 상한+매치상한으로 ReDoS 완화.

결과. +의존성 0. −일부 비정상 HTML 추출 누락(폴백 메타로 graceful). 추적: SH-A5
