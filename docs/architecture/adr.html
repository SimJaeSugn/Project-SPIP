ADR-001 — 외부 npm 의존성 0개, Node 내장 모듈만 사용

상태: 승인 · 관련 요구: N-04

맥락 (Context)

N-04는 "외부 npm 의존성 최소화(가능하면 Node 표준 모듈 위주), 빌드/프레임워크 없음"을 요구한다. 가벼운 설치·낮은 운영 비용·프라이버시(공급망 위험 최소)가 제품 가치(§2)다. 동시에 디렉터리 순회·git 호출·웹서버·JSON 처리가 필요하다.

고려한 대안

대안장점단점/트레이드오프
A. 내장 모듈만 (fs/path/child_process/http) ← 채택의존성 0, 공급망 위험 0, 설치 즉시 동작, N-04 완전 충족일부 편의(라우팅·glob)를 직접 구현 → 코드량 소폭 증가
B. 최소 유틸 몇 개 추가 (express, chalk, fast-glob 등)개발 편의·코드량 감소의존성 트리 수십 개 유입, 공급망·취약점 표면 증가, N-04 위배 소지
C. 번들러/TS 도입타입 안전·모듈 관리빌드 단계 추가 = 확정 제약("빌드 없음") 정면 위배

결정 (Decision)

A 채택. 외부 npm 의존성 0개를 목표로 하고 fs·path·child_process·http·os·crypto·node:test 등 내장 모듈만 사용한다. 라우팅·정적 서빙·순회는 lib/에서 직접 구현한다. 의존성 추가가 불가피해지면 개별 ADR로 근거를 남긴다.

근거 (Rationale)

  • 요구 범위(라우트 소수·로컬 전용·정적 UI)가 작아 직접 구현 비용이 의존성 관리 비용보다 낮다.
  • 공급망 취약점·라이선스 리스크 0 → 프라이버시 우선 제품 성격과 정합.
  • 설치 = clone/link 즉시 동작. node_modules 자체가 본 도구가 경고하는 디스크 잠식 대상이라는 점에서도 일관.

결과 (Consequences)

  • + 가장 가벼운 설치·실행, 보안 표면 최소.
  • + 빌드 파이프라인 부재로 진입 장벽 낮음.
  • − 라우팅·MIME·순회를 직접 구현 → lib/ 코드량 증가, 테스트로 품질 보전 필요.
  • − 고급 기능(예: 정교한 glob) 필요 시 직접 작성 부담.

ADR-002 — Git 상태 수집은 git CLI(child_process, shell:false, 절대경로 고정)로

상태: 승인 (rev.2 — 보안 H-2 반영) · 관련 요구: R-08 R-13 N-03 N-04 · 보안 H-2·M-4

맥락 (Context)

R-08은 저장소 여부·브랜치·미커밋 변경·ahead/behind 수집을 요구한다. 방법은 (1) git CLI 호출, (2) .git 디렉터리 순수 파싱, (3) 외부 git 라이브러리 중 하나다. N-04(의존성 최소)·N-03(인젝션 방지)·R-13(부재 시 graceful)을 동시에 만족해야 한다.

고려한 대안

대안장점단점/트레이드오프
A. git CLI via child_process.spawn(shell:false) ← 채택정확·견고(git이 직접 계산), 의존성 0, ahead/behind까지 신뢰성 있게 취득git 미설치 시 사용 불가 → 단, R-13 graceful로 'N/A' 처리하면 됨. spawn 호출 비용
B. .git 디렉터리 순수 파싱(JS)git 미설치여도 동작, 외부 프로세스 0packed-refs·인덱스·원격 추적 등 구현 복잡·취약, ahead/behind 정확 계산 난이도 높음, 유지보수 부담 큼
C. isomorphic-git 등 라이브러리JS만으로 git 동작외부 의존성(트리 큼) → N-04 위배, 번들 무게

결정 (Decision)

A 채택 + 보안 강화(H-2). collectors/git.js가 common/safeExec를 통해 git 상태를 취득한다. 인자는 항상 배열로 전달하고 경로·브랜치명을 셸 문자열로 연결하지 않는다(N-03). git 미설치/비저장소/타임아웃이면 status:'na'로 반환해 전체 스캔을 중단시키지 않는다(R-13).

H-2 반영(필수): git 실행 파일은 이름만 고정하지 않고 기동/스캔 시작 시 resolveBin('git')으로 절대경로를 1회 해석해 고정한 뒤 그 절대경로로 spawn한다. git collector가 cwd:projectPath(부분신뢰 폴더, TB-B)에서 돌더라도 폴더 내 위장 git.exe/git.bat가 PATH 우선순위로 선택돼 RCE로 이어지는 경로를 막는다. 가능하면 git -C <projectPath> 인자를 써서 cwd 의존을 제거하고, Windows에서는 PATHEXT 기반 .bat/.cmd 자동 실행을 차단한다(절대경로 .exe만). M-4 반영: 호출에 타임아웃+강제 kill·출력 버퍼 상한을 둔다.

근거 (Rationale)

  • git CLI는 ahead/behind·dirty 판정을 정확하게 제공 — 직접 파싱 대비 정확성·견고성 우위.
  • shell:false+인자 배열은 N-03의 인젝션 방지 규칙을 구조적으로 충족(셸 해석 없음).
  • 절대경로 고정(H-2)은 PATH 하이재킹·위장 바이너리 선택을 차단 — 이름만 고정하면 부분신뢰 cwd에서 RCE 표면이 남기 때문(보안검증 1회차 높음).
  • 의존성 0 유지(ADR-001 정합). 부재는 제품 가정상 graceful 처리로 충분(R-13).
  • git 호출은 프로젝트당 1회로 묶어 성능 비용(N-01·RA2) 통제, 타임아웃으로 행(hang) 차단(M-4).

결과 (Consequences)

  • + 정확한 git 인사이트, 인젝션 안전, 의존성 0.
  • + code 실행(R-12)과 동일한 safeExec 경로 재사용 → 보안 통제(절대경로·타임아웃) 일원화.
  • + 절대경로 고정으로 부분신뢰 폴더 내 위장 바이너리 RCE 차단(H-2).
  • − 실행 파일 절대경로 해석(resolveBin) 단계가 추가됨(기동/스캔 시작 시 1회) — 해석 실패 시 graceful 'na' 처리 필요.
  • − git 미설치 환경에서는 git 인사이트가 'N/A'(기능 일부 비활성, 단 사용자 안내).
  • − 대량 프로젝트에서 spawn 누적 비용 → 묶음·타임아웃·격리로 완화 필요.

ADR-003 — 데이터/설정 저장은 단일 OS 앱 데이터 폴더의 JSON 파일

상태: 승인 (rev.2 — P1-2 위치 확정·M-2 권한) · 관련 요구: N-06 N-04 PRD §10 · 검증 P1-2 · 보안 M-2

맥락 (Context)

스캔 결과(~200개 프로젝트 메타데이터)와 사용자 설정을 영속화해 서버/스캐너가 읽어야 한다. N-06은 결과 JSON의 OS 사용자 캐시 폴더 저장·프로젝트 폴더 비오염·단일 사용자·인증 없음을 요구하나, 설정 파일 위치는 N-06이 규정하지 않아 본 ADR에서 확정이 필요하다(검증 P1-2). 동시 쓰기 없음(스캔은 단일 프로세스), 읽기 위주다. 스냅샷은 전체 프로젝트 절대경로·디렉터리 구조·git 브랜치 등 프라이버시 정보를 담으므로 파일 권한 결정도 필요하다(보안 M-2).

고려한 대안

대안장점단점/트레이드오프
A. JSON 파일 + 원자적 쓰기, 단일 앱 데이터 폴더(config/·cache/) ← 채택의존성 0, 사람이 읽기 쉬움, 서버 1회 로드로 충분, 설정·캐시 위치를 한 폴더로 단순화(paths.js 계약 명료)대용량·복잡 쿼리엔 부적합(본 규모엔 무관)
A'. 설정은 OS 설정 폴더, 캐시는 OS 캐시 폴더로 분리OS 관례에 가장 부합(cache는 비우기 가능 의미)경로 베이스가 OS별로 2종씩 → paths.js 분기·문서 표기 증가. 200개 규모엔 이득 미미
B. SQLite (내장 모듈 아님)쿼리·인덱스·증분 갱신외부 의존성 또는 네이티브 바인딩 → N-04 위배, 200개 규모에 과설계
C. 프로젝트 폴더에 결과 파일경로 단순N-06 위배(폴더 오염), 다수 위치 산재

결정 (Decision)

A 채택 (P1-2 확정). 설정과 캐시를 단일 OS 앱 데이터 디렉터리(Win %APPDATA%\spip\ / macOS ~/Library/Application Support/spip/ / Linux $XDG_CONFIG_HOME/spip/) 아래 용도별 하위로 둔다 — 설정은 config/spip.config.json, 스냅샷은 cache/projects.json. common/paths.js가 appDir()·configPath()·cachePath()로 단일 해석한다. serializer는 임시파일 쓰기 후 rename으로 원자적 교체하고(부분 쓰기 손상 방지), 두 파일과 임시 파일을 소유자 전용 권한(POSIX 0600/디렉터리 0700; Windows ACL)으로 생성한다(M-2). 스키마는 schemaVersion을 가진다.

A'(설정·캐시 베이스 분리) 대신 A(단일 앱 폴더)를 택한 이유는 paths.js 계약과 문서 표기를 단순화하고 PM 결정(단일 앱 데이터 디렉터리 통일)과 정합하기 위함이다. cache 의미를 보존하려는 A'의 이점은 200개 규모·단일 사용자에서 미미하다.

근거 (Rationale)

  • 200개 메타데이터는 수백 KB 수준 → 서버가 1회 로드해 메모리 보관해도 가볍다(N-01).
  • JSON은 의존성 0·디버깅 용이·스키마 진화 쉬움.
  • 원자적 쓰기로 스캔 중단/충돌 시에도 손상 스냅샷 방지 → 서버 graceful과 결합해 견고.
  • 필터/검색은 클라이언트 메모리 처리이므로 DB 쿼리가 불필요.

결과 (Consequences)

  • + 의존성 0, 프로젝트 폴더 비오염, 사람이 검사 가능.
  • + 설정·캐시 위치가 단일 앱 폴더로 확정되어 paths.js 계약(appDir/configPath/cachePath)이 명료(P1-2 해소).
  • + 소유자 전용 권한(0600/ACL)으로 멀티유저 OS의 경로 프라이버시 노출 표면 축소(M-2, RC-1 심층방어).
  • − %APPDATA%는 동기화 로밍 대상일 수 있어 캐시가 로밍될 소지 — 필요 시 캐시만 %LOCALAPPDATA%로 옮기는 것은 M4 재검토 여지(현 규모엔 무시 가능).
  • − 데이터가 수천~수만 규모로 커지면(전체 드라이브 R-05/M4) 단일 JSON 로드가 무거워질 수 있음 → 그 시점에 분할/스트리밍 재검토(M4 결정 사항).

ADR-004 — 웹서버·정적 서빙은 내장 http 모듈로 직접 구현

상태: 승인 (rev.2 — M-1·L-1·P1-1 반영) · 관련 요구: R-10 N-02 N-03 N-04 · 보안 M-1·L-1 · 검증 P1-1

맥락 (Context)

대시보드를 위해 정적 자산(HTML/CSS/JS)과 소수의 데이터/액션 API를 localhost에 제공해야 한다. 엔드포인트는 7개 미만이고 로컬 단일 사용자다. 프레임워크 도입 여부가 N-04에 직결된다.

고려한 대안

대안장점단점/트레이드오프
A. 내장 http + 직접 라우터/정적 핸들러 ← 채택의존성 0, 바인딩·헤더·CSRF·정적 경로 차단을 완전 제어, N-04 충족라우팅/MIME 직접 구현(소량)
B. Express/Koa 등 프레임워크라우팅·미들웨어 편의의존성 트리 유입 → N-04 위배, 본 규모엔 과함
C. 정적 파일 서버 패키지(serve 등)정적 서빙 즉시API·액션·CSRF·화이트리스트 통합 어려움, 의존성

결정 (Decision)

A 채택. server.js가 http.createServer로 서버를 만들고 listen(port, '127.0.0.1')로 루프백 전용 바인딩한다(N-02). lib/server/router가 메서드·경로로 디스패치하고, 정적 자산 URL 규약은 P1-1로 통일한다 — GET / → public/index.html, GET /static/<asset> → public/<asset>, /api/* 외 모든 GET은 staticHandler가 처리하며 매핑 후 public/ 절대경로 하위로 정규화해 이탈 요청을 404 처리(N-03)한다. index.html은 자산을 /static/*로 참조한다.

M-1 반영(CSRF/DNS 리바인딩): 상태변경 POST는 Content-Type: application/json 요구에 더해 Host 헤더 allowlist 정확 비교(기대 127.0.0.1:<port>/localhost:<port>)와 Origin 폐쇄 기본값(부재/불일치 거부), 기동 시 1회 생성한 세션 토큰 동봉 검증을 적용한다. L-1 반영: index.html에 CSP 메타(인라인/외부 스크립트 제한)를 둔다.

근거 (Rationale)

  • 엔드포인트가 적어 직접 라우팅 비용이 프레임워크 의존성 비용보다 낮다.
  • 바인딩 주소·Host allowlist·CSP·경로 정규화·세션 토큰을 프레임워크 추상화 없이 명시적으로 제어 → 보안 통제(§11)가 명확(M-1·L-1).
  • 정적 URL 규약을 /static/*→public/ 단일 규약으로 통일해 staticHandler 정규화 기준과 index.html 자산 참조가 일치(P1-1, 구현 분기 제거).
  • ADR-001과 정합(의존성 0).

결과 (Consequences)

  • + 보안 통제(Host allowlist·세션토큰·CSP)를 코드로 직접·명시적으로 보장, 의존성 0.
  • + 정적+API+액션을 한 서버에서 일관 처리, URL 규약 통일로 구현 임의해석 제거.
  • − 라우팅·MIME·파싱·Host 검증·토큰 주입 보일러플레이트를 직접 작성·테스트해야 함.

ADR-005 — 스캔(쓰기)과 서빙(읽기)을 별도 프로세스로 분리

상태: 승인 · 관련 요구: R-14 R-10 R-16 N-01 · package.json 골격

맥락 (Context)

확정 골격은 scan.js(bin spip)와 server.js를 별도 스크립트로 두고 있다(scan/start/dev). 스캔은 무거운 일회성 I/O 배치, 서빙은 가벼운 상시 읽기다. 두 책임을 한 프로세스에 합칠지, 분리할지 결정해야 한다.

고려한 대안

대안장점단점/트레이드오프
A. 별도 프로세스 + JSON 스냅샷 연결 ← 채택책임 분리, 서버 항상 경량, CLI 스캔 독립 실행, 확정 골격과 일치'재스캔'(R-16) 시 서버가 자식 프로세스로 scan을 호출하는 연동 필요(M4)
B. 단일 서버 프로세스가 요청 시 스캔프로세스 1개스캔 중 서버 블로킹/메모리 급증, 30초 작업이 HTTP 타임아웃·UX 악화, 관심사 결합
C. 상시 watch 데몬항상 최신MVP 범위 밖(§9.2 Could/M4), 복잡·자원 상주

결정 (Decision)

A 채택. scan.js는 결과를 스냅샷으로 쓰고 종료한다. server.js는 스냅샷을 읽어 서빙만 한다. 둘은 OS 캐시의 projects.json으로만 연결된다(공유 메모리·IPC 불필요). 대시보드 재스캔(R-16, M4)은 서버가 scan.js를 자식 프로세스로 호출하고 스냅샷을 다시 로드하는 방식으로 자리만 확보한다.

근거 (Rationale)

  • 스캔(무거운 배치)과 서빙(경량 상시)의 성격 차이를 프로세스 경계로 분리 → 서버 응답성·견고성 유지(N-01).
  • 스냅샷 파일 인터페이스는 가장 단순한 결합(공유 메모리·DB·IPC 불필요).
  • 확정된 package.json 스크립트 골격과 1:1 일치 → 추가 설계 부채 없음.
  • 서버가 스캔 미실행 상태로 떠도 graceful(200 + hasSnapshot:false + 빈 배열로 통일, P2-5) → 실행 순서 강제 불필요.

결과 (Consequences)

  • + 서버는 어떤 스캔 부하에도 영향받지 않고 가볍게 동작.
  • + CLI 스캔을 자동화/크론처럼 독립 사용 가능.
  • − 스냅샷이 시점 데이터라 최신성은 사용자의 재스캔에 의존(실시간 watch는 M4).
  • − R-16 재스캔 시 서버→scan 자식 프로세스 연동 코드 필요(M4 구현).

ADR-006 — 스캔 대상은 부분신뢰 입력으로 취급하고 적대적 입력을 방어

상태: 승인 (신규 rev.2) · 관련 요구: N-03 N-05 R-13 · 보안 H-1·H-2·H-3·M-3·L-1 · PM 위험수용 RC-2

맥락 (Context)

기존 설계는 환경 부재·손상 파일에 대한 graceful 격리(R-13/N-05)는 갖췄으나, 고의로 만든 악성 입력에 대한 방어는 명세하지 않았다. 보안 검증 1회차는 스캔 대상(package.json·.git·폴더명·심링크)이 신뢰 경계 밖(TB-B)임을 지적하고 높음 3건(H-1 심링크/realpath 우회, H-2 PATH 하이재킹, H-3 JSON 폭탄·ReDoS)을 게이트 차단 항목으로 제시했다. PM은 RC-2를 "인정"으로 확정(스캔 대상에 악성 외부 클론 리포가 섞일 수 있음)했으므로, 이들 방어는 권고가 아니라 필수다. graceful 격리(try)만으로는 무한 루프·메모리 폭발·경로 경계 우회·RCE를 막지 못한다는 점이 핵심이다.

고려한 대안

대안장점단점/트레이드오프
A. 신뢰 경계 밖 전제 + 입력 게이트(크기·깊이·realpath·절대경로·이스케이프) ← 채택RC-2 인정과 정합, 높음 3건 구조적 차단, 추가 의존성 0(내장 fs/path/child_process로 구현 가능)각 수집기·walker·pathGuard·safeExec에 가드 코드·테스트 추가 → 구현/검증 비용 증가
B. graceful 격리(try)만 유지구현 단순적대적 입력에 무력(DoS·경계우회·RCE 잔존) → 보안 게이트 FAIL, RC-2와 모순
C. 스캔을 별도 샌드박스/저권한 프로세스에서 실행강한 격리OS별 샌드박스 의존·복잡도 급증, 단일 사용자 로컬 도구엔 과설계(N-04 정신 위배), MVP 범위 밖

결정 (Decision)

A 채택. 스캔 대상을 부분신뢰(TB-B)로 명시 전제하고 다음 게이트를 설계 계약에 못박는다(구현 수용 기준): ① realpath 화이트리스트 — walker 등록 시점·pathGuard 검증 시점 모두 canonicalize()(fs.realpath + 대소문자/유니코드/UNC/드라이브상대/확장경로 규칙)로 실경로 정규화 후 정확 일치(H-1). ② 실행 파일 절대경로 고정 — resolveBin으로 1회 해석, git의 cwd 위장 바이너리 차단, Windows PATHEXT 차단(H-2, ADR-002 연계). ③ 입력 파싱 가드 — 파일 크기 상한·JSON 깊이/바이트 가드·선형시간(ReDoS-free) 정규식·문자열 필드 길이절단/제어문자 제거(H-3). ④ 심링크 루프·깊이 — MVP 내장 안전 상한 강제+방문 실경로 추적 가지치기(M-3). ⑤ 출력 이스케이프+CSP — 대시보드 동적 텍스트 textContent/이스케이프·CSP 메타(L-1). 위 가드는 모두 격리(try)보다 앞단에 둔다. 상세는 architecture.html §5 계약·§11.1·11.2·11.5·11.6.

근거 (Rationale)

  • RC-2 인정 하에서 H-2·H-3·L-1은 필수 방어 — 권고로 두면 악성 리포 스캔/열기만으로 RCE·DoS·XSS가 성립한다.
  • 모든 게이트는 Node 내장 모듈로 구현 가능 → ADR-001(의존성 0) 유지.
  • 가드를 격리 앞단에 두는 것이 핵심 — try/catch는 무한 루프·메모리 폭발을 막지 못한다(H-3 시나리오).
  • 샌드박스(C)는 단일 사용자 로컬 도구에 과설계이며, 입력 게이트로 위협 표면을 충분히 축소한다.

결과 (Consequences)

  • + 높음 3건(H-1·H-2·H-3)을 구조적으로 차단해 보안 게이트 PASS 조건 충족.
  • + 적대적 입력에도 스캔이 완주(SLA 유지)하고 경로 경계가 보존됨.
  • − 각 모듈에 가드·테스트 추가 → 구현/검증 비용 증가(test/fixtures에 악성 입력 케이스 포함 필요).
  • − realpath 해소·크기 검사 등으로 스캔에 소폭 오버헤드(제외 가지치기로 상쇄, RA1).
  • ※ RC-1(동일머신 타 사용자 격리)은 MVP 비목표로 수용하되 파일 권한(M-2)·세션 토큰(M-1)은 심층방어로 적용(architecture.html §11.7).
