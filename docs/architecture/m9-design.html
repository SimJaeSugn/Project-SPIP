1. 개요 · 아키텍처 동인 · 품질 속성

M9는 새 제품이 아니라 기존 단일 store + 전체 재렌더(render()=app.replaceChildren()) 아키텍처의 구조적 한계를 보강하는 배치다. 감사가 지목한 단일 공통 근원(전체 DOM 교체)이 한글 IME 깨짐(R-26)·드래그 파괴(R-32)·차트 인스턴스 파괴(R-33)의 원인이며 실시간 갱신(R-31)이 이를 악화시킨다. 따라서 R-25(부분 갱신/재렌더 가드)를 먼저 정립하고 나머지를 그 위에 얹는다.

3. ★ R-25 API surface 명세 (최우선 · F-2 확정)

이 절이 본 설계의 핵심 산출물이다. 기능명세가 의도 4가지만 확정하고 함수명·시그니처·모듈을 아키텍트에 위임(F-2)했으므로, 여기서 확정 계약으로 못 박는다. R-26·R-31·R-32·R-33 설계는 모두 이 API를 참조한다.

3.1 모듈 배치 결정 — 왜 app.js 내부 네임스페이스 + 순수 로직만 분리인가

번들러·ESM이 없고 public/에 빌드 단계가 없다. 따라서 새 가드는 다음 2층으로 둔다.

  • (A) 렌더러 측 가드 객체 RG — public/app.js의 기존 IIFE 내부에 const RG = (function(){ … return {…}; })(); 단순 모듈 패턴으로 둔다. DOM·store·render에 접근하는 부분(부수효과)을 담당. app.js는 자동 단위테스트 비대상(F-3 관례)이므로 여기엔 wiring만.
  • (B) 순수 판정 로직 lib/common/renderGuard.js (CommonJS, 신규) — DOM·타이머 없이 "지금 재렌더를 보류해야 하는가", "coalesce 큐 상태 전이" 같은 순수 함수만. node --test로 헤드리스 검증. app.js의 RG가 window 노출 또는 인라인 복제로 이 로직을 사용(번들러 없으므로 require 불가 → §3.6 로딩 방식 참조).
이 분리는 R-25 수용기준 "가드/coalesce 판정 로직은 app.js DOM과 분리된 순수 함수로 단위테스트 가능"을 직접 충족한다. 마커 RG-1(보류 판정)·RG-2(coalesce)·RG-3(보존)·RG-4(위젯 라이프사이클)를 코드/테스트에 병기한다.

3.2 계약 (1) composition / 재렌더 보류 — RG.composition + 순수 shouldDeferRender()

// lib/common/renderGuard.js — 순수 판정(헤드리스 단위테스트 대상, RG-1)
/** 현재 상태에서 라이브 갱신/재렌더를 보류해야 하면 true. DOM 미접근. */
function shouldDeferRender(f) {  // 모든 보류 플래그를 OR (DOM 미접근, 순수)
  return !!(f && (f.composing || f.dragging || f.overlayOpen
              || f.busyMail || f.busyCommit || f.editing));
}

// public/app.js — RG 네임스페이스(부수효과 wiring, RG-1)
RG.composition = {
  isComposing() { return store._composing === true; },         // 단일 진입점
  bind(inputEl) { /* compositionstart→_composing=true, compositionend→false+flush */ },
};
/** 모든 라이브 갱신 트리거의 단일 게이트. true면 호출측은 render()를 부르지 않는다. */
RG.deferred = () => shouldDeferRender({  // 아래 매핑표와 1:1
  composing:   store._composing === true,
  dragging:    store._dragging === true,
  overlayOpen: !!(store.showSettings || store.showHelp || store.state.selectedId),
  busyMail:    store.busyMailSummary === true,     // [R1] 누락 보완
  busyCommit:  store.busyCommitActivity === true,
  editing:     store.todoAdding === true,
});

계약 의도: 기존에 applyProjectsUpdate·maybeAutoRefreshMail이 제각각 검사하던 보류 조건을 RG.deferred() 한 곳으로 모으고 composing을 새로 추가한다. R-26·R-31은 이 게이트만 호출한다.

[R1 반영] P1 "기존 동작 100% 보존"을 위한 보류 조건 전수 매핑 (누락 0). 설계 검증에서 busyMailSummary 누락이 지적됐다. 현재 코드에 흩어진 모든 보류/가드 조건을 deferred() 입력에 1:1 매핑한다 — 아래 표의 모든 조건이 들어가야 P1이 회귀 0이다.

3.3 계약 (2) 노드 보존 부분 갱신 — RG.preserve

기존 render()의 FOCUS_SEL 복원(노드 교체 후 포커스·캐럿 재설정)은 이미 동작하지만 조합 세션은 복원 불가하다. 그래서 보존은 두 단계로 제공한다.

// public/app.js — RG.preserve (RG-3)
RG.preserve = {
  /** 재렌더 전: 보존 대상 노드의 상태 스냅샷(값·포커스·캐럿·스크롤)을 캡처. */
  capture(rootEl, selectors) { return snapshot; },   // 기존 FOCUS_SEL/SCROLL_SEL 로직 흡수
  /** 재렌더 후: 동일 셀렉터 새 노드에 상태 복원. */
  restore(rootEl, snapshot) { },
  /** [후속 — 1차 범위 제외] 부분 갱신. 본 M9 배치에서는 구현하지 않는다(R5). */
  // patchRegion(...) — 후속 마일스톤. M9는 destroy/recreate(전체 재렌더) + capture/restore로만.
};
[R5 반영] 1차 범위 확정 — patchRegion은 M9에서 명시적으로 제외(후속). 부분 갱신 영역 내 위젯 destroy/mount 책임·보존 노드 adopt 순서가 추가 계약을 요구해 빅뱅 회피 원칙과 상충할 여지가 있다. M9 전 요구(R-25~R-33)는 "전체 replaceChildren + RG.widget destroy/recreate + RG.preserve capture/restore"로만 구현한다. 보존(입력값·포커스·캐럿·스크롤)은 capture/restore로 충분하며, 이것이 현 render()의 인라인 복원 로직을 동작 동일하게 리팩터링한 것이라 회귀 0이다.
점진 도입 핵심: 1차(R-25 자체)는 capture/restore로 현 render()의 인라인 복원 로직을 리팩터링만(동작 동일, 회귀 0). 차트 깜빡임 등 부분 갱신이 필요해지면 patchRegion은 별도 후속 요구로 분리한다(§10·§14 리스크 참조).

3.4 계약 (3) stateful 위젯 라이프사이클 — RG.widget

현재 destroyCardSortable()/initCardSortable()가 render() 진입부와 dashboard 분기에 흩어져 있다. 이를 위젯 레지스트리로 일반화해 SortableJS(R-32)·SVG 차트(R-33)가 동일 규약으로 정리·재생성되게 한다.

// public/app.js — RG.widget (RG-4): render() 1회당 destroyAll → 뷰 빌드 → mount
RG.widget = {
  /** 위젯 정의 등록(앱 1회). { id, init(rootEl)->instance, destroy(instance) } */
  define(spec) { },
  /** render() 진입부에서 호출: 살아있는 모든 인스턴스 destroy(노드 교체 전). */
  destroyAll() { },
  /** render() 종료부에서 호출: 현재 DOM에 매칭되는 위젯만 init(중복 방지). */
  mountAll(rootEl) { },
};
// 예: 카드 정렬(기존), 홈 섹션 정렬(R-32), 커밋 차트(R-33)
RG.widget.define({ id: 'cardSortable', init: initCardSortable, destroy: destroyCardSortable });

render()는 RG.widget.destroyAll()(맨 위, 기존 destroyCardSortable 대체) → app.replaceChildren() + 뷰 build → RG.widget.mountAll(app)(맨 아래, 기존 initCardSortable 대체) 순으로 단순화된다. 중복 인스턴스·핸들러 누적·누수 0 보장.

[R4 반영] Sortable onEnd × destroy/flush 타이밍 충돌 회피. 기존 카드 Sortable의 onEnd는 (1) store._dragging=false 즉시 설정 후 (2) commitReorder를 마이크로태스크(Promise.resolve().then)로 지연해, onEnd 실행 중 render()가 그 Sortable 인스턴스를 destroy하지 못하게 한다(app.js:4914-4923). 홈 섹션 Sortable(R-32)·flushIfPending도 이 패턴을 그대로 따른다: onEnd에서 _dragging=false → 영속(setHomeLayout)·재렌더는 마이크로태스크로 지연 → 그 시점에 RG.coalesce.flushIfPending(). 즉 drag 해제 → 마이크로태스크 안에서 flush 순서를 고정해 "onEnd 도중 destroyAll이 현재 드래그 인스턴스를 파괴"하는 충돌을 막는다.

3.5 계약 (4) 실시간 push coalesce — RG.coalesce + 순수 큐

// lib/common/renderGuard.js — 순수 coalesce 상태기(RG-2, 헤드리스 테스트)
/** 보류 중 들어온 갱신을 누적. flush 시 1회 병합 결과 반환. */
function createCoalescer() {
  let pending = false;
  return {
    mark() { pending = true; },           // 보류 중 갱신 도착
    hasPending() { return pending; },
    flush() { const had = pending; pending = false; return had; }, // 정확히 1회
  };
}

// public/app.js — RG.coalesce: 모든 라이브 트리거의 공통 종착
// debouncedRender = debounce(()=>{ if (RG.deferred()) { _coalescer.mark(); return; } render(); }, 120)
//   ↑ [R2] 디바운스 발화 시점에 deferred() 재검사 — 발화 직전 보류가 시작되면 mark 후 발화 취소.
RG.coalesce = {
  request() {                              // 기존 render() 직접호출을 대체
    if (RG.deferred()) { _coalescer.mark(); return; }  // 보류: 큐에만 기록
    debouncedRender();                       // 비보류: 디바운스 1회 렌더(발화 시 재검사)
  },
  flushIfPending() {                        // 보류 해제 지점에서 호출. 아직 다른 보류 사유가 남았으면 발화하지 않음.
    if (!_coalescer.hasPending()) return;
    if (RG.deferred()) return;            // [R2] 다중 보류 동시 → 마지막 해제자만 발화
    if (_coalescer.flush()) debouncedRender();
  },
};
[R2 반영] coalesce "정확히 1회" 동시성 계약 확정. 단일 전역 boolean + 단일 디바운스 타이머가 폴링·IME 조합·드래그가 동시에 일어나는 상황에서 정합성이 미정의라는 지적을 다음 3규칙으로 확정한다.
  1. 디바운스 발화 시 deferred() 재검사: request()가 비보류로 판단해 타이머를 걸어도, 120ms 안에 보류 사유(조합 시작 등)가 들어올 수 있다. 따라서 타이머 콜백 내부에서 다시 deferred()를 검사해, 보류면 실제 render() 대신 mark()만 하고 발화를 취소한다(중복·조합 중 렌더 0).
  2. 다중 보류 사유의 단일 flush 책임: 동시에 여러 보류 사유(예: 드래그 + 조합)가 걸렸다가 하나가 풀려도 다른 사유가 남아 있으면 flushIfPending()가 deferred() 재검사로 발화하지 않는다. 마지막으로 모든 보류를 해제하는 호출자만 실제 1회 render를 트리거한다 — flush 책임이 자동으로 단일화된다.
  3. pending은 단조: 보류 중 N번 갱신이 와도 mark()는 boolean을 true로 유지(누적 횟수 무관). flush()가 1회 소비하면 false로 — 따라서 해제 후 정확히 1회.
DoD 게이트(R2): createCoalescer() 순수 테스트(마커 RG-2)에 동시성 케이스 명세: ① mark×3 후 flush 1회만 had=true ② flush 후 즉시 flush=false ③ "request(비보류)→타이머 대기 중 보류 시작→발화 콜백이 deferred=true 감지→mark" ④ "보류A+보류B 동시→A해제 시 deferred 여전히 true→발화 안 함→B해제 시 발화 1회". (③④의 deferred 부분은 주입 가능한 deferred 함수로 헤드리스 검증.)

핵심 흐름: R-31의 메일/커밋 폴링·R-24의 projectsUpdated push는 모두 render()를 직접 부르지 않고 RG.coalesce.request()를 부른다. 보류 상태면 mark()만 하고, 보류가 풀리는 순간(조합 종료·드래그 종료·모달 닫힘) RG.coalesce.flushIfPending()가 누적분을 정확히 1회 반영한다(위 동시성 규칙으로 다중 보류에서도 1회 보장). 수용기준 "중복 렌더 0" 충족.

3.8 [R3] R-26 검색 핸들러와 기존 debounce(render,120)의 관계 — 대체

현재 검색 input 핸들러(app.js:3533-3534)는 자체 const debounced = debounce(()=>render(), 120)를 만들어 매 입력마다 호출한다. R-26은 이를 중첩이 아니라 대체한다.

// 변경 전: store.state.search 갱신 + 자체 debounce(render)
searchInput.addEventListener('input', (e) => { store.state.search = e.target.value || ''; debounced(); });

// 변경 후(R-26): 자체 debounce 제거 → 공통 게이트만. composing이면 render 안 함.
RG.composition.bind(searchInput);   // compositionstart/end 추적 + end 시 flushIfPending
searchInput.addEventListener('input', (e) => {
  store.state.search = e.target.value || '';  // store는 항상 갱신(IME 중에도)
  RG.coalesce.request();                    // 보류면 mark, 비보류면 debouncedRender(공통 120ms)
});
  • 대체 관계: 검색 전용 debounce(render,120)를 삭제하고 RG.coalesce.request()의 공통 debouncedRender(동일 120ms)로 일원화한다. 디바운스가 2겹이 되지 않는다.
  • 한글 조합 중: store._composing===true면 request()가 deferred()로 보류→mark()만. render()가 안 불려 input 노드가 교체되지 않아 조합이 안 깨진다. compositionend 시 RG.composition.bind가 flushIfPending() 호출→1회 반영.
  • 영문/숫자 입력 회귀 0 검증 경로: 영문은 isComposing이 안 뜨므로 deferred()=false→기존과 동일하게 120ms 디바운스 후 1회 render. 검증: ① 영문 빠른 타이핑 시 화면 갱신 타이밍·결과가 기존과 동일(수동 스모크) ② matchShortcut과 무관(검색 입력은 단축키 비대상) ③ 한글 "프로젝트" 입력 시 자모 분리 0건(수동) — 헤드리스로는 request()가 composing=true에서 render 미호출임을 deferred 순수테스트로 검증.

6. uiState 신규 스키마 — homeLayout (R-32)

현 order는 프로젝트 id 해시 전용(ID_RE=/^[0-9a-f]{1,64}$/)이라 홈 섹션 순서를 담을 수 없다. 새 키 homeLayout(섹션 id 화이트리스트 배열)을 신설한다.

섹션 id 화이트리스트 (고정 enum — 실행/경로 의미 없음)

// lib/common/uiStateStore.js (신규 상수)
const HOME_SECTION_IDS = ['attention', 'productivity', 'activity',
                          'todos', 'mail', 'disk', 'featureAdd']; // 현 7섹션
const MAX_HOME_SECTIONS = HOME_SECTION_IDS.length;
const HOME_SECTION_SET = new Set(HOME_SECTION_IDS);

정규화 (normalizeHomeLayout)

/** 섹션 순서 정규화: 화이트리스트 외 id 제거, 중복 제거, 누락 섹션은 기본 순서로 보충. */
function normalizeHomeLayout(input) {
  const out = [], seen = new Set();
  if (Array.isArray(input)) for (const id of input) {
    if (typeof id !== 'string' || !HOME_SECTION_SET.has(id) || seen.has(id)) continue;
    seen.add(id); out.push(id);
  }
  for (const id of HOME_SECTION_IDS) if (!seen.has(id)) out.push(id); // 신규 섹션 자동 합류(끝)
  return out;
}
설계 결정: 누락 섹션을 거부하지 않고 기본 순서로 보충한다. 향후 섹션이 추가돼도 기존 사용자의 저장값이 그 섹션을 자동 포함하므로 마이그레이션 불필요. 수용기준 "미지 id·중복·길이초과 정리"는 화이트리스트·Set·고정 enum으로 충족(별도 길이 상한 불요 — enum 자체가 상한).

defaultState 확장

{ schemaVersion, favorites, order, sortMode, names, theme, todos, langTrend,
  homeLayout: [...HOME_SECTION_IDS] }   // 기본 = 현재 하드코딩 순서

normalizeState()에 homeLayout: normalizeHomeLayout(obj.homeLayout) 추가, SCHEMA_VERSION은 유지(폴백이 graceful이라 bump 불요).

[C-M-1 반영 · 중요] write()의 normalizeState 재적용으로 인한 "조용한 키 누락" 차단. uiStateStore.write()는 저장 직전 normalizeState()를 무조건 재적용하고, normalizeState()는 자신이 명시적으로 반환하는 고정 키만 남긴다(uiStateStore.js:101-111). 따라서 IPC 핸들러가 write({...state, homeLayout})를 호출해도 normalizeState에 homeLayout 키가 없으면 그 값은 조용히 버려진다 — 핸들러는 ok:true를 반환하는데 실제 영속·이후 read에는 반영되지 않아 응답과 영속이 어긋난다. 이를 막기 위해 다음 3곳을 반드시 동반 수정한다(하나라도 빠지면 버그):
  • ① normalizeState() 반환 객체에 homeLayout: normalizeHomeLayout(obj.homeLayout) 추가(필수 — 누락 시 write가 키를 버림).
  • ② defaultState()에 homeLayout: [...HOME_SECTION_IDS] 추가(read 부재/손상 시 graceful 기본).
  • ③ ipc/uiState.js의 toResponse()에 homeLayout: state.homeLayout 추가(getUiState 응답에 노출 — 누락 시 렌더러가 못 읽음).

13. ADR (M9) — 마일스톤 정본 수록

기존 관례(ADR-M8-1이 m8-design.html에 수록)에 따라 M9 ADR은 본 정본에 둔다. adr.html에는 인덱스 한 줄만 append.

ADR-M9-1 — R-25 가드를 app.js 내부 네임스페이스 + 순수 로직 동형 분리로 구현 (vs preload 노출 vs 전면 VDOM)

상태: 승인 · 관련: R-25

맥락: 4가지 가드 계약을 단위테스트 가능하게 두되, 번들러·ESM이 없고 CSP가 강하며 app.js는 5404줄이다.

대안: (a) 전면 가상DOM/프레임워크 — 비범위·규약 위반. (b) 순수 로직을 preload가 window에 노출 — 데이터 채널 외 유틸 노출로 관례 이질. (c) 순수 판정 로직을 lib/common/renderGuard.js(Node 테스트)와 app.js RG(렌더러)에 동형 구현, DOM 부수효과는 RG에만.

결정: (c). 판정 로직은 작고 순수해 드리프트 위험이 낮고, 기존 "VM은 lib에서 test·app.js는 wiring" 패턴(F-3)과 일치.

결과: 좋은 점 preload·CSP 불변, 헤드리스 테스트 가능, 점진 도입(동작 보존 리팩터링). 감수 순수 로직이 2곳(테스트로 동기 보장).

ADR-M9-2 — 라이브 갱신은 직접 render() 대신 coalesce 게이트 단일 경유

상태: 승인 · 관련: R-25/R-31/R-26

맥락: 현재 메일 폴링·projectsUpdated·검색이 제각각 보류 조건을 검사하고 render()를 직접 호출. R-31로 빈도가 오르면 입력/드래그/차트가 깨진다.

대안: (a) 각 호출처에 보류 조건 중복 유지 — 드리프트·누락. (b) 모든 트리거가 RG.coalesce.request() 단일 게이트 경유, 해제 지점에서 flush.

결정: (b). 보류 판정·디바운스·coalesce를 한 곳으로. [R2] 동시성 정합성을 3규칙으로 확정(§3.5): ① 디바운스 발화 시 deferred() 재검사(발화 직전 보류 시작 차단) ② 다중 보류 사유가 동시에 걸리면 flushIfPending이 deferred() 재검사로 마지막 해제자만 1회 발화 ③ pending은 단조 boolean(누적 횟수 무관, flush가 1회 소비).

결과: 좋은 점 폴링+IME+드래그 동시 발생에서도 중복 렌더 0·보류 누락 0·헤드리스 검증 용이. 감수 모든 호출처를 게이트로 바꾸는 1회 리팩터링 비용 + 디바운스 콜백 내 재검사 한 줄.

ADR-M9-3 — 홈 섹션 순서는 새 uiState 키 homeLayout(섹션 enum 배열)로 영속 (vs order 재사용)

상태: 승인 · 관련: R-32

맥락: 기존 order는 프로젝트 id 해시 전용(ID_RE). 홈 섹션은 고정 enum.

대안: (a) order에 섞기 — 검증 정규식 충돌·의미 혼선. (b) 새 키 homeLayout + 화이트리스트 정규화 + 신규 채널 setHomeLayout.

결정: (b). enum 화이트리스트라 별도 길이 상한·해시 검증 불요, 누락 섹션 자동 보충으로 마이그레이션 프리.

결과: 좋은 점 향후 섹션 추가에 강건, 보안 표면 극소(enum). 감수 채널·preload·스키마 1세트 신설.

ADR-M9-4 — 커밋 차트 실시간은 5분 폴링 + 뷰/가시성 정지 (vs onProjectsUpdated 편승 vs fs.watch)

상태: 승인 · 관련: R-31

맥락: git log 수집은 비용이 커 현재 폴링 제외. 사용자 확정: 5분 폴링 + 홈 이탈 시 git 0.

대안: (a) StateWatcher에 편승 — git log는 freshness 폴링보다 무겁고 주기가 다름. (b) fs.watch(.git) — ADR-M8-1과 동일 사유로 보류(Windows·자원). (c) 메일 60초 폴링 패턴 복제: 별도 5분 타이머 + 뷰=home·가시성 가드, 이탈 시 clearInterval.

결정: (c). 사용자 확정값과 일치, 기존 패턴 재사용.

결과: 좋은 점 홈 이탈 시 git 호출 0 검증 가능, 단순. 감수 최대 신선도 지연 5분(허용).
