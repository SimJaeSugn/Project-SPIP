1. 개요 · 아키텍처 동인 · 품질 속성

M9는 전체 replaceChildren + RG.widget destroy/recreate + RG.preserve capture/restore 조합으로 stateful 상호작용(IME·드래그·모달)을 보호했다. 그러나 매 render마다 DOM 전체를 교체하기 때문에 5분 폴링 시 차트가 깜빡이고, 툴팁이 막대 위치를 따라가지 않으며, deferred 보류로 건너뛴 폴링 틱이 최대 ~10분까지 지연되는 구조적 한계가 남았다(코드 리뷰 m9_p8·m9_p6 QA 메모). M10은 이 4가지 후속을 설계한다.

핵심 원칙: patchRegion은 기존 전체 render 경로를 완전히 교체하지 않고 옵트인으로 특정 호출처만 전환한다. M9의 RG 계약(deferred·coalesce·widget·preserve)과 정합을 유지하며, 점진 도입 중 어느 시점에도 전체 render 폴백이 살아 있다.

2. ★ patchRegion API surface (핵심 기반)

이 절이 M10 설계의 핵심 산출물이다. M9 §3.3이 "후속 마일스톤"으로 명시 위임한 patchRegion을 여기서 확정 계약으로 못 박는다. 이후 차트·툴팁·폴링 설계는 모두 이 API를 참조한다.

2.1 시그니처 확정

// public/app.js — RG.preserve 확장 (M9 계약 정합, 기존 capture/restore 메서드 보존)
RG.preserve = {
  capture(rootEl) { /* M9 기존 — 전체 render 전 스냅샷 */ },
  restore(rootEl, snap) { /* M9 기존 — 전체 render 후 복원 */ },

  /** [M10 신규] 특정 영역만 부분 갱신. 전체 replaceChildren 대신 지정 컨테이너만 교체. */
  patchRegion(containerEl, builderFn, opts) {
    // containerEl : 교체 대상 DOM 요소(살아있는 노드). null/미발견이면 전체 render 폴백.
    // builderFn   : () => Node | Node[] | null. 새 콘텐츠 생성 순수 빌더.
    // opts        : { widgets?: string[], preserveFocus?: boolean, fallback?: () => void }
    //   widgets     — 이 영역 안에 마운트된 RG.widget id 배열(영역 내 위젯만 destroy/mount).
    //   preserveFocus — 영역 안 포커스된 입력 캐럿 보존 여부(기본 true).
    //   fallback    — containerEl 부재·빌더 오류 시 호출할 전체 render 대리(기본 render).
  },
};

2.2 내부 실행 순서 (계약)

patchRegion은 다음 5단계를 순서대로 실행한다. 단계 중 복구 불가 오류가 발생하면 opts.fallback()으로 전체 render 폴백한다.

  1. 가드 검사: RG.deferred()가 true이면 RG.coalesce.request()를 호출하고 즉시 반환(보류 중에는 부분 갱신도 수행하지 않는다). — M9 deferred 계약과 동일한 규칙.
  2. 영역 내 위젯 destroy: opts.widgets 배열에 지정된 id만 RG.widget._destroyById(id)로 정리. 전체 destroyAll이 아닌 영역 범위 위젯만 파괴한다.
  3. 포커스/스크롤 캡처(opt-in): opts.preserveFocus !== false이면 RG.preserve.capture(containerEl)로 영역 내 포커스·캐럿 스냅샷.
  4. 영역 교체: builderFn() 호출 → 반환된 노드로 containerEl.replaceChildren(...nodes). builderFn이 null 반환하면 영역을 비움(삭제 의미).
  5. 복원 + 영역 내 위젯 mount: RG.preserve.restore(containerEl, snap) → RG.widget._mountById(id, containerEl) for each opts.widgets.

2.3 영역 내 위젯 destroy/mount — 신규 내부 메서드

기존 RG.widget.destroyAll/mountAll은 전체 위젯 레지스트리를 순회한다. patchRegion용으로 id 단위 조작 메서드를 RG.widget 내부에 추가한다(공개 API는 아님, 밑줄 접두어로 내부 표기).

// RG.widget 내부 확장 — 공개 API 추가 없음(기존 define/destroyAll/mountAll 보존)
widget = {
  // ... 기존 define/destroyAll/mountAll 그대로 ...

  /** patchRegion 전용: 특정 id 위젯만 destroy. id 불일치 시 no-op. */
  _destroyById(id) {
    const s = _specs.find((s) => s.id === id);
    const inst = _instances[id];
    if (s && inst != null && typeof s.destroy === 'function') {
      try { s.destroy(inst); } catch (_) {}
    }
    _instances[id] = null;
  },
  /** patchRegion 전용: 특정 id 위젯만 mount. 이미 살아있으면 skip. */
  _mountById(id, rootEl) {
    if (_instances[id] != null) return;
    const s = _specs.find((s) => s.id === id);
    if (!s || typeof s.init !== 'function') return;
    let inst = null;
    try { inst = s.init(rootEl); } catch (_) {}
    _instances[id] = (inst != null) ? inst : null;
  },
};

2.4 전체 render 폴백 규칙

폴백 설계 의도: patchRegion 자체가 실패해도 앱이 빈 상태로 남지 않는다. 전체 render가 폴백으로 동작하므로 사용자에게는 약간의 깜빡임이 있을 뿐 정보 손실이 없다. 점진 도입 중 안전망 역할을 한다.

2.5 deferred / coalesce 상호작용 계약

  • patchRegion 진입 시 RG.deferred()가 true이면 부분 갱신을 수행하지 않는다. 대신 RG.coalesce.request()를 호출해 보류 해제 시 전체 render가 1회 반영되게 한다.
  • patchRegion이 정상 완료되면 coalesce pending을 건드리지 않는다. 이미 pending이 쌓인 상태라면 보류 해제 시 전체 render가 추가로 1회 발화하는 것이 정상(영역 외 변경 반영).
  • patchRegion은 coalesce.release()를 호출하지 않는다. release는 "즉시 1회 전체 render"를 뜻하므로 patchRegion 목적과 충돌한다. 부분 갱신 완료 후 추가 전체 render가 필요한 경우는 호출처가 명시적으로 release()를 부른다.

2.6 보존 노드 adopt 순서 (focus/caret/scroll)

영역 교체 후 복원 순서는 다음을 따른다. 이 순서가 바뀌면 브라우저가 focus 이후 setSelectionRange를 무시할 수 있다.

  1. containerEl.replaceChildren (새 노드로 DOM 교체 완료)
  2. scrollTop 복원 (새 노드가 DOM에 붙은 뒤 레이아웃이 확정된 시점)
  3. focus({ preventScroll: true }) 호출
  4. setSelectionRange(start, end, dir) 호출
보안 채널 안전성 확인: patchRegion이 보존·adopt하는 것은 캐럿 위치(정수)·scrollTop(정수)·셀렉터(하드코딩 상수)뿐이다. 스캔/커밋 데이터에서 읽은 문자열은 adopt 경로에 진입하지 않는다(L-1 불변식 영향 없음).

2.7 옵트인 점진 도입 — 기존 render() 경로 보존 + 노드 소유권 단일화 (F-2)

patchRegion은 기존 render()를 건드리지 않는다. 각 호출처가 독립적으로 patchRegion으로 전환한다.

// 기존 전체 render 경로 — 변경 없음
function render() {
  RG.widget.destroyAll();
  const snap = RG.preserve.capture(app);
  app.replaceChildren(); // 전체 교체 — patchRegion 도입 후에도 유지
  // ... 뷰 빌드 ...
  RG.preserve.restore(app, snap);
  RG.widget.mountAll(app);
}
[F-2 반영 · 중요] 노드 소유권 = 위젯-소유 모델로 단일화. 현재 commitChart 위젯의 init(app.js:5727~5737)은 스스로 chartBars()로 차트 노드를 생성해 host.appendChild(chart.node)로 삽입하고, destroy에서 호버 핸들러를 detach한다 — 즉 차트 노드의 생성·삽입·파괴 전권을 위젯이 가진다. 따라서 patchRegion의 builderFn이 차트 노드를 또 만들어 반환하면 노드가 이중 삽입되고 핸들러가 누수된다(patchRegion이 막으려던 문제를 역유발). 이를 막기 위해 소유권을 다음 한 모델로 못 박는다:
  • builderFn은 차트 노드를 만들지 않는다. builderFn은 빈 호스트 컨테이너만(.commit-chart-host div) 반환한다. 실제 차트 노드 생성·삽입·destroy는 오직 RG.widget('commitChart')가 _mountById(init이 host.appendChild)·_destroyById(destroy가 핸들러 detach)로 단독 책임진다.
  • patchRegion 실행 순서(§2.2)가 이 모델과 정확히 맞물린다: ② _destroyById('commitChart')로 기존 차트·핸들러 정리 → ④ builderFn이 빈 호스트를 replaceChildren → ⑤ _mountById('commitChart', containerEl)가 빈 호스트 안에 차트를 1회 삽입. 이중 삽입·핸들러 누수 0.
  • commitChart 위젯의 init은 현재대로 (root||document).querySelector('.commit-chart-host')로 호스트를 찾되, patchRegion 경로에서는 _mountById(id, containerEl)가 containerEl을 root로 전달하므로 스코프가 영역으로 한정된다(전역 querySelector 의존 제거 — m9_p8 사소 항목도 동반 해소).
// patchRegion 전환 예시 — 차트 호스트만 교체(옵트인). 위젯-소유 모델.
function patchCommitChart() {
  const region = document.querySelector('.commit-chart-region'); // 차트 호스트를 감싸는 영역
  RG.preserve.patchRegion(region, () => {
    // builderFn은 빈 호스트 컨테이너만 반환 — 차트 노드는 만들지 않는다(소유권은 위젯).
    return el('div', { cls: 'commit-chart-host', attrs: { 'aria-label': '최근 7일 커밋 빈도 차트' } });
  }, {
    widgets: ['commitChart'],   // ②_destroyById → ⑤_mountById 가 차트 노드를 단독 생성/삽입/파괴
    preserveFocus: false,        // [P2 경미] 차트 영역엔 포커스 입력 없음 — 캡처/복원 생략
    fallback: () => render(),    // 영역 부재·빌더 오류 시 전체 render 안전망
  });
}
호스트 vs 영역 구분: 기존 .commit-chart-host는 차트 위젯이 노드를 꽂는 컨테이너다. patchRegion의 교체 대상(containerEl)은 그 호스트를 감싸는 한 단계 바깥의 영역(.commit-chart-region)으로 두어, builderFn이 호스트 자체를 새로 만들어 갈아끼우고 위젯이 그 안을 채우게 한다. 이렇게 하면 builderFn 결과 노드(빈 호스트)와 위젯 소유 노드(차트)의 경계가 명확해진다. renderHomeProductivity()는 .commit-chart-host를 .commit-chart-region > .commit-chart-host 2단 구조로 변경(P4 동반 — §10 파일 변경 지도).

M9 §10 리스크와 m9_p8 QA 모두 "5분 폴링 시 전체 render → chart destroy/recreate → 깜빡임"을 예고했다. 두 접근을 비교하고 권장안을 확정한다.

4.1 대안 비교

접근장점단점 / 트레이드오프적합성
A. patchRegion 적용
(차트 호스트 영역만 부분 갱신)		 데이터 변경 시에도 차트 영역 외 DOM 불변 → 깜빡임 0. 범용 patchRegion 기반 닦음. patchRegion 구현 필요(M10 핵심 기반). commitChart 위젯을 _destroyById/_mountById로 정밀 제어. 권장
B. diff 가드
(데이터 동일 시 재생성 skip)		 구현 단순. 폴링으로 동일 데이터가 오면 render()는 호출하되 chart를 skip. 데이터가 바뀌면 전체 render → 깜빡임 여전. 카드 스크롤 흔들림도 해결 안 됨. 가드 상태 관리 필요. 보완책
C. A + B 병합
(patchRegion + 데이터 동일 skip)		 데이터 동일 → skip, 변경 → patchRegion. 최소 재작업. 두 메커니즘의 조합 — 이해하기 쉽고 구현 비용도 낮음(A를 기반으로 데이터 hash 비교만 추가). 선택적 강화
결정: A(patchRegion) + 조건부 B(데이터 동일 skip) 병합. patchRegion을 기반으로 하되, commitActivity 데이터가 이전과 동일하면 patchRegion 자체도 호출하지 않는 단순 직렬화 키 비교를 추가한다. 단계별 효과를 정확히 구분한다:

  • P2(diff 가드 단독) — 데이터 무변경 폴링 틱에서 불필요한 갱신을 skip(완료부 render/release 미호출). 데이터 변경 시에는 여전히 전체 render → 깜빡임 잔존. 즉 P2의 효과는 "깜빡임 0"이 아니라 "무변경 시 불필요 갱신 0"이다.
  • P4(patchRegion 전환) 이후에 비로소 데이터 변경 시에도 차트 영역만 교체 → 깜빡임 0 + 홈 카드 스크롤 흔들림 0이 달성된다.
[F-1 반영 · 중요] 진입부 로딩 render()가 깜빡임의 진짜 원인. 현재 refreshCommitActivity는 진입부(app.js:2204~2205)에서 store.busyCommitActivity=true 직후 if (view==='home') render()로 로딩 표시를 위해 fetch 완료 전에 이미 전체 render를 1회 한다. 따라서 "완료부만 patchRegion으로 바꿔도" 이 진입 로딩 render 때문에 깜빡임이 남는다. P4에서 다음을 동반해야 깜빡임이 실제로 0이 된다:
  • 진입부 로딩 render() 제거 또는 영역 한정: 폴링 경로(maybeAutoRefreshCommit→refreshCommitActivity)에서는 진입 로딩 render()를 호출하지 않는다(차트 호스트는 fetch 동안 이전 차트를 그대로 유지 — 폴링은 백그라운드 갱신이라 로딩 스피너가 불필요). 최초 로드(maybeLoadCommitActivity 경유, commitActivityLoaded===false)에서만 로딩 상태를 표시하되, 그조차 차트 영역에 한정한 patchRegion(빈 호스트 + "집계 중…" 텍스트)으로 처리한다.
  • 이를 위해 refreshCommitActivity(opts)에 { silent } 플래그를 둬, 폴링 호출(maybeAutoRefreshCommit·maybeFlushCommitRefresh)은 silent:true(진입 render 생략), 최초/수동 호출은 기본(로딩 표시)으로 분기한다.

4.2 commitActivity 데이터 동일성 판정 (epoch-day prefix 포함 — Q-M10-1 확정)

// public/app.js — 폴링/완료부 diff 가드 (순수 키 함수는 헤드리스 테스트 대상)
let _lastCommitActivityKey = '';

/** [Q-M10-1] 키 = 마지막 날짜의 epoch-day 정수 + count 시퀀스.
 *  날짜 문자열이 아니라 정수만 키에 넣어 L-1/M-2 불변식 유지하면서,
 *  자정 경계로 7일 창이 밀리면(count가 같아도) 키가 달라져 갱신 누락 0. */
function commitActivityKey(ca) {
  if (!ca || !Array.isArray(ca.days) || ca.days.length === 0) return '';
  const last = ca.days[ca.days.length - 1];
  // epoch-day: Date.parse → ms → 일 단위 정수. 파싱 실패(NaN)면 0. 데이터 문자열은 키에 미진입.
  const ms = Date.parse(last && last.date);
  const epochDay = Number.isFinite(ms) ? Math.floor(ms / 86400000) : 0;
  const counts = ca.days.slice(-7).map((d) => {
    const n = Number(d && d.count);                  // M-2 동일 규칙(Number+isFinite, 음수/NaN→0)
    return (Number.isFinite(n) && n >= 0) ? n : 0;
  }).join(',');
  return epochDay + '|' + counts;                       // 예: "20263|0,1,3,0,12,5,2"
}

// 완료부 — fetch 직후. silent 여부와 무관하게 동일.
function onCommitActivityFetched() {
  const newKey = commitActivityKey(store.commitActivity);
  if (newKey === _lastCommitActivityKey) return; // [P2] 데이터·날짜창 무변경 → 갱신 skip
  _lastCommitActivityKey = newKey;
  // [P1 단계] RG.coalesce.release()로 전체 render. [P4 단계] patchCommitChart()로 영역만 교체.
  patchCommitChart();
}
L-1 / M-2 안전성: 키는 정수(epoch-day)와 정수(count)만 사용한다 — Date.parse는 문자열을 ms 정수로 환원하고 실패 시 0, count는 Number+isFinite로 강제한다. 날짜·라벨 문자열은 키 계산에 직접 들어가지 않으므로 신뢰불가 스캔/커밋 데이터가 비교 경로에 흐르지 않는다. 라벨은 textContent 렌더 경로에만 흐른다(불변).

4.3 전체 render 스크롤 흔들림도 함께 완화 (P4 이후)

P4에서 (1) 진입 로딩 render 생략(폴링 silent) + (2) 완료부 patchCommitChart(영역만 교체)가 함께 적용되면, 폴링 시 전체 render()가 한 번도 호출되지 않으므로 홈 카드의 스크롤 위치 흔들림이 자동으로 해결된다. 이것이 diff 가드(P2)만으로는 얻을 수 없는 patchRegion의 추가 이점이다.

5. 툴팁 막대 추적 설계

m9_p8 코드 리뷰 개선 항목: 현재 툴팁 CSS는 left:50% 고정(wrap 전체 중앙). 막대 위치에 관계없이 항상 차트 중앙에 떠서 양끝 막대와의 시각적 대응이 모호하다. M9 §10은 "툴팁은 SVG 외부 div로 포지셔닝"을 명시했으나 1차 구현에서 미완으로 남겼다.

5.1 좌표 계산 설계 (F-3a 정정 — 막대별 클로저)

차트 모델(commitChartModel)은 각 막대의 b.x(SVG 좌표계 left), b.w(너비)를 이미 계산한다. 현 chartBars는 model.bars.forEach((b)=>{...}) 안에서 막대마다 rect를 만들고 showTip/hideTip 클로저로 on(rect,'mouseenter',showTip) 등을 등록한다(app.js:1297~1326). 툴팁 위치 계산을 이 클로저 안에 두어 b(막대 모델)·rect를 그대로 캡처한다 — 별도 barRectEl 인자 없이.

// public/app.js — chartBars 내부, model.bars.forEach((b) => { ... }) 안의 showTip
// svgEl·tip·wrap 은 chartBars 스코프 변수(이미 존재). b·rect 는 forEach 클로저 캡처.
const showTip = () => {
  rect.setAttribute('fill', CHART_PALETTE.barHover);                 // 강조(기존 동작 보존)
  tip.textContent = (b.label || '·') + ' · ' + b.count + ' 커밋'; // [M-2/L-1] textContent 만
  tip.style.display = 'block';                                      // offsetWidth 측정 위해 먼저 표시

  // 1. 막대 중심 비율 (SVG viewBox 기준, 0~1)
  const centerRatio = (b.x + b.w / 2) / model.viewW;
  // 2. 실제 렌더 치수(px). 표시 직후라 offsetWidth 실측 가능(0이면 80 추정).
  const svgRect = svgEl.getBoundingClientRect();
  const wrapRect = wrap.getBoundingClientRect();
  const tipWidth = tip.offsetWidth || 80;
  // 3. wrap 기준 left + 4. 경계 클램핑 [0, wrapWidth - tipWidth]
  const rawLeft = (svgRect.left - wrapRect.left) + centerRatio * svgRect.width - tipWidth / 2;
  const maxLeft = Math.max(0, wrapRect.width - tipWidth);
  const left = Math.max(0, Math.min(rawLeft, maxLeft));
  // 5. [F-3b] 가로 위치만 left 픽셀로. 세로 띄움 translateY(-100%)는 transform 으로 유지.
  tip.style.left = String(Math.round(left)) + 'px';
  tip.style.transform = 'translateY(-100%)'; // 막대 위로 띄움 — 절대 'none' 으로 덮지 않는다
};
const hideTip = () => {
  rect.setAttribute('fill', baseFill); // 막대별 baseFill 클로저 복원(기존 동작 보존)
  tip.style.display = 'none';
};
on(rect, 'mouseenter', showTip); on(rect, 'mouseleave', hideTip);
on(rect, 'focus', showTip);      on(rect, 'blur', hideTip);
[F-3b 반영 · 회귀 방지] 세로 띄움 보존. 기존 CSS는 툴팁을 막대 위로 띄우기 위해 transform(또는 top 오프셋)으로 세로 위치를 잡는다. JS에서 가로 위치를 바꾸려고 transform:'none'으로 덮으면 세로 띄움까지 제거돼 툴팁이 막대와 겹친다(F-3b 버그). 따라서 가로는 style.left(픽셀)로, 세로 띄움은 transform:'translateY(-100%)'로 분리해 가로만 바꾸고 세로는 항상 유지한다. left/translateX를 transform에 섞지 않는다.

5.4 getBoundingClientRect 호출 빈도 고려

getBoundingClientRect는 레이아웃 강제 리플로우를 발생시킨다. 이를 최소화하기 위해:

  • showTip은 mouseenter/focus 시만 호출(사용자 상호작용 이벤트). 애니메이션 루프에서 호출하지 않는다.
  • wrap.getBoundingClientRect()는 캐시하지 않는다: 창 리사이즈·스크롤 후 틀릴 수 있어 매 호버마다 최신값이 필요하다. 호버는 사용자 상호작용 빈도라 성능 문제 없다.
  • [P2 경미] 첫 호버 정확도: §5.1 설계는 tip.textContent 설정 + tip.style.display='block'을 좌표 계산보다 먼저 수행한다. 따라서 첫 호버에서도 tip.offsetWidth가 실측값을 반환해 || 80 폴백에 의존하지 않는다(display:none 상태에서 측정하던 종전 우려 해소). 만약 폰트 로딩 지연 등으로 0이면 80px 추정으로 graceful — 두 번째 호버부터 정확.
L-1 확인: tipEl.textContent에만 데이터가 흐른다. tipEl.style.left에는 Math.round(left) + 'px'(정수 + 리터럴 문자열)만 들어간다. 스캔/커밋 데이터 문자열이 스타일 속성에 진입하지 않는다.

6.3 설계 — pendingCommitRefresh 플래그 + silent 분기

refreshCommitActivity는 진입부에서 로딩 표시용 전체 render(app.js:2205)를 한다. 폴링/재시도 경로에서는 이 진입 render가 깜빡임을 만들므로(§4.1 F-1), { silent } 인자로 진입 render를 분기한다.

// public/app.js — refreshCommitActivity 진입부 분기(F-1 정합)
async function refreshCommitActivity(opts) {
  opts = opts || {};
  if (!bridgeHas('getCommitActivity') || store.busyCommitActivity) return;
  store.busyCommitActivity = true;
  // [F-1] silent(폴링/재시도)면 진입 로딩 render 생략 — 이전 차트를 fetch 동안 유지(깜빡임 0).
  if (!opts.silent && store.state.view === 'home') render();
  // ... await ipc('getCommitActivity') ... 완료 후 onCommitActivityFetched()(§4.2) ...
}

// public/app.js — 커밋 폴링 보류 재시도 보강
let _pendingCommitRefresh = false; // 보류로 건너뛴 틱이 있음을 추적

function maybeAutoRefreshCommit() {
  if (store.state.view !== 'home' || !bridgeHas('getCommitActivity')) return;
  if (store.busyCommitActivity) return;
  if (RG.deferred()) {
    _pendingCommitRefresh = true; // ← 변경: 건너뜀을 기록
    return;
  }
  _pendingCommitRefresh = false;
  refreshCommitActivity({ silent: true }); // [F-1] 폴링은 silent — 진입 로딩 render 생략
}

/** 보류 해제 지점에서 호출. 건너뛴 폴링 틱이 있으면 즉시 1회 커밋 갱신 시도. */
function maybeFlushCommitRefresh() {
  if (!_pendingCommitRefresh) return;
  if (store.state.view !== 'home') { _pendingCommitRefresh = false; return; }
  if (store.busyCommitActivity || RG.deferred()) return; // 아직 보류 중이면 대기(다음 해제 지점에서 재시도)
  _pendingCommitRefresh = false;
  refreshCommitActivity({ silent: true }); // 즉시 1회 실행(폴링 보강이므로 silent)
}

6.4 maybeFlushCommitRefresh 호출 지점 — 모든 보류 해제 지점 (F-3d 정정)

[F-3d 반영] 일부 해제 지점만 덮으면 지연이 남는다. RG.deferred()의 보류 사유는 6종(composing·dragging·overlayOpen·busyMail·busyCommit·editing)이다. compositionend·드래그 onEnd·모달 닫힘만 다루면 busyMail 해제·todoAdding(editing) 해제로 마지막 보류가 풀리는 경우 폴링 재시도가 누락된다. 따라서 코드에서 RG.coalesce.release() 또는 flushIfPending()을 호출하는 모든 지점(현 8곳)에 maybeFlushCommitRefresh()를 동반한다. release/flushIfPending이 이미 "보류 해제 시점"의 단일 표지이므로, 그 옆에 한 줄 추가하는 규칙으로 누락을 구조적으로 막는다.

7. 적용 호출처 지도 — patchRegion 적용 vs 전체 render 유지

patchRegion은 모든 호출처에 적용하지 않는다. 아래 기준으로 분류한다:

  • patchRegion 적합: 고빈도 갱신(폴링) + 영역 명확(단일 컨테이너) + 영역 외 DOM에 사이드이펙트 없음 + 위젯 범위 국소.

CSPinnerHTML/insertAdjacentHTML/eval/new Function 미사용. 인라인 스크립트 0. 외부 fetch 0. index.html 불변.
patchRegion 보존 노드 채널adopt 경로에 진입하는 값은 캐럿 위치(정수)·scrollTop(정수)·셀렉터(하드코딩 상수 FOCUS_SEL/SCROLL_SEL)뿐. 스캔/커밋 데이터 문자열은 adopt 경로에 진입하지 않는다.

Security reviewer 입력 안내: M10에서 신뢰 경계 변화 0, 새 IPC 0, 새 외부 의존성 0. 신규 위협 모델 항목 없음. 기존 신뢰 경계 다이어그램(m9-design.html §12) 유효. 검토 초점: patchRegion builderFn 내부에서 텍스트 데이터가 innerHTML 경로로 흐르지 않는지 코드 리뷰 체크리스트 항목 추가.

13. ADR (M10)

ADR-M10-1 — patchRegion을 RG.preserve 메서드로 추가 (vs 독립 함수 vs 별도 모듈)

상태: 승인 · 관련: M10 전 항목

맥락: M9 §3.3이 patchRegion을 RG.preserve의 메서드로 예정했다. 번들러 없음, app.js IIFE 구조, DOM 접근 필요.

고려한 대안:
(a) 독립 함수 patchRegion(el, fn, opts) — app.js 내 독립 선언. 단순하나 RG 네임스페이스 일관성 깨짐.
(b) lib/common/에 신규 파일 — DOM 접근 불가 원칙 위반(renderGuard.js는 순수 함수 전용).
(c) RG.preserve.patchRegion 메서드 — M9 설계 예고 충족. capture/restore와 자연스러운 동거. RG 네임스페이스 일관성 유지.

결정: (c). M9 §3.3 계약 정합, lib/common 순수 원칙 불변.

결과: + M9 계약과 자연스러운 연속성. + lib/common 오염 없음. − RG.preserve가 capture/restore/patchRegion 3가지 책임을 가짐(단, 모두 "보존 관련" 응집도 높음).

ADR-M10-2 — 차트 깜빡임 완화: patchRegion + diff 가드 병합 채택 (vs diff 가드 단독)

상태: 승인 · 관련: §4

맥락: diff 가드만으로는 데이터 변경 시 전체 render → 깜빡임이 여전하다. patchRegion 단독은 구현 비용이 있으나 스크롤 흔들림까지 해결한다.

고려한 대안:
(a) diff 가드 단독 — 동일 데이터 skip만. 변경 시 깜빡임 잔존.
(b) patchRegion + diff 가드 병합 — 동일 → skip, 변경 → patchRegion. 모든 경우 깜빡임 0.
(c) CSS transition으로 깜빡임 마스킹 — 근본 해결 아님.

결정: (b). P1에서 diff 가드 먼저, P4에서 patchRegion 완성. 점진 도입으로 각 단계 독립 회귀 검증 가능.

결과: + 깜빡임·스크롤 흔들림 동시 해결. + patchRegion이 향후 다른 영역 확장 기반이 됨. − P4까지 단계가 있어 완전 해소에 시간 필요.

ADR-M10-3 — 폴링 재시도: pendingCommitRefresh 플래그 + 해제 훅 연결 (vs 폴링 주기 단축)

상태: 승인 · 관련: §6

맥락: deferred 보류로 건너뛴 폴링 틱이 최대 ~600초 지연될 수 있다. ADR-M9-4(5분 폴링 근거)를 파기하지 않으면서 개선이 필요하다.

고려한 대안:
(a) 폴링 주기 단축(300s → 60s) — git 호출 비용 5× 증가. ADR-M9-4 근거 파기.
(b) _pendingCommitRefresh 플래그 + 보류 해제 훅 3곳에 maybeFlushCommitRefresh() 연결 — ADR-M9-4 불변. 보류 해제 직후 즉시 1회 fetch.
(c) coalesce 내부에 커밋 fetch 콜백 등록 — coalesce가 render 이외의 부수효과를 담당하게 돼 단일 책임 위반.

결정: (b). ADR-M9-4 정합, 구현 최소, 기존 보류 해제 지점 재사용.

결과: + 최대 지연을 폴링 주기 → 보류 지속 시간으로 단축. + git 호출 빈도 불변. − 보류 해제 지점 3곳에 호출 추가(코드 산포, 단 지역적).

ADR-M10-4 — 툴팁 좌표: getBoundingClientRect 기반 동적 계산 (vs 막대 인덱스 기반 비율)

상태: 승인 · 관련: §5

맥락: SVG viewBox 비율로 left를 계산하려면 실제 렌더 크기가 필요하다. SVG는 width:100%라 고정 픽셀 계산 불가.

고려한 대안:
(a) 막대 인덱스 기반 CSS nth 매핑 — CSS로 해결하나 index ↔ DOM 순서 의존. 경계 클램핑 CSS 어려움.
(b) 막대 인덱스 × (100/n)% 방식 — 간단하나 막대 간격/패딩을 무시. 양끝 오차 큼.
(c) getBoundingClientRect(svgEl) + viewBox 비율 × svgWidth + wrapEl 기준 offset + 클램핑 — 정확. 사용자 상호작용 빈도(hover)라 리플로우 비용 허용.

결정: (c). 정확성과 경계 처리 완성도가 (a)·(b)보다 우수. hover 빈도라 성능 이슈 없음.

결과: + 7칸 양끝 막대에서도 정확 포지셔닝. − hover마다 getBoundingClientRect(2회) — hover 빈도라 무시 가능.

14.2 가정

  • builderFn은 동기 함수여야 한다. M10에서는 비동기 builderFn을 지원하지 않는다.
  • patchRegion은 단일 컨테이너 노드를 대상으로 한다. 다중 컨테이너 동시 갱신은 M10 범위 밖.
  • 차트 영역(.commit-chart-region)과 그 안의 호스트(.commit-chart-host)는 홈 뷰에서 항상 단일 인스턴스다(P4 2단 구조).
  • 차트 노드의 생성·삽입·파괴는 오직 commitChart 위젯이 소유한다(builderFn은 빈 호스트만 생성 — F-2).
  • SVG preserveAspectRatio="none"·width:100% 구조는 M9에서 확정됐으며 변경하지 않는다.
