기존 Electron 데스크톱 앱(electron-migration.html v1.1, m6-design.html rev.2 기준)에 사용자 요청 2건을 점진 확장으로 반영하는 설계 델타다. ① 즐겨찾기 UI를 메인 대시보드 창 내부 오버레이(ADR-M6-5)에서 대시보드와 독립된 별도 프레임리스 토스트형 BrowserWindow로 전환(우측 하단 등장, 대시보드가 트레이로 숨어 있어도 동작)하고, ② 대시보드 카드 reorder 시 부드러운 이동 애니메이션(FLIP)을 추가한다. 기존 프로세스 모델·IPC 계약·보안 베이스라인(EC-1~6·EM·M6 통제)을 0 약화로 보존하고, 새 창에도 메인 창과 동등한 보안 통제를 통째로 적용한다. 본 문서는 설계이며 실행 가능한 구현 코드를 포함하지 않는다(IPC/창 계약·데이터 흐름·의사코드·SVG·ADR까지만).

1. 개요 · 아키텍처 동인 · 변경 범위

3. 창 구조 (메인 + 위젯) · 신뢰 경계

M7 이후 앱은 최대 2개의 BrowserWindow를 가진다 — 메인 대시보드 창(기존)과 즐겨찾기 위젯 창(신규). 두 창은 동일한 보안 컨텍스트(app:// origin·동일 CSP·동일 하드닝)를 갖되, preload allowlist는 위젯이 더 좁다. 둘 다 같은 main 프로세스의 register.js guard로만 IPC하며, guard의 senderFrame 검증은 app:// origin 정확 일치라 두 창 모두 신뢰 발신자로 통과한다(별도 분기 불필요).

신뢰 경계 변화 요약: 새 렌더 컨텍스트(위젯 창)가 1개 추가된다. 이것이 M7의 유일한 신규 보안 표면이며, §5에서 메인 창과 동등한 통제로 봉합한다. IPC 채널은 신규 push 채널 1개(spip:favorites-changed)만 추가되고, invoke 채널은 기존 것을 재사용한다(새 invoke 0). 따라서 main 재검증·guard 표면은 거의 불변이다.

// electron/favoritesWidget.js — 위젯 창 생성(설계 의사코드, 구현 아님)
// ★SEC-H1: applyCspHeaders는 앱 1회만 등록(main.js:151, default session). 위젯은 같은 default
//   session을 공유하므로 위젯 생성 시 재호출하지 않는다(아래 deps에 applyCspHeaders 없음).
const PRELOAD_FAVORITES = path.join(__dirname, 'preload-favorites.js');
let widget = null;   // 단일 위젯 인스턴스(재사용)
let graceTimer = null;

function ensureWidget(deps) {                 // deps: { hardenWebContents, TRUSTED_ORIGIN, screen } — ★applyCspHeaders 불요(SEC-H1)
  if (widget && !widget.isDestroyed()) return widget;
  widget = new BrowserWindow({
    width: 360, height: 220, show: false,
    frame: false, transparent: true, resizable: false,
    minimizable: false, maximizable: false, fullscreenable: false,
    movable: false,                               // ★SEC-H2: 비드래그·고정 우측하단(MQ7-2)
    skipTaskbar: true, alwaysOnTop: false,         // ★SEC-H2: 생성 시 off, show 시에만 floating
    webPreferences: { preload: PRELOAD_FAVORITES, contextIsolation:true,
                      nodeIntegration:false, sandbox:true, webSecurity:true },
  });
  // ★SEC-H1: applyCspHeaders(session) 재호출 없음 — default session에 main이 이미 1회 등록.
  //   (onHeadersReceived는 세션당 1핸들러만 유효하므로 중복 등록 금지. favorites.html 메타 CSP가 2차 방어.)
  deps.hardenWebContents(widget.webContents, { trustedOrigin: deps.TRUSTED_ORIGIN }); // ★per-wc 필수: will-navigate·openHandler deny
  widget.loadURL('app://favorites.html');            // ★app:// 로 별도 페이지 로드(file:// 아님)
  widget.on('blur', () => {                            // ★SEC-H2+R1(MQ7-1): blur 시 floating 즉시 해제 + 100ms grace 후 hide
    widget.setAlwaysOnTop(false);                    // 표시 중에만 floating — 비포커스 위젯이 위에 떠 입력 가로채지 않게
    clearTimeout(graceTimer);
    graceTimer = setTimeout(() => { if (widget && !widget.isFocused()) widget.hide(); }, 100);
  });
  widget.on('focus', () => clearTimeout(graceTimer)); // 재포커스(위젯 내부 클릭 경합) 시 grace 취소 — 오작동 닫힘 방지
  widget.on('closed', () => { widget = null; clearTimeout(graceTimer); });
  return widget;
}

// show: 위치재계산 → floating ON → show → focus. hide: floating OFF 후 hide. Esc·헤더 ×는 명시 hide().
function show(deps) {
  const w = ensureWidget(deps);
  positionBottomRight(w, deps.screen);

// 우측하단 위치 계산(설계) — screen.getDisplayNearestPoint 사용
function positionBottomRight(win, screen) {
  const cursor = screen.getCursorScreenPoint();
  const disp = screen.getDisplayNearestPoint(cursor);   // 커서가 있는 모니터
  const wa = disp.workArea;                            // {x,y,width,height} — 작업표시줄 제외 영역
  const [w, h] = win.getSize();
  const margin = 16;
  const x = wa.x + wa.width  - w - margin;             // 우측 여백
  const y = wa.y + wa.height - h - margin;             // 작업표시줄 위(하단 여백)
  win.setPosition(Math.round(x), Math.round(y), false);
}

9. R-23 — 카드 드래그 reorder 애니메이션 (FLIP)

현재 reorder는 commitReorder가 store.state.order를 바꾸고 render()로 카드 DOM을 전부 재구성(app.js:2588)하므로 카드가 새 위치로 즉시 점프한다. FLIP(First-Last-Invert-Play)로 재배치 전/후 좌표차를 트랜지션으로 보간해 부드럽게 이동시킨다. 렌더러 전용 — IPC·main·보안 무관(DOM transform/transition만, 새 입력·새 권한 없음).

9.1 FLIP vs CSS-only transition (대안 비교)

대안장점단점 / 적합성
A. CSS transition만구현 단순(transition: transform). 코드 최소.현 구조에 부적합 — render()가 DOM을 새로 만들어(노드 교체) 기존 노드의 위치 트랜지션이 성립 안 함. transition은 같은 노드의 속성 변화에만 적용된다.
B. FLIP (채택)render()로 DOM이 재구성돼도 id 기준 이전 좌표를 기억해 새 좌표와의 차이를 invert 후 transition으로 play. 현 "전체 재렌더" 패턴과 호환. 라이브러리 0(getBoundingClientRect+requestAnimationFrame).좌표 캐시·rAF 1단계 추가. 키 매핑(card의 기존 data-card-id 재사용·R3) 필요. 복잡도 소폭 증가하나 외부 dep 0 원칙 유지.
C. DnD 라이브러리(SortableJS 등)드래그+애니메이션 일괄 제공.과설계·런타임 dep 추가 — "런타임 추가 의존성 electron 외 0" 원칙(EM ADR-EM-1) 위배. 기존 네이티브 HTML5 DnD(R-19)와 중복. 거부.

권고: B(FLIP). 현 전체 재렌더 구조와 유일하게 호환되고 런타임 의존성 0을 지킨다. ADR-M7-4.

9.2 FLIP 적용 설계 (의사코드)

// public/app.js — render() 전후를 감싸는 FLIP 헬퍼(설계 의사코드, 렌더러 전용)
// ★R3: 카드 요소는 이미 data-card-id를 가짐(실측 app.js:1870 card.dataset.cardId = vm.id).
//   신규 data-id를 부여하지 않고 기존 data-card-id/dataset.cardId를 그대로 키로 재사용(중복 속성 금지).
function flipReorder(containerSelector, mutate) {
  const reduce = window.matchMedia('(prefers-reduced-motion: reduce)').matches;
  const first = new Map();                              // First: cardId → 이전 rect
  if (!reduce) {
    for (const elx of document.querySelectorAll(containerSelector + ' [data-card-id]'))
      first.set(elx.dataset.cardId, elx.getBoundingClientRect());
  }
  mutate();                                          // = commitReorder의 order 변경 + render() (Last)
  if (reduce) return;                               // 모션 민감 사용자 — 즉시 적용(전이 0)
  for (const elx of document.querySelectorAll(containerSelector + ' [data-card-id]')) {
    const prev = first.get(elx.dataset.cardId);
    if (!prev) continue;                            // 신규 카드 — 페이드인 별도(선택)
    const now = elx.getBoundingClientRect();
    const dx = prev.left - now.left, dy = prev.top - now.top;   // Invert
    if (!dx && !dy) continue;
    elx.style.transform = `translate(${dx}px,${dy}px)`;
    elx.style.transition = 'none';
    requestAnimationFrame(() => {                    // Play
      elx.style.transition = 'transform 180ms cubic-bezier(.2,.8,.2,1)';
      elx.style.transform = '';
    });
  }
}
// commitReorder/reorderByDrop/reorderByMove가 render() 직접 호출 대신 flipReorder('.cards', () => render())로 위임.
// 드래그 중인 카드는 .is-dragging(기존 클래스)으로 FLIP 제외(포인터 추종은 네이티브 DnD가 처리).
항목설계
적용 범위 (R5)cards 밀도 카드 컨테이너만(.cards). R5 실측 확인: table 밀도(renderTable app.js:1985~)는 행에 draggable·reorderByDrop/reorderByMove 핸들러가 전혀 없다 — 즉 드래그 reorder 경로가 table에는 존재하지 않는다(행 클릭은 드로어 열기). 따라서 FLIP을 cards에만 적용하는 것은 누락이 아니라 reorder 트리거 부재에 정합한다(table는 reorder 자체가 없으므로 애니메이션 대상 아님 — UX 비대칭 없음). R-23 "카드 reorder" 요구 범위와 일치. 향후 table에 행 reorder가 도입되면 동일 FLIP 확장.
드래그 vs 드롭드래그 중 포인터 추종은 네이티브 HTML5 DnD(.is-dragging). 애니메이션은 드롭 확정 후 재배치(render) 시점의 위치 전이에 적용(다른 카드가 자리를 비켜주는 모션).
키보드 이동동일 FLIP 적용 — reorderByMove(위/아래 버튼·화살표, N-07)도 render를 flipReorder로 위임해 시각적 일관성 확보. 접근성 사용자도 이동을 시각적으로 추적.
prefers-reduced-motion존중(필수). matchMedia('(prefers-reduced-motion: reduce)') 시 transition 생략하고 즉시 적용. styles.css에도 @media (prefers-reduced-motion: reduce){ .card{ transition:none !important } } 방어.
성능transform/opacity만 애니메이트(레이아웃 리플로 회피, GPU 합성). 180ms 단발. rAF 1프레임. 카드 수 많아도 transform 일괄이라 부담 작음.
보안영향 0. 새 IPC·새 입력·새 권한 없음. DOM 노드의 transform 속성만 변경. textContent(L-1) 렌더 불변. §5 위젯 보안과 무관.

ADR-M7-1 — 즐겨찾기 위젯: 메인창 내 오버레이 → 별도 프레임리스 BrowserWindow (ADR-M6-5 번복)

상태: 제안(승인 권고) · 대체: ADR-M6-5(메인창 내 오버레이)를 Superseded로 전환

맥락: ADR-M6-5는 즐겨찾기 슬라이더를 메인창 내 position:fixed 오버레이로 결정했고, 그 근거는 "별도 창은 webPreferences·preload·CSP·senderFrame·하드닝을 한 벌 더 구성·검증(보안 표면·복잡도 2배)"이었다. 그러나 사용자 요구가 명확해졌다 — 대시보드 창을 닫아 트레이에만 상주해도 트레이 '즐겨찾기'로 위젯이 떠야 한다(독립성). 오버레이는 메인 renderer에 종속(renderFavoritesSlider)이라 메인창이 hide면 표시 불가(m6-design §9.3 가정에 이미 명시된 한계). 요구를 충족하려면 호스트가 메인창과 분리돼야 한다.

고려한 대안:

  • A. 오버레이 유지 + 트레이 시 항상 win.show: 트레이 '즐겨찾기'가 메인창을 강제로 띄움. 그러나 "대시보드를 닫아둔 채 즐겨찾기만 보고 싶다"는 의도(작업 방해 최소)와 모순 — 대시보드 전체가 떠버림. 독립성 요구 미충족.
  • B. 별도 프레임리스 BrowserWindow(채택): 메인창과 생명주기·렌더 컨텍스트 분리. 메인창 hidden이어도 위젯만 우측하단에 토스트로 등장.
  • C. OS 네이티브 토스트/알림: 즐겨찾기 카드의 인터랙션(열기·복사·해제·슬라이딩)을 알림 UI로 못 담음. 부적합.

결정: B. 즐겨찾기를 별도 frameless·우측하단·토스트형 BrowserWindow(favoritesWidget.js·app://favorites.html)로 구현. 트레이 '즐겨찾기'→favoritesWidget.show()(메인창 무관).

번복 근거(ADR-M6-5의 "보안 표면 2배" 우려 재평가): M6 이후 실측 코드에서 보안 통제가 이미 창 무관 재사용 단일 원천으로 모듈화됐다 — registerAppProtocol(app:// public/ 매핑·이탈 차단), security.js의 applyCspHeaders·hardenWebContents·CSP_POLICY, register.js의 isTrustedSender(app:// 정확 일치, 창 무관). 따라서 새 창의 보안 구성은 "한 벌 더 작성"이 아니라 동일 헬퍼를 2줄 호출 + app://favorites.html 1페이지 + 축소 preload 1개로 끝난다. "복잡도 2배"는 실제로 성립하지 않으며, 위젯 preload를 6개로 좁혀 표면은 오히려 메인보다 작다(§5). 독립성 요구를 충족하는 단순한 구조다.

결과: (+) 대시보드 닫혀도 즐겨찾기 동작(요구 충족)·토스트 UX·표면 축소 preload. (−) 렌더 컨텍스트 1개·push 채널 1개 추가(§5에서 동등 통제로 봉합)·메인↔위젯 동기화(broadcast) 필요·favoriteViewModels 등 일부 렌더 로직 위젯으로 이전(중복 최소화 위해 공유 모듈 권장). ADR-M6-5는 Superseded.

ADR-M7-2 — 위젯 닫기: hide(재사용) vs destroy

상태: 제안

맥락: blur/Esc/× 시 위젯을 destroy(매번 재생성)할지 hide(인스턴스 유지·재사용)할지.

고려한 대안:

  • A. destroy: 메모리 즉시 회수. 그러나 재호출마다 BrowserWindow 생성·app:// 로드·CSP/하드닝 재설정 비용 → 토스트 등장이 느려지고 깜빡임.
  • B. hide(재사용, 채택): 최초 1회 생성 후 hide/show. 재등장 즉시(<200ms). 메인창 close-to-tray(hide)와 동일 패턴(일관). 완전 종료에서만 destroy.

결정: B(hide 재사용). doFinalQuit()에서만 favoritesWidget.dispose()로 destroy. 단일 인스턴스(모듈 변수).

근거: 반응성·일관성(메인창 hide 패턴과 동형). 위젯은 작은 창이라 상주 메모리 부담 작음. 보안은 hide여도 동일 컨텍스트 유지라 약화 없음.

결과: (+) 빠른 재등장·구현 단순(생성 1회). (−) 숨은 채 메모리 상주(작음). 완전 종료 시 정리 1줄 필요(doFinalQuit 멱등 경로에 추가).

ADR-M7-3 — alwaysOnTop 수준: 'floating' (과하지 않은 상위 레벨)

상태: 제안

맥락: 토스트답게 다른 창 위로 떠야 하나, 모든 것 위에 고정되면(스크린세이버·전체화면 게임 위) 방해.

고려한 대안:

  • A. alwaysOnTop:false: 다른 창에 가려짐 — 토스트 의미 약함.
  • B. alwaysOnTop:true, level 'screen-saver'/'pop-up-menu': 거의 모든 것 위 — 전체화면 앱·시스템 UI도 덮어 과함.
  • C. alwaysOnTop:true, level 'floating'(채택): 일반 창보다 위, 전체화면/스크린세이버보다 아래. 토스트로 적정.

결정: C. show() 시 setAlwaysOnTop(true,'floating'), hide·blur 시 setAlwaysOnTop(false)로 해제(SEC-H2). 즉 표시·포커스 중에만 floating이다.

근거: 토스트의 "보이되 방해 최소" 절충. SEC-H2 보강 — 해제 시점 명세: 비포커스/숨김 위젯이 다른 앱 위에 계속 떠 입력을 가로채는 표면을 제거하기 위해 floating은 표시 중에만 유지하고 blur 즉시 해제한다(이후 100ms grace로 hide). 생성 시 alwaysOnTop:false로 시작.

결과: (+) 적정 가시성·비포커스 시 입력 가로채기 표면 제거. (−) 일부 전체화면 컨텍스트에서 가려질 수 있음(의도된 절충). 필요 시 후속에서 레벨 조정.

ADR-M7-5 — 위젯 외형: transparent 토스트 vs 불투명 패널 (SEC-H2 클릭통과·UI redressing)

상태: 제안

맥락: 보안 검증(SEC-H2)이 transparent:true frameless·alwaysOnTop 위젯의 클릭통과·UI redressing 표면을 지적했다. 둥근 모서리 토스트 외형을 어떻게 내되 클릭통과·오클릭 표면을 만들지 않을지 결정한다.

고려한 대안:

  • A. transparent:true + 클릭통과(setIgnoreMouseEvents forward): 투명 여백 클릭이 아래 앱으로 흐름. 거부 — 위젯이 다른 앱 입력 영역을 시각적으로 덮으면서 일부 클릭만 흘리면 사용자 인지 경계가 흐려지고(클릭재킹 유사), 반대로 아래 앱 입력을 일부 가로채는 모호성 발생.
  • B. transparent:true + 클릭통과 비허용(채택): setIgnoreMouseEvents 미사용. 투명은 모서리 라운드 표현용일 뿐, 위젯 창은 자기 영역의 모든 클릭 좌표를 소유한다(아래 앱으로 흘리지 않음). body 둥근 패널로 시각 외형만 토스트화.
  • C. transparent:false + 불투명 패널 + border-radius(폴백): 투명 미사용. Windows 환경별 transparent 렌더 이슈(그림자·잔상)나 부담 시 동등 외형을 CSS만으로 제공. 보안·UX 동등.

결정: B(클릭통과 비허용)를 기본, 환경 부담 시 C(불투명 폴백). 어느 경우든 setIgnoreMouseEvents 패스스루는 사용하지 않는다.

근거: 클릭통과를 쓰지 않으면 "위젯 위 클릭=위젯, 위젯 밖 클릭=blur(grace hide)"라는 단순·예측 가능한 입력 경계가 성립한다. 부수효과 액션은 focus 게이팅(SEC-H2)으로 추가 보호. B/C는 보안·UX가 동등하므로 ADR-M7-1(별도 창) 결정에 영향 없다.

결과: (+) 클릭재킹/입력 가로채기 표면 제거·환경 호환 폴백 보유. (−) 투명 여백이 클릭을 "삼키는"(아래 앱 미전달) 동작 — 토스트로는 자연스러우며 의도된 설계.

ADR-M7-4 — 카드 reorder 애니메이션 기법: FLIP (CSS-only·라이브러리 아님)

상태: 제안

맥락: reorder가 render() 전체 재구성이라 카드가 점프. 부드러운 이동 필요. §9.1 대안 비교.

고려한 대안: A. CSS transition만(노드 교체로 미성립) · B. FLIP(재렌더 호환·dep 0) · C. DnD 라이브러리(런타임 dep·과설계).

결정: B(FLIP). render 전 id별 rect 캐시 → render(Last) → invert(transform) → rAF로 play. prefers-reduced-motion 시 생략. 키보드 이동에도 동일.

근거: 현 "전체 재렌더" 구조와 유일 호환. 런타임 의존성 0(EM ADR-EM-1) 유지. transform/opacity만 애니메이트(성능). 보안 영향 0(렌더러 전용).

결과: (+) 부드러운 이동·dep 0·접근성 존중. (−) 좌표 캐시·rAF·기존 data-card-id 키 재사용(R3) 소폭 추가. table 밀도는 reorder 경로 부재로 미적용(R5).
