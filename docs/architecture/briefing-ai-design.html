Project-SPIP — 브리핑 AI 연동 아키텍처

홈 대시보드의 정적 "오늘의 브리핑"을, 로컬/온프레미스 OpenAI 호환 LLM이 생성하는 맥락 기반 능동 브리핑으로 확장한다. 데이터 변경 이벤트 → 순수 필요성 판정 → coalesce/fast-path → 메인 프로세스 LLM 호출(Langchain) → 스트리밍 토큰을 IPC 단방향 push → 홈 브리핑 영역 부분 갱신(RG/patchRegion). 모델 미구동 시 기존 정적 브리핑으로 폴백. 코드가 아니라 청사진이다 — 시그니처·스키마·의사코드·다이어그램까지만 정의하고 구현은 개발팀에 위임한다.

1. 개요 · 아키텍처 동인

이 설계가 좌우된 핵심 동인을 요구에 추적한다. 모든 결정은 아래 동인 중 하나 이상에 묶인다.

#아키텍처 동인출처설계 영향
D1네트워크 egress는 메인 프로세스 단독, 렌더러 CSP connect-src 'none' 불변R-34·N-08LLM 클라이언트는 lib/ai/에 두고 메인에서만 호출. 렌더러는 IPC만.
D2온프레미스 OpenAI 호환 서버 교체 가능(벤더 비고정)R-34baseURL/model/apiKey 런타임 설정 가변. ChatOpenAI 인스턴스를 호출마다 현재 config로 생성.
D3"거의 즉각 반응" = 인디케이터 ≤300ms + 급함 fast-path (추론 TTFT와 분리)R-36·N-09필요성 판정 → 인디케이터 push를 LLM 호출과 분리한 2단 경로.
D4과도 호출 억제 — 유의 변화일 때만, 폭주는 1회로 병합R-36순수 필요성 판정 함수 + m9 coalesce 큐 재사용·확장.
D5미처리 항목 carry-over + 상승 세션 write 보류R-38·m12uiStateStore 신규 키 + elevationState 게이트 일관.
D6신뢰 불가 입력(스캔/메일/커밋)의 프롬프트 인젝션 방어 + 출력 표시 전용N-08·L-1데이터/지시 영역 구조 분리·길이 상한·출력 textContent only.
D7모델 미구동/실패 시 무회귀 폴백R-40정적 브리핑 경로 보존 + 에러 분류 + 비방해 안내.
D8런타임 의존성 최소 규약(유일 의존 electron-updater)CLAUDE.mdLangchain 채택을 ADR로 정당화·범위 최소화(메인 전용).
D9기존 RG/patchRegion 부분 갱신 인프라 재사용(VDOM 비도입)R-35·m9/m10스트리밍 텍스트는 브리핑 영역만 patchRegion 교체.
설계 철학(단순성 우선). 새 프로세스·새 DB·새 메시지 브로커를 도입하지 않는다. 기존 composition root(context.js)·IPC register guard·RG·uiStateStore·config 0600·elevationState를 그대로 확장해 표면을 최소화한다. 유일하게 추가하는 외부 의존성은 Langchain이며, 그 정당화가 ①ADR이다.

3. 시스템 아키텍처(상위) · 신뢰 경계

시스템 컨텍스트와 신뢰 경계. 빨간 점선이 신뢰 경계다. 모든 LLM egress는 메인 프로세스 한 곳에서만 발생하며, 렌더러는 IPC를 통해서만 메인과 통신한다. 이 다이어그램은 security-reviewer의 위협 모델링 입력이다.

메인(신뢰·egress) 렌더러(샌드박스) 외부/비신뢰 영속 0600 빨간 점선 = 신뢰 경계

4. 소프트웨어 아키텍처 · 모듈 경계

레이어와 의존성 방향. 화살표는 "의존" 방향이며, 역방향 의존 금지(순수 코어는 Electron/DOM/IO를 모른다).

핵심 추상화 · 의존성 규칙. L4 순수 코어는 require('electron')·DOM·fs를 절대 import하지 않는다(헤드리스 단위테스트 보장, F-3 패턴). L2 오케스트레이터가 데이터·타이머·네트워크를 L4 순수 함수에 주입한다. 이는 기존 renderGuard.js(정본 순수 로직) ↔ app.js RG(얇은 어댑터) 구조와 동형이다.

① ADR-BR-1 — Langchain 런타임 의존성 채택

ADR-BR-1   Langchain(@langchain/openai) 메인 프로세스 런타임 의존성 채택   제안 → 승인 권고

맥락(Context)

CLAUDE.md는 "런타임 의존성 최소화 — 유일한 런타임 의존성은 electron-updater"를 규약으로 둔다. 그런데 FeatureSpec R-34와 ref.md는 Langchain 사용을 명시하며 OpenAI 호환 온프레미스 확장을 요구한다. 규약과 요구가 충돌하므로 결정을 기록한다.

고려한 대안

대안장점단점/리스크
A. @langchain/openai + @langchain/core(권고)ref 명시 충족 · OpenAI 호환 추상화 · 스트리밍/abort 1급 지원 · 향후 온프레미스 확장(다른 ChatModel) 용이의존성 트리 증가(@langchain/core 전이) · asar 번들 크기 +수 MB · CommonJS require 호환 확인 필요
B. 경량 fetch 직접 호출(/v1/chat/completions SSE 파싱 자체 구현)의존성 0(Node 18 내장 fetch) · 번들 영향 0 · 규약 완전 부합ref "Langchain 이용" 요구 위배 · SSE 파싱·취소·재시도·에러 매핑을 직접 구현(표면 증가) · 온프레미스 확장 시 추상화 부재
C. openai 공식 SDKOpenAI 호환 표준 · 스트림 지원ref는 Langchain 지정 · 추가 의존성은 A와 동급

결정(Decision)

대안 A 채택. 단, 의존성을 @langchain/openai + (전이) @langchain/core 2개로 최소화한다. langchain 메타 패키지(에이전트·체인·툴·벡터스토어 전체)는 설치하지 않는다. Langchain import는 lib/ai/llmClient.js 단일 파일로 격리하고, 나머지 모듈은 외부 의존성 0을 유지한다(교체·제거 시 영향 1파일).

근거(Rationale)

  • 요구 추적성: R-34·ref.md가 Langchain을 명시 — 임의 대체는 PM 결정 위배.
  • egress는 메인 단독(D1·N-08): Langchain은 메인 프로세스에만 적재되며 렌더러 번들(public/, 번들러 없음)에는 전혀 들어가지 않는다 → 렌더러 CSP/공격 표면 0 증가.
  • 격리로 규약 정신 보존: 단일 파일 격리 + 패키지 2개 한정으로 "최소화" 정신을 지키고, 무게 문제가 커지면 대안 B로의 교체가 1파일 수정으로 가능하도록 인터페이스를 클라이언트 경계 뒤에 둔다.

패키징 검증(필수 — 구현 P1 게이트)

  • CommonJS require: 프로젝트는 "type":"commonjs". @langchain/openai는 ESM/CJS dual package지만 일부 하위 의존이 ESM-only일 수 있다 → P1에서 require('@langchain/openai')가 Node 18 CJS에서 정상 로드되는지 스모크 검증. 실패 시 동적 import()(CJS에서 ESM 로드) 폴백 또는 대안 B 전환을 ADR 갱신으로 기록.
  • asar 포함: electron-builder.yml의 files allowlist는 electron/**·lib/**·public/**·package.json만 명시하고 node_modules는 명시하지 않는다. electron-builder는 production dependencies의 node_modules를 기본 자동 포함(devDependencies는 제외)하므로 dependencies에 등재한 Langchain은 asar에 들어간다. 그러나 allowlist를 명시한 경우의 동작을 P1 빌드에서 실측 확인하고, 누락 시 node_modules/@langchain/**/*를 allowlist에 명시 추가한다.
  • asar 내부 로드: main.js의 app:// 핸들러는 fs로 직접 읽지만, require는 asar 패치된 Node fs로 정상 동작. 네이티브 애드온이 없는 순수 JS 패키지인지 확인(있으면 asarUnpack 필요).

결과(Consequences)

④ 이벤트 파이프라인

오케스트레이터가 기존 발신원을 메인에서 직접 구독한다(렌더러 경유 아님 — 메인 자체 이벤트). 발신원은 이미 메인에 존재: StateWatcher onUpdate(projectsUpdated), MailWatcher onNewMail, ScanController 완료. 할 일/디스크 변경은 IPC 핸들러 성공 시 오케스트레이터 notify.

// 두 스냅샷 비교 → { trigger:bool, urgent:bool, signals:[...] } (DOM/타이머/IO 미접근)
function evaluate(prevSnap, curSnap, const) {
  const signals = [];
  // 유의 변화 표(FeatureSpec R-36) — 예시 일부
  if (newDirtyRepos(prev,cur).length > 0) signals.push({type:'dirty', urgent: newDirty >= const.MASS_DIRTY});
  if (becamePositive(cur.behind)) signals.push({type:'behind', urgent:true});   // 급함
  if (deadlineEnteredWindow(cur, const.DEADLINE_H)) signals.push({type:'deadline', urgent:true});
  ... // ahead·new mail·disk·attention·scan
  return { trigger: signals.length > 0, urgent: signals.some(s=>s.urgent), signals };
}

⑤ carry-over 상태 스키마(uiStateStore 신규 키)

상태 전이·규칙

  • open → done/dismissed: 사용자가 R-41 액션(resolveItem). "다시 본 것"은 처리 아님.
  • carry-over 선별: 재생성 시 (현재 유의 변화 신규 항목) + (status==='open'인 이전 항목). done/dismissed 제외.
  • auto-resolve: 근본 조건이 해소되면 해당 open 항목을 resolved로 자동 전이해 carry-over에서 제외. 판정은 현재 스냅샷 vs 항목 키의 signalType 대조(순수). 신호유형별 해소 정의는 아래 매트릭스.
  • 만료·상한: items 개수 상한(예 200, MAX_TODOS와 동급) + dismissed는 N일(예 14일) 후 정리. uiStateStore 1MB 상한·깊이 가드 그대로 적용.
  • m12 상승 세션: elevationState.isElevated()면 uiStateStore write()가 이미 no-op(메모리 유지) — carry-over·done/dismiss는 해당 세션 메모리에만 유지. 중복 경고 금지(기존 elevation 배너로 충분). 오케스트레이터는 메모리 캐시를 진실 원천으로 삼고 write 결과(정규화 메모리)를 그대로 사용.

분류 소유권(중요). 3분류·urgent 여부는 순수 정책(briefingPolicy.evaluate)이 결정하고 모델에는 "이 항목을 설명·가이드하라"만 요청하는 것을 기본으로 한다. 모델이 분류를 제안해도 fast-path/우선순위 같은 동작에 영향 주는 분류는 정책 값을 신뢰(프롬프트 인젝션 ③ 무력화). 모델 출력은 표현(title/reason/guide)만 채운다.

⑦ 렌더러 스트리밍 UI(R-35/R-41)

현재 히어로의 정적 안내 문장(app.js renderHome heroL의 <p>)을 브리핑 영역으로 승격한다. 영역을 2단 구조로 만든다(m11 메일/차트 패턴 동형): .briefing-region(patchRegion 교체 대상) > 카드. 스트림 delta가 올 때마다 브리핑 영역만 patchRegion 교체해 홈 다른 섹션·라이브 갱신을 끊지 않는다.

렌더 동작

  • 스트리밍 텍스트: delta 누적 버퍼를 textContent로 표시(L-1). 마크다운→HTML 변환 금지(M-3) — innerHTML·insertAdjacentHTML 절대 금지, el()+textContent 노드 구성만. 강조가 필요하면 평문 토큰을 별도 노드로 분리(HTML 파싱 경유 없음).
  • 인디케이터(≤300ms): state:generating 수신 즉시 "생성 중" + 중단 버튼 표시. LLM 첫 토큰과 무관(D3).
  • RG 정합: patchRegion은 deferred()(IME 조합·드래그·오버레이·busy) 중이면 자동 보류 후 coalesce로 1회 병합(기존 계약). 검색 한글 조합 중 delta가 와도 입력 안 깨짐. delta는 고빈도이므로 오케스트레이터가 메인에서 토큰을 약하게 배치(예 ~80ms flush)해 push 빈도를 낮춘다(렌더 부하·IPC 폭주 완화).
  • 세대(gen) 가드: 렌더러는 현재 gen과 다른 delta/done을 무시(취소된 이전 호출 잔여 토큰 섞임 방지, R-34 수용 기준).
  • 항목 UI(R-41): done/dismiss 버튼 + 분류 배지(must/good/urgent 색 구분, urgent 강조) + 가이드 접힘/펼침(urgent 기본 펼침). 키보드 접근(role/tabindex/Enter·Space, N-07 연속). done/dismiss → spip.briefing.resolveItem(UI→상태 단방향, R-38 소유).
  • 중단: 중단 버튼 → spip.briefing.abort. 마지막까지 받은 부분 결과 보존.
RG.widget('briefingActions')로 done/dismiss·펼침 토글 핸들러 라이프사이클을 관리(render 1회당 destroy/mount — 핸들러 누적 0, m9 RG-3 패턴). 항목 노드는 전부 el()+textContent로 구성하며 모델 출력의 어떤 문자열도 클릭 핸들러·href 실행에 쓰지 않는다(N-08 출력 표시 전용).

리스크 · 가정 · 미해결 질문

리스크

RK1 — Langchain CJS/asar 호환. ESM-only 전이 의존이 있으면 CJS require 실패 가능. 완화: P1 게이트에서 스모크 검증 → 실패 시 동적 import() 폴백 또는 ADR 대안 B(경량 fetch) 전환. 인터페이스를 llmClient 경계 뒤에 둬 1파일 영향.
RK2 — 로컬 모델 환각. 7.8B가 없는 기능 안내 가능. 완화: 분류는 정책이 소유·출력 표시 전용·자동 실행 0 → 안전 영향 0(품질 영향만).
RK3 — delta 고빈도 IPC 부하. 토큰마다 push 시 IPC/렌더 폭주. 완화: 메인에서 ~80ms 배치 flush + patchRegion 영역 격리.
RK4 — ui-state 키 추가. briefing 키 추가(schemaVersion 유지). 완화: normalizeBriefing이 누락/구버전 파일을 graceful 기본값으로 채움 — 기존 normalize 패턴과 동일(파괴적 마이그레이션 없음·무손실).
RK5 — 인디케이터 vs 추론 혼동. 300ms와 TTFT를 한 지표로 보면 저사양에서 오판. 완화: 두 지표를 구조·테스트로 분리(D3·N-09).

가정

  • 사용자가 OpenAI 호환 로컬 서버(LM Studio 등)를 별도 구동(앱은 연결만).
  • StateWatcher/MailWatcher/ScanController가 메인에서 구조화 데이터·이벤트를 이미 제공(재사용).
  • RG/patchRegion/설정 2-pane/elevationState가 그대로 재사용 가능(m9~m12 완료).
  • electron-builder가 production dependencies node_modules를 asar에 자동 포함(P1에서 실측 확정).
