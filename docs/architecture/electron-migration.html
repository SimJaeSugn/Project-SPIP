Project-SPIP — Electron 설치형 데스크톱 이전 설계서

현재 "터미널 npm start + 브라우저 별도 접속" 형태의 로컬 HTTP 웹앱을, 더블클릭 실행 + OS 네이티브 폴더 대화상자를 제공하는 Electron 설치형 데스크톱 앱으로 이전하기 위한 마이그레이션 청사진이다. 핵심 전략은 검증된 lib/ 도메인 코드(스캐너·수집기·scanController·pathGuard·safeExec)를 그대로 보존하고, HTTP 전송 계층(server.js·router·staticHandler)만 Electron main/preload/renderer + IPC로 교체하는 점진 이전이다. 본 문서는 설계이며 구현 코드를 포함하지 않는다.

1. 개요 / 이전 동인

기존 시스템은 스캔(쓰기) / 서빙(읽기)을 프로세스로 분리한 파이프라인-앤-대시보드 구조다(architecture.html §4). 데스크톱 이전은 이 핵심 설계를 유지한 채, "서빙" 경로만 HTTP에서 Electron IPC로 바꾼다. 스캔 도메인(lib/scan/*)·공용 유틸(lib/common/*)·스냅샷/액션/재스캔 도메인(lib/server/snapshotStore·scanController 등)은 전송 방식과 무관하므로 무손상 재사용한다.

보존되는 핵심 보안 전제(RC-2, 그대로 유효): 스캔 대상 폴더에는 신뢰할 수 없는 외부 클론 리포가 섞일 수 있다. 따라서 부분신뢰(TB-B) 입력 방어(pathGuard realpath·safeExec 절대경로·H-3 입력 가드·M-3 깊이/심링크·L-1 textContent)는 전송 계층과 무관하게 전부 유지한다. HTTP→IPC 전환은 TB-A(네트워크 경계)만 바꾸고 TB-B는 손대지 않는다.

2. 프로세스 모델 (main · preload · renderer)

Electron은 3개 실행 컨텍스트로 나뉜다. 무엇이 어디서 도는가가 이전의 핵심이다 — 무거운 I/O(스캐너·수집기·scanController)·OS 접근(dialog·spawn·파일)은 main(Node 전권)에서, UI는 renderer(Chromium, Node 없음)에서 돌고, 그 사이는 preload가 화이트리스트한 IPC API로만 연결된다. renderer는 직접 require·fs·child_process에 접근할 수 없다.

3. 재사용 / 교체 / 드롭 매핑

이전의 핵심 원칙: 전송(HTTP)과 도메인을 분리해 도메인은 손대지 않는다. 아래 분류 — 그대로 재사용 코드 무수정 · IPC로 감싸 재사용 도메인 코드는 유지하되 호출자를 ipcMain 핸들러로 교체 · 드롭 HTTP 전송 전용이라 제거 · 신규 Electron 골격.

4. IPC 계약 (api-contract.md 대체)

renderer↔main 통신은 두 패턴을 쓴다 — invoke/handle 요청/응답(ipcRenderer.invoke ↔ ipcMain.handle, Promise 반환) · on/send main→renderer 단방향 푸시(webContents.send ↔ ipcRenderer.on). 진행 표시는 기존 1초 폴링(GET /api/scan-status)을 폐기하고 main→renderer 이벤트 푸시로 단순화한다(R-15). 모든 채널은 preload가 window.spip에 화이트리스트 함수로만 노출한다.

EM-H-1 / PM 확정 Q2 — browseDir 채널 드롭: 이전 초안의 인앱 디렉터리 브라우저 채널(spip:browseDir(dir))은 제거(드롭)한다. renderer가 임의 디렉터리 경로를 던져 파일시스템 트리를 열람하는 채널은, RC-2(부분신뢰 리포)+저장형 XSS 가정에서 화이트리스트 밖 임의 경로 정보노출 표면이 된다. 임의 디렉터리 열람 채널을 둔다는 발상 자체를 폐기한다. 폴더 선택은 (1) 네이티브 dialog.showOpenDialog(§5) 또는 (2) 경로 직접 입력(addRoots) 두 경로만 두고, 둘 다 main에서 pathGuard.canonicalize/realpathSync.native·존재·디렉터리·시스템 폴더 제외 검증을 통과한 것만 config에 영속한다.

// === main: ipc/folders.js — addRootsResolve 얇은 래퍼 (의사코드, F-5 해소) ===
// normalizeScanRoots는 거부분을 안 돌려주므로(config.js:90~127), 래퍼가 입력 대비
// 채택/거부를 직접 산출한다. normalizeScanRoots는 무수정 유지.
function addRootsResolve(rawPaths, currentRoots, logger) {
  if (!Array.isArray(rawPaths)) return { ok:false, code:'INVALID_PATH' };
  const added = [], rejected = [];
  for (const raw of rawPaths.slice(0, LIMITS.maxScanRoots)) {
    if (typeof raw !== 'string' || raw.length === 0 || raw.length > 4096) { rejected.push({ path:String(raw).slice(0,256), reason:'NOT_FOUND' }); continue; }
    const real = canonicalizeDir(raw);              // resolve→realpathSync.native→statSync.isDirectory
    if (!real)        { rejected.push({ path:raw, reason:'NOT_FOUND' }); continue; }
    if (isSystemDir(real)) { rejected.push({ path:raw, reason:'SYSTEM_DIR' }); continue; }  // 드라이브 루트·WINDIR·ProgramFiles 제외
    if (currentRoots.includes(real) || added.includes(real)) { rejected.push({ path:raw, reason:'DUP' }); continue; }
    added.push(real);
  }
  // 최종 영속은 normalizeScanRoots(merge)로 한 번 더 통과시켜 단일 화이트리스트 원천 유지
  const roots = config.normalizeScanRoots([...currentRoots, ...added], logger);
  persistConfig({ ...currentConfig, scanRoots: roots });   // 0600 원자적(M-2)
  return { ok:true, added, rejected, roots };
}

5. 네이티브 폴더 선택 (R-04)

브라우저가 절대경로를 못 얻던 근본 제약을 OS 네이티브 대화상자로 해소한다. 대화상자 호출과 경로 정규화·영속은 전부 main에서 수행하고, renderer는 트리거만 한다(경로 문자열을 renderer가 생성·전송하지 않음 → 신뢰 출처 유지).

// === main: ipc/folders.js (의사코드) ===
ipcMain.handle('spip:pickFolders', async (evt) => {
  const { canceled, filePaths } = await dialog.showOpenDialog(win, {
    title: '스캔할 프로젝트 폴더 선택',

  // canonicalize/realpath·디렉터리·시스템폴더 제외·dedup을 거친 것만 영속(L-2/H-1/M-2)
  const r = addRootsResolve(filePaths, currentConfig.scanRoots, logger);
  return { ok:true, added: r.added, roots: r.roots };
});

원격 콘텐츠 미로드loadFile('public/index.html')만. http(s)·외부 URL 로드 금지. webPreferences.webSecurity 기본 유지
file:// 자산 경로 전략 (EM-M-1)loadFile로 로컬 로드 시 현행 index.html의 /static/styles.css·/static/app.js 절대경로는 file 루트(드라이브 루트) 기준이 되어 로드 실패한다(staticHandler가 사라지므로 /static 매핑 부재). 정정: 자산 참조를 ./static/ 상대경로(또는 public/ 평탄화)로 재작성하거나, 안정성을 위해 커스텀 프로토콜 protocol.handle('app://')로 public/ 루트를 매핑하고 loadURL('app://index.html')로 로드한다(권장 — origin이 app://로 명확해져 CSP 'self' 의미가 분명). 둘 중 택1을 단계 1에서 확정
탐색/팝업 차단webContents.on('will-navigate', e=>e.preventDefault()) + setWindowOpenHandler(()=>({action:'deny'})) + will-attach-webview 거부(sub-frame·webview 생성 금지). EM-L-1: shell.openExternal은 기본 금지(미사용) — 임의 URL 스킴 실행 표면 차단. 향후 도입 시 https: 스킴 allowlist + 사용자 확인을 전제로만
IPC 발신자 검증 (EM-M-2)모든 ipcMain.handle에서 event.senderFrame(또는 webContents)이 메인 윈도우의 신뢰 로컬 origin(app:// 또는 file:// 우리 index)인지 확인하고, 불일치 시 무시. M-1(Origin/토큰) 제거가 만든 공백의 심층방어 — will-navigate 차단이 실패하거나 미래에 외부 프레임이 유입돼도 main이 무차별 수신하지 않게 함
renderer CSP 강화 (EM-M-1)index.html <meta http-equiv="Content-Security-Policy">를 현행 전 디렉티브 보존+강화(축약 금지 — 누락은 약화다). 권장 정책:
default-src 'none'; script-src 'self'; style-src 'self'; img-src 'self' data:; font-src 'self'; connect-src 'none'; object-src 'none'; base-uri 'none'; form-action 'none'; frame-ancestors 'none'
(IPC만 쓰므로 connect-src는 'none' 가능). 추가로 Electron session.webRequest.onHeadersReceived로 동일 CSP를 응답 헤더로도 주입해 메타 누락/우회를 이중 차단. staticHandler의 CSP 상수 기준이 사라지므로 단일 출처를 index.html 메타 + 세션 헤더로 재정의. L-1 저장형 XSS 2차 방어선 유지·강화

6.3 유지되는 통제 (TB-B 부분신뢰 — 코드 무수정 보존)

  • pathGuard canonicalize/realpath 화이트리스트 정확 일치(H-1) — IPC open 핸들러가 여전히 호출. id→경로 역참조·realpath 검증 동일
  • safeExec resolveBin 절대경로·shell:false·타임아웃·maxBuffer(H-2·M-4) — code/git 실행 그대로
  • H-3 입력 가드 collectors의 크기·깊이·ReDoS-free·문자열 절단 — 스캐너가 main에서 동일 동작
  • M-3 walker 내장 안전 깊이 상한·방문 realpath 추적(심링크 루프 차단)
  • L-1 textContent 렌더 — app.js 보존. 오염된 name/description/branch 저장형 XSS 차단(RC-2)
  • M-2 config/cache 0600·0700 원자적 쓰기 — serializer/paths 그대로
  • L-3 에러 메시지 정책 — IPC 반환 객체도 고정 code/message만, 절대경로·스택 비노출
  • M4-H-1 ScanProgress.currentPath shortenPath 축약 — 진행 이벤트 push에도 동일 적용
RC-2 그대로 유효: 데스크톱 전환은 신뢰 불가 리포 위협을 줄이지 않는다. 오히려 nodeIntegration:false·sandbox로 renderer 측 폭발 반경을 추가로 축소한다. main은 여전히 부분신뢰 파일을 파싱·실행하므로 TB-B 통제가 1차 방어선이며 필수 보존.

    }

12. ADR (아키텍처 결정 기록)

ADR-EM-1 — Electron 채택 + 외부 의존성 0 원칙 의도적 완화

상태: 승인(PM 확정)

맥락: 사용자는 (1) 터미널+브라우저 이중 실행, (2) 브라우저의 폴더 절대경로 미획득으로 불편. 기존 N-04(외부 의존성 0)는 데스크톱 전환과 충돌.

고려한 대안:

  • A. 로컬 HTTP 유지 + 런처 스크립트/PWA: 의존성 0 유지하나 폴더 절대경로 문제(②) 미해결, 더블클릭 설치 경험 부족.
  • B. Electron: 기존 Node 백엔드·vanilla 프론트 재사용 최적. dialog로 ② 해결. 번들 크기 큼·런타임 추가.
  • C. Tauri(Rust): 경량 번들이나 Rust 백엔드 재작성 필요 → lib 재사용 불가, 검증 자산 폐기.

결정: B(Electron). 단 electron·electron-builder는 devDependencies, 런타임 dep 0 유지.

근거: 핵심 자산(검증된 Node lib 213 테스트·vanilla UI)을 무손상 재사용하면서 두 불편을 동시 해결. C는 재작성 비용이 과대, A는 ②를 못 풂.

결과: (+) lib 무수정 재사용·즉시 더블클릭·네이티브 dialog. (−) 설치본 크기 증가(~100MB대)·Electron 버전 추적/보안 패치 부담. 단일 사용자 로컬 도구라 수용.

ADR-EM-2 — 전송 계층: HTTP 폐기 후 IPC 전면 채택 (로컬 HTTP 병존 안 함)

상태: 승인

맥락: Electron 내에서 renderer↔main 통신을 (a) 기존 로컬 HTTP 서버 유지(BrowserWindow가 127.0.0.1 로드) vs (b) IPC 전환 중 선택.

고려한 대안:

  • a. 로컬 HTTP 유지: 코드 변경 최소(router/security/fetch 그대로). 그러나 리스닝 소켓·DNS 리바인딩·CSRF·세션토큰 통제를 계속 짊어짐. Electron 안에서 불필요한 네트워크 표면.
  • b. IPC 전환: 네트워크 표면 0. preload allowlist로 노출 최소화. 진행은 이벤트 푸시로 단순화. 전송 어댑터 교체 비용 발생.

결정: b(IPC 전면 채택). server.js/router/staticHandler/security 드롭, ipcMain 핸들러로 대체.

근거: 데스크톱 전환의 보안 이득(네트워크 표면 제거)을 온전히 취함. 폴링→푸시로 진행 로직도 단순화. 도메인 무수정이라 어댑터 교체 비용이 작음.

결과: (+) 보안 표면 축소·진행 단순화·테스트 순수함수화 용이. (−) app.js 전송부·HTTP 테스트 이관 필요. 과도기 HTTP는 완료까지 보존 후 삭제(PM 확정 Q1).

ADR-EM-3 — Electron 보안 베이스라인 (contextIsolation·sandbox·preload allowlist)

상태: 승인

맥락: RC-2(신뢰 불가 리포)가 유효한 상태에서 renderer가 오염 메타데이터를 다룬다. Electron 기본 설정은 안전하지 않을 수 있음.

고려한 대안: (a) nodeIntegration:true 편의 우선 — renderer에서 직접 fs/spawn(위험, 거부). (b) contextIsolation+nodeIntegration:false만. (c) (b)+sandbox:true+preload 최소 allowlist.

결정: (c). contextIsolation:true·nodeIntegration:false·sandbox:true(필수 MUST — EM-L-2 격상)·preload는 window.spip 화이트리스트만(ipcRenderer 원본·범용 invoke 비노출)·원격 미로드·will-navigate/setWindowOpenHandler/webview 차단·renderer CSP 강화(default-src 'none' + 전 디렉티브) + 세션 헤더 이중주입(EM-M-1).

근거: 저장형 XSS가 발생해도 renderer에서 OS로 승격 불가(폭발 반경 최소). TB-B 통제(pathGuard·safeExec·H-3)는 main에서 그대로 1차 방어선. sandbox는 본 설계가 이미 호환하므로 "권장"이 아닌 "필수"로 강제할 수 있다(비용 0).

결과: (+) RCE 승격 차단·심층 방어. (−) sandbox 채택 시 preload에서 우리 모듈 require 불가 → 모든 로직 main 강제(본 설계가 이미 준수). dialog 결과 외 renderer발 경로 전부 검증 필요.

ADR-EM-6 — IPC 노출면 최소화: 전 채널 main 재검증 + browseDir 드롭 (EM-H-1)

상태: 승인(PM 확정 Q2 — 보안 게이트 EM-H-1 해소)

맥락: 초안은 preload allowlist를 "원칙"으로 선언하고 §4.2 검증을 일부 채널만 명세했다. 특히 인앱 디렉터리 브라우저(spip:browseDir(dir))는 renderer가 임의 디렉터리 경로를 던지는 신규 임의경로 채널로, RC-2(부분신뢰)+저장형 XSS 가정에서 화이트리스트 밖 파일시스템 정보노출 표면이 된다.

고려한 대안:

  • A. browseDir 유지 + scanRoots 하위로만 열람 제약: 구현·검증 복잡, 여전히 임의경로 입력 표면이 존재.
  • B. browseDir 드롭 + 네이티브 dialog·직접입력만: 임의경로 열람 채널 자체를 제거. 폴더 선택은 OS 대화상자(신뢰 출처)와 검증된 직접 입력으로 충분.

결정: B. browseDir 채널을 드롭하고, 전 IPC 채널의 main 재검증을 §4.2에 못박는다(인자 스키마·길이/개수 상한·화이트리스트·고정 실패 code·반환 shape). renderer는 신뢰하지 않으며(XSS 가정), 경로는 canonicalize/realpath·디렉터리·시스템 폴더 제외를 통과한 것만 영속한다. 발신자 검증(senderFrame, EM-M-2)을 공통 통제로 추가한다.

근거: 임의 디렉터리 열람 채널을 두지 않는 것이 가장 단순하고 안전하다. 네이티브 dialog는 경로 출처가 OS라 신뢰 가능하고, 직접 입력은 동일 검증을 거친다. preload allowlist는 표면 축소일 뿐, 신뢰 근거는 main 재검증에 둔다.

결과: (+) 임의경로 정보노출 표면 제거·보안 게이트 해소. (−) 인앱 트리 탐색 UX 부재(네이티브 대화상자로 대체). 모든 채널에 검증 코드 필요(어댑터 얇게 유지).

ADR-EM-4 — 앱 데이터 경로: paths.js 유지 + app.getPath 정합 결정

상태: 제안(PM/구현 확인 필요)

맥락: 기존 paths.appDir()는 OS 규약 경로(%APPDATA%\spip)를 직접 계산. Electron은 app.getPath('userData')로 %APPDATA%\<productName>를 제공. 두 경로가 다르면 CLI(scan.js)와 앱이 다른 캐시를 봄.

고려한 대안: (a) paths.js 그대로 — CLI/앱이 spip 폴더 공유, app.getPath 미사용. (b) paths.js를 app.getPath로 전환 — Electron 관례 정합하나 CLI는 Electron 없이 동작해야 해 분기 필요.

결정(권고): (a) paths.js 유지 + electron-builder productName을 경로와 무관하게 두고, CLI와 앱이 동일한 spip 앱 폴더를 공유. app.getPath는 로그/임시 등 비공유 용도로만.

근거: CLI 스캔 결과를 앱이 즉시 읽어야 함(단일 출처). paths.js는 Electron 비의존이라 CLI 호환 유지.

결과: (+) CLI↔앱 캐시 일관·재사용 무수정. (−) Electron 관례(userData)와 약간 어긋남 — 문서화로 해소. 구현 시 실제 경로 일치 테스트 필요(Q3).

ADR-EM-5 — 패키징: electron-builder · NSIS + 포터블

상태: 승인(서명 정책 제외)

맥락: Windows 설치형 배포 필요. 단일 사용자 도구.

고려한 대안: (a) electron-builder(NSIS/포터블/다양한 타깃·서명·업데이트 통합). (b) electron-packager(저수준, 설치본 직접 구성). (c) Squirrel.Windows(자동업데이트 강하나 설정 복잡).

결정: (a) electron-builder, Windows 타깃 NSIS + portable 2종. Electron fuses(RunAsNode off·OnlyLoadAppFromAsar·asar integrity)·ELECTRON_RUN_AS_NODE 무력화를 베이스라인으로 포함(EM-M-3). 미서명 배포는 위험 수용으로 확정(PM Q5).

근거: 설정 간결·설치/언인스톨·포터블 동시 산출·향후 서명/업데이트 확장 용이. asar는 무결성 보장이 아니므로 fuses·integrity로 런타임 악용 표면을 줄인다.

결과: (+) 더블클릭 설치 경험·런타임 악용 표면 축소. (−) 미서명 시 SmartScreen 경고(수용 — 설치 안내 포함). 외부 배포 확대 시 코드 서명 재검토. 자동 업데이트는 범위 외.
