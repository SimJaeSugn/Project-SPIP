M12 — 자동 업데이트 후 스캔 데이터 미표시 버그

근본 원인 조사 · 최소·안전 수정안 설계 (조사/설계 단계, 코드 미수정)

대상 Project-SPIP (Electron 데스크톱 앱) · electron-builder 24.13.3 + electron-updater 6.8.9 · Windows NSIS+portable · CommonJS
증상 앱 내 자동 업데이트(electron-updater) 직후에만 스캔 목록이 빈다. 일반 재시작은 정상이며, 업데이트 후 빈 앱을 완전히 종료하고 다시 열면 데이터가 돌아온다(삭제 아님).
결론(요약) 원인 확정 업데이트 설치가 관리자 권한(elevated)으로 진행된 케이스에서, NSIS의 설치 후 재실행이 관리자 권한 컨텍스트로 앱을 띄워 그 세션의 %APPDATA%가 관리자 프로필을 가리킨다 → 빈 %APPDATA%\spip → "초기화"처럼 보임. 다음 일반 실행은 원래 사용자라 데이터 복귀.

① 근본 원인 조사 — 근거

1.1 데이터 경로는 전적으로 환경변수에 종속

스캔 스냅샷·설정·UI 상태는 lib/common/paths.js의 appDir()가 결정한다. Windows에서:

base = process.env.APPDATA || path.join(os.homedir(), 'AppData', 'Roaming');
return path.join(base, 'spip');   // → %APPDATA%\spip

lib/server/snapshotStore.js가 기동 시 paths.cachePath()(= <appDir>/cache/projects.json)를 1회 읽어 메모리에 적재하고, 렌더러 public/app.js load()가 getProjects로 그 스냅샷을 복원한다. 버전 게이팅·마이그레이션·캐시 삭제 로직은 어디에도 없다. 즉 데이터가 사라지는 게 아니라, 읽는 %APPDATA%가 다른 프로필을 가리키면 빈 폴더로 보인다.

핵심: os.homedir()·process.env.APPDATA·Electron app.getPath('appData')는 모두 현재 프로세스를 띄운 사용자 토큰을 따른다. 앱이 관리자(Administrator) 토큰으로 뜨면 %APPDATA% = C:\Users\Administrator\AppData\Roaming (또는 UAC 가상화/다른 프로필) → 거기엔 spip\cache\projects.json이 없다.

1.2 electron-updater의 재실행 경로 (소스 확인)

electron/autoUpdate.js는 quitAndInstall(false, true)(isSilent=false, isForceRunAfter=true)를 호출한다. node_modules 소스 추적 결과:

① electron-updater/out/BaseUpdater.js — quitAndInstall

quitAndInstall(isSilent = false, isForceRunAfter = false) {
  // 비-사일런트면 isForceRunAfter 대신 this.autoRunAppAfterInstall(기본 true) 사용
  const isInstalled = this.install(isSilent, isSilent ? isForceRunAfter : this.autoRunAppAfterInstall);
  ...
}

→ 우리 호출은 install(false, true)와 동치(autoRunAppAfterInstall 기본 true). 설치 후 앱을 재실행한다.

② electron-updater/out/NsisUpdater.js — doInstall

const args = ["--updated"];
if (options.isSilent) args.push("/S");          // ← isSilent=false 이므로 /S 없음
if (options.isForceRunAfter) args.push("--force-run");
...
if (options.isAdminRightsRequired) {            // latest.yml의 isAdminRightsRequired(기본 false)
  callUsingElevation();  // process.resourcesPath/elevate.exe 로 인스톨러 실행
  return true;
}
this.spawnLog(installerPath, args)...           // 평소엔 비-elevated 로 spawn
관찰 1. 인스톨러는 비-사일런트(/S 없음) + --force-run로 실행된다.
관찰 2. 우리 빌드는 publish에 isAdminRightsRequired를 넣지 않으므로 electron-updater 단계에서 elevate.exe는 쓰이지 않는다 — 인스톨러는 현재(비-상승) 사용자 토큰으로 시작한다.

1.3 NSIS 인스톨러 내부의 권한 상승 & 재실행 (소스 확인)

oneClick:false 이므로 assisted installer 경로다(app-builder-lib/templates/nsis/installer.nsi):

!ifdef INSTALL_MODE_PER_ALL_USERS   ; ← perMachine:false 라 이 심볼 미정의
  RequestExecutionLevel admin
!else
  RequestExecutionLevel user        ; ← 우리 케이스: 인스톨러 자체는 user 레벨로 시작
!endif

재실행(launch-after-install) 경로 — assistedInstaller.nsh & installSection.nsh

비-사일런트 assisted 인스톨러는 두 가지 재실행 지점이 있다:

  • 완료 페이지(MUI_PAGE_FINISH) "Run app" 체크박스 → StartApp 함수가 ${StdUtils.ExecShellAsUser}로 실행. StdUtils.nsh 주석: "ShellExecute() as NON-elevated user from elevated installer" — explorer.exe 경유 de-elevation.
  • installSection.nsh doStartApp → ${if} ${isForceRun} ${andIf} ${Silent} 조건. 우리는 비-사일런트라 이 분기는 실행 안 됨.
de-elevation이 정상 동작하는 경우: 사용자가 완료 페이지까지 보고 "실행"으로 떴다면 ExecShellAsUser가 원래 사용자로 de-elevate → %APPDATA% 정상 → 데이터 정상. (그래서 항상 재현되지 않고 "자동 업데이트 직후에만" 나타난다.)

1.4 그렇다면 언제 elevated 컨텍스트로 뜨나 — 가설 검증 결과

인스톨러가 RequestExecutionLevel user로 시작하더라도, 다음 경로에서 UAC 상승이 일어난다(multiUserUi.nsh, multiUser.nsh, installer.nsi):

  • 기존 설치가 per-machine(HKLM)인 경우: initMultiUser가 HKLM InstallLocation을 읽어 $hasPerMachineInstallation=1 → 설치 모드 "all"로 강제 → UAC_RunElevated로 상승된 inner 인스턴스 생성. 사용자가 과거에 "모든 사용자용" 또는 allowToChangeInstallationDirectory:true로 Program Files 등 보호 경로에 설치했다면 이 경로를 탄다.
  • 보호 경로 쓰기 실패: 디렉터리 변경으로 보호 경로에 설치 시도 시 쓰기 권한이 필요해 상승 흐름으로 이어진다.
버그가 트리거되는 정확한 조합: 상승된 inner 인스턴스(Administrator 토큰)에서 설치가 끝난 뒤, 그 인스턴스가 앱을 재실행할 때 ExecShellAsUser를 통과하지 못하면(완료 페이지를 건너뛰는 흐름이거나, 상승 컨텍스트에서 launch가 일어나는 흐름이면) 앱은 관리자 토큰으로 뜬다. 그 세션의 %APPDATA%는 관리자 프로필 → %APPDATA%\spip 빈 폴더 → 빈 목록.

1.5 증상과의 정합성 (가설 확정 근거)

관찰된 증상가설로 설명
자동 업데이트 직후에만 빔업데이트 설치가 상승 컨텍스트를 거치고, 재실행이 그 토큰을 상속할 때만 발생. 일반 실행은 항상 로그인 사용자 토큰.
일반 재시작은 정상바탕화면/시작메뉴 단축아이콘 실행은 로그인 사용자 토큰 → 원래 %APPDATA%\spip 읽음.
완전 종료 후 다시 열면 데이터 복귀데이터는 원래 사용자 프로필에 그대로 있고 삭제되지 않았다. 단지 직전 세션이 잘못된 프로필을 읽었을 뿐.
마이그레이션·캐시삭제 코드 없음에도 발생코드가 데이터를 지우는 게 아니라, 읽는 경로(프로필)가 바뀐 것이라는 가설과 완전 정합.
확정 vs 잔여 불확실성. 메커니즘(상승 컨텍스트 → %APPDATA% 프로필 분기 → 빈 spip)은 소스 근거로 확정이다. 다만 "사용자의 실제 설치가 per-machine/보호경로였는지"는 해당 PC의 레지스트리·설치 경로 1회 확인이 필요하다. 검증 방법은 §⑤에 둔다. 이 확인은 원인 확정에 영향을 주지 않으며(B안의 방어는 원인과 무관하게 안전망), A안 적용 여부 판단에만 쓰인다.

② A · B · C 설계 결정과 트레이드오프

A) 설치 방식 교정 — 권장

업데이트가 권한 상승 없이 per-user로 진행·재실행되도록 한다. 상승이 안 일어나면 재실행은 항상 로그인 사용자 토큰 → %APPDATA% 정상.

후보효과트레이드오프
allowToChangeInstallationDirectory: false 권장. 디렉터리 변경 페이지를 없애 보호경로 설치 자체를 차단. 기본 per-user 경로(%LOCALAPPDATA%\Programs\Project-SPIP)로 고정 → 상승 불필요. 사용자가 설치 위치를 못 고름(로컬 단일사용자 도구라 영향 미미). 기존 보호경로 사용자는 별도 마이그레이션 필요(§④).
perMachine: false 유지 이미 설정됨. per-machine 강제 상승 회피의 전제. 변경 없음.
oneClick 전환(false→true) 완료 페이지 제거·자동 진행. 그러나 oneClick에서도 isForceRun 재실행 분기 존재. 비권장. UX 변화 크고 원인(상승)과 직접 무관. 회귀 위험.
코드 서명 도입 SmartScreen 완화. 상승 문제와는 무관. 범위 밖(PM이 미서명 수용).
A 결정: allowToChangeInstallationDirectory: false 1줄 변경. perMachine:false 유지. 그러면 신규/재설치 사용자는 항상 per-user 경로로 상승 없이 업데이트된다. (이미 보호경로에 깔린 기존 사용자는 이 변경만으로는 자동 교정되지 않음 — §④ 마이그레이션으로 처리.)

B) 메인 프로세스 방어 — 권장: b3 (경고 + 쓰기 차단)

A를 적용해도 (ⅰ) 이미 보호경로에 깔린 기존 사용자, (ⅱ) 사용자가 앱을 "관리자 권한으로 실행"한 경우엔 여전히 빈 목록이 보일 수 있다. 안전망으로 상승 감지(SID) → 경고 + 영속 write 차단을 둔다. 핵심 목표는 상승 세션이 관리자 프로필에 새 데이터·설정·비밀번호를 영속화해 오염시키지 않기.

옵션구현/신뢰성판단
b1 로그인 사용자의 실제 spip 경로 해석 상승 세션에서 "원래 사용자"를 신뢰성 있게 알아내려면 WTSQueryUserToken/explorer.exe 소유자 조회 등 네이티브 호출 필요. Node 내장만으로는 불가하고 새 의존성·복잡도↑. 잘못 해석 시 엉뚱한 프로필에 쓸 위험. 비채택 (신뢰성·의존성 위반)
b2 de-elevated로 재실행 Windows에서 자식 프로세스를 de-elevate하려면 explorer 토큰 도용/COM(ShellExecute via shell) 트릭 필요. Electron 단독으론 까다롭고 재실행 루프·UX 깜빡임 위험. 비채택 (구현난이도·회귀위험)
b3 경고 + 영속 write 차단 상승 감지는 가벼움(아래). 상승 세션의 디스크 write를 no-op으로 만들어 관리자 프로필 오염을 막고, 사용자 명시 재스캔은 조기 거부. "관리자 권한 실행 — 일반 실행으로 다시 여세요" 안내. 채택 — 최소·안전·의존성0
위협 모델 교정(reviewer P1). 실제 코드 확인 결과 부팅/주기 자동 스캔 트리거는 존재하지 않는다(electron/main.js onReady는 rescan을 부르지 않고, lib/server/stateWatcher.js는 메모리 오버레이·렌더러 push만으로 디스크 write 없음 → 보류 대상 아님). 따라서 위협은 "자동 스캔"이 아니라 상승 세션이 영속 write(전부 사용자 명시 동작)로 관리자 프로필에 새 0600 파일을 만드는 것"이다. b3의 목표를 "자동 스캔 보류"가 아니라 "상승 세션의 영속 write 차단"으로 교정한다.

b3 — 차단해야 할 영속 write 3경로 (전부, security M12-S-02)

실제 코드 확인 결과 상승 세션에서 관리자 프로필에 새 0600 파일을 만드는 write는 정확히 세 곳이다:

#write 함수 (실제 코드)저장물 / 민감도
1lib/scan/serializer.js writeSnapshot (scanController 경유)스캔 스냅샷 cache/projects.json
2lib/common/uiStateStore.js write (0600 원자적)UI 상태 — 즐겨찾기·정렬·테마·할일·homeLayout·별칭
3lib/common/config.js persistConfigKeys (0600 원자적)설정 — 메일 계정 비밀번호 포함(정보노출상 가장 민감)
가드 단일화(권장 구조). 시작 시 elevationGuard.detectElevation() 결과를 중앙 elevated 플래그 1곳에 둔다(출처: electron/main.js onReady에서 1회 판정 → ctx/모듈 단일 게터). 위 3개 write 함수(또는 그를 부르는 IPC mutation 핸들러 경로)는 플래그 true면 디스크 write를 no-op(메모리만 유지) 하는 단일 규칙을 따른다. 추가로 사용자 명시 재스캔은 electron/ipc/actions.js rescan() 진입부에서 controller.acquire()/start() 이전에 elevated면 { ok:false, code:'ELEVATED' } 조기 반환(명확한 UX). → 플래그 출처 1곳·가드 규칙 1개로 산재 방지.

상승 감지 방법 (Windows, 내장 + safeExec) — SID 권위·보수적 폴백:

  • 최종 권위 = SID. safeExec(whoami, ['/groups']) 출력에서 S-1-16-12288(High Mandatory Level) 포함 여부가 유일한 차단 근거. High ML이 확인된 경우에만 elevated:true.
  • 휴리스틱은 "검사 트리거"로만. 기대 %APPDATA%(path.join(os.homedir(),'AppData','Roaming','spip')) vs 실제 불일치는 whoami 검사를 유발하는 신호일 뿐, 단독 차단 금지(APPDATA 리다이렉트·로밍 프로필 false positive 방지, reviewer/security #3).
  • 보수적 폴백. whoami 실행/파싱 실패 등 판정 불가 시 비상승(정상)으로 간주 — 정상 사용자의 가용성·write를 절대 막지 않는다. (오탐으로 데이터 저장을 막는 쪽이 더 큰 피해.)
B 결정(b3): detectElevation()이 SID로 High ML을 확인한 경우에만 elevated:true. 그러면 메인이 고정 문구 경고를 렌더러에 push하고, 위 3경로 write를 no-op + rescan 조기거부한다. 데이터 위치·권한은 그대로(§C 불변).

C) 회귀 안전 (불변)

  • 일반(비상승) 실행·portable·dev 모드 동작은 완전 불변. B 감지는 "상승"일 때만 분기.
  • 데이터 이동/위치 변경 금지 — %APPDATA%\spip 유지가 정답.
  • 보안 모델 불변: 0600 권한, per-user 격리, pathGuard, safeExec(절대경로·shell:false) 유지. b3의 whoami 호출도 safeExec 규약 준수.

B — 메인 프로세스 (신규 모듈 1개 + 진입점 훅)

파일변경 내용
lib/common/elevationGuard.js (신규) 내장 모듈 + safeExec만 사용. detectElevation() — (트리거) 기대 %APPDATA% vs 실제 불일치 등으로 검사 유발 → (권위) safeExec(whoamiBin, ['/groups']) 출력에서 S-1-16-12288(High ML) 포함 시에만 elevated:true. whoami는 PATH 검색 금지 — path.join(process.env.SystemRoot || 'C:\\Windows', 'System32', 'whoami.exe') 고정 절대경로를 resolveBin(absPath)로 검증 후 shell:false 실행(H-2, security M12-S-01). 인자 ['/groups'] 고정. 판정 실패 시 비상승 폴백. { elevated:boolean, reason } 반환(경로·프로필명 등 민감값은 호출측 비노출). 판정만, 부작용 없음.
electron/main.js onReady에서 detectElevation() 1회 호출 → 중앙 elevated 플래그 설정(단일 출처). elevated===true면 렌더러로 spip:elevation:warning push(고정 토큰만). 자동 스캔 보류 로직은 없음(애초에 자동 스캔 트리거가 없으므로). 비상승이면 기존 흐름 완전 불변.
electron/ipc/actions.js rescan() 진입부(controller.acquire()/start() 이전)에서 중앙 플래그 elevated면 { ok:false, code:'ELEVATED' } 조기 반환. 락 획득·스캔 시작·write를 일절 하지 않음.
electron/preload.js onElevationWarning(cb) 단방향 수신 API 노출(기존 onScanProgress/onUpdateStatus와 동일 패턴, textContent 렌더 원칙 유지).
lib/scan/serializer.js writeSnapshot (또는 scanController 저장 직전) write 보류 #1. 중앙 elevated 플래그 true면 디스크 write를 no-op(메모리 유지). 읽기는 허용. (rescan 조기거부와 이중 안전.)
lib/common/uiStateStore.js write write 보류 #2. elevated true면 0600 디스크 write no-op(정규화 메모리 결과만 반환). 즐겨찾기·정렬·테마·할일·homeLayout이 관리자 프로필에 안 떨어짐.
lib/common/config.js persistConfigKeys write 보류 #3 (가장 민감). elevated true면 0600 디스크 write no-op. 메일 계정 비밀번호가 관리자 프로필에 새로 기록되는 것을 차단(정보노출 방지).
public/app.js / public/index.html / public/styles.css onElevationWarning 구독 → 상단 경고 배너 표시. 고정 문구 textContent만(L-1, security #4) — 실제 %APPDATA% 경로·프로필명·whoami 출력은 배너에 넣지 않음(정보노출 차단). "일반 실행으로 다시 여세요" 안내 + 재스캔 버튼 비활성/경고.
가드 단일 지점 정리: 플래그 출처는 main.js onReady 1회 판정(중앙 플래그). 차단 규칙은 ① 3개 write 함수 no-op + ② actions.js rescan() 조기거부. 산재한 if 분기 대신 출처 1·규칙 2로 고정.

autoUpdate.js는?

electron/autoUpdate.js의 quitAndInstall(false, true)는 변경하지 않는다. 상승 자체를 A안으로 없애므로 인자 변경 불필요. (대안으로 isForceRunAfter=false로 두어 자동 재실행을 끄고 사용자가 단축아이콘으로 직접 여는 방안도 있으나, UX 후퇴라 비권장.)

④ 기존 사용자 마이그레이션

allowToChangeInstallationDirectory:false는 신규 설치 경로만 바꾼다. 이미 보호경로(예: Program Files)에 per-machine으로 깔린 사용자는 새 인스톨러가 여전히 HKLM 항목을 읽어 per-machine 상승 흐름을 탈 수 있다. 대응:

  • 1순위(자동 안전망): B안(b3)이 이미 적용되어 있으므로, 마이그레이션 전이라도 빈 화면에서 데이터를 망치지 않고 안내가 뜬다. 사용자는 일반 실행으로 데이터를 정상 확인 가능 — 실질 피해 없음.
  • 2순위(클린 재설치 안내): 릴리즈 노트·설정 화면 안내로 "기존 버전 제거 후 신규 설치(기본 per-user 경로)" 1회 권장. 데이터는 %APPDATA%\spip(Roaming)에 있어 제거/재설치와 무관하게 보존된다. NSIS deleteAppDataOnUninstall 미설정이라 제거 시에도 삭제 안 됨(현행 유지).
  • 주의: 데이터 이동 스크립트는 만들지 않는다(§C: 위치 불변). Roaming 경로는 그대로이므로 재설치만으로 충분.
리스크: per-machine 기존 설치가 남아 있는 동안 자동 업데이트가 상승 흐름을 또 탈 수 있다. 이때도 b3가 오염을 막는다. 완전 해소는 사용자의 1회 클린 재설치가 필요 — 강제는 불가(로컬 단일사용자·미서명 환경).

⑤ 테스트 전략

헤드리스 가능 (node --test)

  • SID 파싱(M12-1): safeExec 모킹으로 whoami /groups 출력 주입 — S-1-16-12288 포함 시 elevated:true, Medium(S-1-16-8192)이면 false.
  • 휴리스틱은 트리거만(M12-1b): APPDATA 불일치만으로는 차단 안 됨 확인(SID Medium이면 elevated:false).
  • 보수적 폴백(M12-1c): whoami 실행 throw/출력 파싱 불가 시 elevated:false(가용성 우선).
  • write 보류 3경로(M12-2): elevated true에서 (a) serializer.writeSnapshot, (b) uiStateStore.write, (c) config.persistConfigKeys 각각 디스크 write no-op(fs 모킹으로 openSync/rename 미호출 확인). 비밀번호 키가 디스크에 안 떨어짐 검증.
  • rescan ELEVATED(M12-2b): actions.js rescan()이 elevated 시 {ok:false,code:'ELEVATED'} 반환 + controller.acquire 미호출.
  • 비상승 회귀(M12-3): elevated:false면 3경로 write·rescan 흐름 전부 기존과 동일(불변).

수동 스모크 (실 설치본)

  • 원인 확인: 증상 PC에서 HKLM\Software\<APP_GUID> InstallLocation 존재 여부 + 설치 경로가 Program Files류인지 확인(가설 §1.4 검증).
  • A 적용 빌드로 신규 설치 → 설치 경로가 %LOCALAPPDATA%\Programs인지, UAC 프롬프트가 뜨지 않는지.
  • 기존 per-user 설치 → 자동 업데이트 시나리오: 이전 버전을 per-user(기본) 경로에 깔고 → 자동 업데이트 수행 → 재실행된 앱에서 스캔 목록·UI상태·설정이 보존되는지(상승 없는 정상 경로 회귀 확인).
  • (보강) 기존 per-machine/보호경로 설치 → 자동 업데이트 → 상승 발생 시 b3 배너 + 3경로 write 보류 동작 확인.
  • B 검증: 앱을 "관리자 권한으로 실행" → 경고 배너(고정 문구) + 재스캔 ELEVATED + 3경로 write 미반영 확인. 이후 일반 실행 시 데이터 정상.
  • portable 빌드: 상승 무관·동작 불변 확인.
