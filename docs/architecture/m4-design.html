0. 개요

본 문서는 MVP(M1~M3) 완료 이후의 M4 후속 6건에 대한 설계 델타다. 기존 docs/architecture/architecture.html(§9 API·§11 보안 8중 통제·§14 추적성)와 docs/api-contract.md를 단일 출처로 삼아, 변경분만 정의한다. 코드는 작성하지 않으며, 인터페이스 시그니처·계약 표·의사코드·ADR만 제시한다. 기존 보안 8중 통제(127.0.0.1 바인딩·Host allowlist·Origin·세션토큰 X-SPIP-Token·Content-Type·realpath pathGuard·safeExec·textContent+CSP)와 외부 의존성 0 제약은 약화 없이 확장한다.

정합 원칙: M4 항목은 모두 기존 모듈의 점진 확장으로 설계한다. 새 신뢰 경계를 만들지 않고, 새 통제는 기존 8중 통제 위에 덧붙이는(additive) 형태다. 계약 변경은 전진 호환(forward-compatible) — MVP의 size.status:'skipped'·rescan 404 같은 자리표시는 그대로 실값으로 승격된다.

2. R-16 — POST /api/rescan (상태변경)

브라우저에서 재스캔을 트리거한다. MVP의 404 NOT_FOUND 자리표시(actionHandlers.rescan)를 활성 핸들러로 승격한다. 동시성은 전역 단일 in-flight 락(프로세스 1개만 백그라운드 스캔, 중복은 즉시 거부). 완료 시 snapshotStore를 무중단 갱신한다.

2.1 동시성 모델 — 단일 in-flight 락

  • 서버 프로세스 내 전역 ScanController(신규, lib/server/scanController.js)가 boolean 락 running과 진행 스냅샷 state를 보유한다.
  • /api/open의 id별 in-flight(safeExec maxInflight)와 달리, 재스캔은 전역 1개다(스캔은 캐시 전체를 재작성하므로 동시 2개는 무의미·위험).
  • 락이 잡혀 있으면 새 요청은 409 SCAN_IN_PROGRESS로 즉시 거부(대기열 없음 — 단순성).
  • 스캔은 백그라운드 비동기로 실행(요청 핸들러는 즉시 202 ACCEPTED 반환). 완료/실패는 진행 라우트(R-15)로 관측한다.
  • 락 해제 견고성(M4-M-3 — 데드락 방지): 백그라운드 start()는 try/finally로 성공·실패·예외 무관 항상 running=false 해제한다. 추가로 ScanController는 watchdog 타임아웃(config.scan.watchdogMs, 기본 절대 상한)을 두어, 그 시간을 넘기면 스캔을 강제 종료 신호 + phase="error" + 락 강제 해제한다. 즉 처리되지 않은 예외·무한 순회로 백그라운드가 죽어도 락이 영구 점유되어 이후 재스캔이 409로 영구 거부되는 가용성 데드락이 발생하지 않는다(N-05 "재스캔 중 0 다운" 정합). phase="error" 진입 시에도 락은 해제되어 idle로 복귀한다.

2.2 보안 — 기존 M-1 그대로

약화 없음 (단, 라우터 정정 필수 — P1-2). /api/rescan은 상태변경 POST이므로 라우터가 /api/open 분기(router.js:56-64)와 동형으로 security.checkStateChange(Host allowlist + Origin + X-SPIP-Token + Content-Type)를 선통과시켜야 한다(M-1). 현재 router.js:66-69는 rescan을 메서드 무관 404로 처리하며 checkStateChange를 호출하지 않으므로, §10의 router.js 정정이 선행되어야 이 보안 주장이 성립한다(이전 rev는 이 라우터 수정을 누락해 모순이었다 — 정정 완료). 실패 시 기존과 동일 403 FORBIDDEN_ORIGIN. 본문은 {}(또는 옵션) — 기존 readJsonBody 16KB 상한·JSON-only 파싱 가드 재사용. 재스캔은 새 파일을 읽지만 scanRoots/excludes/depthLimit는 모두 서버측 config에서 가져오며 요청 본문으로 경로를 받지 않는다(경로 조작 표면 0 — H-1 정합).

2.5 ScanController 인터페이스 (설계 시그니처)

// lib/server/scanController.js (신규) — 전역 단일 인스턴스
class ScanController {
  // acquire(): 락 시도. 성공 시 scanId 발급, 실패(이미 running) 시 null.
  //   [M4-L-1] scanId = crypto.randomBytes(8).toString('hex')(16자, 충분 엔트로피).
  //   scanId는 인가 토큰이 아니라 "표시·상관용 식별자"다(프론트가 자기 scanId와 대조해 타 스캔 무시).
  acquire(opts) -> { scanId } | null

  // start(opts): 백그라운드 scanner.scan 시작. scanner의 onProgress(ScanProgress)로 this.state 갱신.
  //   state에 scanId·startedAt·elapsedMs·note를 합성하고, scanner가 준 currentPath 절대경로는
  //   currentPathAbs(서버 메모리 전용)로 보관(응답엔 §3.4 shortenPath만 노출).
  //   완료: scanner.scan() 반환 → phase="finalizing" → serializer.writeSnapshot → store.load() → phase="done".
  //   [M4-M-3] 전 과정을 try/finally로 감싸 성공·실패·예외 무관 finally에서 running=false.
  //            예외 시 phase="error"(락 해제 후 idle 복귀). watchdog(watchdogMs) 초과 시 강제 error+해제.
  start({ config, withSize, allDrives, store, logger }) -> void  // 비동기, 대기 안 함(202 즉시 반환)

  // status(): 현재 진행 스냅샷(R-15 ScanProgress). 락 없을 땐 phase="idle".
  //   currentPath는 shortenPath(currentPathAbs)로 basename 축약(M4-H-1). scanId·elapsedMs 합성.
  status() -> ScanProgress

  // running: boolean (락 상태) — finally·watchdog로 영구 true 방지(M4-M-3).
}
// start() 데드락 방지 골격(의사코드 — M4-M-3)
async function start(opts) {
  const wd = setTimeout(() => { this._forceError('WATCHDOG_TIMEOUT'); }, opts.config.scan.watchdogMs);
  try {
    this.state.phase = 'scanning';
    const snap = await scanner.scan({ ...mapped, onProgress: (p) => this._merge(p) });
    this.state.phase = 'finalizing'; this.state.counts = snap.counts;
    serializer.writeSnapshot(snap, { logger: opts.logger });   // 원자 0600(M-2)
    opts.store.load();                                          // graceful·idempotent(P2-5)
    this.state.phase = 'done';
  } catch (err) {
    this.state.phase = 'error'; this.state.note = '스캔 실패';   // 내부정보 비노출(L-3)
    opts.logger.error('background scan failed', err);
  } finally {
    clearTimeout(wd);
    this.running = false;        // ★ 항상 해제 — 영구 점유(409 데드락) 방지
  }
}

(a) scan-status에도 M-1 검증 적용 (읽기 게이트)

4.1.1 collector status vs data.status — 2레이어 분리 (P2-3)

1회차 레이어 충돌 정정: 검증이 지적한 대로 두 개의 status가 충돌한다. 5.3 시스템 디렉터리 제외 — canonicalize 실경로 + 세그먼트 단위 (M4-H-2)

보안 게이트 차단 사유(높음) 해소: 이전 rev는 시스템 제외를 "경로 prefix 기반"으로만 명세해, raw 문자열 prefix 비교가 H-1 canonicalize 규칙과 어긋나는 잔여를 남겼다. all-drives 활성 시 다음으로 C:\Windows 등 제외 대상을 우회 순회할 수 있었다: (1) Windows 8.3 단축명 C:\PROGRA~1, (2) 다른 위치의 junction/심링크가 시스템 폴더를 가리킴, (3) 대소문자 c:\windows, (4) UNC/확장경로 \\?\C:\Windows·\\127.0.0.1\C$\Windows. 이는 시스템 경로 정보노출(I)·거대 시스템 트리 순회 자원고갈(D)로 이어진다.

5.4 스캔 시작 전 시간 고지 (R-05 수용기준 — P3-2)

PRD §12 R-05 수용기준 후단("시작 전 사용자에게 시간이 오래 걸릴 수 있음을 고지")을 충족하기 위해, all-drives 활성 스캔은 시작 전 고지 단계를 둔다(채널별).

채널고지 방식
CLI(scan.js)--all-drives 인자가 설정되고 config.allowAllDrives===true이면, 루트 열거 직후 스캔 시작 전 콘솔에 경고 출력(예: "전체 드라이브 스캔은 디스크 규모에 따라 수 분 이상 걸릴 수 있습니다. 진행 상황은 콘솔/대시보드에서 확인하세요."). --quiet여도 이 1줄 경고는 출력(안전 고지).

6. R-03 — walker depthLimit (config 기반)

현재 walker는 하드코딩 SAFE_MAX_DEPTH=24를 사용자 설정과 무관하게 강제한다(M-3). M4에서 config 기반 depthLimit를 도입하되, MVP 안전상한을 하한(floor)으로 유지해 약화하지 않는다.

6.1 정책

  • config.depthLimit 신설(기본 SAFE_MAX_DEPTH=24). 사용자가 더 얕게 줄이는 것은 허용(성능 목적).
  • 약화 금지: 사용자가 24보다 큰 값을 줘도 walker는 min(이론상 무한, 절대 상한)이 아니라, 절대 상한 ABS_MAX_DEPTH(예: 64)를 새로 도입해 그 위로는 못 넘게 한다. 즉 유효 깊이 = clamp(config.depthLimit, 1, ABS_MAX_DEPTH). 기존 24는 기본값으로 강등되고, 안전 의미는 ABS_MAX_DEPTH가 승계한다.
  • all-drives 연계(자원 예산 재검증 — M4-M-2): all-drives 모드에서는 (1) depthLimit 미지정 시 더 낮은 기본 12를 강제 적용하고, (2) all-drives에서는 ABS 절대 상한도 더 낮게(ABS_MAX_DEPTH_ALL_DRIVES = 24) 별도 적용한다 — 즉 all-drives 유효깊이 = clamp(depthLimit ?? 12, 1, 24). 일반 스캔의 64까지 허용하지 않아 디스크 전체 결합 시 순회 폭발을 추가 억제한다.
  • walker 레벨 전역 자원 상한(신규 — M4-M-2): size 측정에만 있던 entry/시간 상한을 walker 순회 전체에도 둔다 — config.scan.maxDirs(전역 순회 디렉터리 상한)·config.scan.timeBudgetMs 도달 시 순회를 안전 중단(phase는 done이되 note에 "자원 상한 도달로 일부만 스캔"). 깊이와 독립적으로 작동해, 얕고 넓은 거대 트리(예: 수백만 파일의 평면 디렉터리)도 N-01/N-05를 깨지 않는다. ABS_MAX_DEPTH=64 + all-drives + maxEntries 조합이 예산을 깨지 않음은 구현 게이트 벤치(V-2 연장)로 실증한다.
  • 심링크 미추적·visited 루프 차단(M-3)은 그대로 — depthLimit은 깊이만 다룬다.
// walker.js 시그니처 변경(전진 호환) — 실제 walk(roots, opts) 정합

7. N-07 — 접근성 (WCAG 2.1 AA)

9. ADR (아키텍처 결정 기록)

ADR-M4-1 — 재스캔 동시성: 전역 단일 in-flight 락 (대기열 없음)

승인(권고)

맥락: R-16 재스캔은 캐시 전체를 재작성한다. 브라우저에서 버튼 연타·다중 탭으로 중복 트리거될 수 있다. /api/open은 id별 in-flight(safeExec maxInflight)지만, 스캔은 전역 1개여야 의미가 있다.

고려한 대안: (a) 전역 단일 락 + 즉시 거부(409), (b) 락 + 대기열(다음 스캔 예약), (c) 무제한 동시 스캔.

결정: (a) 전역 단일 in-flight 락, 중복은 409 SCAN_IN_PROGRESS 즉시 거부. 백그라운드 실행 + 202 즉시 응답.

근거: (c)는 캐시 파일 동시 쓰기 경쟁·자원 2배 → 위험. (b) 대기열은 단일 사용자 로컬 도구에 과설계(재스캔 결과는 어차피 최신 1개면 충분, 중복 예약은 잉여). (a)가 가장 단순·견고하며 기존 in-flight 패턴과 개념 일관.

결과: + 단순·예측 가능, 캐시 경쟁 0. − 진행 중 재요청은 거부됨(프론트가 버튼 비활성·"진행 중" 표시로 흡수 — N-07 aria-live와 정합).

ADR-M4-2 — 진행 전달 메커니즘: 폴링(GET /api/scan-status) 채택, SSE 기각

승인(권고)

맥락: R-15 진행(폴더수·발견수·현재경로·경과·완료)을 브라우저에 전달해야 한다. 외부 의존성 0·단순성·견고성이 결정 기준.

고려한 대안: (a) 폴링 GET /api/scan-status 1초 간격, (b) SSE text/event-stream long-lived 스트림, (c) WebSocket(즉시 기각 — 내장 http로 비자명·과설계).

결정: (a) 폴링 채택.

근거: 스캔은 단발성·수초~수십초 작업이라 1초 지연은 UX 무해. 폴링은 서버를 무상태 GET 1건으로 유지 → 구현 복잡도·연결 누수·DoS 표면이 SSE보다 낮고, 연결이 끊겨도 다음 폴링에서 자동 복구(견고성 우위). SSE의 실시간성 이점은 이 작업 특성상 잉여. 둘 다 의존성 0이나 (a)가 단순성·견고성·보안 표면에서 우위.

결과: + 단순·견고·보안 표면 최소. − 최대 1초 지연·폴링 트래픽(로컬·주기≥1s라 무시 가능). 미래에 실시간성이 정말 필요하면 동일 state를 SSE로 노출하는 확장은 가능(현재는 불필요).

ADR-M4-3 — size 수집: opt-in + 예산/상한 + node_modules 지연 측정

승인(권고)

맥락: R-09 size는 node_modules(수만 파일) stat이 필요해 N-01 30초 예산을 위협(R3). node_modules는 walker 순회 제외 대상이지만 크기는 재야 한다.

고려한 대안: (a) 항상 전체 측정(정확·느림), (b) opt-in + 시간/깊이/entry 상한 + node_modules top-level 근사(기본), 전체는 옵션, (c) du/외부 명령 위임(의존성·보안 위반).

결정: (b). deps/devDeps는 package.json을 이미 읽으므로 항상 수집(저비용). 용량 측정은 opt-in, 상한 도달 시 partial.

근거: (a)는 N-01 위반·적대적 트리에서 자원 고갈(H-3 위배). (c)는 외부 명령 = ADR-001/N-04·safeExec 정책 위반. (b)는 기본 스캔 성능을 보존(opt-in)하면서, 켰을 때도 상한으로 예산을 지킨다. status 확장은 계약 전진 호환.

결과: + N-01 보존·예산 안전·계약 무파손. − 큰 트리는 근사/partial(정확도 trade-off, 명시적 라벨로 사용자 인지).

ADR-M4-4 — all-drives: opt-in 게이트 + 5중 안전 가드, 기본 차단

승인(권고)

맥락: R-05 전체 드라이브 스캔은 거대 트리(R1)·신뢰불가 입력(RC-2) 노출이 가장 크다. 외부 명령 없이 OS별 드라이브/마운트를 열거해야 한다.

고려한 대안: (a) 기본 활성(편의 우선), (b) opt-in config 게이트(기본 차단) + 시스템 제외 + BUILTIN_EXCLUDES + depthLimit + realpath 화이트리스트 + 진행/격리, (c) 미구현.

결정: (b). config.allowAllDrives 기본 false, 요청 본문만으로 켤 수 없음.

근거: (a)는 안전 기본값 위반·성능/보안 사고 위험. (c)는 R-05 요구 미충족. (b)는 위험을 명시적 opt-in으로 한정하고 기존 통제(excludeRules·pathGuard·walker M-3·N-05)를 재사용·강화 — 새 신뢰 경계 없이 최소 추가로 충족.

결과: + 안전 기본값·기존 통제 재사용. − 사용자가 명시 활성화 필요(편의 약간 ↓, 안전과 trade-off). 드라이브 열거는 OS별 분기 코드 추가.

ADR-M4-5 — depthLimit: config화하되 MVP 안전상한을 절대 상한으로 승계

승인(권고)

맥락: R-03은 깊이 설정 정교화를 요구. 그러나 walker의 하드코딩 SAFE_MAX_DEPTH=24는 무한재귀/심링크 폭발 방어(M-3)의 일부라 단순 제거는 보안 약화.

고려한 대안: (a) 하드코딩 유지(요구 미충족), (b) config.depthLimit + 별도 절대 상한 ABS_MAX_DEPTH 신설(약화 금지), (c) config로 완전 위임(상한 제거).

결정: (b). 유효 깊이 = clamp(config.depthLimit ?? 24, 1, ABS_MAX_DEPTH=64).

근거: (c)는 사용자가 무한 깊이 지정 가능 → M-3 약화·자원 고갈. (b)는 사용자 유연성(얕게/깊게 조정)을 주면서 절대 상한으로 안전 의미를 승계 — "MVP 안전상한은 하한으로 유지"라는 요구를 충족.

결과: + 유연성 + 안전 불변. − 절대 상한(64)은 여전히 하드코딩(의도된 안전 floor/ceiling).

보강(M4-M-2): 절대 상한이 24→64로 완화되는 것이 all-drives와 결합 시 자원 위험을 키운다는 지적을 반영해, all-drives 모드에서는 별도 상한 ABS_MAX_DEPTH_ALL_DRIVES=24 + 기본 깊이 12를 강제하고(§6.1), 깊이와 독립적인 walker 전역 entry/시간 상한(config.scan.maxDirs·timeBudgetMs)을 추가했다. 즉 일반 스캔은 64까지, all-drives는 24까지로 제한해 MVP 안전상한(24)을 all-drives에서 사실상 유지한다.
