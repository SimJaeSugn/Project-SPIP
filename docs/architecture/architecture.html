1. 개요

Project-SPIP는 파이프라인-앤-대시보드 구조의 로컬 도구다. 한 번의 CLI 스캔이 디스크를 순회해 프로젝트를 탐지하고 인사이트(언어/스택·신선도·git·규모)를 수집한 뒤 결과를 OS 캐시 폴더의 JSON 스냅샷으로 영속화한다. 별도로 실행되는 로컬 웹서버는 그 스냅샷을 읽어 정적 대시보드에 데이터 API로 제공하고, 'VS Code로 열기' 같은 액션을 화이트리스트 검증 후 실행한다. 스캔(쓰기)과 서빙(읽기)을 시간·프로세스로 분리한 것이 핵심 설계 결정이며, 이는 단순성·견고성·성능 요구를 동시에 만족한다.

설계 원칙 적용: 외부 npm 의존성 0개(Node 내장 모듈만), 빌드 도구 없음, 프레임워크 없음. 요구를 충족하는 가장 단순한 구조를 택했고, 분산화·마이크로서비스 등 과설계를 배제했다(N-04 / ADR-001).
핵심 보안 전제(PM 확정 RC-2 = 인정): 스캔 대상 폴더에는 신뢰할 수 없는 외부 클론 리포가 섞일 수 있다고 전제한다. 따라서 스캔 대상 파일시스템(package.json·.git·폴더명·심링크)은 부분신뢰(TB-B) 입력으로 취급하며, 실경로(realpath) 정규화·실행 파일 절대경로 고정·적대적 입력 파싱 방어·대시보드 출력 이스케이프를 필수 방어로 설계에 반영했다(§11 / 보안검증 H-1·H-2·H-3·L-1). "로컬이라 안전"이라는 가정은 명시적으로 배제한다.

4. 시스템 아키텍처 (상위 수준)

시스템은 5개 구성 요소(① CLI 스캐너, ② 데이터/설정 저장소, ③ 로컬 웹서버+API, ④ 브라우저 대시보드, ⑤ 외부 CLI 도구)로 이뤄진다. 데이터는 단방향 파이프라인으로 흐른다: 스캔 → 캐시 JSON → 서버 읽기 → 브라우저 렌더. 액션('열기'·재스캔)만 브라우저 → 서버 → OS 방향의 역흐름을 가진다.

실행 흐름 요약

  1. 스캔 페이즈(spip / npm run scan): 설정 로드 → 루트 순회(제외 규칙) → 프로젝트 탐지 → 인사이트 수집(언어·신선도·git·규모) → projects.json을 OS 캐시에 원자적 쓰기 → 콘솔 요약. R-01·02·04·06·07·08·14
  2. 서빙 페이즈(npm run start): projects.json 읽어 메모리 적재 → 127.0.0.1 고정 포트 바인딩 → 콘솔에 URL 출력(자동 오픈 없음). R-10 N-02·N-06
  3. 표시 페이즈(브라우저): 페이지 로드 → GET /api/projects·/api/stats → 카드/통계 렌더 → 클라이언트 필터/정렬/검색. R-10·R-11
  4. 액션 페이즈: '열기' 클릭 → POST /api/open(Host allowlist+세션토큰) → realpath 화이트리스트 검증 → spawn(absoluteCodeBin,[path],{shell:false}) → 피드백. R-12·R-13 N-03 · H-1·H-2·M-1

5. 소프트웨어 아키텍처 (추상화)

두 실행 진입점(scan.js, server.js)이 공용 라이브러리(lib/)를 공유한다. 의존성은 진입점 → 도메인 모듈 → 공용 유틸 한 방향으로만 흐르며, 공용 유틸은 상위를 참조하지 않는다(순환 의존 금지). 인사이트 수집기는 공통 인터페이스(계약)를 따르는 교체 가능한 모듈로 격리해, 새 수집기 추가나 graceful 실패가 다른 수집기에 영향을 주지 않게 한다.

핵심 추상화 (계약, 설계 수준 시그니처)

아래는 모듈 간 계약을 보여주는 설계 표현이며 구현 코드가 아니다. 실제 시그니처·세부는 구현팀이 확정한다.

인사이트 수집기 계약 (collectors/* — 보안검증 H-3 반영)

// 모든 수집기는 동일한 형태를 따른다(교체 가능·격리). 실패는 throw가 아니라 결과에 담는다.
// 스캔 대상은 부분신뢰(TB-B)이므로 graceful 격리만으로는 부족 — 적대적 입력 방어를 계약에 포함한다.
Collector = {
  name: string,                       // 'language' | 'freshness' | 'git' | 'size'
  mvp: boolean,                       // size(R-09)는 false → M4
  limits: {                           // H-3: 적대적 입력 가드(수집기별 상한, 공통 기본값 제공)
    maxFileBytes,                     // 읽기 전 fs.stat로 크기 검사. 초과 시 읽지 않고 status:'na'+warning (예: package.json 1MB)
    maxReadBytes,                     // 스트림/부분 읽기 바이트 한도(파일 핸들 가드)
    maxJsonDepth,                     // JSON.parse 후 깊이 가드(과중첩 거부) — 깊이 초과 시 skip
    maxStringField                    // name/description/branch 등 문자열 필드 길이 절단
  },
  collect(projectPath, context) -> {  // context: { config, signals, logger, limits }
    ok: boolean,                      // 수집 성공 여부
    data: object | null,              // 성공 시 인사이트 필드(문자열은 길이절단·제어문자 제거 후)
    status: 'ok' | 'na' | 'error',    // 'na' = 환경 부재 또는 입력 한도 초과(graceful), R-13/H-3
    note: string | null               // 사용자 안내용 메모
  }
}
H-3 부분신뢰 입력 파싱 규칙(필수): ① 파싱 전 fs.stat로 파일 크기 상한 검사 → 초과 시 읽지 않고 skip+warning(거대 package.json 메모리 고갈 차단). ② JSON.parse 입력은 바이트 한도·깊이 가드를 거친다(JSON 폭탄·과중첩 차단). ③ 언어 추정 등 모든 정규식은 선형시간(ReDoS-free) 패턴만 사용하거나 입력 길이를 먼저 제한한다(역추적 폭발 금지). ④ 수집한 문자열 필드(name·description·branch)는 길이 절단 + 제어문자 제거 후 스냅샷에 담는다(저장형 XSS 표면 1차 축소, L-1 보완). 단일 항목 try 격리는 무한 루프/메모리 폭발을 막지 못하므로, 위 가드를 격리보다 앞단에 둔다.

안전 프로세스 실행 (safeExec — 보안 핵심, 보안검증 H-2·M-4 반영)

resolveBin(name) -> absolutePath | null  // 기동/스캔 시작 시 1회만 호출해 결과를 고정(캐시)
  // 'code'/'git'을 신뢰 PATH 기준(where/which 또는 알려진 설치 경로)으로 절대경로 1회 해석
  // 이후 절대경로로만 spawn → 이름만 고정(name)이 아니라 경로를 고정(H-2)

safeExec(absoluteBin, argsArray, opts) -> { code, stdout, stderr, timedOut }
  // 항상 spawn(absoluteBin, argsArray, { shell:false })  — 사용자/스캔 입력을 셸 문자열로 연결 안 함
  // opts.timeoutMs: 기본 타임아웃 + 초과 시 강제 kill(M-4)
  // opts.maxBuffer: stdout/stderr 출력 버퍼 상한(M-4)
  // Windows: .bat/.cmd 자동 확장(PATHEXT) 실행 금지 — 절대경로의 .exe만 실행(H-2)
  // git collector는 cwd:projectPath로 돌더라도 실행 파일은 절대경로 → cwd 내 위장 git.exe/.bat 선택 차단
H-2 PATH 하이재킹 차단(필수): code·git은 이름만 고정하지 않고 기동/스캔 시작 시 resolveBin()으로 절대경로를 1회 해석해 고정한다. git collector가 cwd:projectPath(부분신뢰 폴더)에서 실행되더라도 실행 파일을 절대경로로 지정해, 스캔 대상 폴더 내에 심어진 위장 git.exe/git.bat가 선택돼 RCE로 이어지는 경로를 막는다. 가능하면 git 호출도 -C <projectPath> 인자 사용으로 cwd 의존을 제거한다.

  //   · 방문 실경로(realpath) 집합 추적 → 재방문 가지치기(심링크 루프 a→b→a 차단)

경로 가드 (pathGuard — 보안 핵심, 보안검증 H-1 반영)

canonicalize(rawPath) -> string        // 정규화 1차 함수 — 등록·검증 양쪽이 동일 규칙 사용
  // 1) path.resolve로 '../' 접기  2) fs.realpathSync로 심링크/Windows 접합점(junction) 실경로 해소
  // 3) 대소문자 비민감 FS(Windows/macOS)는 비교용 폴드(fold) 적용
  // 4) 유니코드 NFC 정규화로 NFC/NFD 차이 흡수
  // 5) UNC(\\\\server\\share)·드라이브 상대경로(C:foo)·확장경로(\\\\?\\)는 정규화 또는 거부

isAllowed(requestedPath, whitelist) -> boolean
  // whitelist = 스냅샷 등록 시점에 canonicalize()로 만든 "실경로(realpath)" 집합
  // 요청 경로도 canonicalize() 후 실경로끼리 정확 일치만 허용 (등록·검증 동일 규칙 = TOCTOU 축소)
  // '../' 순회·심볼릭/접합점 우회·접두사 부분일치 모두 차단 (N-03, H-1)
  // 실경로 해소 실패(경로 소멸 등)는 false 반환 → 호출부가 PATH_GONE/403 처리
H-1 핵심 규칙: 화이트리스트는 walker가 등록하는 시점과 pathGuard가 검증하는 시점 모두 동일한 canonicalize()(= fs.realpath 기반)로 만든 실경로를 기준으로 구성·대조한다. path.resolve만으로는 심링크/접합점이 해소되지 않아 스캔 루트 밖을 가리키는 경로가 "정확 일치"로 통과할 수 있으므로, 실경로 정규화를 단일 진실의 원천으로 강제한다.

11.1 경로 화이트리스트 — realpath 기준 (N-03 / 보안 H-1)

  • 화이트리스트 원천: 스냅샷 projects[].path를 canonicalize()(= path.resolve + fs.realpath)로 해소한 실경로 집합만 신뢰. 그 밖의 경로는 어떤 API도 접근/실행 안 함.
  • 등록·검증 동일 규칙(H-1 핵심): walker가 화이트리스트를 등록하는 시점과 pathGuard가 검증하는 시점 모두 같은 canonicalize()로 실경로를 만들어 대조한다. path.resolve만으로는 심볼릭 링크/Windows 접합점(junction)이 해소되지 않아, 등록 경로 자체가 심링크이거나 하위에 외부를 가리키는 링크가 있으면 "정확 일치"로도 스캔 루트 밖이 열린다. 실경로 정규화를 단일 진실의 원천으로 강제해 이를 차단한다(TOCTOU 표면도 축소).
  • 정규화 규칙(필수 포함): ① 대소문자 비민감 FS(Windows/macOS)는 비교용 폴드 적용 ② 유니코드 NFC 정규화(NFC/NFD 차이 흡수) ③ UNC(\\server\share)·드라이브 상대경로(C:foo)·확장경로(\\?\)는 정규화 또는 거부.
  • 검증 방식: 요청은 id로 받고 서버가 스냅샷에서 역참조한다(클라이언트 경로 조작 표면 제거). 실경로 해소 실패(경로 소멸)는 거부 → PATH_GONE/403.
  • 정적 서빙: staticHandler는 P1-1 규약(/static/<asset> → public/<asset>)으로 매핑 후 public/ 절대경로 하위로 정규화, 이탈 시 404. 프로젝트 폴더는 정적 서빙 대상이 절대 아니다.

11.2 외부 프로세스 인젝션 / PATH 하이재킹 방지 (N-03 / 보안 H-2·M-4)

  • 모든 외부 호출(code, git)은 common/safeExec를 통해 spawn(absoluteBin, argsArray, {shell:false})로만 실행. 경로·브랜치명 등 사용자/스캔 유래 입력을 셸 문자열로 연결하지 않는다.
  • 인자는 항상 배열 요소로 전달(예: ['/path with space']) — 공백·한글·특수문자 안전(F-2 엣지).
  • 실행 파일 절대경로 고정(H-2 핵심): 실행 파일명을 고정하는 것만으로는 부족하다. 기동/스캔 시작 시 resolveBin('code'|'git')으로 신뢰 PATH 기준 절대경로를 1회 해석해 고정하고, 이후 그 절대경로로만 spawn한다. git collector가 cwd:projectPath(부분신뢰 폴더)에서 돌더라도 절대경로 지정으로 폴더 내 위장 git.exe/git.bat가 선택되지 않는다. 가능하면 git -C <path>로 cwd 의존 자체를 제거한다.
  • Windows 특이사항: .bat/.cmd의 PATHEXT 자동 확장 실행을 막고 절대경로 .exe만 실행한다.
  • 자원 통제(M-4): safeExec에 기본 타임아웃+초과 시 강제 kill, 출력 버퍼 상한을 둔다. /api/open은 id별 in-flight 상한으로 중복 spawn을 차단한다(클라 디바운스는 서버 보호가 아님).

11.3 로컬 API 노출 / CSRF / DNS 리바인딩 (N-02 / 보안 M-1)

  • 서버는 server.listen(port, '127.0.0.1')로 루프백 전용 바인딩. 0.0.0.0 금지.
  • Host allowlist(M-1 핵심): 상태 변경 POST(/api/open)는 Host 헤더를 기대 127.0.0.1:<port>/localhost:<port> allowlist와 정확 비교한다 — DNS 리바인딩(공격 도메인을 127.0.0.1로 재바인딩)으로 Origin만으로는 막히지 않는 우회를 차단.
  • Origin 폐쇄 기본값: Origin이 없거나 불일치하면 거부(FORBIDDEN_ORIGIN/403). Content-Type: application/json 요구는 보조 수단으로 유지.
  • 세션 토큰(M-1 강화): 서버 기동 시 1회 생성한 세션 토큰을 index.html에 주입하고, 상태변경 POST에 동봉·검증한다(동일머신 타 페이지/프로세스의 액션 트리거 표면 축소).
  • 인증은 단일 사용자 원칙상 두지 않는다(§9.1 Won't). 대신 위 바인딩+Host allowlist+토큰으로 노출 표면을 한정한다.

11.4 민감 데이터 보호 / 파일 권한 (N-06 / 보안 M-2)

  • 스캔 결과(경로·git 상태 등)는 OS 앱 데이터 폴더(config/·cache/)에만 저장, 외부 전송 0(N-02). 프로젝트 폴더 비오염.
  • 파일 권한(M-2): serializer/paths는 캐시·설정·임시 파일을 소유자 전용 권한(POSIX 0600, 디렉터리 0700; Windows는 사용자 ACL)으로 생성한다. 원자적 쓰기의 임시 파일도 동일 권한·동일 디렉터리(rename 원자성)·실패 시 정리. RC-1(멀티유저 격리)이 MVP 비목표여도 본 권한은 적용한다.
  • 스냅샷에는 파일 내용을 담지 않는다 — 메타데이터(경로·언어·시각·git 요약)만 보관해 노출 시 영향을 최소화.

11.5 부분신뢰 입력 파싱 방어 (TB-B / 보안 H-3·M-3 · RC-2 인정)

  • 전제: 스캔 대상(package.json·.git·폴더명·심링크)은 신뢰 경계 밖(TB-B)이며 악성 입력이 섞일 수 있다(RC-2 인정). graceful 격리(R-13/N-05)는 환경 부재·손상에 대응하지만 적대적 입력에는 부족하므로, 다음 가드를 격리보다 앞단에 둔다.
  • 크기·깊이·바이트 가드(H-3): 파싱 전 fs.stat로 파일 크기 상한(예: package.json 1MB) 검사 후 초과 시 읽지 않고 skip+warning. JSON.parse 입력은 바이트 한도·깊이 가드를 거친다(JSON 폭탄·과중첩 차단).
  • ReDoS 방어(H-3): 언어 추정·확장자 비율 등 모든 정규식은 선형시간(역추적 폭발 없는) 패턴만 사용하거나 입력 길이를 먼저 제한.
  • 문자열 정제(H-3·L-1): 수집한 name·description·branch는 길이 절단 + 제어문자 제거 후 스냅샷에 담는다.
  • 심링크 루프/깊이(M-3): walker는 MVP에서 내장 안전 깊이 상한(고정값)을 항상 강제하고, 방문한 실경로(realpath) 집합을 추적해 재방문을 가지치기한다. 심링크는 기본 미추적(따라갈 경우 실경로가 루트 실경로 하위인지 재확인). 사용자 설정 depthLimit·심링크 정책 정교화는 M4(R-03).

11.6 대시보드 출력 이스케이프 / CSP (보안 L-1 — 저장형 XSS 차단, 필수)

  • 저장형 XSS 표면: 악성 리포의 package.json name/description, git 브랜치명에 <img onerror> 등 페이로드가 들어가면 스캔→스냅샷→/api/projects→DOM 경로로 저장형 XSS가 가능하다(RC-2 인정). 이를 필수 방어로 확정한다.
  • 이스케이프(필수): app.js — view의 모든 동적 텍스트는 textContent/명시적 이스케이프만 사용하고 innerHTML 직접 결합을 금지한다(§6.3 계약). §11.5의 문자열 정제(제어문자 제거)와 이중 방어.
  • CSP(필수): index.html에 CSP 메타를 두어 인라인/외부 스크립트를 제한한다(예: default-src 'self', 인라인 스크립트 금지). 자산은 P1-1 규약대로 동일 출처 /static/*에서만 로드.

11.7 수용된 리스크 (PM 위험 수용 확정)

RC-1 (멀티유저 OS 동일머신 타 사용자 격리) = MVP 비목표: 제품 가정은 "로컬 단일 사용자 PC"다. 127.0.0.1 루프백은 동일 머신의 다른 로컬 사용자/프로세스에게 도달 가능하나(루프백은 사용자 격리를 하지 않음), 동일 머신 타 사용자로부터의 격리는 MVP 비목표로 수용한다(공용 PC·터미널 서버는 지원 범위 밖). 단, 저비용 심층방어는 적용한다 — 캐시/설정 파일 권한(M-2, 0600/ACL)은 MVP에 적용하고, 세션 토큰(M-1)도 함께 둔다. 이 환경을 지원 범위에 포함하려면 인증 부재(§9.1 Won't)를 재검토해야 하며 이는 향후 결정 사항이다.
RC-2 (스캔 대상 부분신뢰) = 인정: 스캔 대상에 신뢰할 수 없는 외부 클론 리포가 섞일 수 있음을 명시 인정한다. 이에 따라 보안 H-2·H-3·L-1은 단순 권고가 아니라 필수 방어로 확정되어 §11.2·11.5·11.6 및 §5 계약에 반영됐다.

13. 배포 / 실행 구성

배포 토폴로지는 단일 PC 로컬 프로세스 2종이다. 서버 인프라·컨테이너·CI 파이프라인이 없으므로 "설치=npm link 또는 전역 설치, 실행=두 스크립트"로 끝난다.

단계명령동작
설치/링크npm link 또는 전역 설치bin spip → scan.js 등록
스캔(쓰기)spip / npm run scan루트 순회·수집 → OS 캐시 projects.json 생성 → 콘솔 요약
서빙(읽기)npm run start127.0.0.1:<포트> 바인딩 → 콘솔에 URL 출력(자동 오픈 없음)
개발 일괄npm run devscan.js && server.js 순차 — 스캔 후 바로 서빙
실행 순서 의존: 서버는 스냅샷이 없어도 graceful 기동한다(R-13). 즉 start 전 scan 미실행이어도 크래시하지 않고, 데이터 조회는 200 + hasSnapshot:false + 빈 배열로 응답하며(P2-5 통일, 503 폐기) 프런트가 "먼저 스캔하세요" 빈 상태를 표시한다.

15. 리스크 및 가정 / 미해결 질문

가정

  • Node 16+ 설치·본인 권한 실행. code/git CLI는 있을 수도 없을 수도 있음(없으면 graceful, R-13).
  • 고정 포트 기본값은 일반적으로 비어 있음. 점유 시 명확한 에러를 출력하고 설정으로 변경 가능(PRD §10).
  • 스냅샷은 단일 PC 기준의 시점 데이터다. 실시간 watch·자동 재스캔은 범위 밖(§9.2 Could/M4).
  • 스캔 대상은 부분신뢰(RC-2 인정) — 악성 클론 리포가 섞일 수 있다고 가정하고 H-2·H-3·L-1 필수 방어를 적용(§11.5·11.6).
  • 동일머신 타 사용자 격리는 MVP 비목표(RC-1 수용) — 단일 사용자 PC 가정. 단 파일 권한(M-2)·세션 토큰(M-1)은 적용(§11.7).

리스크 및 완화

RA1 — 스캔 성능(N-01). 깊은 트리·다수 git 호출로 30초 초과 위험. 완화: 제외 가지치기 우선, git 호출은 프로젝트당 1회로 묶기, 용량(R-09)은 M4 분리, test/fixtures로 회귀 벤치.
RA2 — git 수집 비용. 프로젝트마다 git CLI spawn은 누적 비용이 큼. 완화: 저장소 폴더만 대상, 단일 명령으로 필요한 상태를 한 번에 취득, 타임아웃·실패 격리.
RA3 — 경로 보안(N-03). 화이트리스트 검증 누락 시 임의 파일 노출/실행. 완화: id 기반 역참조 + realpath 정규화 정확 일치(H-1), staticHandler public 격리, security-reviewer 검토 반영.
RA6 — 부분신뢰 입력으로 인한 RCE/DoS(H-2·H-3). 악성 클론 리포의 위장 바이너리·JSON 폭탄·ReDoS·심링크 루프. 완화: 실행파일 절대경로 고정+cwd 의존제거(H-2), 크기/깊이/바이트 가드+ReDoS-free 정규식(H-3), 방문 실경로 추적 루프차단(M-3), safeExec 타임아웃(M-4).
RA7 — 저장형 XSS(L-1). 오염된 메타데이터가 대시보드 DOM에 삽입. 완화: view 전면 textContent/이스케이프·innerHTML 결합 금지, CSP 메타, 문자열 정제(§11.6).
RA4 — 포트 점유. 고정 포트가 이미 사용 중일 수 있음. 완화: EADDRINUSE를 잡아 안내, config.port로 변경 지원.
RA5 — 스냅샷-실제 불일치. 스캔 후 폴더 삭제·이동 시 '열기' 실패. 완화: 액션 시 경로 존재 재확인 → "재스캔 권장" 안내(F-2).

미해결 질문 (설계 블로커 아님 · 구현팀 확정 사항)

  • 고정 기본 포트 번호 구체값(설계는 예시 7421 사용, 충돌 가능성 낮은 값으로 구현팀 확정).
  • 언어/스택 추정의 구체 우선순위(package.json deps vs 확장자 비율 가중치) — 탐지율 ≥95% 벤치로 튜닝.
  • [해소됨 — P1-2] 설정 파일 위치는 단일 OS 앱 데이터 디렉터리 하위(%APPDATA%\spip\config/, 캐시는 cache/)로 확정. paths.js의 appDir()/configPath()/cachePath() 계약 결정(§8.2). 본 항목은 미해결에서 내림.
