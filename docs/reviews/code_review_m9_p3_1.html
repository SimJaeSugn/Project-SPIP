수식키 판정 — F5만/Ctrl+O/Ctrl+R, 조합 충돌 회피 정상 app.js:579-587 — F5는 !ctrl && !shiftKey && !altKey로 무수식만 매칭, Ctrl+O/R은 ctrl=(ctrlKey||metaKey)로 Cmd 동등. 수식키 없는 o/r은 null(텍스트 입력 충돌 차단). 테스트가 F5+수식·무수식 o/r·미지정 키까지 망라(shortcuts.test.js:44-72).
ESC 충돌·중복 등록 정상 두 개의 document.keydown 리스너가 책임을 분리한다 — ESC 핸들러(app.js:5480)는 e.key!=='Escape'면 즉시 return, 단축키 핸들러(:5491)는 matchShortcut이 ESC를 의도적으로 null 반환(:586)하므로 같은 이벤트를 두 핸들러가 모두 소비하는 경로가 없다. 모달·드로어 내부의 지역 ESC/Enter 핸들러들도 자체 stopPropagation 또는 별도 엘리먼트 스코프라 전역과 분리.
F5 브라우저 기본 reload 차단 정상 matchShortcut이 action을 줄 때만 e.preventDefault()(:5493-5494). F5→'refresh'이므로 Electron 렌더러의 F5 전체 reload 기본동작이 차단되고 앱 내 refreshDashboard()만 호출 → 상태 소실 없음. 보안 리뷰 권고(m9_design_security_review_1 R-28)와 일치.
refreshDashboard 안전성 정상 app.js:4560-4563 — 스캔 중(view==='scanning')엔 push 갱신에 양보하고 조기 return, 그 외엔 reloadAfterScan() 재사용. 새 IPC·부수효과 없음.
'about' 죽은 매핑의 정확성 영향 없음 dispatchMenuAction의 case 'about'(:603)은 호출되지 않는다 — 유일 발신원이던 onMenu가 제거됐고, keydown 핸들러는 dispatchMenuAction을 거치지 않고 matchShortcut이 내는 3개 토큰만 처리(:5495-5500)한다. 'about'은 도달 불가 → 정확성 무영향. (단, 죽은 코드로서 §2-A 개선 대상)
