M12-S-01 whoami 절대경로 해석 — 중간

위협(STRIDE): 위장(Spoofing)·권한상승. 악용 시나리오: elevationGuard가 상승 판정에 whoami /groups를 실행할 때, 실행 파일을 PATH 검색으로 찾으면(resolveBin('whoami')) 공격자가 부분신뢰 cwd나 사용자 쓰기 가능한 PATH 디렉터리에 whoami.exe를 심어 위장 바이너리가 선택될 수 있다. 또한 %SystemRoot% 환경변수로 경로를 조립하면 환경변수 변조 표면이 생긴다.

현행 safeExec 분석(증거): lib/common/safeExec.js의 resolveBin은 cwd(.)·빈 항목을 PATH 탐색에서 제외하고 .exe만 허용하며, safeExec는 절대경로·shell:false·문자열 인자배열을 강제한다 — H-2 불변식은 이미 견고하다. 문제는 whoami가 신뢰 경로에서 해석되도록 강제하는지다. resolveBin('whoami')는 사용자 PATH 전체를 순회하므로, PATH 앞쪽에 비신뢰 디렉터리가 있으면 위장 바이너리를 집을 여지가 이론상 존재한다 (System32보다 앞선 사용자 PATH 항목).

권고 조치(구현 단계):

  • whoami를 PATH 검색이 아니라 고정 절대경로로 직접 검증한다. path.join(process.env.SystemRoot || 'C:\\Windows', 'System32', 'whoami.exe')를 만들되, resolveBin(absPath)(절대경로 분기 — 확장자·존재 검증만 수행)를 거쳐 safeExec에 넘긴다. 이러면 PATH 순회 자체가 일어나지 않아 PATH 하이재킹 표면이 0이 된다.
  • process.env.SystemRoot를 신뢰하기 어려우면 그대로 C:\Windows 폴백을 두되, 파일이 없으면 판정 실패가 아니라 보수적 분기(M12-S-03 참조)로 처리한다.
  • 인자는 ['/groups'] 고정 배열 — 사용자/스캔 입력을 절대 인자로 넣지 않는다(이미 설계 의도와 일치).
  • 호출은 설계대로 기동 시 1회·timeout 고정으로 제한(safeExec 기본 5s·maxBuffer 1MB가 자동 적용).

판정: 설계 문구("절대경로 C:\Windows\System32\whoami.exe·shell:false")는 올바른 방향이다. 다만 "resolveBin('whoami') PATH 검색"이 아니라 고정 절대경로 검증"임을 구현 계약으로 못박아야 중간 등급에서 안전으로 내려간다. 1차 휴리스틱(APPDATA 비교)만으로도 대부분 커버되어 whoami는 보강용이라 실위험은 낮으나, 호출 시 표면을 0으로 닫는 것이 정석.

M12-S-02 write 보류 범위 — 중간

위협(STRIDE): 변조·정보노출. 악용/오작동 시나리오: 설계 §3의 변경 지점은 write 보류를 scanController.js의 스냅샷 저장(serializer.writeSnapshot) 한 곳에 둔다. 그러나 상승 세션에서 사용자가 즐겨찾기·정렬·테마·할 일을 만지거나 설정을 바꾸면 uiStateStore.js·config.js의 persistConfigKeys가 관리자 프로필에 새 0600 파일을 생성한다. 이는 설계 목표("관리자 프로필 오염 방지")의 부분 누수다.

증거: lib/common/uiStateStore.js(0600 원자적 쓰기), lib/common/config.js persistConfigKeys(0600 원자적 쓰기)는 scanController와 무관하게 IPC 핸들러에서 직접 디스크에 쓴다. 상승 플래그가 scanController 저장 가드에만 걸리면 이 경로들은 보류 대상에서 빠진다. (참고: stateWatcher.js는 메모리·renderer push만 하고 디스크 write를 하지 않으므로 보류 대상 아님 — 양호.)

권고 조치:

  • 상승 플래그를 중앙 1지점(예: ctx 또는 elevationGuard 모듈)에서 보관하고, 모든 영속 write 진입부가 이를 조회하도록 한다: ① serializer.writeSnapshot 호출부, ② uiStateStore write, ③ config.persistConfigKeys(메일 계정 비밀번호 포함 — 정보노출 측면에서 가장 민감).
  • 대안(더 단순·안전): 상승 세션이면 UI를 읽기 전용으로 비활성화(재스캔·즐겨찾기·설정 저장 버튼 disable) + 배너 안내. 그러면 write 진입 자체가 줄어 가드 누락 리스크가 감소한다.
  • 구현 후 테스트: 상승 플래그 set 상태에서 스냅샷·UI상태·설정 write가 모두 스킵되는지 단위 검증 (설계 §5의 M12-2를 세 경로로 확장).

판정: "기존 데이터 무손상"은 충족된다 — write를 보류하면 디스크를 건드리지 않으므로 원래 사용자 프로필의 데이터는 그대로 보존된다(읽기 허용·위치 불변). 단 오염 방지의 완전성을 위해 가드 범위를 세 write 경로로 넓혀야 한다.

M12-S-03 감지 false negative — 보수적 기본값 — 낮음

위협: 서비스거부(데이터 가용성). 시나리오: 1차 휴리스틱(process.env.APPDATA vs os.homedir() 기대값)은 환경에 따라 빗나갈 수 있다(로밍 프로필·UAC 가상화·도메인 계정 등). whoami 호출이 실패(파일 부재·timeout·파싱 실패)하면 감지 로직이 무엇을 반환하느냐가 중요하다.

권고 조치: 감지의 안전 기본값을 명시한다. 핵심 원칙은 "감지 실패가 보안 약화로 이어지지 않게"다. 다만 M12에서 "상승 의심 시 write 보류"는 가용성(빈 화면)을 희생하므로, false positive(정상인데 상승 판정)는 데이터를 안 망치되 사용성만 떨어뜨리고, false negative(상승인데 정상 판정)는 오염을 허용한다. 따라서:

  • 두 신호(APPDATA 불일치 OR whoami High ML) 중 하나라도 상승을 가리키면 보류(보수적 OR 결합).
  • whoami 호출 자체 실패 시 — APPDATA 휴리스틱 결과를 단독 채택(추가 위험 없음). 둘 다 불확실하면 비상승(정상 흐름)으로 폴백해 정상 사용자의 가용성을 보장한다(false negative보다 정상 사용자 피해가 더 흔하므로). 이 트레이드오프를 설계 문서에 1줄로 명시 권장.
  • 감지는 설계대로 부작용 0·판정만 유지 — 어떤 새 권한·신뢰 경계도 만들지 않는다(설계 충족, 양호).

M12-S-04 경고 배너 L-1 · preload 단방향 — 낮음

위협: 변조(XSS). 시나리오: 경고 배너가 동적 문자열(프로필명·실제 APPDATA 경로 등 시스템 문자열)을 innerHTML로 렌더하면 L-1 위반·정보노출이 된다.

증거(설계는 올바른 방향): 설계 §3·§6은 "고정 문구 textContent만"을 명시하고, preload는 기존 onScanProgress/onUpdateStatus와 동일한 단방향 on() 중계 패턴(콜백만, ipcRenderer 원본 미노출)을 따른다 — electron/preload.js 기존 패턴과 일치해 안전하다.

권고 조치(구현 게이트):

  • 배너는 하드코딩 고정 문구("관리자 권한으로 실행됨 — 일반 실행으로 다시 여세요")만 표시. 실제 APPDATA 경로·프로필명·whoami 출력을 화면에 절대 넣지 않는다(정보노출 차단 + L-1 단순화). 디버깅 정보가 필요하면 main 로그에만(logger는 clampString 사용).
  • preload onElevationWarning(cb)은 payload를 받더라도 신호용(boolean/없음)으로 최소화. 만약 reason을 전달하면 renderer는 그것을 고정 메시지 매핑으로만 쓰고 직접 렌더하지 않는다.
  • 채널명 하드코딩(spip:elevation:warning) — preload allowlist 원칙 준수.

M12-S-05 per-user 격리·위치 불변 — 정보(긍정)

allowToChangeInstallationDirectory: false + perMachine: false는 설치를 per-user 경로(%LOCALAPPDATA%\Programs)로 고정해 UAC 상승 유발 자체를 제거한다. 보안상 명백히 긍정이다:

  • 상승 흐름이 사라지면 관리자 토큰으로 앱이 뜰 표면이 줄어 공격·오염 표면 감소.
  • 데이터 위치(%APPDATA%\spip)·권한(0600/0700)·per-user 격리가 전부 불변(설계 §C, 증거: paths.js·serializer.js 미변경). 데이터 이동 스크립트를 만들지 않는 결정도 옳다(이동은 권한·경합 리스크를 새로 만든다).
  • 설치 위치 선택 불가의 부정적 보안 영향 없음 — 로컬 단일사용자 도구라 기능 손실 미미.

유의: 이미 보호경로에 per-machine으로 설치된 기존 사용자는 A안만으로 자동 교정되지 않으므로 b3 안전망이 반드시 함께 적용되어야 한다(설계 §4가 이미 인지·반영).

M12-S-06 whoami 출력 파싱 = 신뢰 경계 — 정보

whoami stdout은 OS가 생성하지만 파싱 코드는 방어적이어야 한다. S-1-16-12288(High Mandatory Level) 부분 문자열 포함 여부 검사는 단순·안전하다. 권고:

  • stdout 부재·빈 문자열·timeout(timedOut:true)·non-zero exit는 "상승 미확인"으로 처리하고 M12-S-03의 폴백 규칙을 따른다.
  • safeExec의 maxBuffer(1MB)·timeout(5s)·SIGKILL가 자동 적용되므로 출력 폭주·hang DoS는 이미 차단됨(양호).
  • 파싱은 정규식/indexOf 단순 매칭만 — 동적 로직·eval 금지(자명하나 명시).

③ 점검 항목별 평가 요약

요청 점검평가근거
1. whoami safeExec(H-2)조건부 양호 safeExec는 절대경로·shell:false·인자배열 강제로 견고. 단 whoami를 PATH 검색이 아닌 고정 절대경로 검증으로 구현해야 PATH 하이재킹 0(M12-S-01).
2. 상승 감지의 안전성양호 감지는 부작용 0·판정만 — 새 신뢰 경계·권한 생성 없음. 감지 실패의 보수적 기본값만 명시하면 됨(M12-S-03).
3. 데이터 오염 방지 효과부분 양호 write 보류는 기존 데이터 무손상·위치 불변을 보장(읽기 허용). 단 가드를 스냅샷 외 UI상태·설정 write로 확장해야 완전(M12-S-02).
4. 경고 배너 L-1양호 고정 문구 textContent + 기존 단방향 on() 패턴 재사용. 시스템 문자열을 배너에 넣지 않는 것만 게이트로 확인(M12-S-04).
5. per-user 격리·위치 불변양호(긍정) 상승 제거로 표면 감소. 0600/0700·per-user·위치 전부 불변(M12-S-05).
구현 단계 보안 게이트(요약):
  • ① whoami = %SystemRoot%\System32\whoami.exe 고정 절대경로 검증(PATH 검색 금지).
  • ② write 보류 가드 = 스냅샷 + UI상태 + 설정(persistConfigKeys) 세 경로 전부.
  • ③ 감지 실패 폴백 규칙 명시(보수적 OR + whoami 실패 시 휴리스틱 단독, 둘 다 불확실 시 정상 폴백).
  • ④ 배너는 고정 문구 textContent만, 시스템 경로·프로필명 비노출.
  • ⑤ 단위 테스트: 감지 판정(M12-1) + 세 write 보류(M12-2 확장) + 비상승 회귀 불변(M12-3) + whoami High ML 파싱.
