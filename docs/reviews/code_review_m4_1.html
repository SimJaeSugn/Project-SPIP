0. 종합 판정

판정: PASS (조건부)

M4 변경분은 설계 델타(rev.2)에 충실히 정합한다. 신규/수정 17개 파일 + 테스트 3종(70개)을 검토했고, 테스트는 전부 통과한다(node --test 실측). 실행 전 보이는 정확성 결함(P1)은 1건으로, 이 1건은 "기능 오동작"이 아니라 설계가 약속한 watchdog 안전망이 실제로는 작동하지 않는 가용성 결함이다 — 정상 경로 동작에는 영향이 없어 PASS로 두되, RM-6/M4-M-3의 안전 보증을 실제로 성립시키려면 다음 회차 전 수정 권고(P1)다. 나머지는 P2 6건·P3 5건으로, 신호 대 잡음을 지켜 핵심만 보고한다.

강점(잘 쓰인 부분):
  • 모듈 경계 명료. scanController(동시성/진행) ↔ actionHandlers(게이트/응답) ↔ scanner(순회) ↔ walker(가지치기)의 책임이 깨끗이 분리. 의존 방향(server→scan) 불변, 순환 없음.
  • 락 finally 보장. scanController.start의 try/finally + 래퍼 run().catch() 이중 안전망으로 정상·예외 경로 모두 running=false 해제. 데드락(영구 409) 1차 방어는 견고.
  • size 2레이어 status 분리 명확. 계약 status(res.status = ok/na/error)와 도메인 status(res.data.status = ok/partial/skipped/error)를 §4.1.1대로 분리, 주석으로 의도 명문화. scanner.js:60 병합과 충돌 없음.
  • 경계 처리 정확. walker maxDirs·size maxEntries 상한 비교에 off-by-one 없음(실측 확인).

1. P1 — 정확성 결함

P1-1 watchdog 타임아웃이 락을 "강제 해제"하지 못한다 — 설계 보증 미성립

항목내용
무엇watchdog 발화 시 _forceError()로 phase='error'만 설정하고, running은 풀지 않으며 진행 중인 scanner.scan()도 중단시키지 않는다. 실제 락 해제는 scanner.scan()이 자연 완료되어 finally에 도달해야 일어난다.
위치lib/server/scanController.js:97-102 (_forceError), 108-165 (start/run)
왜 문제설계 §2.1·§2.5·§3.3·RM-6은 "watchdog 초과 시 스캔을 강제 종료 신호 + phase=error + 락 강제 해제"를 명시한다. 그러나 코드의 watchdog는 타이머 콜백이 phase만 바꾸고, 락 해제와 스캔 중단을 못 한다. watchdog가 막으려던 시나리오(무한 순회·블로킹으로 background가 죽음)에서는 scanner.scan()이 끝나지 않으므로 finally에 영영 도달하지 못해 running이 영구 true로 남는다 — 즉 watchdog는 정확히 그 시나리오에서 무력하다. 더구나 walker/scanner 순회 루프는 동기 제너레이터를 for...of로 소비하는 구간이 길어 await 양보가 드물고, 진짜 블로킹이면 setTimeout 콜백(watchdog) 실행 자체가 지연될 수 있다. 결과적으로 RM-6의 "영구 409 데드락 방지" 보증이 코드에서 성립하지 않는다.
왜 PASS정상/유한 트리에서는 maxDirs·timeBudgetMs 전역 상한(M4-M-2)이 순회를 유한 시간에 끝내므로 락은 결국 풀린다. 일반 기능 동작에는 결함이 드러나지 않는다(테스트 watchdog 케이스도 작은 트리라 scanner가 곧 끝나 통과 — 즉 테스트가 강제 해제를 실제로 검증하지 못함). 안전망이 "있는 것처럼 보이나 실제로는 1차 finally에만 의존"하는 상태.
권고(1) watchdog 발화 시 this.running=false를 즉시 풀어 후속 재스캔(409 영구거부)을 막거나, (2) scanner에 협조적 취소(cancel token/AbortSignal류 또는 watchdog가 set하는 cancelled 플래그를 walker 루프가 maxDirs처럼 체크)로 순회를 실제 중단시킬 것. 최소한 watchdog 콜백에서 락을 푸는 것이 설계 의도(가용성 우선)에 부합. 동반: watchdog 강제해제를 실제로 발화시키는 테스트(scanner가 끝나기 전 watchdog가 running을 푸는지)를 추가. scanController.js:117-120 참고.
// 현재(scanController.js) — watchdog는 phase만 바꿈, running/scan은 그대로
this._wd = setTimeout(() => {
  watchdogFired = true;
  this._forceError('WATCHDOG_TIMEOUT');   // ← phase='error'만. running=true 유지
}, watchdogMs);
// ...scanner.scan()이 끝나야 finally 도달 → 그게 안 끝나는 게 watchdog가 막으려던 상황
설계 인용: m4-design.html §2.5 의사코드는 this._forceError('WATCHDOG_TIMEOUT')로만 적혀 있어 설계 자체가 "락 강제 해제"를 산문으로만 주장하고 의사코드엔 누락했다. 즉 구현은 설계 의사코드에 충실하나, 설계 산문(§2.5 "강제 종료 신호 + 락 강제 해제", RM-6)과는 어긋난다. 구현·설계 동시 정정이 필요한 항목 — tech-lead 판단 권고.

2. P2 — 개선 (정확성에 직접 영향은 없으나 잠재 결함·중복·복잡도)

P2-1 size: nodeModulesBytes 주석-코드 불일치 + entry 카운터 공유로 측정 누락

무엇주석은 "node_modules 측정분을 totalBytes에 가산"이라 하나 실제 코드는 가산하지 않는다. 또한 measureNodeModules가 budget.state(entry 카운터)를 공유하므로, 앞선 measureTree(projectPath)가 maxEntries를 거의 소진하면 nodeModulesBytes는 0에 가깝게 측정된다(아무 안내 없이).
위치lib/scan/collectors/size.js:152-170, 86-98
왜 문제(1) 주석이 실제 동작과 달라 유지보수자가 오해(size.js:161-162 "가산" 문구). (2) measureTree(projectPath)는 walker excludeRules를 거치지 않아 node_modules도 포함 측정하므로 totalBytes에 이미 포함됨 → nodeModulesBytes는 순수 "표시용 부분집합". 그러나 entry 예산을 본문 측정이 다 써버리면 nodeModulesBytes가 실제보다 크게 과소측정되며 partial 플래그만으로는 "어느 값이 부분인지" 구분 불가. UI는 nodeModulesBytes를 비중 표시에 쓰는데 신뢰도가 들쭉날쭉.
권고주석을 실제 동작(가산 안 함, 표시용 부분집합)에 맞게 수정. node_modules 측정에 별도 entry 예산을 두거나, 본문/nm 측정 순서·예산 배분을 명시적으로 문서화. 정확성은 깨지지 않으므로 P2.

P2-2 재스캔 finalizing 시 writeSnapshot·store.load가 cachePath를 전달하지 않음

무엇start()가 serializer.writeSnapshot(snap, { logger })와 opts.store.load({ logger })를 cachePath 없이 호출 → 둘 다 paths.cachePath() 기본 경로 사용.
위치lib/server/scanController.js:141-144 · 대비 server.js:40(store.load에 opts.cachePath 주입)
왜 문제운영(기본 경로)에서는 writeSnapshot/load/초기 load가 모두 기본 경로라 일관 — 결함 없음. 그러나 커스텀 cachePath로 기동한 경우(테스트·다중 인스턴스) 초기 적재는 커스텀 경로, 재스캔의 write/load는 기본 경로로 갈려 재스캔 결과가 보이지 않는 비일관이 발생한다. 현재 ScanController는 cachePath를 모르므로 구조적으로 ctx의 cachePath를 전파할 길이 없다.
권고ctx/start opts에 cachePath를 전달해 write/load 양쪽에 넘길 것. 또는 createServer가 cachePath를 config/ctx에 보관해 scanController.start가 참조. 단일 사용자 운영 기본값에선 무해하므로 P2.

P2-3 security: checkStateChange / checkReadAccess 1·2·3요소 중복

무엇checkReadAccess가 checkStateChange의 Host·Origin·토큰 3블록을 그대로 복붙하고 CT 검증만 뺀 형태(약 20줄 중복).
위치lib/server/security.js:68-96 vs 107-130
왜 문제두 함수가 공유하는 검증 규칙이 한쪽만 바뀌면 보안 게이트 비대칭이 silently 생긴다(예: Host 비교 강화 시 한쪽 누락). 설계 §3.4도 "1·2·3요소 공유"라고 명시 — 공유가 의도이므로 공통 헬퍼로 추출하는 게 옳다.
권고공통 checkOrigin3(req, ctx)(Host+Origin+토큰)를 추출하고 checkStateChange는 그 뒤 CT만 추가. DRY + 단일 출처. 동작 동일하므로 회귀 위험 낮음.

P2-4 driveEnum/excludeRules에 시스템 제외 가지치기 미연동(현재 walker 진입 전 제외가 루트엔 미적용)

무엇시스템 제외(isUnderSystemDir)는 walker가 자식을 canonicalize한 뒤에만 적용된다(walker.js:119). all-drives에서 driveEnum이 반환한 루트 자체(예: POSIX /)는 제외 판정을 거치지 않고 walkRoot에 바로 들어가, 루트 직하 첫 레벨에서 비로소 자식 단위로 걸러진다.
위치lib/scan/walker.js:178-186 (루트 루프), 119-121 (자식만 제외)
왜 문제기능상 자식 단위 제외로 시스템 디렉터리 진입은 차단되므로 보안/정확성 결함은 아니다(security 영역). 다만 루트가 곧 시스템 경로인 경우(예: / 직하의 /proc) 첫 yield로 루트 자신은 후보가 되며, detector가 무의미한 루트를 한 번 본다. 설계 §5.3은 "진입 전 앞단 가지치기"를 강조 — 루트 레벨엔 그 원칙이 적용 안 됨. 일관성/명료성 개선 여지.
권고all-drives일 때 루트 루프에서도 isUnderSystemDir(rootCanonical) 1회 체크. 또는 설계상 "루트는 driveEnum이 마운트 루트만 주므로 시스템 루트 자체는 없음"을 주석으로 명시해 의도 확정.

P2-5 idle 응답에 폴링이 무한 continue될 수 있는 좁은 경쟁

무엇프론트 폴링이 phase:'idle'를 받으면 continue(무한 폴링). 정상 흐름은 acquire 직후 phase='scanning'이라 거의 안 일어나나, 락이 풀린(done 후 누군가 reset) 또는 acquire 이전 타이밍의 폴링에서 idle continue가 종료 조건 없이 반복될 수 있다.
위치public/app.js:1490-1522 (pollOnce), 364-387 (nextPollAction)
왜 문제리소스 누수(타이머가 안 멈춤) 잠재. 단, document.hidden 시 네트워크 생략·1초 주기라 영향은 작다. 자기 scanId 발급 후 시작하므로 실사용 경로에선 거의 무해.
권고idle continue에 안전 카운터(예: 연속 idle N회 또는 시작 후 X초 경과 시 stopPolling)를 둘 것. RM-4 "idle 시 폴링 중단"과 정합.

P2-6 scanner: onProgress 객체 생성 코드가 onDir와 프로젝트확정 두 곳에 중복

무엇{ phase:'scanning', dirs, found: projects.length, currentPath } 객체 조립 + try/catch 호출이 emitProgress()(스로틀)와 프로젝트 확정 블록(scanner.js:195-199)에 각각 존재.
위치lib/scan/scanner.js:149-157, 194-199
왜 문제ScanProgress shape가 바뀌면 두 곳을 동기화해야 함. 프로젝트 확정 블록은 스로틀(250ms)을 우회해 매 프로젝트마다 호출 — 의도일 수 있으나 onDir 경로와 빈도 정책이 갈려 currentPath/dirs가 호출마다 의미가 미묘하게 다르다.
권고단일 emit(extra) 헬퍼로 통일하고 스로틀 정책을 한 곳에서 결정. 동작 동일하게 리팩터 가능.

5. 테스트 품질 평가

3개 스위트 70 테스트 전부 통과(node --test 실측). 락·폴링전이·축약·시스템제외·예산을 대체로 의미있게 검증한다.

검증 대상평가
락 / 409 SCAN_IN_PROGRESS충분 — acquire 직접 점유 후 409·scanId 일치 검증.
watchdog 강제해제불충분 — watchdogMs=1 케이스가 작은 트리라 scanner가 곧 끝나 finally로 락이 풀린다. watchdog가 scanner보다 먼저 락을 푸는지(P1-1)를 검증하지 못함. 강제해제 보증을 실증하려면 scanner를 인위적으로 지연시키는 스텁 필요.
폴링 전이충분 — scanning→finalizing→done 시퀀스, foreign(scanId 불일치), null graceful 모두 케이스화.
currentPath 축약(C-3)충분 — shortenPath 단위 + 직렬화에 'secret'/'E:' 미포함 단언.
시스템 제외 세그먼트(C-2)대체로 충분 — WindowsApps prefix 우회 차단·대소문자 폴드. 단 walker 통합 실차단은 isUnderSystemDir 단위로만 모사(테스트 주석도 인정: systemKeySet 주입 불가). walker가 buildSystemExcludeKeySet을 실제로 쓰는 end-to-end 경로는 미검증.
size 예산/partial(C-5)충분 — budgetMs/maxEntries 양쪽 partial 유발, 계약/도메인 status 분리 단언.
축약·MVP 회귀충분 — withSize 미지정 skipped + totalBytes null 회귀 케이스 존재.
테스트 보강 권고: (1) watchdog 강제해제 실증(P1-1 직결) — scanner를 지연 스텁으로 두고 watchdog가 running을 먼저 푸는지. (2) all-drives 시스템 제외 end-to-end — walker에 platform 주입 + 실디렉터리로 시스템키가 실제 가지치기하는지(현재는 단위 모사). (3) P2-2 cachePath 비일관 — 커스텀 cachePath 재스캔 후 그 경로에 써지는지.

6. P1-1 결함 흐름 (watchdog vs 실제 락 해제)

7. 확인 필요 (설계 의도)

  • watchdog 강제해제(P1-1): 설계 산문(RM-6)과 의사코드(§2.5)가 어긋난다. "watchdog는 phase 표시만, 락 해제는 finally에 위임"이 의도라면 RM-6 문구를 약화해야 하고, "강제 해제가 진짜 필요"라면 코드 수정 필요. tech-lead/설계자 확정 요망.
  • size nodeModulesBytes 의미(P2-1): totalBytes에 포함된 부분집합(표시용)인지, 가산 대상인지 — 주석/코드 중 어느 쪽이 의도인지 확정.
  • 커스텀 cachePath 재스캔(P2-2): 다중 인스턴스/테스트에서 cachePath 분리 운영을 지원할지, "기본 경로 단일"만 보장할지.

8. 마무리

판정: PASS(조건부). P1 1건은 기능 오동작이 아니라 안전망(watchdog 강제해제) 미성립이며 정상 경로에는 영향이 없어 통과시키되, 다음 회차 전 수정(또는 설계 문구 정정)을 권고한다. P2 6건·P3 5건은 품질 개선 항목이다. 코드는 직접 수정하지 않았다.
