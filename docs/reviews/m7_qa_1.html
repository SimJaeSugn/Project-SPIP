9. 사용자 PC 확인 필요 (헤드리스 한계 — 추측 PASS 아님)

아래는 실 GUI 디스플레이/입력이 있어야 체감·확정 가능한 항목이다. 코드/단위는 통과했으나 런타임 시각·OS 거동은 미검증으로 분리한다.

  • 토스트 등장 위치 — 트레이 '즐겨찾기' → 위젯이 실제로 우측 하단(작업표시줄 위, margin 16)에 뜨는지. 멀티모니터/작업표시줄 위치 변경 시 커서 모니터 기준 배치.
  • blur 자동 닫힘 + grace — 다른 앱 클릭(blur) 시 100ms 후 hide, 위젯 내부 클릭으로 인한 순간 blur는 닫히지 않는지(재focus 취소). 특히 × 버튼 window.blur() 폴백이 실제로 닫는지(O-1).
  • 대시보드 닫힘(트레이 상주) 중 독립 동작 — 메인창 X→hide 상태에서 트레이 '즐겨찾기'로 위젯 정상 등장(R-22 핵심 시나리오). 구조는 확인됨, 실 트레이 클릭 체감 미확인.
  • FLIP 시각 부드러움 — 카드 드래그/키보드 이동 시 다른 카드가 180ms로 자연스럽게 비켜주는지, 점프/깜빡임 없는지. reduced-motion 켠 OS에서 즉시 적용.
  • focus 게이팅 체감(SEC-H2) — 비포커스 위젯의 액션 버튼 비활성, 첫 클릭은 포커스만(액션 미실행).
  • transparent/둥근 모서리 외형 + 폴백 — Windows transparent+frameless 렌더, 환경 부담 시 불투명 폴백(ADR-M7-5).
  • 양 창 동시 열림 동기화 — 메인·위젯 동시 표시 중 한쪽 별 토글 시 다른 쪽 즉시 반영(broadcast). 깜빡임 없는 idempotent 수렴.
  • alwaysOnTop floating 전이 — show 시 floating, hide/blur 시 해제가 다른 앱 입력을 가로채지 않는지.
