Project-SPIP — M7 설계 보안 검증 (새 BrowserWindow 공격 표면 위협 모델링)

검증 대상: docs/architecture/m7-design.html(R-22 즐겨찾기 독립 위젯 창 · R-23 카드 FLIP 애니메이션). 기준: electron_design_security_review_2.html·m6_design_security_review_2.html(EC-1~6·senderFrame·preload allowlist·contextIsolation/sandbox·CSP·EM 베이스라인) 및 실측 코드(main.js·security.js·register.js·preload.js·tray.js). 1. 자산 · 신뢰 경계 · 위협 모델 요약

보호 자산: 사용자 프로젝트 메타데이터(경로·이름)·ui-state(즐겨찾기/순서)·외부 툴 실행 권한(R-12 open)·클립보드. M7이 추가하는 신뢰 경계는 위젯 렌더 컨텍스트(app://favorites.html) 1개다. 위젯은 메인 창과 동일 app:// origin·동일 main 프로세스 guard를 공유하므로, 검증의 핵심은 "위젯이 메인 창과 동등한 봉합을 빠짐없이 받는가"와 "새 창·새 창 속성(frameless·transparent·alwaysOnTop)·새 push 채널이 기존 통제를 약화·우회하는 경로를 만드는가"이다.

3. 발견 항목 (Findings)

높음 SEC-H1 — applyCspHeaders 세션 모델 미명세: 위젯 창의 default session 중복 등록 / 누락 모호

위치: m7-design.html §4.1 의사코드(deps.applyCspHeaders(widget.webContents.session)) · §5 "동일 applyCspHeaders(widget.webContents.session)" · 실측 security.js:40~46(session.webRequest.onHeadersReceived) · main.js:151(applyCspHeaders(win.webContents.session))

위협(STRIDE): Tampering / Information Disclosure — CSP 미적용/약화 시 위젯에 인젝션·외부 리소스 로드 표면 발생

악용 시나리오: Electron에서 별도 partition을 지정하지 않으면 위젯 창은 메인 창과 동일한 default session을 공유한다. 실측 applyCspHeaders는 session.webRequest.onHeadersReceived(...)에 핸들러를 등록하는데, 이 API는 세션당 최후 등록 핸들러 1개만 유효하다(Electron은 onHeadersReceived 리스너를 덮어쓴다). 따라서 위젯 생성 시 같은 default session에 다시 호출하면 ⓐ 기존 메인 핸들러를 덮어쓰기만 해 무해할 수도 있으나, ⓑ 만약 위젯을 별도 partition으로 분리하는 의도였다면 그 session에는 CSP가 전혀 적용되지 않는 누락이 발생할 수 있다. 설계가 "session" 객체를 무엇으로 보는지(default 공유인지 신규 partition인지)를 명시하지 않아, 구현자가 partition 분리를 택하면 CSP 헤더 주입이 조용히 빠질 수 있다. CSP 메타 태그(favorites.html)가 2차 방어로 남지만 EM-M-1의 "이중주입" 불변이 깨진다.

권고 조치: ① 설계에 위젯이 메인과 동일 default session을 공유함을 명시하고, 따라서 applyCspHeaders 재호출이 불필요/중복임을 적시(또는 멱등성 보장). ② 만약 partition 분리를 택한다면 해당 session에 applyCspHeaders를 반드시 호출하고 onHeadersReceived 단일 등록 제약을 문서화. ③ 어느 경우든 favorites.html의 CSP 메타 태그는 security.js CSP_POLICY 단일 원천과 문자열 동일해야 함을 명세(드리프트 방지). ④ 구현 게이트에 "위젯 응답 헤더에 CSP 존재" 스모크 추가.

높음 SEC-H2 — transparent + frameless + alwaysOnTop 위젯의 UI redressing / 입력 가로채기 / 클릭통과 표면

위치: §4.1(frame:false, transparent:true, alwaysOnTop:true, level 'floating') · ADR-M7-3 · §13 리스크표(transparent 클릭통과)

위협(STRIDE): Tampering / Elevation(UI) — frameless·alwaysOnTop 창은 사용자 인지 경계를 흐려 오클릭·입력 가로채기·클릭재킹 유사 표면을 만든다

악용 시나리오: ① alwaysOnTop:true + transparent:true 위젯은 우측하단에 다른 앱 위로 떠 있어, 위젯 투명 영역이 클릭을 가로채거나(클릭통과 미보장) 반대로 아래 창 입력을 막을 수 있다. ② 위젯 안에는 R-12 실행(open)·copyText 같은 부수효과 액션 버튼이 있는데, frameless라 OS 신뢰 표시(타이틀바)가 없고 사용자가 어느 앱의 UI인지 혼동할 수 있다. ③ 본 위젯 자체는 app:// 로컬 페이지라 외부 공격자가 직접 위젯 DOM을 조작하진 못하나(웹콘텐츠 격리됨), 위젯 위에 다른 악성 창이 겹치거나(역 클릭재킹) 위젯이 다른 앱 입력을 가로채는 데스크톱 UX 안전 문제는 남는다. §13이 클릭통과·blur 오작동을 리스크로만 적었을 뿐 통제로 확정하지 않았다.

권고 조치: ① setIgnoreMouseEvents 정책 명시 — 투명 여백은 클릭통과(forward) 또는 불투명 패널로 완전 차단 중 택일을 설계에 확정(현재 "body 배경 패널로 채움"은 권고일 뿐 불명확). ② 부수효과 액션(open/copyText)은 위젯이 focus 상태일 때만 활성(blur 시 비활성)으로 오클릭 방지. ③ alwaysOnTop은 show 시에만, hide 시 해제 명세(ADR-M7-3은 'show 시 적용'만 적음 — 해제 시점 누락). ④ MQ7-2(드래그 이동·-webkit-app-region:drag)는 후속으로 미루되, 도입 시 drag 영역이 액션 버튼과 겹치지 않도록 별도 검토 표시.

보통 SEC-M1 — hardenWebContents의 will-navigate가 favorites.html 내부 상대 이동을 과도 차단하거나, deny 핸들러 회귀 미검증

위치: §4.1(deps.hardenWebContents(widget.webContents, {trustedOrigin})) · 실측 security.js:56~74

위협(STRIDE): Tampering — 위젯이 외부 탐색/팝업/webview로 탈출하는 경로

악용 시나리오: 설계는 hardenWebContents를 위젯에 호출한다고 명시했고 실측 함수는 will-navigate(app:// 외 차단)·setWindowOpenHandler:deny·will-attach-webview 거부를 모두 적용하므로 구조는 올바르다(PASS 방향). 그러나 ⓐ 위젯은 per-webContents 등록이라 메인과 달리 반드시 명시 호출되어야 하며 누락 시 조용히 약화된다(§13이 "ensureWidget이 생성 시 강제 호출"로 완화 — 양호). ⓑ setWindowOpenHandler:deny가 모든 새 창을 거부하므로, 위젯에서 R-12 open이 새 창이 아닌 main의 shell.openExternal/툴 실행 경로(IPC)임이 보장돼야 한다(위젯 DOM이 window.open/<a target>으로 외부를 못 염). 설계상 open은 IPC invoke라 정합하나 명문 확인 권고.

권고 조치: ① "위젯의 모든 외부 동작은 IPC invoke만, 렌더러 직접 네비/팝업 0"을 §5에 단언으로 추가. ② 구현 게이트에 EC-5 정합 스모크(위젯에서 window.open·외부 location 시도 차단 확인)를 명시(§13에 이미 언급 — 통제로 승격).

보통 SEC-M2 — broadcast 대상 webContents 지연평가의 파괴 후 send 안전성·임의 대상 금지 보장

위치: §6.3 의사코드([mainWc, widgetWc].forEach(wc => wc && wc.send(...))) · §5 "양 창의 webContents로만 send" · §7

위협(STRIDE): Tampering / Information Disclosure — push 대상 오염·파괴된 wc로 send 시 예외

악용 시나리오: broadcast는 main→renderer 단방향이고 payload가 {favorites:string[]}(id 배열, 형식 검증된 값)뿐이라 데이터 노출 위협은 낮다(설계 방향 양호). 잔여 위험: ⓐ getFavoritesWidgetWc()가 destroy된 wc를 반환하면 wc.send가 throw — 의사코드는 try/catch로 감쌌으나(양호), isDestroyed() 가드를 명시해 main.js의 makeProgressSender 패턴(지연평가+파괴 가드)과 동형화 권고. ⓑ payload에 경로·실행 인자가 절대 실리지 않음을 단언으로 고정(현재 "id 배열뿐"은 서술 — 계약으로 명문화). ⓒ broadcast 대상이 메인 wc + 위젯 wc 2개로 한정됨을 보장(임의 BrowserWindow.getAllWindows() 순회 금지 — 향후 창 추가 시 누설 방지).

권고 조치: ① wc && !wc.isDestroyed() && wc.send(...)로 가드 명시. ② push payload 스키마를 "id 배열만, 경로/인자/내부상태 금지"로 §6.3 계약에 고정. ③ 대상 wc 화이트리스트(메인·위젯 2개)를 명문화.

보통 SEC-M3 — preload-favorites.js의 generic invoke 비노출·MUST 명문화 및 onFavoritesChanged 수신 검증

위치: §5·§6.2 의사코드(preload-favorites.js) · 실측 기준 preload.js(채널명 하드코딩·ipcRenderer 원본 비노출 MUST)

위협(STRIDE): Elevation — 위젯 preload가 generic invoke나 원본 ipcRenderer를 흘리면 6채널 축소가 무의미해짐

악용 시나리오: §6.2 의사코드는 채널명을 하드코딩하고 ipcRenderer 원본을 노출하지 않으며 인자를 String()/!!로 강제해 메인 preload와 동일 패턴이다(PASS 방향). 잔여 점검: ⓐ onFavoritesChanged가 ipcRenderer.on으로 수신하는데, push 채널 1개만 구독하고 콜백에 payload를 그대로 전달한다 — 위젯 렌더러는 이 payload를 신뢰하여 textContent 렌더해야 하며(L-1), main이 보낸 형식 검증된 favorites만 싣는다는 전제가 깨지지 않도록 main 측 단언(SEC-M2)과 묶임. ⓑ 메인 preload의 강력 채널(setToolPath·pickToolExecutable·setOrder·rescan·addRoots·removeRoot·setSortMode·pickFolders)이 위젯 preload에 단 하나도 없어야 한다(설계상 6개 한정 — 표면 축소 양호).

권고 조치: ① "preload-favorites.js는 ipcRenderer 원본·generic invoke 비노출(MUST)·채널명 하드코딩"을 메인 preload와 동일하게 §6.2에 단언으로 명기. ② 위젯 onFavoritesChanged 수신 payload는 main 신뢰값으로 간주하되 위젯 렌더는 textContent(L-1) 유지. ③ 강력 채널 비노출을 추적성 표에 체크리스트로 고정.

보통 SEC-M4 — 위젯 open(R-12 실행)·copyText가 메인과 동일한 main 재검증을 받는지 명문 확인(IDOR/실행 표면)

위치: §5(전 채널 main 재검증) · §6.1(open/copyText 재사용) · 실측 register.js:118,130(openInVsCode·copyText guard)

위협(STRIDE): Elevation / Tampering — 위젯이 실행·복사 채널을 동일 검증 없이 호출하면 우회 경로

악용 시나리오: 위젯이 호출하는 spip:openInVsCode·spip:copyText·spip:setFavorite는 실측 register.js에서 동일 guard로 1회만 등록되어 있고, 위젯도 같은 guard를 통과하므로 id 형식·toolId 화이트리스트·pathGuard·resolveBin force·safeExec(.exe·shell:false) 재검증이 그대로 적용된다(EC-1·H-1·H-2 보존 — PASS 방향). 위젯이 새 핸들러를 만들지 않고 기존 채널을 재사용한다는 설계가 이 동등성을 구조적으로 보장한다. 잔여: 위젯 렌더러가 보내는 id가 위젯 표시 교집합(favorites∩스냅샷) 밖의 임의 id일 수 있으나, main 핸들러가 어차피 id 형식·존재를 재검증하므로 IDOR로 확대되지 않음(렌더러 신뢰 0 원칙). 명문 확인만 권고.

권고 조치: ① "위젯의 open/copyText/setFavorite은 신규 핸들러 0, 기존 register guard 핸들러 100% 재사용 → EC-1·H-1·H-2 재검증 자동 승계"를 §5에 단언. ② 위젯 렌더러가 보내는 모든 인자는 비신뢰로 간주(main 재검증이 유일 권위)임을 명기.

낮음 SEC-L1 — onTray action 축소(favorites 제거) 시 잔여 채널·tray.js TRAY_ACTIONS 정합

위치: §8.1(preload.js onTray action을 dashboard만으로 축소) · 실측 preload.js:80~90(actions=['dashboard','favorites']) · tray.js:18(TRAY_ACTIONS)

위협(STRIDE): 표면 정리(미정리 시 죽은 수신 채널) — 보안 위협은 낮으나 표면 위생

악용 시나리오: M7은 트레이 '즐겨찾기'를 메인창 push가 아닌 favoritesWidget.show로 바꾸므로 spip:tray:favorites push가 사라진다. 설계는 onTray를 dashboard만으로 축소(또는 favorites 분기 무시)한다고 했다. favorites 분기를 남기면 아무도 send하지 않는 죽은 수신 채널이 되어 표면 위생만 저하(권한 상승은 아님). tray.js TRAY_ACTIONS는 호환 위해 유지 가능하다고 했으나, 화이트리스트와 실제 push 채널 집합이 불일치하면 추적성 혼란.

권고 조치: onTray actions에서 favorites 제거를 확정(권장)하거나, 유지 시 "수신만·미사용" 주석 명시. TRAY_ACTIONS는 메뉴 라벨 정의용으로만 쓰이면 무해 — 용도 분리 명기.

낮음 SEC-L2 — favorites.html CSP 메타와 security.js CSP_POLICY 문자열 드리프트 위험

위치: §5(favorites.html meta http-equiv에 "메인과 동일 CSP") · 실측 security.js:18~29(CSP_POLICY 단일 원천)

위협(STRIDE): Tampering — 메타 CSP가 헤더 CSP와 달라지면 약한 쪽 약점 발생

악용 시나리오: CSP는 헤더(applyCspHeaders)와 메타 태그 이중주입이 베이스라인(EM-M-1)이다. favorites.html 메타가 손으로 작성돼 security.js CSP_POLICY와 문자열이 어긋나면(예: img-src data: 누락, script-src 완화) 위젯에서 메타 기준이 적용돼 미묘한 약화 발생. 설계는 "security.js의 CSP_POLICY 단일 원천 재사용"이라 했으나 메타는 빌드타임 정적 문자열이라 동기화 보장 수단이 불명확.

권고 조치: index.html 메타와 favorites.html 메타가 동일 문자열임을 구현 게이트에서 비교(또는 빌드 시 CSP_POLICY로부터 생성). 두 페이지 메타가 문자 단위 일치하는지 테스트 추가.

낮음 SEC-L3 — 위젯 hide 재사용 시 stale 렌더 컨텍스트의 정보 잔존 / dispose 멱등성

위치: §4.3(hide 재사용·doFinalQuit에서만 destroy) · ADR-M7-2 · 실측 main.js:260(doFinalQuit 멱등)

위협(STRIDE): Information Disclosure / DoS(좀비) — hide된 위젯에 즐겨찾기 데이터가 잔존, dispose 누락 시 좀비

악용 시나리오: hide 재사용은 위젯 DOM에 즐겨찾기(경로·이름)가 메모리에 남는다 — 같은 사용자 세션 내라 위협 낮음(외부 도달 0, app:// 격리). dispose는 doFinalQuit에서만 호출되며 실측 doFinalQuit은 _disposed 멱등 가드가 있으나 favoritesWidget.dispose() 자체의 멱등성(이미 destroy/null)은 위젯 모듈 책임 — 설계 §6.4가 "멱등"이라 명시(양호). 잔여: 메인창 close-to-tray 후 위젯만 떠 있고 메인창이 destroy되지 않은 상태에서 broadcast 대상 mainWc가 hidden wc로 send되는 경우 — send 자체는 hidden wc에도 유효하므로 무해.

권고 조치: ① favoritesWidget.dispose()의 멱등성(중복 호출·미생성 시 no-op)을 계약으로 명문화(이미 §6.4 언급 — 단언 승격). ② doFinalQuit 순서에 위젯 dispose가 기존 scanController dispose·tray.destroy·app.exit 순서를 깨지 않음을 명기(P2-2 단일 종료 경로 불변 — 설계 §4.3에 이미 있음, 확인).

  • MQ7-3 (메인 UI 위젯 트리거): 메인창에서 위젯을 띄우려면 신규 invoke spip:showFavoritesWidget 1개가 추가될 수 있다 — 추가 시 register guard 등록·senderFrame 검증 자동 승계(저위험)이나 "신규 invoke 0" 설계 주장이 깨지므로 PM 결정 필요.

6. 종합 판정

판정: 조건부 PASS (CONDITIONAL). 치명 0건. 높음 2건(SEC-H1·SEC-H2)은 설계 반영 필수 — 보안 게이트는 두 항목이 설계 델타에 명문 반영된 뒤 통과한다.

발견 건수: 치명 0 · 높음 2 · 보통 4 · 낮음 3 (총 9).

요약: M7의 신규 표면은 위젯 렌더 컨텍스트 1개 + push 채널 1개로 정확히 한정되며, 설계는 메인 창과 동등한 통제(동일 webPreferences·app:// 로드·하드닝·senderFrame·textContent)와 더 좁은 preload(6채널)를 의도해 표면을 축소한다 — 방향성은 견고하다. senderFrame 동등 통과(T4)·기존 main 재검증 승계(T7)·오버레이 제거 무약화·FLIP 무관(R-23)은 실측 코드 대조로 PASS. 다만 ⓐ CSP 세션 주입 모델(SEC-H1 — default 공유/멱등 vs partition 누락)이 모호하고, ⓑ transparent·frameless·alwaysOnTop 위젯의 UI redressing/클릭통과/alwaysOnTop 해제 시점(SEC-H2)이 리스크로만 적혀 통제로 확정되지 않았다. 두 높음 항목과 보통 4건(broadcast 가드·preload MUST 명문화·open/copyText 재검증 단언·will-navigate 단언)을 설계에 반영하면 PASS 전환 가능하다. 직접 수정하지 않고 시정 항목만 제시한다.
