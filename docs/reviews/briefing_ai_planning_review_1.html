우선순위 P1 (진행 전 반영 권고)

F-1 P1  "거의 즉각 반응"(ref goal)과 ~2초 coalesce 디바운스의 정합이 명시적으로 화해되지 않음

  • 무엇: ref.md goal은 "새로운 이벤트가 발생하면 거의 즉각적으로 반응해야 한다"가 핵심 비기능 요구다. 반면 명세는 R-36/N-09에서 coalesce 디바운스 기본 ~2초 + TTFT≤3초를 둔다. 디바운스 2초 + 추론 지연 3초 = 사용자가 체감하는 최초 반응이 최대 5초 이상이 될 수 있다.
  • 위치: R-36 "coalesce 디바운스(권장 기본 ~2초)" · N-09 "TTFT 중앙값 ≤3초" · 미해결질문표 "coalesce 디바운스 창 ~2초" · §1 성공지표 TTFT≤3초.
  • 문제: "거의 즉각"이라는 원본 의도가 별도 측정 가능한 요구로 승격되지 않았고, 디바운스 값과의 트레이드오프가 한 줄("과도 호출 억제 vs 거의 즉각 균형")로만 언급될 뿐 "즉각"의 정량 정의·상한이 없다. 검증 단계에서 "이 기능이 ref 의도를 충족했는가"를 판정할 기준이 없다.
  • 권고: (a) "이벤트→사용자 가시적 반응 시작(스피너/스트리밍 인디케이터 표시)"을 TTFT(첫 토큰)와 분리해 별도 지표화 — 예: "이벤트→인디케이터 표시 ≤ (디바운스창+200ms)". (b) "급함(deadline 경과 등)" 신호는 디바운스를 우회하거나 더 짧은 창을 적용하는 fast-path를 요구로 명시(현재 모든 신호가 동일 ~2초). (c) 미해결질문표의 디바운스 결정에 "ref '즉각' 의도와의 트레이드오프"를 명시적 근거로 기록.

F-4 P1  프롬프트 인젝션 방어(N-08) 수용 기준이 단일 프로브에 그쳐 검증 강도 부족

  • 무엇: 신뢰 못 할 스캔/메일 데이터(커밋 메시지·메일 본문·프로젝트명)가 LLM 프롬프트에 들어가는 것은 이 기능의 가장 큰 신규 신뢰 경계 위반 위험이다. N-08은 "데이터는 인용/구분자 블록으로 주입"한다고 명세하나, 수용 기준은 "이전 지시 무시하고…" 삽입이 시스템 지시를 덮어쓰지 못함을 프로브로 확인 1건뿐이다.
  • 위치: N-08 수용 기준 3번째 항목 · R4 리스크.
  • 문제: 인젝션은 단일 페이로드로 대표되지 않는다(가짜 도구 호출 유도, 가이드에 악성 URL/명령 삽입 유도, 분류 조작, 시스템 프롬프트 누출 등). "구분자 블록"만으로는 방어가 불완전한데, 방어가 성공했다고 볼 판정 기준(어떤 케이스 집합을 통과해야 PASS인가)이 모호하다. 또한 메일 본문은 길고 적대적 제어 가능성이 가장 높은 입력인데 별도 취급이 없다.
  • 권고: (a) 인젝션 프로브를 케이스 집합으로 구체화 — ① 지시 덮어쓰기 ② 가이드에 외부 URL/쉘 명령 삽입 유도 ③ 분류 강제(모두 "급함"으로) ④ 시스템 프롬프트/키 누출 시도, 각각 "모델 출력이 시스템 정책을 위반하지 않음"을 수용 기준으로. (b) "가이드 내 URL/경로/명령은 렌더 시 실행 불가 텍스트로만, 클릭 가능 링크화 금지(또는 화이트리스트만)"를 N-08/R-37 수용 기준에 추가. (c) 입력 데이터 길이 상한·잘라내기 정책을 명시(메일 본문 전체 투입 시 토큰 폭주·인젝션 표면 확대).

우선순위 P2 (아키텍처 단계 위임 가능, 명세 보강 권고)

F-2 P2  "필요성 판정" 임계값이 모두 "권장 기본값·조정 가능"으로 열려 있어 수용 기준 재현성이 약함

  • 무엇: R-36 표의 트리거 조건은 검증 가능하게 잘 정의됐으나, 마감 임박 ≤24h·디스크 +1GB·디바운스 2초 등 수치가 "권장·아키텍처에서 조정 가능"으로 명시됨.
  • 위치: R-36 필요성 판정 표 하단 note · 미해결질문표.
  • 문제: 수용 기준("유의 변화 1건 발생 시 1회 트리거")은 임계값이 고정돼야 테스트 픽스처로 재현된다. 값이 전부 가변이면 "트리거됨/안 됨"의 경계 케이스가 비결정적이다.
  • 권고: 각 임계값을 "상수 단일 출처"로 고정(이미 방향 제시됨)하되, 기능명세 수준에서 기본 상수값을 확정 고정값으로 못 박고 "설정 노출 여부"만 아키텍처에 위임. 경계 테스트(임계 직전/직후)를 R-36 수용 기준에 추가.

F-5 P2  가이드 환각 억제·"실존 액션만 참조"의 검증 방법이 모호

  • 무엇: R-37 수용 기준 "가이드 내 앱 기능 안내는 실제 존재하는 기능만 참조한다(환각은 프롬프트로 억제)".
  • 위치: R-37 수용 기준 4번째 항목.
  • 문제: "프롬프트로 억제"는 비결정적 LLM 출력에 대한 검증 불가능한 기준이다. 7.8B 로컬 모델이 없는 기능을 안내할 가능성은 R2 리스크로도 인정됨. "환각 0"을 수용 기준으로 두면 사실상 통과 불가.
  • 권고: 수용 기준을 결정 가능한 형태로 재작성 — "가이드 텍스트는 자동 실행되지 않으며(이미 명시), 환각 액션이 있어도 사용자가 클릭/실행할 수단이 없다"를 안전 불변식으로(검증 가능). 환각 빈도는 "0" 대신 "안전 영향 없음(자동 실행 0)"으로 목표 재설정.

우선순위 P3 (경미 · 가독성/완결성)

F-3 P3  R-38↔R-41 의존 표기가 양방향이라 순환 인상

  • 무엇: R-38 "의존: R-41(상태 입력)" + R-41 "의존: R-38(상태)". 다이어그램에도 R-41→R-38 엣지 존재.
  • 위치: R-38·R-41 dep 표기 · §3 다이어그램.
  • 문제: 구현 순서(R-38 상태모델 먼저 → R-41 UI가 액션 발신)는 합리적이나, 양방향 의존 표기는 "어느 것을 먼저 만드나"를 흐린다. 실제로는 R-38(상태 스키마/순수 로직) → R-41(UI가 done/dismiss 호출)의 단방향에 가깝다.
  • 권고: "R-41은 R-38의 상태 전이 API를 호출(UI→상태)"로 방향 명확화. 순환이 아니라 R-38이 계약을 정의하고 R-41이 소비하는 단방향임을 명시.

F-6 P3  carry-over 영속 저장소가 m12 상승 세션 write 보류와 충돌하는데 동작 정의가 "보류 안내"로 미확정

  • 무엇: R-38은 done/dismiss 상태를 uiStateStore(0600)에 영속. 그런데 m12-design에 따르면 권한 상승 세션에서는 uiStateStore write가 no-op(보류)된다. R-40은 이를 "보류 안내"로 언급만.
  • 위치: R-38 영속 · R-40 5번째 항목 · R5 리스크.
  • 문제: 상승 세션에서 사용자가 done 처리해도 영속이 안 되면 다음 재생성 시 항목이 되살아난다(carry-over 수용 기준 "다시 나타나지 않는다"와 충돌). 이 충돌의 기대 동작이 명세에 결정되지 않음.
  • 권고: 상승 세션에서의 done/dismiss 기대 동작을 명시 — 메모리 반영 + "이 세션에선 처리 상태가 저장되지 않음" 안내, 또는 상승 세션에선 처리 액션 비활성. 수용 기준에 1줄 추가.

4. 가장 위험한 약점 (요약)

① (F-1) ref의 핵심 의도 "거의 즉각 반응"이 측정 가능한 요구로 승격되지 않음. 이 기능의 존재 이유 중 하나가 "이벤트와 동시에 브리핑"인데, 명세의 디바운스 2초 + 추론 3초 설계가 이 의도와 정면으로 트레이드오프됨에도 화해/정량 기준이 없다. 검증 단계에서 "ref 충족"을 판정할 수 없고, 구현 후 "느리다"는 회귀가 사양 위반인지 판별 불가.
② (F-4) 프롬프트 인젝션 방어의 수용 기준이 단일 프로브로 과소. 신뢰 불가 데이터(특히 메일 본문)를 LLM에 넣는 것은 이 배치가 새로 들이는 가장 큰 신뢰 경계 위험이다. "구분자 블록 + 프로브 1건"은 방어 성공의 판정 기준으로 불충분하며, 가이드 출력의 링크/명령 처리 정책이 비어 있다.
두 약점 모두 설계를 막는 결함은 아니다(구조·추적성은 견고). 그러나 기획 산출물이 "검증 가능한 수용 기준"을 제공해야 한다는 기준에서, 이 기능의 정체성(즉각성)과 최대 위험(인젝션)에 대한 수용 기준이 모호한 것은 P1로 본다. 두 항목을 보강하면 아키텍처 단계로 진행 가능.
