5. 구현 게이트 이월 (설계 PASS ≠ 코드 PASS)

설계 청사진은 PASS이나, 다음은 코드 보안 검증 단계에서 설계 의도대로 구현되었는지 반드시 실증해야 한다(본 검증의 범위 밖, 추적 항목).

  • C-1 (H-1 해소 실증): 라우터에 rescan=POST한정+checkStateChange, scan-status=GET한정+checkReadAccess가 실제 적용되고, 토큰 누락/오류 시 403이 나는지(특히 checkReadAccess 신규 함수가 safeEqual 상수시간 비교를 쓰는지) — V-1과 함께.
  • C-2 (H-2 해소 실증): isUnderSystemDir가 canonicalize 직후·push 전에 호출되고, 8.3 단축명(PROGRA~1)·junction·\\?\·대소문자·UNC 입력 각각에 대해 C:\Windows/C:\Program Files 순회가 실제 차단되는지 — V-2 적대적 입력 실측 케이스에 추가.
  • C-3: currentPathAbs가 응답·로그(L-3) 어디에도 새지 않는지(절대경로 누설 0), 프론트가 ScanProgress 전 문자열 필드를 innerHTML 없이 textContent로만 렌더하는지.
  • C-4: start() 예외/watchdog 경로에서 running이 반드시 false로 복귀하는지(데드락 회복) 회귀 테스트(N-05).
  • C-5: all-drives + ABS 24 + 전역 entry/시간 상한 조합이 N-01(30초)/N-05를 깨지 않음을 벤치로 실증(M4-M-2, V-2 연장).
PM/아키텍트 메모: §8.5대로 docs/api-contract.md는 본 설계 2회차 PASS 직후 갱신(rescan 202/409·scan-status 신규·size.status 확장·totalBytes 승격·옵션 게이트 비대칭). 설계 확정 전 선반영 금지(계약 표류 방지)는 적절하다.
