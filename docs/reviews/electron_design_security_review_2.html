onProgress 1점 추가 (F-1) — 보안 영향 평가

보안 무영향 확인(코드 접지). v1.1은 "무수정 중계"가 불가함을 인정하고 scanController.start(opts)에 선택적 opts.onProgress 1개를 추가하는 최소 수정으로 정정했다. 실측 검증 — ① 167행 onProgress:(p)=>this._merge(p,myScanId)가 실재(설계 인용 정확). ② v1.1 의사코드는 내부 _merge를 그대로 호출한 뒤 opts.onProgress(this.status())를 추가 호출 — push 페이로드가 status()를 거치므로 shortenPath 축약된 currentPath만 전달, 절대경로(currentPathAbs)는 push되지 않는다(M4-H-1 자동 보존). ③ this._isActive(myScanId) 가드로 버려진(orphaned) run의 push도 차단(L113/116/183/196 세대 가드와 정합). ④ 사적 _merge를 main이 직접 호출하지 않아 캡슐화 위반(F-9) 회피. 결론: 새 정보노출·새 신뢰경계 침범 없음 — push는 main→renderer 단방향이며 페이로드는 기존 status() 출력과 동일 shape.

RC-2 (부분신뢰 스캔 대상) 유효성

RC-2 그대로 유효. §1 danger-box·§6.3 danger-box·§13 가정에 "데스크톱 전환은 신뢰 불가 리포 위협을 줄이지 않으며 TB-B 통제가 1차 방어선·필수 보존, nodeIntegration:false·sandbox로 renderer 폭발반경 추가 축소"가 유지된다. v1.1은 오히려 §4.2 전제에서 "renderer 저장형 XSS 성립 가정"을 명문화해 RC-2 인식이 더 강해졌다. RC-1(동일 머신 타 사용자 비격리)도 userData/설치 디렉터리 분리·0600 심층방어 유지로 베이스라인 수용 그대로.

4. 신규 위협 재도출 (STRIDE) — 결과 없음

app:// 커스텀 프로토콜프로토콜 핸들러가 public/ 밖 경로 traversal 허용 시 임의 파일 서빙(I/T)신규 아님(구현 게이트 이월). 설계가 public/ 루트 매핑을 명시. 단 핸들러의 .. traversal 차단·정규화는 EC-2에 이미 포함(file:// 자산 로드+CSP 실차단 실증). 설계 청사진 결함 아님
senderFrame 검증검증 우회·신뢰 origin 오판(S/E)신규 아님. 단일창·원격 미로드·will-navigate/webview 차단 전제에서 심층방어 추가. 실효성은 EC-3 실증 이월
fuses / RUN_AS_NODE 차단차단 미적용 빌드 산출(T)신규 아님. 설계 베이스라인 고정. 빌드 산출물 반영 여부는 EC-4 실증 이월
addRoots 텍스트 입력 잔존임의경로 입력으로 화이트리스트 밖 정보노출(I)신규 아님. canonicalize/realpath·디렉터리·시스템폴더 제외·dedup 통과분만 영속·rejected는 고정 토큰만(절대경로 외 내부정보 비노출). browseDir과 달리 열람이 아닌 등록이며 H-1 모델 정합
onProgress pushrenderer로 절대경로/내부 상태 유출(I)신규 아님(§3). status() shortenPath·_isActive 가드로 차단
재도출 결론: v1.1 변경으로 인한 신규 위협 0건. 위 가설들은 모두 설계가 이미 통제 방향을 잡았거나 구현 게이트(EC-2~EC-4)에서 실증할 항목으로, 설계 청사진 수준의 결함이 아니다. 따라서 PASS 상향을 막는 잔여 항목이 없다.

5. 구현 게이트 이월 (설계 PASS ≠ 코드 PASS) — 명시

설계 보안 게이트 PASS는 청사진의 위협 모델 정합을 뜻할 뿐, 통제의 실효성은 코드 보안 검증에서 실증해야 한다. 아래 항목은 v1.1 §13·§10에 이미 이월 명시돼 있으며, 본 검증은 이를 차단 아닌 이월로 재확인한다.

  • EC-1 (EM-H-1 실증): 전 IPC 핸들러가 인자 스키마(타입·길이·개수·키 화이트리스트)를 main에서 거부, addRootsResolve가 시스템폴더/비디렉터리/중복을 rejected 토큰으로 거부, removeRoot가 canonicalize 정확 일치만 제거, preload가 범용 invoke/ipcRenderer 미노출인지 회귀·스모크.
  • EC-2 (EM-M-1 실증): app://(또는 file://) 자산 로드 성공 + 프로토콜 핸들러의 public/ 밖 traversal 차단 + CSP 메타+헤더가 인라인/외부/data 스크립트를 실제 차단(오염 메타 픽스처로 XSS 미발화).
  • EC-3 (EM-M-2 실증): will-navigate/setWindowOpenHandler/will-attach-webview 차단 동작 + ipcMain의 senderFrame 검증이 비신뢰 프레임 거부.
  • EC-4 (EM-M-3 실증): fuses(RunAsNode off·OnlyLoadAppFromAsar·asar integrity) 빌드 반영 + ELECTRON_RUN_AS_NODE 기동 차단.
  • EC-5 (EM-L-2 실증): sandbox:true 상태 전 기능 동작 스모크(preload가 우리 모듈 require 안 함).
  • EC-6: CLI(scan.js)↔앱 캐시 경로 일치(ADR-EM-4/Q3) 통합테스트.
  • V-1 / V-2 (베이스라인 유효): TOCTOU(V-1)·적대적 입력 실측(V-2)은 도메인 코드 보존으로 그대로 유효. scanController onProgress 1점 추가에 대한 단위테스트(_merge 동작 보존·콜백 전달·orphaned run push 차단)를 추가할 것.
의존성 비고: 런타임 dep 0이나 electron·electron-builder(devDep)는 공급망/취약점 추적 대상이다. 구현 착수 후 npm audit를 코드 게이트에 포함하고 Electron 보안 패치 추적을 운영 항목으로 둘 것(현 단계는 설계 청사진이라 audit 미실행).

6. 확인 필요 (위험 수용 결정) — 모두 v1.1에 명시 수용됨

  • browseDir(Q2): 1회차 "드롭 권고/PM 명시 결정 필요" → v1.1에서 드롭 확정(ADR-EM-6·§13 Q2). 위험 수용 결정 불요(채널 자체 제거).
  • 미서명 배포(Q5): 1회차 "PM 명시 수용/서명 시점 결정 필요" → v1.1에서 위험 수용 확정(ADR-EM-5·§7.2 warn-box·§13 Q5), fuses·integrity·RUN_AS_NODE 무력화로 런타임 악용 표면 축소·SmartScreen 안내 포함·외부 배포 확대 시 서명 재검토 명시.
  • RC-1/RC-2: 베이스라인 수용 그대로 유효. 추가 조치 불요(RC-1 userData/0600 심층방어·RC-2 TB-B 통제 보존).
본 2회차에는 보안 게이트 통과의 전제가 되는 미결 위험 수용 결정이 없다(차단 항목 0). 따라서 AskUserQuestion으로 상신할 중대 결정 사항도 없다. Q3(앱 데이터 경로 정합)은 무결성 관련 구현 확인 항목(EC-6)으로 차단 아님.
