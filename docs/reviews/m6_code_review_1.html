2. 개선 항목 (P2)

P2-1 — setFavorite/setOrder/setSortMode의 read-modify-write 동시쓰기 경쟁

무엇uiState IPC 3개 핸들러가 모두 store.read() → 메모리 수정 → store.write() 패턴이다. 동일 채널이 빠르게 연속 호출되면(드래그 직후 즐겨찾기 토글 등) 두 핸들러가 같은 디스크 상태를 읽고 각자 다른 키를 갱신·쓰기 → 나중 쓰기가 먼저 쓰기의 변경을 덮어쓸 수 있다(lost update).
위치electron/ipc/uiState.js:47-95
왜 문제render 측은 낙관적 반영 후 응답으로 재동기화하므로 화면은 곧 일관되나, 디스크의 ui-state.json이 일시적으로 한 필드를 잃을 수 있다. IPC handle은 직렬화되지만 read가 동기 statSync/readFileSync라 단일 호출 내 race는 없고, 서로 다른 채널 호출 사이에 메모리 캐시가 없어 매번 디스크를 재독한다. 현재 모든 핸들러가 동기이고 Electron ipcMain.handle은 순차 처리하므로 실제 손상 가능성은 낮으나(P1 아님), in-memory 단일 상태 캐시가 없는 설계가 미래 비동기화 시 취약하다.
권고uiState IPC가 ctx 레벨에서 1개 in-memory 상태를 보유하고 write 후 그 캐시를 갱신하거나(read는 캐시 우선), 최소한 본 동기 가정(핸들러가 동기 read-write)을 모듈 주석에 불변식으로 명시. 현재 구조 유지 가능하나 향후 async store 도입 시 lost-update 회귀 위험을 문서화.

P2-2 — resolveBin 절대경로 + {force:true} 시 _binCache 오염

무엇resolveBin(absPath,{force:true})는 캐시 읽기는 건너뛰지만 결과를 _binCache.set(name, resolved)(safeExec.js:89)로 여전히 캐시에 기록한다. resolveTool/tools.js는 항상 force라 영향 없지만, 향후 비-force 호출자가 이 절대경로 키를 조회하면 등록 시점의 stale 결과를 받는다.
위치lib/common/safeExec.js:82-90
왜 문제M6-H-1의 취지(TOCTOU·캐시 신뢰 제거)에 비추면 사용자 지정 절대경로는 캐시에 남기지 않는 편이 안전하다. 현재 M6 경로는 전부 force라 결함이 발현되지 않으나(P1 아님), 캐시에 사용자 경로가 남는 것 자체가 설계 의도(매 실행 fs 재확인)와 미묘하게 어긋난다. setToolPath가 clearBinCache()로 전체를 비우므로 누적은 방지되지만, 그 사이 비-force 조회는 stale.
권고safeExec.js의 절대경로 분기에서 force일 때는 _binCache.set을 생략(또는 절대경로는 아예 캐시 키로 쓰지 않음). 단일 라인 변경. lib 무수정 원칙과 충돌하나 설계가 "사용자 지정 경로는 캐시 신뢰 금지"를 명시하므로 정합 개선.

P2-3 — commitReorder의 setOrder + applySortMode 중복 영속

무엇commitReorder가 applySortMode('reorder')(→ setSortMode('manual') IPC) 호출 후, 곧이어 setOrder(merged) IPC를 호출한다. setOrder는 이미 서버에서 sortMode='manual'을 동반 영속하므로 setSortMode 쓰기가 불필요한 추가 디스크 쓰기다.
위치public/app.js:2586-2589 (applySortMode→2549), uiState.js:79
왜 문제매 드래그/이동마다 ui-state.json에 2회 원자적 쓰기(setSortMode + setOrder)가 발생. 정확성 결함은 아니나 불필요한 쓰기·중복 경로(P2-1 동시쓰기 경쟁 표면도 키운다). 또한 두 쓰기 사이에 P2-1 lost-update 창이 생긴다.
권고reorder 흐름에서는 메모리 sortMode만 'manual'로 두고 IPC는 setOrder 한 번만 호출(setOrder가 manual 동반). applySortMode('reorder')의 영속 호출을 생략하거나, reorder trigger 시 setSortMode IPC를 스킵.

P2-4 — mergeToolPath의 normalizeTools 재통과가 비-known id를 무음 소거

무엇mergeToolPath가 새 맵을 normalizeTools(next)로 한 번 더 통과시킨다. setToolPath는 진입부에서 이미 isKnownToolId로 거부하므로 안전하지만, 만약 기존 config.tools에 (과거 버전·수동 편집으로) known이 아닌 항목이 섞여 있으면 이 재정규화가 그것을 조용히 드롭한다.
위치electron/ipc/tools.js:89-99
왜 문제이는 설계상 의도된 단일 화이트리스트 원천(known id만 영속)과 정합하므로 동작은 옳다(P1 아님). 다만 "다른 항목 보존"이라는 함수 주석(line 86)과 실제 동작(비-known 드롭)이 미묘하게 어긋나 오해 소지가 있다.
권고주석을 "known id 항목만 보존·정규화(비-known은 화이트리스트 정책상 드롭)"로 정정. 동작 변경 불필요.

P2-5 — favoriteViewModels / applyOrder 구조 중복

무엇favoriteViewModels(app.js:789)와 applyOrder(app.js:726)가 "id 배열 → byId Map 구성 → used Set으로 순서대로 픽" 패턴을 거의 동일하게 반복한다.
위치public/app.js:726-744, 789-800
왜 문제중복 로직. applyOrder는 뒤에 신규 append가 추가되는 차이만 있다. 유지보수 시 한쪽만 고칠 위험.
권고"id 순서대로 vm 픽" 공통 헬퍼(pickByIds(vms, ids, {appendRest}))로 추출. 둘 다 호출. 순수 함수라 단위테스트 그대로 통과.

판정: PASS — 정확성 결함(P1) 0건. M6 신규/변경 코드는 설계 rev.2의 게이트를 충실히 구현했고, 도메인/전송 분리·순수함수+주입·테스트 품질이 양호하다. P2 5건·P3 7건은 통과를 막지 않는 개선·정리 항목이다. 가장 가치 있는 후속은 P2-3(reorder 중복 영속 제거)과 P2-1(uiState 동시쓰기 불변식 명시/단일 캐시), P2-2(절대경로 force 캐시 오염)다.
