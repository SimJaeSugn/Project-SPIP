BUG-1 중간 계약 에러응답 표에 CODE_CLI_NOT_FOUND 누락

무엇이: 구현이 반환하는 {ok:false, code:"CODE_CLI_NOT_FOUND"}·{code:"OPEN_FAILED"}가 계약 문서 에러표에 명시되지 않음.

어디서: docs/api-contract.md §에러응답 ↔ lib/server/actionHandlers.js:108-135

기대 vs 실제: 기대=계약이 단일 출처로서 모든 응답 code를 망라 / 실제=200 ok:false 분기 2종이 미문서화.

재현: code CLI 미설치 환경에서 POST /api/open → 200 CODE_CLI_NOT_FOUND (시나리오 2-5).

영향: 동작 정상(크래시 없음). 계약-구현 동기화 측면의 문서 부채.

조치: 계약 에러표에 200 ok:false code 2종 추가(문서 갱신, PM 중재 P3-3).

BUG-2 낮음 프론트 describeError가 CODE_CLI_NOT_FOUND 미매핑

무엇이: code 미설치 시 사용자 토스트가 친절한 한국어 대신 default 분기로 노출됨.

어디서: public/app.js:373-382 (describeError switch) ↔ openProject 359행.

기대 vs 실제: 기대="VS Code CLI를 찾을 수 없습니다" 류 안내 / 실제=res.ok && data.ok가 false이므로 "열기 실패: HTTP 200" 형태로 표기될 수 있음(서버 message는 사용되지 않고 status 기반 default로 빠짐).

재현: code CLI 미설치 PC에서 카드 "VS Code로 열기" 클릭 → 토스트 문구 확인.

영향: UX 사소(기능 동작엔 무관). 사용자 혼란 가능.

조치: describeError에 CODE_CLI_NOT_FOUND/OPEN_FAILED case 추가 또는 서버 message 직접 표시. → frontend-dev.

BUG-3 낮음 CSP 정책이 두 출처(meta vs 응답 헤더)에서 미세 상이

무엇이: index.html 내 CSP meta와 staticHandler가 주입하는 Content-Security-Policy 헤더의 지시문이 완전 동일하지 않음(헤더는 img-src 'self' data:, font-src 'self' 포함 / meta는 img-src 'self', font-src 미지정).

어디서: public/index.html:7-8 ↔ lib/server/staticHandler.js:49-59

기대 vs 실제: 기대=동일 정책 / 실제=브라우저는 둘 중 더 엄격한 쪽을 합집합으로 적용 → 현재 자산(self만 사용)엔 실문제 없음.

재현: 두 파일의 CSP 문자열 비교. 현재 앱 자산은 모두 'self'라 렌더 차단 없음.

영향: 현시점 동작 무해. 향후 자산 추가 시 혼선 소지. (세부 보안 평가는 security-reviewer 위임)

조치: 두 CSP를 단일 상수로 일원화 권고.

debugger 회부 권고: 해당 없음. 간헐적/경쟁조건/원인불명 결함은 관측되지 않았다(테스트·E2E·벤치 모두 결정적 재현).

7. 미검증·범위 외 항목
