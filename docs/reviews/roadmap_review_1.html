1. 판정 및 요약

PASS

요약: 로드맵은 설계 트리 22개 모듈을 빠짐없이 8단계에 배치하고, MVP 16건 전부를 구현 단계에 매핑하며, 단방향 의존(진입점→도메인→공용) 순서를 위반 없이 지킨다. S4 계약 게이트→S5/S6 병렬 분기, 보안 게이트(H/M/L/V) 배치, 측정 가능한 출시 게이트(95%/30s/200ms) 모두 타당하다. 그대로 S0 세부 계획에 착수 가능하다. 다만 실행 정밀도를 높일 경미한 보완 항목(P2 2건·P3 4건)이 있으며, 이는 게이트 차단 사유가 아니다(권고).

우선무엇 / 위치문제권고
P2-1 R-13 단계 귀속 모호 / §1 요약표 S4행·§8 추적표 R-13(graceful)이 §1 요약표에서 S2·S4·S5 세 단계에 모두 표기되나, S4는 "코드 없음" 계약 게이트다. R-13이 S4에서 무엇을 산출하는지(계약상 status/오류코드 명세인지) 불명확해 추적 혼선 소지. S4의 R-13은 "오류·status 계약 명세(코드 아님)"임을 명시하고, 실제 구현 귀속은 S2(수집기 na)·S5(snapshotStore graceful)로 한정해 §8 추적표와 어휘를 통일하라.
P2-2 safeExec 골격↔실측 책임 분할의 의존성 명시 / §1 요약 S0·S2 및 §4 S2 safeExec가 S0(골격)와 S2(실측)로 쪼개져 H-2/M-4가 두 단계에 걸친다. S2 git collector 작성 시 resolveBin 실측이 선행돼야 하는데, S2 내부 작업 순서(safeExec 실측 → git.js)가 단계 내 의존으로만 암시될 뿐 명문 순서가 없다. S2 DoD에 "safeExec 실측 완료가 git.js 통합의 선행"임을 작업 순서로 명시하라(현재 ④ resolveBin 실측이 ② git 시뮬레이션 뒤에 나열돼 순서 오해 가능).
P3-1 S7 탐지율 벤치의 모집단 정의 / §4 S7·§7.3 탐지율 ≥95%는 "실제 N + 미끼 M" 픽스처로 측정한다고만 적혀 N·M 규모와 미끼 구성 기준이 미정. DoD가 측정 가능하려면 분모가 고정돼야 한다. S7 또는 S0 픽스처 정의 시 N·M 최소 규모와 미끼 유형(빈 폴더·node_modules 잔재·중첩 프로젝트 등)을 픽스처 명세로 확정하라(PRD §3 기준과 연결).
P3-2 Node 16 node:test 폴백 트리거 기준 / §7.1 "S0에서 실측, 미흡 시 순수 assert 폴백"은 방향은 옳으나 "미흡"의 판정 기준(어떤 기능 부재·어떤 플래그 동작 시 폴백)이 정성적이다. S0 DoD에 폴백 결정 기준을 명문화하라(예: --test 러너가 Node 16.x에서 플래그 없이 스위트 실행·종료코드 반영에 실패하면 폴백). 결정 시 ADR로 근거 남김.
P3-3 S5/S6 병렬 시 계약 변경 처리 절차 부재 / §3·§4 S4 S4 게이트 통과 후 S5·S6 병렬 진행 중 계약(shape/오류코드)을 부득이 변경해야 할 때의 재합의 절차가 없다. 병렬 단계에서 경계면 드리프트가 재발할 위험. S4 산출물(api-contract.html)에 "버전/변경 시 PM 재중재 + 양측 재합의" 경량 변경관리 한 줄을 추가하라.
P3-4 N-01 "품질기준" 단계 표기 일관성 / §1 요약표 S7행 §1 요약표 S7 구현 요구 ID에 "N-01(품질)"로 표기돼 마치 S7에서 N-01을 신규 구현하는 것처럼 읽힐 수 있으나, 실제 N-01은 S1·S2·S3의 구조적 선택(스캔/서빙 분리·제외 가지치기)의 누적 결과를 S7에서 측정·게이트하는 것이다. S7 요구 표기를 "N-01(측정·게이트)"로 명확화하고, N-01의 구조적 충족 근거가 S1~S3에 분산됨을 §8 추적표 비고로 남기라(현재 §8은 N-01을 S7 단독 배치).

4. 확인 필요 (사용자/PM 판단)

  • 중대 쟁점 없음. 위 반영 항목은 모두 게이트 비차단 권고로, PM 상신이 필요한 블로커는 발견되지 않았다.
  • 참고: 보안 검증 2회차는 V-1·V-2를 "설계 PASS를 막지 않는 추적 항목"으로 분류했으나, 로드맵은 이를 "MVP 출시 차단 게이트"로 강화했다. 이는 더 보수적인 선택으로 모순이 아니며, PM이 출시 리스크 관리 차원에서 의도된 강화임을 한 번 확인해 두면 좋다.
