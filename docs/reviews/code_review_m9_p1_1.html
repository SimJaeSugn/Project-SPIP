판정 · FAIL (정확성 결함 1건)

요약. 순수 정본(renderGuard.js)과 테스트는 견고하고 잘 문서화돼 있으며 node:test 12건 전부 통과한다. 그러나 app.js 통합부에서 설계가 명시한 "보류 해제 지점 → flushIfPending()" 규칙을 일부만 구현해 coalesce 큐가 처리되지 않는 경로가 있다(정확성 결함 1건 — 높음). 품질 개선 4건(중 2 / 낮 2), 확인 필요 2건.

1. 정확성 결함 (높음)

항목분석
결함
F-1
flush 미호출 보류 해제 경로		 무엇. 설계 §3.4·§3.5 및 데이터플로 다이어그램(m9-design.html:332-333)은 보류 해제 지점 = compositionend · Sortable onEnd · closeSettings/closeHelp/closeDrawer 모두에서 RG.coalesce.flushIfPending()를 부르도록 규정한다. 구현은 compositionend 단 한 곳에서만 호출한다 (app.js:5242).
위치. 누락: 드래그 onEnd app.js:4892-4893(_dragging=false 후 flush 없음, 설계 §3.4 R4가 명시적으로 "onEnd → 마이크로태스크 안에서 flushIfPending" 패턴을 요구) · closeDrawer app.js:4070-4081 · closeSettings app.js:2060-2069 · refreshMailSummary busy 해제 app.js:1951 · refreshCommitActivity busy 해제 app.js:1974 · todoAdding=false app.js:1830.
왜 문제. 보류 중(드래그/모달/busy) applyProjectsUpdate가 들어오면 데이터는 viewModels에 병합되지만 render()는 보류되고 pending=true로 큐에 남는다(app.js:4614-4615). 해제 지점에서 flush를 부르지 않으면 누적분이 즉시 화면에 반영되지 않는다. close* 류는 자체 render()를 직접 부르므로 데이터는 따라오지만(우연한 보정), pending·예약 타이머는 정리되지 않은 채 남는다. 특히 드래그 onEnd는 자체 render도, flush도 없어 보류 해제 후 다음 트리거(다음 push/검색 입력)까지 갱신이 지연된다 — 수용기준 "보류 해제 시 정확히 1회 병합 반영"(설계 line 256) 위반.
권고. 각 보류 해제 지점에 RG.coalesce.flushIfPending() 추가. 드래그는 설계 §3.4대로 onEnd에서 _dragging=false 직후 Promise.resolve().then(() => { commitReorder(...); RG.coalesce.flushIfPending(); }) 순서로(현 마이크로태스크 패턴에 흡수). close*/busy 해제 지점은 기존 직접 render() 호출을 flushIfPending()로 치환하거나, 데이터 즉시성이 필요하면 render() 후 큐 정리를 위해 flushIfPending()(이미 render 됐으면 pending 소비)를 보조 호출. 대안(권장): coalesce 래퍼에 정본의 hasPending()를 노출하고, 보류 해제 헬퍼 1개(RG.coalesce.releaseAndFlush())로 단일화해 호출처 누락 위험 자체를 제거.

2. 동형 드리프트 위험 (R7) 평가

정본 renderGuard.js의 shouldDeferRender/createCoalescer와 app.js의 인라인 동형 복제를 라인 단위로 대조했다.

대조 항목결과
shouldDeferRender OR식완전 동형. 6플래그·OR순서·!!(f && ...) 가드까지 동일(renderGuard.js:24-25 ↔ app.js:5216-5217).
request/onFire/flushIfPending 상태전이동형. cancelTimer만 인라인화됐을 뿐 분기·순서 동일(renderGuard.js:62-89 ↔ app.js:5321-5342).

평가: 현 시점 드리프트 없음(동형 일치). 규칙 주석도 명시돼 신호는 충분하다. 다만 주석 규칙은 강제력이 없다 — 동형 동치를 자동 검증하는 테스트가 없어, 한쪽만 바뀌어도 CI가 잡지 못한다. 설계 R7("동형 테스트")이 요구한 동치 회귀 테스트가 누락됐다(아래 D-3).

D-1
코드 중복(어댑터가 복제)		 무엇/위치. app.js의 _coalescer(app.js:5321-5342)는 정본 createCoalescer 본문을 통째로 재작성한 것이다. 정본은 타이머 주입(setTimer/clearTimer)을 지원하므로 app.js는 래핑만으로 재사용 가능한데 굳이 알고리즘을 복제했다.
왜 문제. 동형 드리프트 표면적의 근원. 주석으로 "변형 금지"를 거는 것보다 코드로 단일 출처를 강제하는 게 견고하다.
권고. app.js에서 renderGuard.js를 직접 쓸 수 있으면(번들 없음 → 불가하면 X) RG._coalescer = createCoalescer({ isDeferred: deferred, flush: render, delay: 120 })로 교체. 순수 프런트엔드 제약상 require 불가라면 D-3(동치 테스트)로 대체 방어. 실현 가능성은 빌드 구조에 달림 — 확인 필요 C-1.

4. 정확성 정밀 점검 (정본 createCoalescer)

  • 타이머 이중 예약: if (timer == null) 가드로 방지됨(renderGuard.js:80). 정상.
  • 타이머 누수: 보류 진입 시 cancelTimer(), onFire/flush에서 timer=null 일관. 정본 내부는 누수 없음. (누수 위험은 app.js teardown 미정리 D-2뿐.)
  • flush 후 pending 리셋: onFire·flushIfPending 모두 pending=false 후 doFlush — 1회 소비 보장(renderGuard.js:71-72, 87-88). 정상.
  • 발화 직전 보류 시작: onFire의 isDeferred() 재검사로 차단(renderGuard.js:69) — 테스트로 검증됨.
  • shouldDeferRender 매핑: 6플래그 ↔ deferred() 수집(app.js:5223-5230) 1:1 일치, 오타·누락 없음. truthy(overlayOpen=selectedId 문자열) 허용도 테스트로 커버.

정본 로직 자체에 정확성 결함 없음. 결함은 전적으로 통합부(F-1, 호출처 누락)에 있다.

판정 근거

정본 로직·테스트는 통과 품질이나, 설계가 규정한 보류 해제 flush 규칙이 통합부에서 일부만 구현돼 "보류 해제 시 정확히 1회 반영" 수용기준을 충족하지 못한다(F-1, 정확성 결함). 따라서 FAIL. F-1 해소(보류 해제 지점 flush 보강) 시 통과 가능하며, D-1~D-3로 동형 드리프트를 코드 차원에서 방어하면 견고성이 크게 오른다.
