4. 항목 3 — 교차 시나리오 (단계 교차, 코드 경로 검증)

#판정시나리오충돌 차단 근거
①PASS 커밋 폴링(R-31) 발화 순간 섹션 드래그(R-32) 진행 중 드래그 onStart가 _dragging=true → deferred()=true. 폴링은 maybeAutoRefreshCommit의 if(RG.deferred())return으로 1회 스킵. push가 들어와도 coalesce가 pending만 누적, 발화 시 재검사로 보류. 드래그 종료(onEnd) 후 마이크로태스크 지연으로 RG.widget.destroyAll이 진행 중 Sortable을 파괴하지 않고, reorder면 commit, 아니면 flushIfPending로 1회 반영. 드래그 세션 비파괴.
②PASS 설정 모달(R-30) 열린 중 폴링/push 도착 overlayOpen이 showSettings를 포함 → deferred()=true. 폴링 스킵·push는 데이터만 병합 후 보류. render() 자체가 모달을 다시 그리되 모달은 showSettings 유지로 닫히지 않으며, 보류로 인해 폴링 중 재렌더 빈도가 0. 모달 안 닫힘.
③PASS 프로젝트 검색 한글 조합(R-26) 중 push 도착 compositionstart → _composing=true → deferred()=true. push는 coalesce.request()로 pending만 적재, 발화 시 재검사로 보류 → input 노드 교체(replaceChildren) 미발생. compositionend에서 _composing=false + flushIfPending()로 1회 반영. 자모 분리 차단 경로 성립.
④PASS SVG 차트(R-33)가 폴링 갱신마다 재생성 — 누수 0 차트는 RG.widget('commitChart')로 등록. render() 1회당 destroyAll(이전 인스턴스 destroy(): 등록된 모든 mouseenter/leave/focus/blur 핸들러 removeEventListener) → mountAll(현재 DOM의 .commit-chart-host에만 1회 부착, 이미 살아있으면 skip). 핸들러 누적·중복 노드 0 경로 성립.
⑤PASS 헤더 통일(R-27)·단축키(R-29)·메뉴제거(R-28) 홈/프로젝트 양 뷰 정합 renderHeader() early-return 제거 — 브랜드→탭→(검색)→공통액션 단일 경로. 뷰별 차이는 headerViewConfig().showSearch 1가지(프로젝트 전용). 단축키는 SHORTCUTS 단일 상수를 안내표(renderShortcutSettings)·keydown(matchShortcut)이 동시 참조. 메뉴는 Menu.setApplicationMenu(null) + preload onMenu/app subscribeMenu 양방향 제거(죽은 채널 0). 두 뷰 정합.
교차 시나리오 충돌 발견: 0건. 5종 모두 단일 RG 게이트가 상호 간섭을 흡수한다. 각 기능이 자체 가드를 만들지 않고 동일 게이트에 수렴해, "한 기능의 라이브 갱신이 다른 기능의 stateful 상태를 깬다"는 통합 리스크가 구조적으로 제거됐다.
