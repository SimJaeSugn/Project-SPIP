요약

M10 P1~P4 구현은 설계 정본(rev.1)의 5단계 patchRegion 계약·위젯-소유 모델(F-2)·8지점 flush(F-3d)·epoch-day diff 가드(Q-M10-1)·툴팁 가로/세로 분리(F-3b)를 충실히 반영했다. 정확성 결함(논리 오류·경계 누락·리소스 누수·이중발화)은 발견되지 않았다. 위젯 소유권은 builderFn=빈 호스트·노드 생성/파괴를 위젯 단독으로 단일화해 이중 삽입·핸들러 누수 0이 코드로 보장된다. 이중 발화는 pending 가드 + busy/deferred 재진입 가드 + 단조 boolean으로 다중 방어된다. 개선 1·사소 4는 모두 테스트 커버리지·가독성 차원이며 동작에 영향 없다.

2. 위젯 소유권 · 이중 발화 평가

2.1 위젯-소유 모델(F-2) — 통과

노드 소유권이 단일 모델로 못 박혀 있다: builderFn = 빈 호스트 컨테이너만, 차트 노드의 생성·삽입·파괴·핸들러는 commitChart 위젯 단독. patchRegion 실행 순서(②destroy→④빈 호스트→⑤mount)가 이 모델과 정확히 맞물려 이중 삽입·핸들러 누수가 구조적으로 0이다. init이 root||document 스코프를 받아 _mountById가 containerEl(region)을 root로 전달 → 전역 querySelector 의존 제거(m9_p8 사소 항목 동반 해소)도 확인.

2.2 이중 발화 — 통과

maybeFlushCommitRefresh가 8지점에서 매번 불려도 실제 fetch는 다음 3중 방어로 정확히 보류분이 있을 때만 1회 일어난다:

  • pending 가드(app.js:4920): if(!_pendingCommitRefresh)return — 건너뛴 틱이 없으면 즉시 반환.
  • 재진입 가드(app.js:4922): busyCommitActivity || RG.deferred()면 반환(아직 보류/in-flight면 다음 해제 지점에서 재시도).
  • in-flight 가드(app.js:2257): refreshCommitActivity 진입부 store.busyCommitActivity 재확인으로 중복 fetch 최종 차단.

또한 coalesce는 단조 boolean pending + 단일 타이머로 render 경로의 이중 발화도 막는다(renderGuard.js와 동형). flush(fetch 트리거)와 release(render 트리거)는 독립 경로라 서로 간섭하지 않는다(설계 §6.4 정합). 이중 발화 결함 없음.

3. 동형 드리프트 · 단순화 — 통과

  • 순수 로직 위치(ADR-M9-1): patchRegionPlan·commitActivityKey·tipLeft는 모두 public/app.js에만 존재하고 lib/common/renderGuard.js는 shouldDeferRender/createCoalescer만 export(불변). patchRegion.test.js:120-123이 renderGuard.js에 patchRegion 미진입을 강제. 동형 드리프트 없음.
  • 새 순수 함수 테스트 커버: patchRegionPlan(3분기)·commitActivityKey(7케이스)·tipLeft(5케이스) 모두 헤드리스 단위 테스트 존재. DOM 미접근 순수 분리가 깔끔하다.
  • 8지점 누락 위험 구조화: "release/flushIfPending 옆 한 줄" 규칙을 patchRegion.test.js:113-118이 maybeFlushCommitRefresh() 등장 횟수 >=8로 회귀 방어한다. 완벽한 구조는 아니나(아래 사소 N-2 참조) 현재 누락 0.
  • 죽은 코드: 발견 없음. dispatchMenuAction 등 기존 정리 흔적만 주석에 남음.

개선 test/patchRegion.test.js:52-59 _destroyById/_mountById 런타임 동작이 정적 소스 매칭으로만 검증됨. 설계 §11.1 M10-WG-1은 "존재 id→destroy 1회·instances[id]=null, 없는 id→no-op, 이미 살아있으면 skip, init throw→null 회복"의 동작 검증을 명시했으나 현재 테스트는 정규식으로 정의 존재·skip 가드 문구만 확인한다. jsdom 0-의존 정책은 이해하나, 이 두 메서드는 fake spec({id, init:()=>obj, destroy})을 define한 뒤 호출하는 식으로 DOM 없이도 라이프사이클 멱등성을 헤드리스로 검증 가능하다(rootEl은 미사용 또는 stub). 권고: RG.widget을 테스트에 노출(이미 일부 RG가 export됨)하거나 _specs/_instances 동작을 작은 순수 헬퍼로 분리해 멱등·throw 회복을 실제 호출로 커버. 정확성 결함은 아님(코드 자체는 정확) — 회귀 안전망 강화 차원.

6. 확인 필요(설계 의도)

  • N-2 관련: 8지점 산포를 그대로 둘지(현 설계 §6.4 "fetch/render 트리거 독립" 의도) vs release 래퍼로 응집할지는 설계 의도 확인 사항이다. 현 구현은 설계 정본을 정확히 따랐으므로 코드 리뷰 판정에는 영향 없음 — tech-lead/아키텍트가 후속 단순화 여부만 결정하면 된다.
위 개선·사소 항목은 전부 동작·정확성에 영향 없는 테스트 커버·가독성·문서 정합 차원이다. M10 코드는 설계 정본(rev.1)의 모든 확정 결정(F-1·F-2·F-3a·F-3b·F-3d·Q-M10-1)을 정확히 구현했다.
