1. 요약

M9 설계는 신뢰 경계를 확장하지 않는다. 새 공격 표면은 IPC 채널 spip:setHomeLayout 1개뿐이며, 이 채널은 경로·실행 인자·해시가 아닌 고정 enum(섹션 id) 배열만 받는다. 설계는 기존 검증 패턴(guard senderFrame 정확 일치 · normalize 화이트리스트 · 0600 원자적 write)을 일관되게 재사용하며, L-1(textContent)·CSP(자작 SVG·외부 0)·죽은 채널 정리(SEC-L1)를 명시적으로 다룬다.

치명·높음 위협은 발견되지 않았다. 다만 설계 문서가 안전을 단언했으나 구현 시 회귀 위험이 있는 두 지점(① normalizeState/toResponse/write에 homeLayout이 일관 반영되지 않으면 데이터 소실·우회, ② chartBars의 SVG text·속성 인젝션 차단을 textContent로 강제)을 중간 등급으로 시정 권고한다. 구현 PR(P7·P8) 게이트에서 테스트로 못 박을 것.

신뢰 불가 입력 메인 검증·영속 preload 경계

STRIDE 요약: 신규 표면에서 의미 있는 위협은 ① 변조(Tampering) — 악의적/손상 상태 파일·임의 IPC 페이로드, ② 정보노출(Info) — 스캔 데이터의 SVG/DOM 표시 시 XSS, ③ 권한상승(EoP) — IPC를 통한 경로/실행 우회. 설계상 ③은 구조적으로 차단(섹션 id는 경로·실행 의미 없음)되고, ①·②는 검증·정규화·textContent로 방어되나 구현 정합성 의존(M-1·M-2).

3. 발견 항목 (등급별)

중간 M-1 — homeLayout 영속 경로(normalizeState/toResponse/write) 일관성 미확정 시 데이터 소실·우회

위치
설계 §6(uiState 스키마), §7(IPC), 파일맵 lib/common/uiStateStore.js·electron/ipc/uiState.js
위협(STRIDE)
변조(Tampering) / 무결성
악용 시나리오
현재 write()는 저장 직전 normalizeState()를 무조건 재적용하고, normalizeState()는 고정 키 집합만 반환한다(현재 homeLayout 없음). 또 toResponse()도 고정 키만 추린다. 설계는 세 곳(defaultState·normalizeState·toResponse)에 homeLayout 추가를 명시했으나, 만약 구현이 setHomeLayout 핸들러에서 write({...state, homeLayout})만 하고 normalizeState에 키를 누락하면, write 내부 normalizeState가 homeLayout을 조용히 버린다 → 저장은 ok 반환하나 영속 실패(무결성 깨짐, 디버깅 난해). 반대로 normalizeState에는 넣고 toResponse에 누락하면 렌더러가 항상 기본 순서로 보임.
심각도 근거
보안 직접 악용은 아니나 검증 우회의 전형(부분 적용)이며, 데이터 무결성 불변식에 해당. 화이트리스트가 한 경로에만 적용되면 다른 경로가 신뢰 못 할 값을 통과시킬 여지.
권고 조치
① normalizeState() 반환 객체에 homeLayout: normalizeHomeLayout(obj.homeLayout)를 반드시 포함(write가 단일 정규화 게이트이므로 여기 누락 시 모든 쓰기가 키를 버림). ② toResponse()에 homeLayout 추가. ③ defaultState()에 homeLayout:[...HOME_SECTION_IDS]. ④ test/uiStateStore.test.js에 "write→read 라운드트립 시 homeLayout 보존" + "미지/중복/길이초과/비배열 입력 정규화" 케이스를 게이트로 추가(설계 §5에 이미 예정 — 라운드트립 보존 케이스를 명시 추가). ⑤ reconcileState()가 homeLayout을 보존하는지 확인(현재 normalizeState 거친 뒤 favorites/order만 손대므로 키만 추가되면 자동 보존되나 테스트로 확인).

중간 M-2 — R-33 SVG chartBars의 <text> 콘텐츠·속성 인젝션을 textContent/숫자형으로 강제(설계 단언의 구현 게이트)

위치
설계 §10(SVG 차트), renderHomeProductivity() → chartBars(days, opts)
위협(STRIDE)
정보노출/스크립트 실행(XSS) — 변조된 클론 리포의 커밋 데이터·언어명이 차트 라벨로 유입
악용 시나리오
차트 데이터(store.commitActivity.days의 label/iso, 언어 추세의 언어명)는 스캔 유래로 신뢰 불가다. 설계는 "라벨은 createElementNS('text') + textContent, 색은 setAttribute/CSSOM, 숫자만 받음"이라 명시해 L-1을 지킨다. 다만 SVG에는 두 가지 추가 인젝션 벡터가 있다: ① <text>에 innerHTML/insertAdjacentHTML로 라벨을 넣으면 <text><a><animate onbegin=...> 등 SVG 내 스크립트·이벤트가 가능(다만 CSP script-src 'self'가 인라인 이벤트 핸들러를 차단하므로 1차 방어 존재). ② 라벨/툴팁 텍스트를 setAttribute로 속성 값에 직접 넣을 때(예: 동적 aria-label, title, 또는 색을 데이터에서 받아 fill에 주입) — 색을 데이터-주도로 받으면 속성 인젝션 표면.
심각도 근거
CSP가 인라인 스크립트·이벤트를 막아 실제 코드실행 가능성은 낮으나, 설계가 "안전"을 단언만 했고 구현 가드(텍스트는 textContent, 수치는 Number 강제, 색은 데이터-주도 아닌 고정 팔레트)가 명세에 못 박히지 않았다. 회귀 시 L-1 위반.
권고 조치
① chartBars는 라벨/툴팁을 오직 el.textContent(또는 text 노드 생성)로만 설정 — innerHTML/insertAdjacentHTML/outerHTML 사용 금지(코드 마커 L-1 병기). ② 수치는 Number()로 강제 변환 후 Number.isFinite 검사하고 막대 높이 계산(비유한·음수 클램프). ③ 색·치수는 데이터에서 받지 말고 고정 팔레트/상수에서 setAttribute. 데이터가 속성 값으로 흐르지 않도록. ④ 툴팁 div(설계상 SVG 외부 div)도 textContent. ⑤ 차트 영역에 foreignObject 미사용 확정(설계가 이미 회피 — 유지). ⑥ 가능하면 신뢰 못 할 라벨(커밋 메시지 등)은 차트에 넣지 않고 날짜(메인 생성 ISO)·수치만 표시.

낮음 L-1 — setHomeLayout graceful 정규화의 침묵 흡수(에러코드 부재) — 모니터링/감사 공백

위치
설계 §7 "잘못된 형태도 정규화가 흡수하므로 에러코드 불필요(graceful)"
위협(STRIDE)
부인(Repudiation)/탐지 공백 — 비정상 페이로드가 로그 없이 흡수
악용 시나리오
설계는 기존 setSortMode/setTheme의 graceful 폴백 패턴과 동형이라 일관적이고, 보안상 위험은 낮다(enum 화이트리스트라 정규화 결과는 항상 안전). 다만 렌더러가 손상되어 대량/비정상 페이로드를 반복 전송해도 탐지 신호가 전혀 없다. guard의 senderFrame 검증이 1차 방어이므로 외부 위협 표면은 거의 없으나, 비정상 입력이 정규화로 흡수되는 사실 자체가 로그에 남지 않으면 침해 분석이 어렵다.
권고 조치
graceful 유지(기존 패턴 일치)하되, normalizeHomeLayout 결과가 입력과 달라진 경우(미지/중복 항목 제거 발생) logger.warn으로 1회 기록 권고(선택). 응답은 정규화된 homeLayout 반환 유지. 페이로드 크기는 preload에서 enum 길이(MAX_HOME_SECTIONS) 이상이면 잘라 전달하는 것도 고려(DoS 표면 0화).

낮음 L-2 — R-32 masonry 재배치 · RG.preserve.patchRegion(adoptNode)가 위험 DOM API를 도입하지 않도록 명시

위치
설계 §3.3(patchRegion ... adoptNode로 이동), §14 리스크(masonry CSS columns/JS 재배치)
위협(STRIDE)
정보노출(XSS) — DOM 재구성 과정에서 신뢰 못 할 데이터의 innerHTML 경유 가능성
악용 시나리오
patchRegion은 "노드를 adoptNode로 이동 후 재배치"라 명시 — adoptNode/appendChild/insertBefore 같은 노드 이동 API는 기존 노드를 옮길 뿐 직렬화/재파싱하지 않으므로 L-1 안전하다(올바른 선택). 위험은 "buildFn"이 새 노드를 만들 때 기존 el() 헬퍼(textContent)가 아닌 innerHTML 기반 템플릿을 쓰는 경우. masonry를 "JS 재배치"로 승격(§14 Q-B)할 때 라이브러리/직접 구현이 innerHTML로 컬럼을 재구성하면 위반.
권고 조치
① patchRegion(containerEl, buildFn, keepSelectors)의 buildFn은 반드시 기존 el() 경로로만 노드 생성(설계 §12에 이미 언급 — 코드 마커로 못 박을 것). ② JS masonry 재배치는 appendChild/insertBefore/adoptNode(노드 이동)만 사용, innerHTML/insertAdjacentHTML 금지. ③ 1차 CSS columns 구현은 DOM 직렬화 없으므로 안전 — 권장 경로 유지. ④ test에는 어렵지만, 코드 리뷰 체크리스트에 "patchRegion 경로 innerHTML 0" 항목 추가.

낮음 L-3 — 드래그 onEnd가 DOM data-id를 읽어 영속할 때 enum 재검증을 렌더러·메인 양쪽에서

위치
설계 §12 "onEnd는 DOM의 data-id를 읽어 enum 검증 후 영속", §7 preload ids.map(String)
위협(STRIDE)
변조(Tampering) — DOM 변조로 화이트리스트 외 섹션 id 주입 시도
악용 시나리오
드래그 후 onEnd가 DOM 노드의 data-id 속성을 순회해 순서 배열을 만든다. 신뢰 못 할 스캔 데이터가 DOM에 들어와 있을 수 있으므로, 렌더러가 모은 배열에 화이트리스트 외 값이 섞일 가능성이 있다. 메인의 normalizeHomeLayout이 최종 방어선(미지 id 제거)이므로 실제 영속은 안전하다 — 신뢰 경계를 올바르게 메인에 둠. preload의 ids.map(String)도 타입 1차 고정으로 적절.
심각도 근거
메인 정규화가 최종 차단하므로 실질 위험 낮음. 다만 렌더러가 비정상 id를 보내도 무시되는 것을 테스트로 보장해야 회귀 방지.
권고 조치
① 메인 정규화(normalizeHomeLayout)를 유일한 신뢰 경계로 명시 — 렌더러 검증은 UX 편의일 뿐 보안 의존 금지. ② test/uiStateIpc.test.js에 "화이트리스트 외 id·과다 길이·비문자열 혼입 → 정규화로 제거" 케이스 추가(설계 §5에 예정 — 적대적 페이로드 케이스 명시). ③ data-id 출처는 섹션 컨테이너의 고정 enum이지 스캔 데이터가 아님을 코드 주석으로.

정보 I-1 — R-28 죽은 IPC 채널 정리(SEC-L1)는 공격 표면 축소 — 양방향 누락 없이 제거 확인

위치
설계 §7·§9.3, 실제 preload.js:160-170(onMenu 화이트리스트 pickFolders|rescan|refresh|about), main.js spip:menu:* 발신
위협(STRIDE)
(축소) — 죽은 수신 채널 잔존은 미래 발신 오용 시 수동 트리거 표면

정보 I-2 — F5 새로고침을 렌더러 keydown으로(globalShortcut 회피) — 올바른 선택

위치
설계 §9.2·§9.3, ADR 부재(F5는 SHORTCUTS 상수)
위협(STRIDE)
(회피) — globalShortcut은 앱 비포커스 시에도 발화 → 의도치 않은 트리거 표면
평가
설계가 F5를 globalShortcut이 아닌 렌더러 keydown(기존 ESC 핸들러 패턴)으로 둔다고 명시 — 앱 비포커스 시 미발화로 안전. matchShortcut(ev) 순수 매핑은 단위테스트 대상(F-3 관례 일치). 새로고침 동작(refreshDashboard)은 기존 핸들러 재사용으로 새 IPC·권한 0.
권고 조치
유지. F5가 브라우저 기본 새로고침(전체 reload)을 호출하지 않고 앱 내 refreshDashboard()만 부르는지 확인(preventDefault) — Electron에서 F5 기본 동작이 전체 페이지 reload면 상태 소실 가능(보안 아닌 UX, 그러나 reload 시 IME/드래그 가드 우회 가능성 점검).
