2. 반영 항목 (FAIL/보완 — P1)

P1-1 정적 자산 서빙 URL 경로가 문서 내에서 자기모순 — 구현 분기 발생

무엇: 대시보드 정적 자산(styles.css·app.js)의 실제 HTTP URL 경로가 설계서 안에서 두 가지로 충돌한다.

위치: §9 API 표(line ~507)는 정적 자산을 GET /static/*로 규정. 그러나 §6.2 staticHandler·§7 디렉터리 트리·§11.1은 자산을 public/ 디렉터리 하위로 두고 "public 루트 밖 차단"만 명시 — URL 프리픽스가 /static/인지 /(루트=public 매핑)인지 불명. 또한 GET /는 index.html(정적)으로 별도 명시되어 있어, index.html이 참조할 <link href="?">·<script src="?">의 경로 규약이 확정되지 않는다.

왜 문제: 구현자가 router에서 /static/styles.css로 매핑할지 /styles.css로 매핑할지 임의 선택하게 되고, index.html의 자산 참조 경로와 staticHandler의 정규화 기준(public 절대경로 하위)이 어긋나면 404 또는 경로 이탈 검증 로직이 불일치한다. R-10 핵심 경로의 구현 분기다.

권고: §9 표·§6.2·§11.1을 하나로 통일하라. 예: "GET / → public/index.html, GET /<asset> → public/<asset>을 public 절대경로 하위로 정규화 후 서빙(/api/* 제외 모든 GET은 staticHandler가 처리)". /static/* 프리픽스를 채택할 경우 디렉터리 트리에 public/static/ 구조를 반영하고 index.html 참조 규약을 1줄 명시.

P1-2 설정 파일(spip.config.json) 저장 위치 결정을 §15 미해결로 미룸 — paths.js 구현 입력 누락

무엇: 설정 파일을 OS 캐시 폴더에 둘지 OS 설정 폴더(%APPDATA% 등)에 둘지가 확정되지 않았다.

위치: §8.2·그림1은 spip.config.json을 캐시 박스 안에 그리고, §7 트리 주석은 "%APPDATA%\project-spip\ (또는 캐시와 동일 베이스)"로 양립, §15 미해결 질문 3번은 "설정 권고는 동일 앱 폴더 하위"라며 결정을 구현팀에 위임.

왜 문제: common/paths.js는 캐시/설정 경로 해석이 핵심 책임(§6 공용 유틸)인데, 그 입력 규약이 미정이면 모듈 시그니처(예: configPath() 반환값)가 흔들린다. N-06은 "결과 JSON만 캐시"로 명시하므로 설정 위치는 N-06이 직접 규정하지 않아 실제로 결정이 필요한 지점이다. "구현팀 결정"으로 두면 paths.js 작성 시점에 다시 막힌다.

권고: 설계 권고("동일 앱 폴더 하위" 또는 "config는 %APPDATA%/XDG_CONFIG_HOME, cache는 %LOCALAPPDATA%/XDG_CACHE_HOME 분리")를 §8.2에 단일 확정값으로 못박아 paths.js의 cacheDir()/configDir() 계약을 결정하라. 미해결 질문에서 본 항목을 내려라(나머지 포트값·언어가중치는 블로커 아님이 맞음).

3. 반영 항목 (보완 — P2)

P2-1 R-11 정렬 키 "용량"이 MVP 데이터에 부재 — 정렬 옵션 표면의 MVP/M4 경계 미표기

무엇: R-11/F-4는 정렬 키로 "이름·최종수정·용량"을 요구하나, 용량(size)은 R-09=M4로 분리되어 MVP 스냅샷에서 size.status='skipped'다.

위치: §6.3 app.js—filter, §9 노트, FeatureSpec F-4 수용기준. 설계서는 "결측값 정렬 말단"은 언급하나 "용량 정렬 키 자체를 MVP UI에 노출할지"는 미명시.

왜 문제: 용량이 전부 skipped인 MVP에서 용량 정렬은 무의미한 컨트롤이 된다. 노출하면 빈 기능, 숨기면 R-11 AC의 정렬 키 목록과 불일치로 보일 수 있다.

권고: §6.3 또는 §9에 "MVP 정렬 키 = 이름·최종수정(stale); 용량 정렬은 size 수집(R-09/M4) 도입 시 활성"을 한 줄 명시하여 R-11 충족 범위를 MVP 경계와 정합시켜라.

P2-2 /api/stats의 totalBytes·총 용량 요약이 MVP에서 항상 null — R-10 "총 용량 요약" AC와의 관계 미설명

무엇: R-10 AC와 F-3은 상단 통계에 "총 용량 요약"을 요구하나, 용량은 M4다. §9 stats shape는 totalBytes|null로 null 허용은 했으나, MVP에서 "총 용량" 칸을 어떻게 표시할지(숨김/"M4"/"—") 미정.

위치: §9 /api/stats 행, §6.3 view, FeatureSpec F-3 수용기준·엣지.

권고: §6.3 또는 §9 노트에 "MVP 통계 요약 = 전체 개수·언어 분포·stale 개수; 총 용량은 R-09/M4 도입 시 추가(MVP에서는 미표시 또는 '미측정')"을 명시. R-10 AC가 M4 항목을 포함한다는 점을 §2 경계처럼 1줄 처리.

P2-3 POST 액션 요청/응답 계약의 비동기·식별 일관성 보강 필요

무엇: (a) /api/open은 {id} 입력(경로 노출 최소화)인데, FeatureSpec F-2 정상흐름은 "식별자(또는 경로)"로 표현 — 설계서가 id로 좁힌 것은 개선이나, id 미존재 시 응답 계약(404? 200 ok:false?)이 §9 오류 표에 없음. (b) R-12 2초 피드백과 §12 "detach 후 즉시 피드백"은 정합하나, 응답이 "VS Code 실행 성공"이 아니라 "여는 중"임이 R-12 AC의 "성공/실패" 표현과 어휘 차이.

위치: §9 API 표·오류 표, §12 성능(열기) 행.

권고: 오류 표에 "id 미존재 → 404 또는 200 ok:false code:'ID_NOT_FOUND'" 행 추가. R-12 "성공"의 정의를 "spawn 시작 성공=실행 중 피드백"으로 1줄 정의해 AC 어휘와 정합(실제 VS Code 창 오픈 확인은 비범위임을 명시).

P2-4 walker 순회 전략(BFS/DFS)과 깊이 안전 상한의 MVP 적용 여부 모호

무엇: §4는 "너비/깊이 순회", §6.1 walker는 "깊이/너비 순회"로 표현이 흔들리고, depthLimit는 §8.2에서 M4 플래그이면서 동시에 "MVP에서도 안전 상한 권고"로 양립.

위치: §4 실행흐름, §6.1 walker, §8.2 depthLimit 행.

왜 문제: R-03(깊이/심링크 안전)은 M4지만 "무한 루프 방지용 안전 상한"은 MVP 견고성(N-05)에 직결된다. M4(설정 가능한 사용자 depthLimit)와 MVP(하드코딩 안전 상한)를 구분하지 않으면 walker 구현 범위가 모호.

권고: "MVP: 무한 루프 방지용 내장 안전 상한(고정값)+방문 inode/실경로 추적은 포함(N-05); 사용자 설정 depthLimit·심링크 정책 정교화는 R-03/M4"로 walker 책임을 2단 분리 명시. 순회 전략 용어도 하나로 통일.

P2-5 snapshotStore 손상/부재 시 응답 코드 일관성 (503 vs 200 빈 상태)

무엇: §9 오류 표는 "스냅샷 없음 → 503 NO_SNAPSHOT", §6.2·§13은 "손상 시 빈 상태로 graceful", FeatureSpec F-3은 스캔 미실행·손상 모두 "빈 상태 안내 + 서버 계속". /api/projects가 스냅샷 부재 시 503을 줄지 200+빈 배열을 줄지가 문서 간 엇갈린다.

위치: §9 오류 표(503), §6.2 snapshotStore, §13 노트, FeatureSpec F-3.

권고: "스냅샷 부재/손상: /api/projects는 200+빈 데이터+상태 플래그(또는 503 일관)" 중 하나로 통일하고, 프런트가 빈 상태 안내를 띄우는 트리거(HTTP 코드 또는 응답 플래그)를 §9에 한 줄 고정. R-13 graceful AC와 정합 확인.

4. 반영 항목 (경미 — P3)

P3-1 입력 산출물에 UserStories.html 참조 — 본 검증 미제공

위치: §2·footer가 docs/UserStories.html을 입력으로 명시하나 본 검증 범위에 포함되지 않았다. 추적성은 PRD/FeatureSpec으로 충분히 성립하므로 블로커는 아니나, UserStories의 스프린트 매핑과 §14 추적성의 정합은 미검증 영역으로 남음.

권고: UserStories 정합은 후속 로드맵 검증 단계에서 함께 확인. 현 설계 판정에는 영향 없음.

P3-2 CSRF/동일출처 검증을 설계가 자체 도입 — 기획 미요구 항목(과부족 아님, 추적 메모)

위치: §11.3·ADR-004·그림3이 POST 동일출처(Origin/Host)+Content-Type 검증을 추가. N-02/N-03은 이를 직접 요구하지 않으나 로컬 API 노출 위협 대응으로 타당. 과설계가 아니라 합리적 강화.

권고: 유지 권장. 상세 적정성·우회 가능성은 security-reviewer 위협모델링 입력으로 넘긴다(설계서도 그렇게 명시 — 적절).

P3-3 /api/rescan(M4)이 MVP 엔드포인트 표에 함께 노출 — 경계 표기는 되어 있음

위치: §9 표에 /api/rescan이 M4 플래그로 포함. ADR-005도 "자리만 확보"로 명시되어 혼입 위험은 낮으나, MVP 라우터 구현 시 미구현 엔드포인트의 응답(404 vs 501 Not Implemented)을 한 줄 정해두면 모호성 제거.

권고: "MVP에서 /api/rescan 미마운트(404) 또는 501 안내" 중 택1 명시(선택 사항).

5. 확인 필요 (사용자/PM 판단)

아래는 생산자(아키텍트)가 결정하면 되는 사항이나, 제품 방향과 닿아 PM 인지가 필요할 수 있다.

  • P1-2 설정 파일 위치: config를 cache와 같은 폴더에 둘지(단순), OS 설정 폴더로 분리할지(관례 부합). N-06은 결과 JSON만 규정하므로 PM/아키텍트 확정 사항.
  • P2-1·P2-2: 용량 정렬 키·총 용량 통계를 MVP UI에 "비활성 노출"할지 "완전 숨김"할지 — R-10/R-11 AC가 M4 항목을 문구상 포함하는 데서 비롯된 UX 결정.

위 항목은 모두 설계 청사진 수준에서 한두 줄로 확정 가능하며, 별도 기획 재검토(요구 변경)를 요하지 않는다. 따라서 본 검증은 AskUserQuestion으로 승격하지 않고 생산자 반영 항목으로 처리한다.
