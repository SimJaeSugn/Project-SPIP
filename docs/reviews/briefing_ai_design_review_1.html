종합 판정

CONDITIONAL PASS

설계는 요구 추적성·실제 코드 정합·단순성 보존이 전반적으로 우수하다. 8개 기능요구 + 3개 비기능요구가 모두 구체적 설계 결정·모듈·테스트로 추적되고, 발신원 이벤트명·elevationState.isElevated()· persistConfigKeys·메일 pass 0600 패턴·RG 순수 상태기 재사용 등 핵심 코드 주장이 실측과 일치한다. ADR-BR-1은 규약 충돌을 정직하게 기록하고 P1 패키징 게이트를 구현 1순위에 박아 RK1을 적절히 통제한다.

다만 실제 코드 구조와 어긋나는 표현 2건(IPC shape 검증 위치 · toResponse 파일 소속/누락)과 설계 내 자기모순 1건(schemaVersion), 비-git 신호 auto-resolve 판정 공백 1건이 구현 착수 전 정정되어야 한다. 이들은 설계 골격을 흔드는 결함은 아니므로 FAIL이 아니라 CONDITIONAL이며, 아래 P1·P2 항목 반영 후 구현 진행 가능하다.

반영 항목 (우선순위순)

P1 F-1 · IPC shape 검증 위치 오기 — 설계 주장이 실제 guard 구조와 불일치

  • 무엇: 설계가 IPC shape(페이로드 화이트리스트) 검증을 "register.js의 guard()가 자동 적용"한다고 표현.
  • 위치: 설계 ③ IPC 계약 §449 ("register.js의 guard()가 senderFrame 정확 일치 + shape 검증을 자동 적용"), ⑧ 설정·보안 §637 ("register.js guard() 경유(senderFrame 정확 일치·shape 화이트리스트)").
  • 문제: 실측상 electron/ipc/register.js의 guard()는 isTrustedSender(event)(senderFrame origin)만 검증하고 try/catch로 감쌀 뿐, 인자 shape 검증은 하지 않는다. shape 검증은 각 핸들러(fn) 내부 또는 preload _mailArgs류 1차 고정에서 수행한다(예: setOrder의 Array.isArray 체크, setFavorite의 ID_RE 체크는 핸들러 본체에 있음). 설계대로 "guard가 shape를 자동 검증한다"고 믿고 구현하면 브리핑 채널의 shape 검증이 통째로 누락되어 N-08 수용기준("senderFrame·shape 검증 통과")을 충족하지 못할 위험이 있다.
  • 권고: ③·⑧ 문구를 "guard는 senderFrame을 검증하고, shape 화이트리스트 검증은 각 briefing 핸들러 본체(또는 preload 1차 고정 + 메인 재검증, _mailArgs 동형)에서 명시적으로 수행"으로 정정. 특히 setSettings의 apiKey/baseURL 타입·null 처리, resolveItem의 key/action 화이트리스트를 핸들러에서 검증하도록 설계 의사코드/계약에 명문화한다.

P1 C-M-1 · toResponse 파일 소속 오인 + 파일변경지도 누락

  • 무엇: carry-over "3곳 반영" 중 toResponse의 실제 위치가 설계 본문/파일변경지도에 정확히 반영되지 않음.
  • 위치: 설계 ⑤ §517·§560(danger-box) — "getUiState 응답(toResponse 경로)"을 uiStateStore.js 맥락에서 서술. ⑩ 파일변경지도 §675 — uiStateStore.js만 "3곳(default/normalize/toResponse)" 수정으로 등재.
  • 문제: 실측상 defaultState()·normalizeState()는 lib/common/uiStateStore.js에 있으나, toResponse(state)는 electron/ipc/uiState.js(line 29-30)에 있는 별도 함수다. 이 파일이 파일변경지도에 수정 대상으로 등재되어 있지 않다(⑩표에 electron/ipc/uiState.js 행 없음). 구현자가 uiStateStore만 고치면 toResponse에 briefing 필드 추가가 누락되어, 정규화·영속까지는 되어도 getUiState 응답에서 briefing이 빠지는 homeLayout류 회귀(C-M-1)가 그대로 재발한다(설계가 막으려던 바로 그 버그).
  • 권고: ⑩ 파일변경지도에 electron/ipc/uiState.js(수정 — toResponse에 briefing 필드 추가) 행을 추가하고, ⑤ danger-box의 "3곳"을 uiStateStore 2곳(default/normalize) + ipc/uiState.js 1곳(toResponse)으로 파일을 명시한다. 다만 브리핑 carry-over는 getUiState가 아니라 spip:briefing:getState 전용 채널로 반환할 수도 있다(③ 계약). 설계가 후자라면 toResponse 수정이 불필요하므로, brieifng 상태를 어느 응답 경로로 노출하는지를 ⑤·③ 간에 일치시켜 확정해야 한다(현재 두 경로가 혼재).

P2 schemaVersion 자기모순 — "1→2"와 "키만 추가(미상승)"가 충돌

  • 무엇: ui-state 스키마 버전 승격 여부에 대한 설계 내부 진술이 상충.
  • 위치: ⑤ §521 SVG("schemaVersion: 1→2"), ⑩ §675("schemaVersion 2"), RK4 §729("ui-state 1→2") vs 미해결질문 §743("기본 키만 추가 — 기존 normalize가 graceful이라 버전 미상승 가능").
  • 문제: 본문·파일지도·리스크는 2로 승격을 단정하는데 미해결질문은 "미상승 가능"을 기본으로 둔다. 구현자가 어느 쪽을 따를지 모호하다. 실측상 uiStateStore.normalizeState는 SCHEMA_VERSION을 무조건 현재값으로 덮어쓰고 누락 키를 graceful 폴백하므로, 키만 추가하면 버전 승격 없이도 무손실이다. 즉 본문의 "1→2 마이그레이션" 서술은 실제로 불필요한 동작을 전제한다.
  • 권고: 한쪽으로 확정한다. 권장은 "키만 추가, schemaVersion 유지"(미해결질문 기본과 실측 정합). 그러면 ⑤ SVG·⑩·RK4의 "1→2" 표현을 "키 추가(버전 유지·normalize graceful)"로 통일한다. 명시 마이그레이션이 정말 필요하다고 판단하면 그 근거(파괴적 변경)를 제시해야 하나, 현재 신규 키 추가만으로는 근거가 없다.

P2 R-38 · 비-git 신호의 auto-resolve 판정 기준 공백

  • 무엇: auto-resolve(근본 데이터 해소 시 carry-over 제외) 판정 로직이 git 계열 신호에만 구체화됨.
  • 위치: 설계 ⑤ §556 auto-resolve("dirty→clean, behind→0, 마감 처리 등 ... 현재 스냅샷 vs 항목 키의 signalType 대조").
  • 문제: FeatureSpec R-36 신호표는 8종(dirty·ahead·behind·attention·new mail·deadline·disk·scan)인데, auto-resolve 예시는 git/마감 위주다. "새 메일"·"디스크 회수 후보"·"스캔 완료(구성 변동)" 같은 신호는 "해소" 개념이 모호하다 — 새 메일을 읽으면 해소인가? 디스크를 비우면? 스캔 항목은 다음 스캔까지 유지인가? FeatureSpec R-38 수용기준("근본 데이터 해소 시 자동 사라짐")을 이들 신호에 어떻게 검증할지 정의가 없어 briefingItems 순수 함수의 경계 테스트를 작성할 수 없다(실행가능성 결함).
  • 권고: ⑤ 또는 ⑫ 테스트표에 신호유형별 auto-resolve 판정 규칙 매트릭스를 추가한다 — (a) 상태형(dirty/ahead/behind/attention/deadline/disk): 현재 스냅샷에서 해당 조건 불성립 시 resolve, (b) 이벤트형(new mail/scan): auto-resolve 비대상으로 두고 사용자 done/dismiss 또는 만료(14일)로만 정리, 등 명시적 정책을 박는다. 각 규칙에 헤드리스 픽스처를 대응시킨다.

P3 경미 · 정정 권고 (구현 차단 아님)

  • patchRegion 네임스페이스 표기: 설계가 일부에서 그냥 patchRegion으로 쓰나 실측 API는 RG.preserve.patchRegion이다(⑦ §610·§615는 정확히 표기, ④·③ 일부는 약식). 모든 언급을 RG.preserve.patchRegion으로 통일하면 구현 혼동 0.
  • category 명명 불일치: FeatureSpec R-37/R-41은 분류를 "알아야 함/알면 좋음/급함"으로, 설계 BriefingItem은 'must'|'good'|'urgent'로 매핑한다. 매핑 자체는 합리적이나 설계 어디에도 한↔영 매핑표가 없어 렌더 배지·프롬프트 지시 간 표기 드리프트 위험. ⑤ 또는 ⑦에 1줄 매핑표 추가 권고.
  • electron-builder allowlist 실측 전제 확인됨: 설계 ADR §349의 "files allowlist에 node_modules 미명시" 주장은 실측(electron-builder.yml: electron/**·lib/**·public/**·package.json만, node_modules 행 없음)과 정확히 일치. P1 게이트에서 production deps 자동 포함을 실측하라는 지시가 적절하다. 정정 불요 — 정합 확인 기록.
  • delta 배치 flush ~80ms는 인디케이터 300ms·TTFT와 독립된 렌더 부하 완화 파라미터로 분리돼 있어 반응성 지표와 충돌 없음(RK3·미해결질문에 적절히 격리). 정정 불요.

실측으로 확인된 강점 (정합 기록)

  • 발신원 이벤트 메서드명 정확: StateWatcher.onUpdate→spip:projectsUpdated, MailWatcher.onNewMail→spip:mailUpdated, isScanning 게이트 — 설계 ④ 주장과 electron/main.js·lib/server/stateWatcher.js·lib/mail/mailWatcher.js 실측 일치. "기존 발신원 메인 직접 구독" 전제 성립.
  • elevationState API 정확: isElevated() 존재, uiStateStore.write()가 상승 세션 시 no-op하고 정규화 메모리 결과만 반환(line 275-278). 설계 ⑤ "메모리 진실원천·중복경고 금지" 전제 성립.
  • 3곳 패턴 실재: defaultState·normalizeState(uiStateStore) + toResponse(ipc/uiState.js) 3함수 모두 실재. C-M-1 교훈 인용이 실제 버그 구조에 정확히 대응(위 P1 항목은 파일 소속 명시 보강일 뿐, 3곳 개념 자체는 옳음).
  • config 키 저장 패턴 정확: persistConfigKeys(patch) 부분 갱신·0600(FILE_MODE=0o600)·메일 계정 pass 평문 0600 선례 — 설계 ⑧ "메일 pass 패턴 동형" 전제 성립.
  • RG 순수 상태기 재사용 가능: lib/common/renderGuard.js의 coalesce는 request/flushIfPending/release 순수 상태기(기본 120ms). 설계가 "메인 측 coalesce 2000ms에 동일 알고리즘 재사용"으로 렌더러 120ms와 명확히 구분(④ §511) — 정합. RG.preserve.patchRegion·RG.widget 라이프사이클·deferred 게이트 실재.
  • ADR P1 게이트 적절: CJS require 스모크 + asar 포함 실측이 P1 구현 1순위로 박혀 RK1을 착수 초기에 노출. 대안 B(경량 fetch) 1파일 교체 경로 + 동적 import() 폴백까지 명시 — 의존성 규약 정신 보존.

확인 필요 (PM/사용자 판단)

  • 브리핑 carry-over 노출 경로: spip:briefing:getState 전용 채널 vs getUiState의 toResponse에 합류 — 설계 ③과 ⑤가 혼재. 전자면 ipc/uiState.js 수정이 불요해진다. 이 선택이 위 P1(toResponse) 반영 범위를 결정하므로 먼저 확정 권장.
  • 분류 소유권: 설계 기본은 "정책 소유(모델은 표현만)"로 인젝션 ③ 방어와 결정성 확보 — 합리적. PM이 "모델 분류 우선"을 원하면 ADR 갱신 필요(설계도 이미 미해결질문으로 올림). 추가 쟁점은 아님.

위 항목은 설계가 이미 미해결질문으로 적절히 표면화했다. 중대도는 보통 — 구현 P3~P4 착수 전 결정으로 충분.
