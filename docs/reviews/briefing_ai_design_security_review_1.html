Project-SPIP — 브리핑 AI 연동 설계 보안 검증

이 배치 최대의 신규 신뢰 경계(LLM 네트워크 egress + 신뢰 불가 데이터의 프롬프트 주입 + API 키)에 대한 적대적 관점 검증. 설계는 청사진이므로 본 보고는 코드가 아니라 설계가 보안 불변식을 약화시키는가와 구현 시 반드시 강제해야 할 게이트를 산출한다. 모든 판정은 실제 코드(config.js·register.js· preload.js·elevationState.js·logger.js·index.html CSP·uiStateStore.js) 대조로 확인했다.

중간M-1baseURL 검증 — scheme 화이트리스트는 명시됐으나 자격증명 URL·리다이렉트·localhost 위장 미명시
위치
lib/common/config.js normalizeBriefing(신규) · spip:briefing:setSettings/testConnection 경로
위협(STRIDE)
정보노출(S/I) — SSRF·자격증명 유출
악용 시나리오
설계는 new URL() 파싱 + protocol∈{http,https}만 명시한다. 그러나 (1) http://user:pass@host/v1 형태 임베디드 자격증명이 통과되면 로그·에러 경로로 새거나 의도와 다른 인증이 붙는다. (2) baseURL이 외부 호스트면 LLM 호출 시 프롬프트(메일 본문·커밋 포함 가능)가 그 외부로 전송된다 — 단일 사용자가 자기 온프레미스를 지정하는 정당 사용이지만, 설정 오타·악성 클립보드 붙여넣기로 외부 유출 가능. (3) localhost 명시 기본이지만 0.0.0.0·IPv6·DNS 리바인딩은 별개 표면.
심각도 근거
로컬 단일 사용자 + opt-in(enabled 기본 false) + 사용자가 직접 입력하는 자기 엔드포인트라 악용 가능성은 낮다. 그러나 신뢰 불가 데이터가 임의 호스트로 나가는 유일 경로이므로 중간.
권고 조치
normalizeBriefing에 (a) url.username/password 비어있을 것 강제(임베디드 자격증명 거부 또는 stripping), (b) protocol 화이트리스트는 거부 시 기본값 폴백이 아니라 명시적 검증 실패 신호(사용자가 외부 host 지정을 인지하도록 testConnection에서 host 표기), (c) 비-localhost baseURL일 때 설정 UI에 "외부 전송됨" 경고 텍스트(L-1). file://·ftp:// 등은 이미 scheme 화이트리스트로 차단됨을 테스트로 못박기.
중간M-2키·엔드포인트 누출 — 로거는 자동 마스킹이 없다(설계 가정과 코드 현실의 간극)
위치
lib/common/logger.js warn(reason, meta) / llmClient.js 에러 분류 · 에러 push payload
위협(STRIDE)
정보노출(I)
악용 시나리오
설계 §8은 "로그 미기록(logger clampString도 키 제외)"이라 적었으나, 실제 Logger.warn은 Object.assign(entry, meta) 후 meta.path를 콘솔에 그대로 출력하며 키/baseURL 자동 redaction 기능이 없다. clampString은 제어문자·길이만 다룬다. 따라서 구현자가 무심코 logger.warn('LLM 호출 실패', { url: cfg.briefing.baseURL, err })처럼 baseURL(자격증명 포함 가능)·키를 meta로 넘기면 콘솔/warnings[]에 평문 적재된다. Langchain ChatOpenAI는 에러 객체에 요청 config(헤더 Authorization 포함 가능)를 실을 수 있어 err 통째 로깅 시 키 누출 위험.
심각도 근거
설계 의도는 옳고(고정 code만 push), 누출은 구현 실수에 한정. 로컬 콘솔이라 외부 노출은 아니나, 상승 세션·화면 공유·로그 첨부 시나리오에서 위험.
권고 조치
(a) 에러 push payload는 설계대로 고정 code만({gen, code}) — baseURL/모델명/메시지 비포함을 테스트로 검증. (b) llmClient 에러 분류는 err.message·err.config를 로그에 싣지 말고 code로 정규화 후만 로깅. (c) baseURL을 meta로 넘길 때는 host만(자격증명·경로 제거). (d) testConnection 응답의 model명은 사용자 입력 반향이므로 textContent 표시 OK이나 키는 절대 반향 금지.
중간M-3프롬프트 인젝션 — 구분자 이스케이프 알고리즘과 프로브 충분성이 청사진 수준
위치
lib/ai/briefingPrompt.js(조립·truncate·구분자) · briefingParse.js(출력 파서)
위협(STRIDE)
변조(T) — 데이터가 지시로 승격
악용 시나리오
설계는 <<<DATA … DATA>>> 고유 구분자 + "데이터에 같은 토큰 있으면 이스케이프"를 명시하나 이스케이프 방식이 미정의다. 메일 본문에 정확히 DATA>>> 문자열을 심으면 데이터 영역을 조기 종료시켜 이후를 지시로 해석시킬 수 있다(영역 탈출). 또한 프로브 4종은 적절하나 구분자 탈출·다국어/유니코드 우회·carry-over 항목을 통한 2차 인젝션(이전 모델 출력이 다음 프롬프트로 재투입)이 빠졌다.
심각도 근거
설계의 분류 정책 소유 + 출력 자동실행 0이라는 이중 근본 방어가 인젝션 성공의 영향을 "표시 텍스트 변조"로 제한 → 권한상승·실행으로 이어지지 않으므로 중간(치명 아님). 그래도 사용자 혼란·허위 가이드 유발 가능.
권고 조치
(a) 구분자 이스케이프를 명확히 정의: 데이터 안의 구분자 토큰을 무해 치환(예 분해/제로폭 삽입)하거나, 구분자 의존을 줄이고 데이터를 JSON 문자열로 인코딩해 구조적으로 영역 분리. (b) 헤드리스 프로브에 ⑤ 구분자 탈출(DATA>>> 본문 삽입), ⑥ carry-over 2차 인젝션(이전 출력 재투입), ⑦ 유니코드/제로폭 우회, ⑧ 초장문 본문으로 시스템 지시 밀어내기(truncate 경계) 추가 — 프로브 4종을 8종 권고. (c) carry-over로 재투입되는 항목 텍스트는 sanitizeTodoText 동형으로 제어문자·길이 정규화(설계가 이미 언급 — 테스트로 강제).
