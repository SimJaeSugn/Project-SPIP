1-2. maybeAutoRefreshMail 본체 유지 판단 — fetch 억제 vs render 보류 구분

구현자는 maybeAutoRefreshMail(app.js:4558)을 deferred()로 치환하지 않고 기존 본체 if (busyMailSummary||showSettings||showHelp||todoAdding) return;를 유지했다. 이 판단은 타당하다:

  • maybeAutoRefreshMail의 가드는 네트워크 fetch(getMailSummary) 시작 억제다 — 모달/입력 중 불필요한 IPC·갱신 churn을 막는 목적. deferred()는 render() 보류(DOM 교체로 인한 포커스/조합 파손 방지)로 의도가 다르다.
  • 두 가드를 합치면 의미가 섞인다: 만약 maybeAutoRefreshMail을 deferred()로 바꾸면 _dragging 같은 render 전용 조건이 fetch 억제로 전이돼 동작이 달라진다(드래그 중 메일 fetch도 막힘 — 불필요).
  • 겹치는 조건(showSettings/showHelp/todoAdding)은 양쪽에 동일하게 존재 — 관측 동작 동일.
경계면 정합 확인: refreshMailSummary(1949·1954)와 refreshCommitActivity(1972·1977)는 fetch 시작/완료 시 직접 render()를 호출한다(coalesce 비경유). 이는 home 뷰 자체의 KPI 렌더로 의도된 경로이며, busyMail/busyCommit 보류 윈도우가 닫히는 지점에서 데이터가 화면에 반영됨을 보장한다(아래 BUG-1의 잔류 pending 분석 참조).

발견 버그 / 관찰

BUG-1 보통 — 직접 render() 경로가 coalescer pending을 소비하지 않아 1회 잉여 렌더 가능

무엇이home 뷰에서 mail/commit fetch 진행 중(busyMailSummary/busyCommitActivity=true) projectsUpdated push가 오면 RG.coalesce.request()가 보류로 pending=true 적재. fetch 완료 시 refreshMailSummary/refreshCommitActivity는 직접 render()를 부르고 flushIfPending()은 부르지 않음 → pending=true가 잔류. 이후 다음 보류해제(compositionend/drag end) 또는 다음 request 시 잉여 render 1회 발생 가능.
어디서public/app.js refreshMailSummary(1949·1954)·refreshCommitActivity(1972·1977) ↔ _coalescer.pending(5322). 경계면: 직접 render 경로 vs coalesce 게이트.
기대 vs 실제기대: 보류 윈도우 종료 시 누적 갱신이 정확히 1회 반영, 잔류 큐 0("중복 렌더 0"). 실제: 데이터는 직접 render로 올바르게 반영되나(merge는 request 전에 store에 완료), coalescer pending 비트가 남아 후속 1회 잉여 render 유발 가능.
재현헤드리스 단정 불가(타이밍/IPC 의존). 시나리오: home 체류 → 60초 폴링 또는 onMailUpdated로 busyMailSummary=true 윈도우 → 그 사이 projectsUpdated push 도착 → fetch 완료 후 검색창 한글 조합 1회 → compositionend의 flushIfPending이 잔류 pending을 소비해 1회 추가 render. 사용자 PC 수동 스모크 필요.
심각도보통 — 정합성/데이터 손실 아님(데이터는 직접 render로 반영). 영향은 "잉여 render 1회"로 수용기준 "중복 렌더 0"의 엄밀 위반 소지. 단 변경 전 동작과 비교하면 회귀 아님(전에는 busyMail을 보류조건으로 안 봐서 push 즉시 render했음). 권고: fetch 완료 직후의 render()를 RG.coalesce.flushIfPending() 호출로 보강하거나, busy 클리어 지점을 보류해제 지점으로 명시 처리.

OBS-1 관찰(비버그) — busyMail/busyCommit 추가로 home의 라이브 push 렌더 타이밍이 이동

변경 전 applyProjectsUpdate는 home에서 mail/commit fetch 중에도 push를 즉시 render했다. 변경 후엔 busyMailSummary/busyCommitActivity가 deferred()에 들어가 그 윈도우 동안 보류된다. 설계 §3.2가 명시적으로 의도한 변경이며(매핑표 라인 185·187), fetch 완료 시 직접 render로 데이터가 반영되므로 관측상 손실은 없다. 다만 "기존 동작 100% 보존"의 문자적 해석에선 home 뷰의 push 렌더 타이밍이 미세 이동한 것이라 사용자 수동 스모크(home 체류 중 프로젝트 변경 watcher 반영 지연 체감 여부)로 확인 권고.

결론

PASS. 핵심 요구인 "기존 동작 100% 보존(회귀 0)"은 render-보류 조건의 완전 흡수(누락 0)와 preserve/widget 로직의 무변형 이동으로 충족된다. coalesce 동시성 4규칙은 코드와 정본이 동형으로 보장하며 12개 헤드리스 테스트로 검증된다. patchRegion은 1차 범위에서 제외됨을 확인했다. 블로커·높음 버그 없음. 보통 1건(BUG-1, 직접 render 경로의 coalescer pending 잔류 — 정합성 손실 아님, 잉여 render 소지)은 후속 보강 권고. IME/드래그/포커스 보존은 헤드리스 검증 불가이므로 사용자 PC 수동 스모크가 통합 회귀의 최종 게이트다.
