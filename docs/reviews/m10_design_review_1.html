1. R-25 정합 · 회귀 방지 — PASS

1.1 확인된 정합 (실제 코드 대조)

  • 시그니처 불변 확인: renderGuard.js의 shouldDeferRender/createCoalescer는 순수 함수로 M10이 건드리지 않음(설계 §9.2·§10 "변경 없음" 확인). app.js의 RG.preserve.capture/restore(5604·5617), widget.define/destroyAll/mountAll(5643·5649·5659), coalesce.request/flushIfPending/release/cancel(5702~5705), deferred 6플래그(5580 editing 등) 모두 M10에서 시그니처 변경 0 — 설계 주장 정확.
  • render() 경로 보존: render()(1375·1381·1407·1409)의 destroyAll→capture→replaceChildren→restore→mountAll 순서를 patchRegion이 전혀 변경하지 않음. 설계 §2.7 의사코드가 실제 render 골격과 일치.
  • 신규 내부 메서드 라이프사이클 정합: 설계 §2.3의 _destroyById/_mountById는 실제 destroyAll(5649)·mountAll(5659)과 동일한 null-체크 + try/catch 흡수 + _instances[id] 갱신 패턴을 그대로 따른다. 리스크 표(§14.1 4행)의 "이중 null 안전"(_destroyById가 null이면 destroyAll no-op) 주장은 실제 코드(5651~5655 inst!=null 체크)로 검증됨.
  • 폴백 안전망: §2.4 표의 4분기(null→fallback / throw→fallback / deferred→request후 반환 / 정상→영역교체)가 빅뱅을 막는 구조로 타당. fallback 기본값이 render()이므로 patchRegion 실패 시 M9 검증 통과 경로로 안전 회귀.
  • ADR append 정합: adr.html 65~69행에 ADR-M10-1~4가 m10-design.html §13과 결정·기각안까지 일치하게 추가됨. 추적성 OK.
이 항목은 "회귀를 막는 골격"에 한해 PASS다. 단 실제 적용(P4)에서의 위젯 노드 정합 문제는 항목 2(F-2)에서 별도로 다룬다 — 그 빈틈이 회귀를 유발할 수 있어 항목 1의 PASS가 P4 안전을 보증하지 않는다.

2. patchRegion 계약 완결성 — CONDITIONAL

F-2 [HIGH] commitChart 위젯의 self-append와 patchRegion 5단계 계약 충돌 — 계약에 빈틈.
  • 무엇: patchRegion의 ④단계는 builderFn() → containerEl.replaceChildren(...nodes)로 영역을 직접 교체하고, ⑤단계는 _mountById('commitChart', containerEl)로 위젯을 mount한다.
  • 위치: 설계 §2.2 ④⑤ + §2.7 patchCommitChart 예시(229행) vs 실제 app.js:5726~5737 commitChart 위젯 init.
  • 문제: 실제 commitChart init은 스스로 const host=querySelector('.commit-chart-host'); host.appendChild(chart.node) 한다(5729·5736). 즉 차트 노드 생성·삽입 책임이 위젯 init 안에 있다. 그런데 설계의 patchCommitChart는 builderFn이 chartBars(...).node를 반환해 patchRegion이 replaceChildren으로 삽입한다고 본다. 두 경로가 동시에 살아있으면 — ④에서 builderFn이 차트를 한 번 삽입하고, ⑤의 _mountById가 init을 호출해 또 한 번 삽입한다(이중 차트). 반대로 builderFn을 빈 노드로 두고 _mountById에만 맡기면 ④의 replaceChildren 대상이 모호해진다. 설계는 "builderFn이 차트를 만들고 위젯도 mount"라는 두 모델을 혼용하고 있어 어느 쪽이 노드를 소유하는지 계약이 불명확하다.
  • 권고: 둘 중 하나로 책임을 못박아 §2에 명시할 것. (A) 위젯-소유 모델: builderFn은 빈 호스트(또는 host 자체)만 반환/통과시키고, 차트 생성·삽입은 전적으로 _mountById(commitChart init)에 맡긴다. 이때 ④는 host의 자식만 비우는 역할(replaceChildren()). 또는 (B) 빌더-소유 모델: commitChart 위젯을 patchRegion 경로에서 widgets 목록에서 빼고, builderFn이 chartBars 생성+핸들러 destroy 책임을 직접 진다. 현재 위젯이 호버 핸들러 정리(destroy)를 담당하므로 (A)가 회귀 위험이 작다. 어느 쪽이든 "_destroyById → replaceChildren → _mountById"가 노드를 정확히 1번만 만들고, init 내부의 host.appendChild가 replaceChildren 결과와 충돌하지 않음을 계약으로 적어야 한다.

3. 차트 깜빡임 접근 · Q-M10-1 — CONDITIONAL

F-1 [HIGH] refreshCommitActivity 진입부의 로딩 render()를 설계가 누락 — P2 효과 과장.
  • 무엇: 설계 §4.2/§8 P2는 "diff 가드로 동일 데이터 폴링 시 render() 미호출 효과"라 주장.
  • 위치: 설계 §4.2 의사코드(364~376행)·§8 P2(655행) vs 실제 app.js:2202~2211.
  • 문제: 실제 refreshCommitActivity는 fetch 시작 직후 busyCommitActivity=true; if(view==='home') render();(2204~2205)로 전체 render를 한 번 호출한다(로딩 상태 표시). 설계 의사코드는 이 진입 render를 전혀 다루지 않고 "완료 후"만 분기한다. 따라서 P2 diff 가드를 완료 후에만 넣어도, 폴링 틱마다 진입 render 1회는 그대로 발생해 차트 destroy/recreate(깜빡임)가 남는다. "동일 데이터면 render 미호출"은 거짓 — 진입 render는 데이터 동일 여부와 무관하게 발화한다.
  • 권고: (1) P2 설계에 진입 render(2205)를 명시하고, 폴링 경로에서는 진입 시 로딩 render를 스킵하는 조건을 추가하라(예: 폴링 자동 갱신이면 commitActivityLoaded가 이미 true이므로 진입 render 생략, 사용자 최초 로드만 진입 render). (2) 또는 진입 render도 patchRegion(P4) 이후에는 host만 "집계 중…" 상태로 부분 갱신하도록 동일 영역에 묶어라. 어느 쪽이든 "P2 단독으로 동일 데이터 시 깜빡임 0"이라는 품질 목표(§1.2)는 P2만으로는 달성 불가임을 설계에 바로잡아야 한다.

3.1 대안 비교·결정의 타당성

§4.1의 A(patchRegion)+조건부 B(diff skip) 병합 결정 자체는 타당하다. patchRegion이 스크롤 흔들림까지 해결한다는 §4.3 주장도 맞다(폴링 시 render() 미호출이면 .dash__main scrollTop 보존 문제가 사라짐). 단 위 F-1 때문에 "P4까지 가야 깜빡임이 실제로 0"이며 P1~P3 단계 동안은 완화가 부분적임을 §1.2/§8에 솔직히 반영해야 한다.

3.2 Q-M10-1 (commitActivityKey 날짜창 미포함) — 리스크 실재 확인

리스크는 실재한다. §4.2 commitActivityKey(354)는 days.slice(-7)의 count 시퀀스만 join한다. 7일 슬라이딩 창이 하루 넘어가면(예: 어제까지 [월..일]=[3,1,0,2,5,1,4] → 오늘 [화..월]=[1,0,2,5,1,4,3]) count 멀티셋이 우연히 동일하거나, 더 흔하게 새 날짜의 count가 직전 마지막 날과 같으면 키가 충돌해 차트가 옛 라벨을 유지할 수 있다. 폴링이 5분 주기라 자정 직후 창이 밀리는 순간을 포착해도 키가 같으면 갱신을 건너뛴다. §14.1 마지막 행·Q-M10-1이 이를 "단순화 채택"으로 미뤘으나, 라벨이 요일 텍스트라 사용자가 즉시 알아챈다(틀린 요일에 데이터 표시).

권장 해소책(Q-M10-1): count 시퀀스에 창의 경계 날짜 식별자를 함께 포함하라. 단 L-1/M-2 원칙(신뢰불가 날짜 문자열을 키에 직접 넣지 않음)을 지키기 위해 원시 date 문자열 대신 정규화된 정수를 쓴다.
  • 1순위 (권장): 키에 days.slice(-7)의 마지막 날짜를 epoch-day 정수(예: Math.floor(Date.parse(lastDate)/86400000), NaN이면 0)로 prefix. 날짜 문자열 자체는 키에 안 들어가고 정수만 들어가므로 보안 채널 불변식 유지. 창이 하루라도 밀리면 lastEpochDay가 바뀌어 키가 달라진다 → 갱신 누락 0. 구현 비용 최소(한 줄).
  • 2순위: count 시퀀스 + 라벨 시퀀스(homeWeekday 결과는 요일 enum 7종으로 신뢰 가능)를 함께 join. 단 라벨은 요일이라 주 단위 반복이면 동일할 수 있어 1순위보다 약함.
  • 이 보강은 P2 구현 시 처음부터 넣을 것(설계는 "P4 이후 재검토"로 미뤘으나, count-only 키는 자정 경계에서 즉시 버그를 내므로 미루면 안 됨). 단위 테스트(M10-DG-1)에 "count 동일·날짜창 이동 → 다른 키" 케이스를 추가.
L-1/M-2 확인: 권장 1순위는 Date.parse 결과(정수)만 키에 넣으므로 신뢰불가 문자열이 키 계산·textContent 외 경로로 흐르지 않는다. 설계 §4.2 secbox의 "count 수치만" 원칙과 양립.

4.2 폴링 무재시도 / 이중발화 — 분석 타당

  • 현재 동작 분석 정확: 설계 §6.1이 인용한 maybeAutoRefreshCommit(실제 4838~4843)의 if(RG.deferred()) return로 틱 통째 건너뜀 → 최대 ~600s 지연 분석이 실제 코드와 일치.
