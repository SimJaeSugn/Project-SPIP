  • 60초 폴링·onMailUpdated push 모두 maybeAutoRefreshMail()→refreshMailSummary({ silent:true })(app.js:4954,4973,4962). silent면 진입 로딩 render 생략(app.js:2268) → fetch 동안 이전 목록 유지.

참고: 폴링이 deferred로 보류된 동안 새 폴링이 또 들어와도 coalesce는 단일 pending/타이머라 잉여 render 없음. 다만 보류 해제는 전체 render()로 반영되므로(coalesce.release/flush) "보류 해제 시 영역만 교체"가 아니라 전체 재렌더 — 깜빡임 0 목표는 정상 흐름(미보류)에서 충족, 보류 해제 1프레임은 기존 M9/M10과 동일한 trade-off.

3. 설정 우측 패널

PASSnav class 토글 + pane patchRegion + bypassDefer
  • nav 버튼 클릭 → switchSettingsTab(cat.id)(app.js:2499). 좌측 nav는 classList.toggle('is-active')+aria-selected만 동기화(app.js:2536-2541) — 모달 골격·nav DOM 보존.
  • 우측 .settings-pane만 patchRegion(pane, …, { bypassDefer:true, preserveFocus:true, fallback:render })로 교체(app.js:2543-2553). builderFn은 임시 frag에 buildSettingsPaneInto 채운 뒤 childNodes 이동.
  • bypassDefer:true 타당성 확인됨: 설정 오버레이는 overlayOpen→deferred()=true이지만 탭 전환은 사용자 명시 액션이라 보류 대상 아님. 우회 안 하면 클릭이 coalesce로 보류되어 즉시 전환 안 됨 → 우회가 정확.
  • 포커스 트랩: getTabbables(dialog)가 dialog 전체를 동적 querySelectorAll(app.js:4504-4507) → 좌측 nav 버튼 + 우측 새 pane 입력 모두 포함. pane 교체 후에도 재계산되어 트랩 유지.
  • 같은 탭 재클릭 store.settingsTab === next no-op(app.js:2533). buildSettingsPaneInto는 renderSettings(app.js:2506)·switchSettingsTab 공용(2회 호출).
관찰(버그 아님): patch 대상 container가 .settings-pane 자신인데 preserve.capture의 SCROLL_SEL은 rootEl.querySelector('.settings-pane')로 자손을 찾아 자기 자신은 캡처 대상이 아님. 즉 탭 전환 시 우측 스크롤은 0으로 리셋된다. 다른 탭으로 전환하면 콘텐츠가 달라 스크롤 톱 시작이 자연스러우므로 의도된 동작으로 판단. preserveFocus:true는 pane 내부 입력(예: 포커스된 텍스트 입력) 캐럿 보존용으로 정상 동작. 심각도: 낮음(설계 의도 일치, 회귀 아님)
