2. 개선 항목 (P2)

P2-1 중복 app.js↔favorites.js 4개 순수 함수 바이트 동일 중복

무엇focusGate·favoritesChangedView·nextSlideIndex 3개가 두 파일에 바이트 동일로 정의됨. favoriteViewModels(app)↔favoriteWidgetViewModels(favorites)는 변수명만 다른 동형 로직.
위치app.js:776,856,866,789 ↔ favorites.js:53,63,69,24
왜 문제한쪽 계약 변경 시 다른 쪽이 자동으로 따라가지 않아 드리프트 위험. 현재는 frontend가 favorites-renderer.test.js의 동형성 단언(app.focusGate(true)===fav.focusGate(true) 등)으로 고정 → 드리프트가 즉시 테스트 실패로 드러나므로 위험도는 낮음. 다만 동형성 테스트는 focusGate·favoritesChangedView 2개만 커버하고 nextSlideIndex·뷰모델 동형은 app↔fav 교차 단언이 없다(각자 단독 검증). 즉 nextSlideIndex가 한쪽만 바뀌면 테스트가 못 잡는다.
권고독립 창이라 ESM/공유 모듈 도입이 부담이면 현 복사 + 동형성 테스트 유지가 합리적. 단 ① 동형성 단언을 nextSlideIndex·뷰모델까지 확장(교차 비교 1~2줄 추가)하거나 ② 두 파일 상단에 "이 함수는 X와 동기화 필수(테스트 고정)" 상호 참조 주석을 넣어 의도를 명문화. 향후 함수가 더 늘면 public/shared/favorites-core.js 같은 단일 원천 추출을 검토(런타임 dep 0 원칙과 무충돌).

P2-2 죽은 코드 app.js 미사용 export(nextSlideIndex·favoriteViewModels)

무엇오버레이 슬라이더 제거 후 app.js 런타임에서 nextSlideIndex·favoriteViewModels는 호출되지 않는다(정의·export만 잔존). focusGate·favoritesChangedView도 app.js 런타임에서 미사용(위젯 전용 로직).
위치app.js:776,789,856,866 (def) / 3015-3023 (export) — grep 결과 런타임 호출부 0
왜 문제설계 §8.2가 "favoriteViewModels·nextSlideIndex 순수 함수는 위젯에서 공유(중복 방지)"라 했으나 실제로는 위젯이 자체 복사본을 갖고 app.js 사본은 오직 테스트(동형성)용으로만 남았다. 코드만 읽으면 "app이 아직 슬라이더를 쓰나?"라는 오해를 부른다. 죽은 코드 신호.
권고P2-1과 함께 결정. 동형성 테스트 기준점으로 의도적으로 남기는 것이라면 각 함수 위에 "@deprecated 런타임 미사용 — 위젯 동형성 테스트 기준점" 주석으로 의도를 박제. 동형성을 favorites.js 단독 검증으로 옮길 수 있으면 app.js에서 4개를 삭제해 표면을 줄이는 편이 더 깔끔하다(슬라이더가 완전히 위젯으로 이전됐으므로).

P2-3 복잡도 favorites.js refresh 중복 트리거(focus + visibilitychange)

무엇show(=포커스 복귀)마다 stale 해소 재조회를 window 'focus'와 document 'visibilitychange' 두 경로로 건다. 한 번의 show에서 두 이벤트가 모두 발화하면 refresh()(getUiState+getProjects 각 1회)가 중복 호출될 수 있다.
위치favorites.js:325-338
왜 문제기능 오류는 아니다(refresh는 idempotent·render 재호출 안전). 그러나 show마다 IPC 2회가 2배(최대 4회)로 나갈 수 있어 불필요한 왕복이고, render() 중복으로 슬라이더 인덱스 clamp가 두 번 일어난다. 의도(둘 중 무엇이 실제 발화 경로인지)가 코드에서 불명확.
권고경합 제거: refresh에 in-flight 가드(예: if(state.refreshing)return)를 두거나, 한쪽 경로를 정규 트리거로 확정하고 다른 쪽은 폴백으로 주석화. 비용이 작아 P2 하위.

P2-4 정확성(경미) doOpen/doCopy 공유 busy 플래그 — 교차 차단

무엇doOpen·doCopy가 같은 state.busy를 공유한다. open in-flight 동안 copy가, copy in-flight 동안 open이 무시된다(다른 동작인데도).
위치favorites.js:99,158-169
왜 문제연타 방지 목적은 달성하나, 서로 무관한 두 액션이 한 플래그를 공유해 한쪽이 다른 쪽을 막는 의도치 않은 상호 차단이 가능. 메인 app.js는 액션별 맵(store.state.opening[id])으로 분리한 것과 일관성도 어긋난다. 위젯은 카드 1장 표시라 실사용 충돌 빈도는 낮음(저심각).
권고액션별 또는 (id,action)별 in-flight 키로 분리하거나, 최소한 open/copy를 각자 플래그로 나눠 교차 차단을 없앤다. setFavorite(doUnfavorite)는 busy를 쓰지 않는데 이는 낙관적 반영이므로 의도적 — 일관성 측면에서 주석 1줄로 차이를 밝히면 가독성 향상.

P2-5 가독성 renderInteractivity가 render의 별칭(이름-구현 불일치)

무엇주석은 "전체 재렌더 없이 버튼 활성/시각만 갱신(가벼운 경로)"라 하지만 renderInteractivity는 사실상 render()를 그대로 호출한다(전체 재렌더).
위치favorites.js:292-293
왜 문제주석-코드 불일치. "가벼운 경로"라는 이름·주석이 실제 비용(전체 재렌더)을 오도한다. 위젯 DOM이 작아 성능 문제는 없으나, 이름이 약속을 어긴다.
권고래퍼를 제거하고 호출부에서 직접 render()를 부르거나, 주석을 "현재는 전체 재렌더로 단순화(위젯 DOM 소형)"로 정정. 둘 중 후자가 변경 최소.
