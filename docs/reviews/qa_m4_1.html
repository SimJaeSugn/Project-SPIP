6. 발견 버그

BUG-M4-01 높음 — deps/devDeps가 기본 스캔에서 항상 null (계약 위반)

무엇이: 설계 계약은 "deps/devDeps는 size.enabled와 무관하게 항상 수집(저비용)"이나, withSize 없는 기본 스캔에서 deps/devDeps가 항상 null.
어디서: 경계면 = lib/scan/collectors/index.js activeCollectors() ↔ lib/scan/scanner.js:133 loadActiveCollectors({includeSize: withSize}). size 수집기는 includeSize===true일 때만 로드되므로, 비활성 시 size.collect()의 deps/devDeps 산출 분기(size.js:111-136)에 도달조차 못 함. buildProject가 skipped+null로 채움.
기대 vs 실제: 기대 {status:'skipped', deps:2, devDeps:1} / 실제 {status:'skipped', deps:null, devDeps:null}.
재현(실측):
// 기본 스캔 (withSize off)
scanner.scan({roots:[root], excludes:[]})
 → alpha.size = {"status":"skipped","totalBytes":null,"nodeModulesBytes":null,"deps":null,"devDeps":null}
// withSize 스캔
scanner.scan({roots:[root], withSize:true, size:{enabled:true}})
 → alpha.size = {"status":"ok","totalBytes":77,"deps":2,"devDeps":1}
// 그러나 직접 호출은 비활성에서도 채움(단위테스트가 통과하는 이유):
size.collect(alphaPath, {config:{size:{enabled:false}}})
 → data = {"status":"skipped","deps":2,"devDeps":1}   ← 모듈은 맞으나 통합경로에서 미호출
근본: 계약 의도("저비용이라 항상")와 구현 게이트("opt-in일 때만 로드")의 불일치. 명확한 단일 결함이므로 debugger 회부 불필요 — 수정 방향: (a) size 수집기를 항상 로드하되 용량측정만 enabled 게이트, 또는 (b) deps/devDeps를 size 수집기 밖(language/detector가 이미 package.json 파싱)으로 이관. 설계 §667이 "detector 단계에서 키 개수만 센다"고 적은 것을 보면 (b)가 설계 본의에 더 부합.
심각도: 높음 — 런타임 크래시 없음(graceful null, UI "미측정"으로 흡수)이나, "의존성 개수"라는 R-09 명시 산출물이 기본 경로에서 미동작하는 계약 위반. 보고 대상: backend-dev.

OBS-M4-02 낮음 (관찰) — size 시간예산이 단일 디렉터리 내부 파일루프에서는 미점검

무엇이: measureTree의 Date.now() > deadlineTs 체크는 스택 pop(디렉터리 경계)에서만 발생. 단일 디렉터리에 수만 파일이 있으면 그 내부 루프는 시간상한을 넘겨도 끝까지 돈다.
완화 존재: maxEntries 상한은 파일루프 내부에서 매 엔트리 점검되므로 stat 폭주는 항상 제한됨. 다중 디렉터리 트리에서는 시간예산 partial이 정상 동작(실측: budgetMs=1, 50서브디렉터리 → partial).
심각도: 낮음 — maxEntries가 진짜 안전망이라 무한정 비용은 없음. 단일 거대 평면 디렉터리(드물다) 한정 시간초과 가능. 출시 비차단, 백로그 메모.
