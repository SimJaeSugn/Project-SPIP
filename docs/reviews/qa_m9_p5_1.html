요약. R-30 설정 2-pane 재구성은 동작·통합·회귀 측면에서 결함 없이 수용 기준을 충족한다. 기존 10개 설정 섹션이 5카테고리에 누락·중복 0으로 정확히 배치되며, 각 섹션은 기존 render*Settings 함수를 그대로 재사용해 저장 로직 변경이 없다(회귀 0). 좌측 카테고리 클릭은 showSettings=true를 유지한 채 전체 render()를 경유하므로 모달이 닫히지 않고 우측만 교체되며, store.settingsTab은 resolveSettingsTab로 재렌더 후에도 복원된다. SCROLL_SEL에 .settings-pane이 선두로 추가되어 우측 스크롤이 복원되고, 포커스 트랩은 좌측 탭 버튼을 자동 포함한다. :has() 미지원 시에도 2-pane flex 레이아웃은 기본 규칙으로 동작(폭만 폴백). 다른 wide 모달(메일·도움말)에는 영향 0. npm test 652건 전부 통과(신규 R-30 7건 포함), 0 fail.

1. 회귀 0 — 섹션 배치·동작 보존 PASS

경계면 교차 검증. SETTINGS_CATEGORIES(app.js:615) ↔ 설계 §8 매핑표(m9-design.html:455-459) ↔ 테스트(settings-categories.test.js:28-48)를 3중 대조했다. 매핑이 정확히 일치한다.

  • 스캔 → roots · exclude · detect · scanOptions (4)
  • 연동 → tools · mail (2)
  • 외관 → theme (1)
  • 단축키 → shortcuts (1)
  • 정보 → info · update (2)  → 총 10개, 누락·중복 0

buildSettingsSection(key)(app.js:2182)의 switch가 10개 키를 각각 기존 renderRootManager/renderExcludeSettings/renderDetectSettings/renderScanOptions/renderToolSettings/renderMailSettings/renderThemeSettings/renderShortcutSettings/renderInfoSettings/renderUpdateSettings로 디스패치한다. 10개 함수 모두 코드에 존재(grep 확인)하며, 빌더는 이들을 호출만 할 뿐 저장·이벤트 로직을 재작성하지 않는다. 따라서 항목별 동작·저장 경로는 변경 전과 동일하다. 우측은 활성 카테고리의 sections만 append하고 나머지는 미렌더 — 설계 §8 명세와 일치.

근거: 단위 테스트 4건(매핑/중복/누락/단일출처 무결성)이 데이터 무결성을 못박고, 디스패치 switch가 동일 키 집합을 커버.

2. 모달 유지 · 카테고리 전환 PASS

모달 비종료. 좌측 버튼 핸들러(app.js:2216)는 store.settingsTab = cat.id; render()만 수행하고 store.showSettings를 건드리지 않는다. render()의 if (store.showSettings) app.appendChild(renderSettings())(app.js:1232) 조건이 여전히 참이므로 모달이 재구성되어 유지된다 — 우측만 교체되는 효과. 결함 없음.

기본값/유지. store.settingsTab 초기값 'scan'(app.js:1067). resolveSettingsTab(627)이 미지정·미지값을 'scan'으로 보정하고 유효값은 그대로 통과 — 재렌더(폴링 포함)를 거쳐도 동일 카테고리 복원. 테스트 2건(56-67)이 기본/정규화/유지 전 경로를 커버.

포커스 트랩. 좌측 nav는 bodyChildren=[layout]로 dialog 내부에 들어가고, getTabbables(dialog)(4185)가 button:not([disabled])를 셀렉트하므로 좌측 탭 버튼이 트랩 순환에 자동 포함된다. Esc는 buildModal keydown 핸들러(2163)가 onClose=closeSettings 호출 — 유지. nav 버튼에 role=tab/aria-selected, nav에 role=tablist, pane에 role=tabpanel 적용(접근성 양호).

3. RG 정합 PASS

R-25 가드 정합. RG.deferred()의 overlayOpen(app.js:5308)이 store.showSettings를 포함 → 설정 열린 동안 5분 폴링 등 라이브 트리거의 render는 보류된다(모달 지속·깜빡임 차단). 한편 좌측 클릭→render()는 사용자 직접 액션으로 deferred 게이트를 거치지 않고 즉시 반영 — 의도된 동작이며 충돌 없음.

SCROLL_SEL. SCROLL_SEL = ['.settings-pane', '.modal__body', ...](app.js:5331)에 .settings-pane이 선두로 추가됨. capture/restore(5337,5350)가 우측 패널 스크롤을 저장·복원하므로 카테고리 전환·재렌더 시 긴 섹션(예: 스캔)의 스크롤 위치가 보존된다. .modal__body도 배열에 남아 폴백 경로(:has 미적용 시 body가 스크롤 컨테이너)에서도 복원이 동작 — 견고.

애니메이션 1회. enter = !store._settingsShown; store._settingsShown = true(app.js:2202)로 최초 열림에만 is-enter 페이드 적용. 카테고리 전환 시 _settingsShown이 이미 true라 재생 안 됨(깜빡임 없음). closeSettings(2122)가 _settingsShown=false로 리셋해 다음 열림에 재적용 — 정확.

4. :has() 폴백 · 타 모달 영향 PASS

폴백 동작. 2-pane의 기능 골격은 .settings-2pane{display:flex} · .settings-nav{...} · .settings-pane{flex:1;overflow-y:auto}(styles.css:717-730)로, :has() 없이도 성립한다. :has(.settings-2pane) 셀렉터(715-716)는 모달 폭(820px)과 body 패딩/overflow 조정만 담당 — 미지원 브라우저에선 .modal--wide 폭(660px)으로 폴백하고 좌우 분할·우측 스크롤은 그대로 작동. 설계 §8 "폭만 폴백" 명세와 일치.

타 wide 모달 영향 0. 메일 본문(app.js:1996)·도움말(4785)도 wide:true지만 bodyChildren에 .settings-2pane를 넣지 않는다. 따라서 :has(.settings-2pane) 셀렉터는 이들 모달과 매칭되지 않아 폭·패딩에 부작용이 없다. 비대상 모달은 기존 660px·패딩 유지 — 격리 확인.

참고: 현 Electron 런타임(Chromium 최신)은 :has()를 지원하므로 실제 앱에서는 820px 정상 적용.

경미 관찰 (블로커 아님) NOTE

무엇이: :has() 미지원 환경 폴백 시 이중 스크롤바 가능성(경미).
어디서: public/styles.css — .modal__body:has(.settings-2pane){overflow:hidden}(716)가 적용 안 되면 .modal__body의 기본 overflow-y:auto(712)와 .settings-pane의 overflow-y:auto(729)가 공존.
기대 vs 실제: 단일 스크롤(우측 패널) 기대 / 폴백 시 body+pane 두 스크롤 컨테이너 존재 가능. 단, 기능 손상은 없고 실제 Electron(Chromium)은 :has() 지원이라 발현 안 됨.
심각도: 낮음 (설계가 명시한 "폭만 폴백" 범위 내 시각적 미세 항목, 현 런타임 미발현).
