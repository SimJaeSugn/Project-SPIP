발견 사항 / 버그

M13-Q-1 — 비-git 신호(mail/마감/디스크) 프로덕션 미발화 [MEDIUM · 관찰]

  • 무엇이: 새 메일·마감 임박·디스크 회수 신호가 실제 앱에서 브리핑 트리거로 이어지지 않을 수 있다.
  • 어디서: electron/context.js deriveSnapshot() ↔ electron/main.js onReady setPush() 경계면.
  • 기대 vs 실제: 기대 — notifyBriefing('mail')(main.js:181) 트리거 시 스냅샷에 mail.unseen이 채워져 mail 신호 발생. 실제 — deriveSnapshot은 store git만 채우고 mail.unseen:0·deadlines:[]·disk.reclaimBytes:0 고정. setPush는 push/isSuppressed만 주입하고 보강 snapshotProvider를 주입하지 않아(context.js:79 주석이 예고한 보강이 main에 없음), policy가 mail/deadline/disk 신호를 산출하지 못한다. 결과적으로 mail 트리거가 와도 git 변동이 없으면 생성이 일어나지 않는다.
  • 재현: enabled=true 상태에서 git 변동 없이 새 메일만 도착 → notify('mail') 호출되나 policy.evaluate signals=0 → 호출 0. 헤드리스로는 snapshotProvider에 mail 데이터를 넣으면 정상 발화(policy 테스트로 신호 생성 자체는 검증됨)되므로, 누락은 배선부(main snapshotProvider 보강 미구현)에 한정.
  • 심각도: MEDIUM — 크래시·보안·회귀 무관. R-36 "새 메일/마감 임박" 트리거 커버리지가 현재 git 신호로 한정됨. R-36 마감/메일 수용기준은 픽스처(직접 스냅샷 주입)로는 PASS이나, 프로덕션 발화는 snapshotProvider 보강 필요.
  • 회부: 원인 명확(배선 누락)하므로 debugger 불필요. backend-dev에게 main.js setPush({ snapshotProvider: () => enhancedSnapshot }) 보강 또는 deriveSnapshot에 mail/todos 소스 연결 권고.

참고(버그 아님) — parse/items 개행 처리 불일치

  • briefingParse.clean은 개행(\n)을 보존하나, orch _mapViews가 거치는 items.sanitizeText는 개행을 제거한다. 최종 표시 텍스트는 개행이 사라지는 cosmetic 불일치 — L-1 안전성·기능엔 영향 없음. 의도라면 무시 가능.

8. 실 LLM 필요 수동 스모크(로컬 모델 구동) — 미검증

아래는 헤드리스로 끝까지 검증 불가 — LM Studio 등 로컬 OpenAI 호환 서버(http://127.0.0.1:1234/v1, exaone-3.5-7.8b 등) 구동 후 수동 확인 필요.
  • S-1 스트리밍 첫 토큰: enabled=true·실서버에서 트리거 → delta가 흐르며 텍스트 증가, 첫 토큰 표시(TTFT 체감 ≤3s 권장사양).
  • S-2 ~80ms 배치 flush: 토큰 폭주 시 끊김 없이 누적 표시되는지(IME 조합 중·다른 섹션 라이브 갱신 동시 발생 시 브리핑 영역만 부분 갱신).
  • S-3 중단 버튼: 스트리밍 중 abort → 토큰 수신 멈춤 + 부분 결과 보존.
  • S-4 실 분류·가이드 출력: 모델이 JSON 구조화로 title/reason/guide 채워 3분류 배지·가이드 토글에 정상 매핑(key 일치). 구조 이탈 시 평문 폴백 확인.
  • S-5 연결 테스트: 설정 패널 testConnection 성공 시 모델명·지연 표시, 서버 끄고 실패 시 CONN_REFUSED 안내.
  • S-6 폴백 진입: 서버 미구동 상태 홈 진입 → 정적 브리핑 + "로컬 모델 확인" 힌트(빈 화면·무한 로딩·다이얼로그 0).
  • S-7 외부 baseURL 경고: 비-localhost baseURL 입력 시 전송 경고 배너 표시.
  • S-8 비-git 트리거(M13-Q-1 연동): snapshotProvider 보강 후 새 메일·마감 임박이 실제 생성을 유발하는지(현재 배선은 미발화).
