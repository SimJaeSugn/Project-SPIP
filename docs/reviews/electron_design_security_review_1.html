Project-SPIP — Electron 이전 설계 보안 검증 보고서

HTTP 로컬서버 → Electron 설치형 데스크톱 이전 설계서(electron-migration.html)의 보안 모델 전환을 적대적 관점으로 위협 모델링했다. 핵심 질문 6가지 — (1) renderer 격리(contextIsolation/nodeIntegration/sandbox), (2) preload 노출면 최소화, (3) IPC 입력의 main 측 재검증, (4) 네비게이션/콘텐츠 잠금, (5) TB-B 부분신뢰 통제(H-1/H-2/H-3/M-3/L-1/M-2/M4-H-1) 보존, (6) M-1 제거 공백·신규 표면(폴더 대화상자·config 영속·패키징/서명) — 을 베이스라인(architecture_security_review_2.html 8중 통제·TB-A/TB-B·RC-1/RC-2 / m4_design_security_review_2.html M4-H-1·H-2)에 대조했다. 중요: 이번 검증 시점에는 lib/·public/·server.js·scan.js가 실제 코드로 존재하므로(과거 아키텍처 검증은 코드 부재 가정), 설계가 "무수정 재사용"을 주장하는 모듈을 Read/Grep으로 교차확인해 설계 주장이 코드에 접지(grounded)되는지까지 점검했다.

전송 계층 전환의 보안 방향성은 옳다 — TB-A(네트워크) 표면을 통째로 제거하고 그 자리를 Electron 보안 베이스라인(contextIsolation·nodeIntegration:false·sandbox·preload allowlist·원격 미로드·will-navigate/팝업 차단·CSP)으로 대체하며, TB-B(부분신뢰 스캔 대상) 통제(H-1 pathGuard realpath·H-2 safeExec resolveBin 절대경로·H-3 입력가드·M-3·L-1·M-2·M4-H-1)는 도메인 코드 무수정으로 보존한다. 코드 교차확인 결과 config.normalizeScanRoots(L90)·safeExec.resolveBin/safeExec(detached 모드 포함)·actionHandlers.open의 H-1/H-2/M-4 검증 체인·git collector의 git -C <path> cwd 의존 제거가 실제로 존재하여 "무수정 재사용으로 통제가 보존된다"는 주장은 접지된다.

그러나 높음 1건이 게이트를 막는다. EM-H-1 — 설계가 preload allowlist를 "원칙"으로만 선언하고, IPC 검증 규칙(§4.2)에 open·rescan·pickFolders만 명세한 채 addRoots·removeRoot·browseDir의 main 측 재검증·인자 스키마·실패 모드를 충분히 못박지 않았다. 특히 browseDir(dir)는 renderer가 임의 디렉터리 경로를 던지는 신규 임의 경로 입력 채널인데, 화이트리스트(scanRoots 하위) 제약·realpath 검증·반환 메타 정보노출 한계가 설계에 없다 — RC-2(부분신뢰) 환경에서 renderer 측 저장형 XSS가 발생하면(L-1은 표시만 막고 페이로드는 메모리에 존재) preload API를 호출해 파일시스템을 열람·열기할 수 있는 정보노출/권한 표면이 된다. 중간 3건은 (1) loadFile+file:// 전환 시 현행 index.html의 /static/* 절대경로·connect-src 'self' CSP가 깨지거나 약화되는 문제, (2) M-1 제거 공백(IPC 발신자 검증·senderFrame 확인 부재), (3) 패키징 무결성(asar·미서명) 미통제다. 낮음 2건은 외부 링크/shell.openExternal 정책과 sandbox "권장(권장)" 표현의 비강제성이다.

1. 갱신된 신뢰 경계 (Electron 전환 후)

HTTP→IPC 전환으로 신뢰 경계가 재편된다. TB-A(네트워크)는 제거되고, 그 자리에 새 경계 TB-C(renderer↔main IPC)가 생긴다. TB-B(스캔 대상 부분신뢰)는 그대로 유지된다. 아래 SVG는 각 경계의 통제·잔여 위협을 요약한다.

2. 보존되는 통제 — 코드 접지 PASS (8건)

설계가 "코드 무수정 재사용으로 보존"을 주장하는 TB-B/공통 통제가 실제 코드에 존재하고 전송 방식과 독립인지 교차확인했다. 8건 모두 보존 가정이 성립한다.

3. 발견 항목 (신규 위협)

EM-H-1 · IPC 노출면 과소명세 — addRoots/removeRoot/browseDir의 main 재검증·스키마·화이트리스트 미고정 높음

위치 설계 §4.1 채널 목록(addRoots·removeRoot·browseDir) · §4.2 검증 규칙(open/rescan/pickFolders만 명세) · §6.2 preload allowlist
STRIDE 정보노출(I) · 권한상승(E) · 변조(T)
분석 설계는 preload 최소 allowlist·"renderer를 신뢰하지 않음"을 원칙으로 선언하나, §4.2 검증표는 open·rescan·pickFolders만 main 재검증을 명세한다. addRoots(paths)·removeRoot(path)·browseDir(dir)는 모두 renderer가 임의 경로 문자열을 main에 주입하는 채널인데, 다음이 설계에 못박히지 않았다: ① addRoots는 §4.2에 "normalizeScanRoots 동일 규칙"이라 적혀 있으나 경로 개수 상한·총 바이트 상한·타입(배열) 강제·거부 사유 누설 한계가 미명세(현 normalizeScanRoots는 64개 상한은 있으나 호출부 인자 타입 가드는 IPC 핸들러 책임). ② browseDir(dir)는 "선택 기능"이라며 남겨두었으나 임의 디렉터리 열람 채널 — scanRoots 하위/화이트리스트 제약·realpath 검증·반환 entries의 정보노출 한계(드라이브 전체 트리 열람 가능?)가 전무. RC-2(부분신뢰)+L-1이 표시만 막고 페이로드는 메모리에 잔존하는 환경에서, renderer 측 XSS가 성립하면 window.spip.browseDir('C:\\')·window.spip.open(임의id)로 파일시스템 열람·VS Code 임의 경로 열기가 가능해진다(nodeIntegration:false로 RCE는 막히나 정보노출/원치 않는 실행은 IPC API 자체로 도달). ③ removeRoot는 화이트리스트를 축소시키는 무결성 영향(가용성/DoS, 사용자 모르게 루트 제거)인데 검증·확인 절차 없음.
악용 시나리오 악성 클론 리포의 package.json name/git branch에 XSS 페이로드 → (L-1 우회 변형 또는 향후 innerHTML 회귀 시) renderer 컨텍스트에서 window.spip.browseDir 반복 호출로 사용자 홈/시스템 디렉터리 구조·파일명 열거(정보노출), 또는 window.spip.open으로 의도치 않은 경로 열기. browseDir에 scanRoots 제약이 없으면 화이트리스트 모델(H-1) 밖의 임의 경로 정보가 renderer로 유출된다.
권고 조치 (1) §4.2 검증표를 전 IPC 채널로 확장: 채널별 인자 스키마(타입·길이·개수 상한)·거부 모드·정보노출 한계를 main에서 강제하도록 명세. (2) browseDir은 드롭(권고) 또는 채택 시 scanRoots 하위 경로로 canonicalize 후 화이트리스트 제약(드라이브 루트/시스템 디렉터리 열람 금지)·반환 entries에 절대경로 미포함(상대/basename). (3) addRoots는 배열·항목 타입·개수 상한(이미 64)·총 길이 상한을 IPC 핸들러에서 선검증 후 normalizeScanRoots 호출, 거부 사유는 고정 code(INVALID_PATH)만(L-3). (4) removeRoot는 입력 path를 canonicalize해 현재 scanRoots와 정확 일치만 제거(임의 문자열로 무관 항목 제거 불가). (5) preload는 ipcRenderer 원본·범용 invoke(channel,...) 비노출, 채널명 하드코딩을 강제(MUST)로 명문화.

EM-M-1 · file:// 로드 시 현행 index.html의 /static 절대경로·CSP가 깨지거나 약화 중간

위치 설계 §3(public/index.html "CSP 유지·강화", staticHandler 드롭) · §6.2 renderer CSP · 실제 public/index.html L11~19·L30
STRIDE 변조(T) · 정보노출(I) (CSP 우회 표면)
분석 현행 index.html은 자산을 /static/styles.css·/static/app.js 절대 경로로 참조(L19·L30)하고, CSP는 default-src 'self'; ... connect-src 'self'; img-src 'self' data: ...로 HTTP origin 기준 'self'를 전제한다. loadFile('public/index.html')+file:// 전환 시: ① /static/* 절대경로는 file 루트(드라이브 루트) 기준이 되어 자산 로드 실패 가능 → 상대경로(./static/ 또는 평탄화) 재작성 필요. ② file://에서 CSP 'self' 의미가 origin 부재로 모호해져 일부 디렉티브가 의도와 다르게 동작(브라우저/Chromium 버전 의존). 설계 §6.2는 CSP를 default-src 'self'; script-src 'self'; object-src 'none'; base-uri 'none'로 축약 제시해 현행의 style-src·img-src·connect-src·form-action·frame-ancestors를 누락 — 강화가 아니라 약화다. RC-2 환경에서 L-1의 2차 방어선(CSP)이 약화되면 저장형 XSS 잔여 위험이 커진다.
악용 시나리오 CSP 디렉티브 누락/약화 + file:// 의미 모호로 인라인/외부 리소스 차단이 느슨해지면, 오염 메타가 DOM에 흘렀을 때(L-1 회귀 시) 외부 비콘·data: 스크립트 등 2차 경로가 열린다.
권고 조치 (1) 자산 참조를 file://에서 동작하는 상대경로로 재작성(설계에 명시). (2) CSP는 현행 전 디렉티브를 보존·강화: default-src 'none'; script-src 'self'; style-src 'self'; img-src 'self' data:; font-src 'self'; connect-src 'none'; object-src 'none'; base-uri 'none'; form-action 'none'; frame-ancestors 'none' 권장(IPC 사용이므로 connect-src는 'none' 가능). (3) Electron session.webRequest/onHeadersReceived로 CSP를 응답 헤더로도 주입해 메타 누락/우회를 이중 차단(설계 옵션으로 명시). (4) staticHandler의 CSP 상수 기준이 사라지므로 단일 출처를 index.html 메타+세션 헤더로 재정의.
게이트 차단 아님(설계 보강 권고). PASS 상향 권장 항목.

EM-M-2 · IPC 발신자 검증(senderFrame/webContents) 부재 — M-1 제거 공백 중간

위치 설계 §4.2(모든 채널 try/catch만 명세) · §6.1 사라지는 통제(세션토큰/Origin 제거)
STRIDE 위장(S) · 권한상승(E)
분석 M-1(세션토큰·Origin·Host)은 소켓 소멸로 타당하게 제거됐다. 그러나 IPC에는 발신 프레임 검증이라는 대체 통제가 있다 — Electron 권장은 ipcMain.handle에서 event.senderFrame의 URL/origin이 신뢰 로컬 콘텐츠인지 확인하는 것이다. 설계는 "renderer는 우리 로컬 콘텐츠뿐, 원격 미로드"라는 전제에 의존할 뿐, will-navigate/setWindowOpenHandler 차단이 실패하거나 향후 <iframe>/sub-frame·webview가 유입될 경우 비신뢰 프레임이 동일 ipcMain 채널을 호출할 수 있다. 현 단일창·원격 미로드 가정에서는 위험이 낮으나, 심층방어로 명세돼야 한다(M-1 제거가 만든 미세 공백).
악용 시나리오 will-navigate 차단 구현 누락 또는 미래의 외부 콘텐츠 임베드 시, 비신뢰 프레임이 spip:open/spip:rescan을 트리거(CSRF 유사). 발신자 검증이 없으면 main이 무차별 수신.
권고 조치 §4.2에 "모든 ipcMain 핸들러는 event.senderFrame(또는 webContents)이 메인 윈도우의 신뢰 로컬 file URL인지 확인 후 처리, 불일치 시 무시"를 추가. webContents.setWindowOpenHandler·will-navigate·will-attach-webview 차단을 MUST로 명문화하고 sub-frame 생성 금지.
게이트 차단 아님(심층방어 보강 권고).

EM-M-3 · 패키징 무결성·배포 신뢰성 미통제(미서명·asar·fuses) 중간

위치 설계 §7.2 패키징(코드 서명 "항목만") · §8 의존성 · ADR-EM-5 · 미해결 Q5
STRIDE 변조(T) · 위장(S)
분석 설계는 NSIS+포터블, asar:true, 미서명(SmartScreen 경고)·자동업데이트 범위 외를 적시한다. 보안 관점 미흡: ① 미서명 배포는 변조·위장(악성 재배포) 표면 — 단일 사용자 내부 도구로 수용 가능하나 위험 수용을 명시해야 한다(현재 Q5로 미결). ② asar는 패킹일 뿐 무결성 보장이 아님(추출·변조 가능) — Electron fuses(RunAsNode 비활성·EnableNodeCliInspectArguments 비활성·OnlyLoadAppFromAsar·EnableEmbeddedAsarIntegrityValidation)와 ELECTRON_RUN_AS_NODE 차단이 설계에 없다 → 패킹된 앱을 Node 런타임으로 악용하는 우회 표면. ③ files allowlist에 test/·docs/ 제외는 적절하나 설정/캐시(0600) 파일이 설치 디렉터리가 아닌 userData에 있는지(쓰기 가능 위치 분리) 명시 필요.
악용 시나리오 미서명 + fuses 미적용 패키지를 공격자가 변조·재배포하거나, ELECTRON_RUN_AS_NODE=1로 앱 바이너리를 임의 Node 실행기로 악용.
권고 조치 §7.2에 (1) Electron fuses 적용(RunAsNode off·OnlyLoadAppFromAsar·asar integrity)·ELECTRON_RUN_AS_NODE 무력화를 베이스라인으로 추가, (2) 미서명 배포는 위험 수용으로 ADR/문서 명시(SmartScreen·변조 위험), (3) 향후 서명 도입 시점·범위를 Q5에 결정 항목으로 고정.
게이트 차단 아님(베이스라인 보강 권고).

EM-L-1 · 외부 링크/shell.openExternal 정책 비고정 낮음

위치 설계 §6.2("외부 링크는 필요 시 shell.openExternal로 명시 처리(현재 불필요)")
STRIDE 권한상승(E) (URL 스킴 악용)
분석 현재 외부 링크가 없어 위험은 낮으나, shell.openExternal은 임의 URL 스킴(file:·smb:·custom:) 실행 표면이다. "필요 시"라는 여지가 남아 향후 추가 시 검증 없이 들어올 수 있다.
권고 조치 "외부 링크 미사용"을 기본 금지로 명문화하고, 향후 도입 시 https: 스킴 allowlist·사용자 확인을 전제로 한다고 §6.2에 못박기.
게이트 차단 아님.

EM-L-2 · sandbox "권장(권장)"의 비강제성 — 베이스라인이 약화될 여지 낮음

위치 설계 §2 SVG·§6.2 표·ADR-EM-3("sandbox:true(권장)")
STRIDE 권한상승(E)
분석 설계는 sandbox를 "권장"으로 표기하고, 본 설계가 모든 로직을 main에 두어 sandbox와 호환됨을 인정한다. 그러나 "권장"은 구현자가 편의상 끌 여지를 남긴다 — RC-2 환경에서 sandbox off는 renderer 폭발반경을 키운다. 호환성이 이미 확보됐으므로 강제할 수 있다.
권고 조치 ADR-EM-3에서 sandbox:true를 "권장"이 아닌 "필수(MUST)"로 격상(본 설계가 이미 호환). 구현 게이트에서 sandbox on 상태 스모크 검증 의무화.
게이트 차단 아님.

5. 구현 게이트 이월 (설계 PASS ≠ 코드 PASS)

베이스라인의 V-1(TOCTOU)·V-2(적대적 입력 실측)는 그대로 유효하며(도메인 코드 보존), Electron 전환으로 다음 항목이 추가 이월된다. 코드 보안 검증에서 실증할 것.

  • EC-1 (EM-H-1 실증): 전 IPC 핸들러가 인자 스키마(타입·길이·개수)를 main에서 거부하는지, browseDir(채택 시) scanRoots 하위로만 열람되는지, removeRoot가 canonicalize 정확 일치만 제거하는지 회귀 테스트. preload가 범용 invoke/ipcRenderer를 노출하지 않는지 스모크.
  • EC-2 (EM-M-1 실증): 패키징본에서 file:// 로 자산 로드 성공 + CSP 메타+헤더가 인라인/외부 스크립트·data 스크립트를 실제 차단하는지(오염 메타 픽스처로 XSS 미발화 확인).
  • EC-3 (EM-M-2 실증): will-navigate/setWindowOpenHandler/webview 차단이 실제 동작, ipcMain 핸들러의 senderFrame 검증이 비신뢰 프레임을 거부하는지.
  • EC-4 (EM-M-3 실증): Electron fuses(RunAsNode off 등) 적용·ELECTRON_RUN_AS_NODE 무력화·asar integrity가 빌드 산출물에 반영됐는지.
  • EC-5: sandbox:true 상태에서 전 기능 동작(preload가 우리 모듈 require 안 함) 스모크 — EM-L-2.
  • EC-6: CLI(scan.js)↔앱 캐시 경로 일치(ADR-EM-4/Q3) — 불일치 시 앱이 오래된/타 경로 캐시를 읽는 무결성 이슈.
비고: 의존성 정책상 런타임 dep 0이나 electron·electron-builder(devDep)는 공급망/취약점 추적 대상이다. 구현 착수 후 npm audit를 코드 게이트에 포함하고, Electron 버전의 보안 패치 추적(ADR-EM-1 결과의 "버전 추적 부담")을 운영 항목으로 둘 것. 현 단계는 설계 청사진이라 audit 미실행.

6. 확인 필요 (위험 수용 결정)

  • RC-1 재확인(추가조치 불필요): 동일머신 타 사용자 비격리는 베이스라인에서 PM 수용·M-2 파일권한 심층방어 적용. Electron 전환은 이를 바꾸지 않는다(userData 0600 보존 전제, EC-6).
  • RC-2 그대로 유효: 부분신뢰 스캔 대상 인정 → TB-B 통제 필수 보존. 본 검증에서 8/8 코드 접지 확인.
  • 신규 위험 수용 후보 — browseDir(Q2): 인앱 디렉터리 브라우저는 보안상 드롭 권고(EM-H-1). 채택 시 화이트리스트 제약 없이는 임의 경로 정보노출 채널이 되므로 PM이 명시 수용/제약 결정 필요.
  • 신규 위험 수용 후보 — 미서명 배포(Q5): SmartScreen·변조 위험을 PM이 명시 수용하거나 서명 도입 시점을 결정해야 한다(EM-M-3).
위 결정은 보안 게이트 통과의 전제가 아니다(차단 항목은 EM-H-1뿐). 다만 browseDir·미서명은 잔여 위험 수용이 명시돼야 추후 책임 소재가 분명해진다.
