종합 판정: CONDITIONAL PASS

근본 원인 분석과 두 방어선(A/B)의 방향은 코드 근거로 타당하다. 다만 위협 모델의 사실관계 한 곳이 실제 코드와 어긋난다: 설계가 보류 대상으로 지목한 "부팅 자동 스캔 / 주기 스캔 트리거"는 electron/main.js에 존재하지 않는다. 오염은 오직 사용자가 명시적으로 재스캔(spip:rescan)을 누를 때 writeSnapshot으로만 발생한다. b3 가드의 실제 적용 지점·문구가 이 사실에 맞게 교정되어야 구현이 정확해진다. 그 외 변경 지점·회귀·테스트 전략은 정합한다.

② 항목 1 — 근본 원인 정합 PASS

설계 §1.1의 주장(appDir()가 process.env.APPDATA || path.join(home,'AppData','Roaming')에 전적으로 종속)은 lib/common/paths.js 37–54행과 정확히 일치한다. 버전 게이팅·캐시 삭제·마이그레이션 로직이 없다는 주장도 snapshotStore.js(부재 시 graceful 빈 적재, 삭제 없음)·serializer.js(write만)에서 확인된다. 증상 4종(자동업데이트 직후만·일반재시작 정상·완전종료후 복귀·삭제코드 없음)은 "읽는 프로필이 바뀐 것"이라는 가설과 모순 없이 정합한다.

확인: os.homedir() 폴백 경로도 토큰 종속이라 상승 시 동일하게 어긋난다. 설계 §1.1 callout이 이를 정확히 짚었다.
잔여 불확실성 (설계도 인정): §1.3–1.4의 "언제 elevated 컨텍스트로 뜨나"는 NSIS 템플릿 소스 추론이며 해당 PC 1회 확인이 필요하다고 정직하게 명시했다. 이 한계 표기는 적절하다 — 메커니즘 자체는 확정, 트리거 조건만 미확정. 원인 판정에는 영향 없음.

③ 항목 2 — A안 타당성 PASS

electron-builder.yml 40–45행은 설계가 인용한 그대로(oneClick:false·perMachine:false·allowToChangeInstallationDirectory:true). 이 한 키를 false로 바꾸면 디렉터리 변경 페이지가 사라져 기본 per-user 경로(%LOCALAPPDATA%\Programs)로 고정 → 신규/재설치 사용자의 보호경로 상승 흐름을 제거한다는 논리는 NSIS assisted installer 동작과 정합한다.

트레이드오프 기술도 정확하다: (ⅰ) 설치 위치 선택 불가(로컬 단일사용자라 영향 미미), (ⅱ) 이미 보호경로에 per-machine으로 깔린 기존 사용자에겐 자동 적용 안 됨(HKLM 항목을 새 인스톨러가 읽어 여전히 상승 가능) — §④로 넘긴 처리도 정직하다. oneClick 전환·코드서명 비채택 근거도 합당.

경미한 확인 권고: allowToChangeInstallationDirectory:false가 기존 per-user 설치본의 자동 업데이트 경로에서도 상승을 유발하지 않는지는 한 번 스모크로 확인하라(설계 §⑤에 이미 포함됨). 신규 설치만 검증하고 "기존 per-user → 신규 자동업데이트"를 빠뜨리면 핵심 회귀를 놓친다.

④ 항목 3 — B안 (b3) 타당성·완결성 CONDITIONAL PASS

감지 수단은 견고하다. lib/common/safeExec.js는 절대경로(isAbsolute)면 resolveBin 없이도 실행을 허용하고 Windows에서 .exe만·shell:false·인자배열을 강제하므로, C:\Windows\System32\whoami.exe /groups 호출은 H-2 규약을 그대로 지킨다. SID(S-1-16-12288) 기준 판정은 한국어 Windows의 그룹명 로컬라이즈에도 영향받지 않아 적절한 선택이다. b1/b2 비채택 근거(네이티브 의존성·de-elevation 난이도)도 프로젝트의 "런타임 의존성 최소화" 원칙과 정합한다.

그러나 다음 세 가지가 설계에 명시되지 않아 구현이 흔들릴 수 있다:

쟁점문제
1차 휴리스틱의 false positive/negative "기대 %APPDATA%(=homedir/AppData/Roaming) vs 실제 불일치"는 APPDATA를 사용자가 비표준 위치로 리다이렉트한 환경(기업 로밍 프로필, 폴더 리다이렉션)에서 false positive를 낸다. 반대로 일부 상승 시나리오에선 APPDATA가 그대로 유지될 수도 있어 false negative. 설계는 "다르면 의심"이라고만 했고 1차 휴리스틱과 whoami SID 판정 중 무엇을 최종 권위로 삼는지가 모호하다.
감지 실패 시 기본동작 whoami.exe가 없거나(드문 환경) safeExec 타임아웃/exit≠0이면 elevated 판정을 어떻게 할지 미정. 안전망이라면 판정 불가 시 보수적으로 "비상승으로 간주(기존 흐름 유지)"가 옳다 — 잘못 elevated로 판정해 정상 사용자의 저장을 막으면 가용성 회귀다. 설계에 이 fallback 규칙이 없다.
오염을 막는 실제 지점(항목 4와 연결) 설계는 "자동 재스캔·저장을 막아 오염 방지"라고 하나, 실제 오염 경로는 사용자가 빈 화면에서 명시적으로 재스캔 버튼을 누를 때뿐이다(아래 항목 4 참조). "관리자 프로필 오염 방지" 목표 자체는 옳고 b3로 달성 가능하나, 위협 시나리오 서술이 부정확해 가드 위치가 잘못 잡힐 위험이 있다.
판정 근거: b3의 방향(감지 → 경고 + write 차단, 데이터 위치 불변)은 안전하고 의존성 0으로 달성 가능 → 채택 타당. 단 위 3개를 설계에 못박지 않으면 구현이 false positive로 정상 사용자를 막거나, 엉뚱한 곳에 가드를 넣어 오염을 못 막을 수 있다. 그래서 PASS가 아닌 CONDITIONAL.

⑤ 항목 4 — 변경 지점 정확성 FAIL

나열된 파일(elevationGuard.js 신규·main.js 훅·preload.js 수신 API·app.js 배너)은 기존 구조·패턴과 정합한다. autoUpdate.js 무변경 판단도 옳다(상승을 A로 제거하므로 인자 변경 불필요). 그러나 핵심 가드 지점 기술에 사실관계 오류가 있다.

핵심 결함: 설계 §B(2.B)·§③(main.js 행)·§⑤(M12 테스트)는 일관되게 "부팅 자동 스캔/주기 스캔 트리거 보류 플래그"를 둔다고 기술한다. 그러나 electron/main.js onReady()(187–354행)에는 scanController.start/acquire 호출이 전혀 없다. 코드 전역에서 스캔 트리거는 사용자 주도 spip:rescan → electron/ipc/actions.js rescan()(218·224행)에서만 발생한다. 즉 "보류할 부팅 자동 스캔"은 존재하지 않는 대상이다.

실제 오염 경로(코드 확인): 상승 세션에서 빈 목록이 뜬 사용자가 직접 재스캔 버튼/Ctrl+R을 누르면 → actions.js rescan() → scanController.start() → scanController.js 203행 serializer.writeSnapshot(...)가 관리자 프로필의 %APPDATA%\spip\cache\projects.json에 빈/잘못된 스냅샷을 쓴다. 이것이 유일한 오염 경로다.

주기 스캔도 없다: main.js가 시작하는 것은 stateWatcher.start(233행)뿐이며, stateWatcher는 git/freshness를 메모리 오버레이로만 갱신한다(snapshotStore.applyLiveState 주석: "디스크 영속은 하지 않는다"). 따라서 stateWatcher는 관리자 프로필을 오염시키지 않으며, "주기 스캔 트리거 보류"도 불필요하다.

무엇위치문제권고
존재하지 않는 "부팅/주기 자동 스캔 트리거 보류" 설계 §②B, §③ main.js 행, §⑤ 헤드리스 M12-2 설명 코드에 해당 트리거 없음 → 보류 플래그가 가공의 대상을 막음. 구현자가 main.js에서 막을 게 없어 혼란 위협 서술을 "상승 세션에서 사용자가 명시적 재스캔 시 write 차단"으로 교정. main.js 훅은 "감지 + 플래그 set + 경고 push"만 책임지고, 자동 스캔 보류 문구는 삭제
write 차단 가드의 정확한 단일 지점 설계 §③ "scanController.js 또는 스캔 트리거 지점" "또는"으로 두 후보를 흐릿하게 제시. 어디에 넣을지 미확정이라 구현이 양쪽에 흩어지거나 누락될 수 있음 단일 지점으로 확정: (권장) actions.js rescan() 진입에서 elevated면 {ok:false, code:'ELEVATED'} 조기 반환 — acquire/start 이전에 막아 write 자체가 시작되지 않음. 또는 차선으로 scanController.start의 writeSnapshot 직전 가드 1곳. 둘 다 넣지 말고 하나만
config/ui-state write 경로 누락 설계 §③ (스냅샷 저장만 언급) 상승 세션에서 사용자가 설정 변경·즐겨찾기·todo를 만지면 configPath()·uiStatePath()도 관리자 프로필에 쓰여 오염. 설계는 cache(projects.json) write만 막음 오염 대상이 cache뿐인지 재검토. 경고 배너로 "일반 실행으로 다시 여세요"를 강하게 유도하면 실질 위험은 낮으나, 상승 시 설정/UI 상태 write도 보류 대상에 포함할지 설계에서 결정·명시
주의 — 가용성 트레이드오프: rescan을 막으면 "관리자 권한으로 일부러 실행해 스캔하려는" 사용자는 스캔을 못 한다. 그러나 상승 세션은 어차피 잘못된 프로필을 읽으므로 스캔 결과가 무의미하다 → 막는 것이 정답. 다만 사용자에게 "왜 막혔는지"가 배너로 명확히 전달되어야 한다(설계 문구 OK).

⑥ 항목 5 — 회귀·기존 사용자 PASS

  • 비상승·portable·dev 불변: B 감지가 "상승일 때만" 분기하고, A는 신규 설치 경로만 바꾸므로 회귀 위험 낮음. 단 항목 3의 "감지 실패 시 보수적 비상승 기본동작"이 보장되어야 이 불변이 실제로 성립한다(미보장 시 false positive로 정상 사용자 회귀).
  • 마이그레이션 현실성: 데이터가 Roaming(%APPDATA%\spip)에 있고 deleteAppDataOnUninstall 미설정이라 제거/재설치와 무관하게 보존 — 코드·yml과 정합. 클린 재설치 권장 + b3 안전망 조합은 강제 불가(미서명·로컬) 환경에서 현실적이다.
  • 위치 불변 원칙: 데이터 이동 스크립트를 만들지 않는다는 결정이 §C·§⑥ 보안표와 일관. 적절.
경미: "1순위 자동 안전망(b3)으로 실질 피해 없음"은 b3가 실제로 모든 write 경로(cache + 가능하면 config/ui-state)를 막을 때만 성립한다 → 항목 4의 write 지점 확정과 연동된다.

⑦ 항목 6 — 테스트 전략 CONDITIONAL PASS

헤드리스(판정·가드 단위) / 수동 스모크(실제 상승) 분리는 타당하다. detectElevation()을 process.env.APPDATA mock으로 순수 검증, whoami는 safeExec 모킹으로 SID 파싱만 검증, 실제 상승은 스모크로 미루는 구획이 합리적이다.

  • M12-2 가드 테스트 위치 의존: "상승 플래그 set 시 스냅샷 저장 스킵"을 어디서 검증할지가 항목 4의 가드 지점 확정에 달려 있다. rescan() 진입 가드로 확정하면 테스트는 actions.js rescan()이 elevated 시 code:'ELEVATED'를 반환하고 acquire가 호출되지 않음을 검증해야 한다(현재 설계 문구는 scanController 저장 가드를 가정 → 가드 지점이 바뀌면 테스트도 바뀜).
  • 누락 권고: "감지 실패(whoami 부재/타임아웃) → 비상승 기본동작" 테스트, "APPDATA 리다이렉트 환경에서 false positive 방지" 케이스를 추가하라. 항목 3의 두 미정 규칙을 테스트로 못박으면 그대로 검증된다.
  • 스모크 강화: §⑤ 스모크에 "기존 per-user 설치 → A 적용 신규버전 자동업데이트 → 상승 안 뜸 + 데이터 보존"을 명시 추가(현재는 신규 설치·관리자 실행만 또렷함).

⑨ 가장 위험한 약점

위협 모델과 코드의 불일치(P1). 설계가 막으려는 "부팅 자동 스캔"은 코드에 없고, 실제 오염은 "사용자 명시 재스캔 write" 하나다. 이 사실을 바로잡지 않으면 구현자가 (ⅰ) main.js에서 막을 게 없어 헤매거나, (ⅱ) scanController write 가드만 넣고 actions.js 진입을 안 막아 acquire/락이 헛돌거나, (ⅲ) config/ui-state write를 빠뜨려 오염이 잔존할 수 있다. 가드 지점을 actions.js rescan() 진입 단일 지점으로 확정하면 깔끔하게 해소된다.
