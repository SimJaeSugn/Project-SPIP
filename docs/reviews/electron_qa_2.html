2. 블로커 #1 — open 채널 정합 (BUG-1 / P1-1) PASS

증상(1회차): preload가 openInVsCode 키로 노출 → renderer의 ipc('open', id)가 window.spip.open 부재로 항상 bridgeMissing() 반환 → R-12 VS Code 열기 전면 불능.

검증 항목방법결과판정
preload가 open: (id)=>invoke('spip:openInVsCode',{id})로 노출정적 + 헤드리스electron/preload.js:23 — 채널명 spip:openInVsCode 유지, 표면 키는 openOK
renderer ipc('open', id)가 핸들러에 실제 도달정적 교차app.js:1988 openProject()가 ipc('open', id) 호출 → spip.open 존재 시 invoke 도달OK
typeof window.spip.open === 'function' (런타임)헤드리스 실측openType: "function"OK
openInVsCode 키 부재 (런타임)헤드리스 실측openInVsCodePresent: falseOK

런타임 노출 키 실측: ["addRoots","getConfig","getHealth","getProjects","getScanStatus","getStats","onMenu","onScanProgress","open","pickFolders","removeRoot","rescan"] — open 포함, openInVsCode 없음.

3. 회귀 테스트 검출력 — electron-preload-contract PASS

devops가 추가한 정합 회귀 테스트가 실제로 메서드명 누락을 잡는지를 신뢰가 아닌 실측으로 검증했다. preload의 open:을 openInVsCode:로 임시 revert 후 테스트 재실행, 그 후 즉시 복원.

상태실행결과

4개 테스트 케이스: ① renderer 호출 ⊆ preload 노출 (집합 대조) ② open 노출 + 채널 정합 + openInVsCode 부재 ③ onMenu action 화이트리스트·채널·unsubscribe ④ menu.js send 집합 ⊆ onMenu 구독 집합(고아 채널 금지). → 검출력 실증 완료.

4. 블로커 #2 — 메뉴 payload 정합 (P2-1) PASS

1회차에서 우려된 "문자열 전달 시 무동작" 불일치는 사라졌다 — preload가 객체를 보내고 frontend도 객체를 기대하므로 4개 액션 모두 디스패치된다.
