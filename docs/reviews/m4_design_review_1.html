5. 실행가능성 (구현 착수 가능 여부)

백엔드/프론트가 즉시 착수하기엔 블로커가 있다. 인터페이스 시그니처(ScanController·driveEnum·walker)는 제시됐으나, 아래가 미결이라 그대로 구현하면 재작업이 발생한다.

  • (블로커) onProgress/dirs 산출 경로 — 위 P1-1. scanner 루프 구조 변경 여부가 정해지지 않아 R-15 백엔드 착수 불가.
  • (블로커) router 수정 누락 — 위 P1-2. rescan POST+M-1 가드, scan-status GET 라우트 추가가 파일 델타에 없어 라우팅 착수 모호.
  • (조정) finalizing/scanId/stats 집계 위치 — P2. 프론트 상태머신·백엔드 집계 지점이 미정.
  • 나머지(size 예산값·depthLimit 24/64/12·driveEnum OS분기)는 의사코드/상수로 충분히 구체적 — 착수 가능.
결론: P1 2건은 "곧바로 구현 착수"를 막는 블로커다. 설계가 의사코드 수준에서 onProgress 데이터 흐름과 router 분기를 명시하면 착수 가능 상태가 된다.

7. 반영 항목 (생산자에게)

P1 — 블로커 (재검 전 필수)

#무엇 / 위치 / 문제 / 권고
P1-1 무엇: scanner onProgress 정합 기술 오류 — dirs 산출 불가.
위치: m4-design §3.2(콜백 확장) · §2.5 ScanController.
문제: 설계는 "MVP onProgress(count), walker yield마다 dirs++, scan.js 1줄 수정"이라 하나, 실제 scanner.js:148-150은 onProgress(projects.length, project)로 프로젝트 확정 시에만 호출된다. 디렉터리 순회 수(dirs)와 currentPath를 현재 호출 지점에서 얻을 수 없어 R-15 진행 표시가 디렉터리 진행을 반영하지 못한다.
권고: dirs/currentPath를 walker 순회 단위로 얻는 방식을 의사코드로 확정하라 — (a) walker.walk에 진행 콜백 인자 추가(시그니처 확장, 전진 호환), 또는 (b) scanner가 candidateStream을 직접 소비하며 카운트(detector 분리). "1줄 수정" 문장은 삭제하고 실제 변경 범위를 §10 파일 델타에 반영(walker.js 또는 scanner.js 루프 변경).
P1-2 무엇: router.js 수정 누락 — rescan M-1 선통과 + scan-status 라우트 추가.
위치: m4-design §2.2(M-1 선통과 주장) · §8.1(엔드포인트) · §10(파일 델타).
문제: 현재 router.js:66-69는 rescan을 메서드 무관 404로 처리하며 checkStateChange를 호출하지 않는다. 또 GET /api/scan-status 라우트가 router에 없다. 설계 §10 파일 델타에는 actionHandlers/apiHandlers 수정만 있고 router.js 수정이 빠져 있어, "라우터가 선통과시킨다"는 주장과 모순된다.
권고: §10 파일 델타에 router.js 수정을 추가하고, rescan을 POST한정 + security.checkStateChange 선통과(/api/open와 동형)로, scan-status를 GET 분기로 추가하는 변경을 명시하라.

P2 — 착수 전 조정

#무엇 / 위치 / 문제 / 권고
P2-1finalizing phase 미정의. 위치: §3.2·§8.3 ScanProgress. 문제: phase 열거에 finalizing이 있으나 진입 시점·counts 유무·프론트 처리 미정. 권고: finalizing 전이 조건(writeSnapshot 진행 중 등)과 그때 dirs/found/counts 값을 정의하거나, 불필요하면 열거에서 제거.
P2-2scanId 형식·생성 주체·대조 규약 모호. 위치: §2.3·§2.5·§8.3. 문제: hex 길이·생성 위치(acquire vs scanner)·202의 scanId와 status.scanId 동일성·프론트가 자기 scanId와 대조하는지 미정. 권고: scanId는 ScanController.acquire가 발급(예 crypto 8~16바이트 hex), 202와 모든 status에 동일 값 반환, 프론트는 자기 scanId와 다르면 무시(타 스캔 혼선 방지)로 규약 고정.
P2-3size collector 계약 status 레이어 충돌. 위치: §4.2 vs architecture.html §5 collectors 계약(H-3). 문제: 공통 collector 계약 status는 'ok'|'na'|'error'(architecture.html:362)인데, size data.status는 'ok'|'partial'|'skipped'|'error'로 다른 값 집합을 쓴다(skipped/partial은 계약 status에 없음). 현재 size.js도 collect의 status='skipped'를 반환해 이미 계약과 어긋남. 권고: collect 반환의 계약 status(ok/na/error)와 size data.status(ok/partial/skipped/error)를 두 레이어로 명확히 구분 서술하고, partial일 때 계약 status는 무엇인지(ok? na?) 명시.
P2-4stats.totalBytes 집계 위치 미지정. 위치: §4.2 주석·§8.4. 문제: "enabled 스냅샷이면 number"라 하나 합계 산출 지점(scanner가 snapshot에 totalBytes 저장 vs getStats가 projects 합산)이 없음. 현재 getStats는 null 고정. 권고: 산출 위치 1곳 확정(예: scanner snapshot.counts/stats에 totalBytes 합계 저장, getStats는 읽기만) — partial 항목 합산 정책(부분값 포함/제외)도 명시.

9. 검증 범위 / 경계

본 검증은 문서·계획의 추적성·일관성·완결성·실행가능성에 한정한다. 보안 취약점·신뢰 경계·인가·시크릿의 적정성은 security-reviewer 소관이며, 본 보고는 추적성 수준(L-1/H-1/H-3/M-1 통제가 설계에서 약화 없이 언급되는가)만 점검했다 — 통제 언급 자체는 일관되게 유지됨. 코드 동작·빌드·수용기준 실측은 qa-engineer, 코드 품질은 code-reviewer 소관. 본 검증자는 산출물을 직접 수정하지 않으며 반영은 생산자가 수행한다.
