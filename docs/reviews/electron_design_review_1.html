FAIL

설계의 핵심 전략(도메인 무수정 보존 + HTTP 전송만 IPC 교체)은 코드 실측으로 대부분 정확히 입증되며, 재사용/드롭 매핑·보안 경계 전환·요구 추적·테스트 보존 전략은 견고하다. 그러나 진행 통지(R-15) 전환 설계가 실제 scanController API와 어긋나 "컨트롤러 무수정" 주장이 성립하지 않는다 — 이것이 단계 5의 착수를 막는 P1 결함이다. 추가로 addRoots IPC 계약의 rejected[]가 재사용 함수로 공급 불가(P2), actionHandlers의 apiHandlers 결합 누락 언급(P2), PM 확정 Q1/Q2/Q4가 본문이 아닌 "미해결 질문"에만 권고로 남아 확정 반영 누락(P2) 등이 있다. P1 1건 해소 + P2 반영 후 재검 필요.

체크리스트 1 — 재사용/드롭 매핑 정확성 PASS

코드 실측으로 매핑이 정확함을 확인했다.

  • "재사용" 검증: snapshotStore.js는 fs·paths·pathGuard·logger만 require, HTTP 의존 0(res/req/http 없음). scanController.js는 crypto·scanner·serializer·logger만 의존, HTTP 0. lib/scan/*·lib/common/*도 전송 무관. → "무수정 재사용" 타당.
  • "드롭" 검증: router/staticHandler/security는 HTTP 디스패치·정적서빙·Host/Origin/토큰 전용. server.js는 http.createServer/listen/token 보유하나 createServer()가 composition root(loadConfig→store→ScanController→ctx)를 분리·export → main 승계 타당.
  • "로직만 이식" 검증: apiHandlers/actionHandlers 모두 (req,res,ctx) 시그니처에 sendJson/res.writeHead/readJsonBody가 결합돼 있고, 그 사이에 도메인 호출(pathGuard.canonicalize/isAllowed·resolveBin/safeExec·controller.acquire/start)이 끼어 있다. HTTP 응답부 분리해 (args,ctx)→result 추출 가능 — 타당.
소결: 설계의 분류 표(§3)는 코드 사실과 일치. 이 부분은 신뢰 가능하다.

체크리스트 2 — IPC 계약 완결성 P1 결함

채널 매핑·shape 불변은 대체로 양호하나, 진행 통지 전환이 실제 컨트롤러 API와 모순된다.

  • 채널 망라: projects·stats·health·open·rescan·scan-status·scanProgress·pickFolders·config·addRoots·removeRoot 모두 매핑 — 누락 없음. Project/ScanProgress shape 불변 명시 — OK.
  • [F-1 / P1] 진행 폴링→푸시 전환이 scanController와 정합하지 않는다. 설계 §4.3 의사코드는 onProgress를 래핑해 scanController._merge(p,scanId); win.webContents.send(...)로 중계하면 "컨트롤러 무수정"이 된다고 주장한다. 그러나 실측: start(opts)는 {config,withSize,allDrives,roots,store,logger,cachePath}만 받고 onProgress를 인자로 받지 않으며, 내부에서 onProgress:(p)=>this._merge(p,myScanId)를 하드코딩(scanController.js:167)한다. 즉 main이 가로챌 onProgress 훅이 존재하지 않는다. 현행 HTTP 모델의 진행 노출은 오직 status() 폴링이며 컨트롤러는 push를 모른다.

무엇이R-15 진행 이벤트 푸시(spip:scanProgress) 설계 §4.3 — "scanController가 이미 갖고 있는 onProgress를 중계" / "컨트롤러 무수정"
어디서electron-migration.html §4.3(라인 291·305~308) / §3 표 scanController 행(라인 243) / §11 R-15 행

권고택1을 명시: (A) start(opts)에 선택적 onProgress 파라미터 추가(컨트롤러 소폭 수정 — 기존 _merge는 유지하고 사용자 콜백을 추가 호출). 이 경우 "무수정" 표현을 "최소 1점 수정"으로 정정. (B) 컨트롤러 진짜 무수정 유지 시, main이 setInterval로 controller.status()를 폴링해 webContents.send push(폴링은 main 내부로만 이동, renderer는 이벤트 구독). 이 경우 "폴링 폐기" 표현을 "renderer 폴링 폐기, main 내부 폴링 또는 이벤트 브리지로 단순화"로 정정. 어느 쪽이든 _merge가 private(_접두)임을 명시하고 의존 여부 결정.

체크리스트 4 — 테스트 이관 전략 PASS P3 경미

  • 도메인/재사용 모듈 테스트 보존: collectors·scanner·walker·detector·config·paths·pathGuard·safeExec·snapshotStore 테스트는 무수정 모듈이라 회귀 0 — 타당.
  • HTTP 테스트(server.test.js·security-h2.test.js·staticHandler.test.js·m4-rescan.test.js·dashboard-m4.test.js)의 IPC 이관: (req,res)→(args,ctx) 순수함수 추출이 mock 제거에 유효 — 현실적. createServer()가 listen 없이 핸들러 검증 가능하게 이미 분리돼 있어 추출 비용 낮음.
  • [F-6 / P3] "213 테스트" 수치 근거 약함. 실측 test/ 18개 파일, test()/it() 호출 약 200개. 213이 subtest 포함 총합인지 명시 안 됨 — 회귀 위험 평가의 분모이므로 출처를 한 줄 보강 권고(예: "node --test 케이스 총계").
  • security.js 테스트 "제거 또는 보관"은 PM Q1(보존 후 삭제)과 정합 — OK.

체크리스트 5 — 마이그레이션 단계 실행가능성 P2

  • 0~8 단계의 의존성 순서(스캐폴드→composition root→preload+읽기→어댑터→액션→진행→폴더→하드닝→패키징)는 점진적이고 각 단계 동작확인 기준이 있어 대체로 착수 가능.
  • [F-2 / P2] 단계 5(진행 푸시)는 F-1 미해결 상태로는 착수 불가 — 단계 진입 전 컨트롤러 수정/폴링 택일 선결 필요. 단계 4(액션)에는 actionHandlers가 require('./apiHandlers')의 sendJson에 결합(actionHandlers.js:25)돼 있어, 로직 추출 시 이 결합을 함께 끊어야 함이 단계 산출에 명시돼야 한다.
  • [F-3 / P2] 헤드리스 환경 GUI 검증 한계 미인지. 단계 0·2·4·6·8의 "빈 창이 뜨는지/카드 렌더/VS Code 열기/설치본 더블클릭" 확인은 모두 디스플레이·OS 대화상자·실제 spawn을 요구한다. 본 설계는 이 검증이 CI/헤드리스에서 불가함을 인지·대안(자동 가능 범위=순수함수/ctx 주입 단위테스트 vs 수동 GUI 스모크 분리, dialog/spawn mock 경계)을 제시하지 않았다. 추가 권고: 각 단계 "검증" 칸에 [자동 가능 / 수동 GUI 필요]를 태깅하고, dialog·webContents·shell 등 Electron 표면은 ctx 주입으로 모킹 가능하게 경계를 그어 둘 것.
  • [F-4 / P2] PM 확정 사항이 본문 결정이 아닌 "미해결 질문" 권고로만 남음. PM은 Q1(이전완료까지 보존 후 삭제)·Q2(인앱 브라우저 드롭)·Q4(종료 시 스캔중 확인 다이얼로그)·Q6(Project-SPIP/com.spip.app)를 확정했는데, §13 미해결 질문은 여전히 "권고"·"필요?" 어투(Q1 "권고", Q2 "드롭 권고", Q4 "권고: 확인 다이얼로그")로 열려 있다. 설계 본문(§3 server.js 행·§5 브라우저 브라우저 행·§7 종료 행)과 확정 결정의 일관성을 위해 해당 항목을 확정(승인)으로 승격하고 미해결에서 제거해야 추적성이 깨지지 않는다. 특히 §7 "진행 중 스캔과 종료" 표는 "watchdog가 정리/조용히 중단" 서술로, 확정된 확인 다이얼로그와 모순된다(Q4 확정 미반영).

체크리스트 6 — 패키징/의존성 정책 PASS

  • electron·electron-builder를 devDeps, 런타임 deps 0(우리 lib+내장만) — 실측 현 package.json deps 부재와 정합. main:"electron/main.js"·start:"electron ." 변경, scan/bin 유지, node:test 유지 — 타당.
  • NSIS+포터블, files allowlist(test/docs 제외), asar:true, appId com.spip.app·productName Project-SPIP — PM Q6 확정과 일치.
  • engines.node>=18은 CLI 기준 유지 명시 — 합리적.
  • ADR-EM-4(paths.js 공유, PM Q3 확정)와 정합. [F-7 / P3] 단 paths.appDir()=%APPDATA%\spip는 productName Project-SPIP와 폴더명이 다르다(spip vs Project-SPIP). app.getPath('userData')를 안 쓰므로 충돌은 없으나, 사용자가 두 위치를 혼동할 소지 → 본문에 "데이터 폴더명은 spip 고정(productName과 무관)"을 한 줄 명시 권고.

F-5P2addRoots rejected[] 공급 불가. §4.1/§5는 spip:addRoots가 {ok:false,code:'INVALID_PATH',rejected:string[]}를 반환한다 하나, 재사용 대상 config.normalizeScanRoots는 거부분을 continue+로그로 버리고 채택분 배열만 반환(config.js:90~126). 즉 어댑터가 rejected를 그 함수 출력만으로 못 만든다. 권고: (a) 입력−출력 차집합으로 어댑터가 rejected 계산하거나, (b) normalizeScanRoots를 {accepted,rejected} 반환으로 확장(소폭 수정·테스트 동반). 계약에 "rejected 산출 방식"을 명시할 것.
