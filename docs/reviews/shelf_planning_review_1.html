요약. 로드맵(SH-1~5)과 IPC 계약 본체는 설계 마일스톤(SH-M1~M5)·실제 코드 구조와 잘 맞물리고 추적성·단계 의존성·DoD가 충실하다. 다만 설계 SSOT가 PM 확정 결정을 따라오지 못해 로드맵/계약과 모순(og:image 96KB↔100KB, 스케줄러 기본값 Q2, 파일 열기 도구 Q4)하고, SCHEMA_VERSION 이행 로직의 실제 코드 변경 지점(normalizeState가 obj.schemaVersion을 안 읽음)이 빌드 단계에서 과소 명세되며, validateHttpUrl 일반화 회귀 테스트가 계획의 테스트 산출물에 명시되지 않았다. 이 세 가지가 구현 착수 전 반영 대상이다. 나머지 핵심 포인트(CSP 무변경·경로 표면·단계 의존성)는 통과.

1. 포인트별 상세

P1 — 2변형 등록 + 기본 숨김 이행 [조건부 PASS]

확인된 정합. 실제 코드에서 HOME_SECTION_IDS는 uiStateStore.js:56과 app.js:646이 완전 동형(8항목: attention/productivity/activity/todos/mail/disk/aiusage/featureAdd). TOGGLEABLE_WIDGET_IDS = HOME_SECTION_IDS - featureAdd이므로 'shelf'·'shelfWide' 추가 시 자동으로 토글 가능 위젯이 되고, normalizeHiddenWidgets·app.js:6461 필터가 둘을 통과시킨다. WIDGET_META(app.js:651)는 featureAdd 미포함이며 계획이 2항목 추가를 명시 — 정합. 로드맵 SH-1이 SCHEMA_VERSION 2 상향 + defaultState().hiddenWidgets 시드 + 기존 사용자 1회 union 이행을 제시하고, shelf-uistate.test.js에 "기본숨김 이행" 테스트를 둔다 — 골격은 옳다.

조건(반영 필요). 실제 normalizeState(uiStateStore.js:177-191)는 obj.schemaVersion을 전혀 읽지 않고 출력값을 항상 상수 SCHEMA_VERSION으로 덮어쓴다. 즉 "구버전 감지 후 union" 이행을 하려면 normalizeState가 입력의 schemaVersion을 먼저 읽어 분기하도록 바꿔야 하는데, 이 코드 변경이 SH-1 빌드 단계 표에는 없고 말미 "정합성 포인트" 주석에만 서술돼 있다. 또한 이행 술어가 obj.schemaVersion < 2 형태면 schemaVersion이 없는(undefined) 레거시 파일에서 undefined < 2 === false가 되어 이행이 건너뛰어진다. 구현자가 이를 놓치면 "기존 사용자에게 즉시 노출" 회귀가 그대로 발생한다.

P2 — og:image 상한 100KB 통일 [FAIL]

로드맵 SH-3(imageCache ≤100KB, line 528·554)과 IPC 계약(≤100KB, line 122·153)은 PM #4를 정확히 반영해 100KB로 통일됐다. 그러나 설계 SSOT 본문은 여전히 96KB다 — §2 품질카드("이미지 ≤ 96KB"), §9 크롤 SVG("image/* · ≤96KB"), §11("크기 상한 96KB"). 로드맵 본인도 line 554에서 이 불일치를 인지하고 "정합화"를 적었지만 SSOT 문서 자체가 미수정이다. "단일 진실"로 선언된 문서가 계약과 모순되면 구현자가 SSOT를 따라 96KB로 구현할 추적성 위험이 남는다. 테스트는 아직 미작성(계획 단계)이라 100KB로 쓰면 되나, 산출 기준 문서가 갈라져 있는 한 "통일됨"으로 판정할 수 없다.

P3 — CSP 무변경 [PASS]

실측 public/index.html:13-14 CSP = default-src 'none'; ... style-src 'self'; img-src 'self' data:; ... connect-src 'none'; .... 배너 <img src="data:...">는 기존 img-src 'self' data: 허용 범위 내 — CSP 무변경 타당. connect-src 'none' 유지로 모든 egress가 main IPC로 강제됨도 정합. 특히 style-src 'self'에 'unsafe-inline'이 없어 마크업 style="" 속성은 CSP가 차단하므로, 로드맵이 렌더 스타일을 node.style.cssText(CSSOM)로 적용하고 표시 문자열은 textContent로만 넣겠다는 계획(line 512·555)은 필요하고 정확하다. L-1(innerHTML 금지) 준수 계획도 계약·로드맵 양쪽에 명시.

경미. 설계 §11/머리말은 CSP를 "실측 electron/main.js CSP"로 지칭하나, 실제 CSP는 index.html meta에 있고 main.js엔 CSP 설정이 없다(grep 무결과). 동작엔 무영향이나 SSOT 참조 정확도 차원에서 정정 권장.

P4 — validateHttpUrl 일반화 회귀 [FAIL]

로드맵 SH-1은 validateBriefingUrl을 validateHttpUrl(input,{maxLen})로 일반화하되 기존 호출은 별칭/래퍼로 무변경 유지한다고 명시 — 설계 방향은 타당하다. 실측상 config.js의 validateBriefingUrl는 normalizeBriefing(config.js:319)에서 호출되고, 기존 회귀 가드로 test/ai/config-briefing.test.js가 이미 존재한다. 문제는 계획이다: SH-1 테스트 산출물 목록은 shelf-uistate·shelf-detectType·shelf-pathPolicy만 열거하고, validateHttpUrl 단위 테스트도, 기존 briefing 회귀 보존 검증도 테스트 deliverable로 명시하지 않았다(line 556의 "회귀 확인"은 검토 체크 문장일 뿐 테스트 항목이 아님). 회귀 안전망이 계획에 못박히지 않으면 일반화 중 시그니처/반환형 드리프트가 briefing을 조용히 깰 수 있다.

P5 — 경로 표면 일관 [PASS]

IPC 계약(line 99·102·154)과 로드맵(line 506·519·530·557)이 일치한다: add는 folder/file → canonicalize + pathPolicy, open은 열기 시점 재-canonicalize + 재-pathPolicy 게이트, file=shell.openPath(OS 기본앱) / folder=safeExec(resolveTool('code'),[real],{shell:false,detached:true}) / url=shell.openExternal+http/https 재검증, 인자 [real] 고정·id별 in-flight 상한. 이는 PM #2와 정확히 합치한다. register.js의 guard()·broadcastFavorites 화이트리스트 패턴이 실재하므로 shelf 채널 guard 등록·broadcastShelf(메인 wc만) 계획도 실현 가능.

주의(SSOT 모순, P2와 연동). 설계 §8 IPC표(line 470)·§10.3(line 591)은 "folder/file → safeExec(code) ... 도구 미설정 시 openPath 폴백"으로 file도 VS Code 우선이라 적혀 있어 PM #2(file=OS 기본앱 항상)와 어긋난다. 이는 설계의 미해결 질문 Q4가 PM에서 확정됐는데 SSOT에 반영되지 않은 잔재다. 로드맵/계약은 올바르나 SSOT가 모순이므로 설계 문서 정정이 필요하다.

P6 — 단계 의존성·DoD·추적 [PASS]

SH-1~5가 설계 SH-M1~M5에 1:1로 추적된다(SH-1↔M1 영속·순수코어, SH-2↔M2 로컬+홈마운트, SH-3↔M3 크롤+og:image, SH-4↔M4 refresh+스케줄러, SH-5↔M5 마감). 의존성 그래프(SH-1 토대 → SH-2/SH-3 병렬 → SH-4 → SH-5, SH-3는 security-reviewer 위협모델링 게이트 후 착수)는 설계의 "안→밖·로컬 우선·egress 게이트" 순서와 정합하며 순환 없음. 각 단계에 DoD·동반 테스트·역할(#3 백엔드/#4 프론트/#5 검증)·태스크 의존이 명시됐고, "인터페이스 먼저"(계약 합의 후 #3-b/#4-a 병렬) 원칙이 경계면 불일치를 예방한다. IPC 계약의 채널표·ShelfBookmarkView shape·고정 에러코드·preload 1차 고정이 설계 §8과 일치.

2. 반영 항목 (우선순위)

우선무엇위치문제 / 권고
P1-필수 SCHEMA_VERSION 이행을 빌드 단계로 승격 로드맵 SH-1 표(uiStateStore.js 행) / normalizeState 문제: 실제 normalizeState는 obj.schemaVersion을 읽지 않고 항상 상수로 덮어쓴다 — "구버전 감지 후 union" 이행이 현재 코드에선 동작 불가. 빌드 단계 표엔 이 변경이 없고 주석에만 있다. 권고: SH-1 표에 "① normalizeState가 입력 obj.schemaVersion을 읽어 분기 ② schemaVersion < 2 또는 부재(undefined)면 ['shelf','shelfWide']를 hiddenWidgets에 union(멱등) ③ 출력 schemaVersion=2"를 명시하고, shelf-uistate.test.js에 (a) 구버전 union (b) schemaVersion 부재 케이스 (c) 사용자가 이미 unhide한 v2 파일은 재숨김 안 함(멱등·의도 보존) 케이스를 못박을 것.
P2-필수 설계 SSOT를 PM 확정으로 정합화 설계 shelf-widget-design.html §2·§9 SVG·§11(96KB) · §8 IPC표·§10.3(file 열기) · §12 Q2 · §16 Q1/Q2/Q4 문제: "단일 진실" 문서가 PM 확정과 모순 — og:image 96KB(↔100KB), file 열기 VS Code 우선(↔OS 기본앱), 스케줄러 기본값 미해결/off 권장(↔기본 ON). 구현자가 SSOT를 따르면 계약과 어긋난 구현이 나온다. 권고: §2·§9·§11의 96KB→100KB, §8/§10.3 file 열기를 shell.openPath(항상)로, §12 스케줄러 기본값을 ON+끄기 토글로 수정하고, §16 Q1/Q2/Q4를 "PM 확정으로 해소(YYYY-MM-DD)"로 마감 표기.
P3-권장 validateHttpUrl 회귀 테스트를 deliverable로 명시 로드맵 SH-1 테스트 행 문제: 일반화 회귀 안전망이 검토 문장에만 있고 테스트 산출물에 없다. 권고: SH-1 테스트에 "validateHttpUrl 단위(scheme/자격/길이/maxLen 경계) + 기존 test/ai/config-briefing.test.js 그린 유지(별칭 시그니처·반환형 {ok,value,external} 불변)"를 명시할 것.
P4-경미 "두 변형 동시 추가" 잔여 미확정 표기 정리 로드맵 line 553 문제: 로드맵이 "둘 다 추가 시 동일 셸프 2개 — PM 확인 필요"로 남겨두었으나 PM #3에서 동시 추가 허용으로 이미 확정. 권고: "PM 확정: 동시 추가 허용(단일 shelfBookmarks 공유)"로 갱신해 미해결 인상 제거.
P5-경미 CSP 출처 참조 정정 설계 §11·머리말 문제: CSP를 electron/main.js로 지칭하나 실제는 index.html meta. 권고: 참조 경로를 public/index.html CSP meta로 정정(동작 무영향, 추적 정확도).

3. 확인 필요 (PM 판단)

  • SSOT 정정 책임 주체. 설계 문서(아키텍트 산출물)를 PM 확정에 맞춰 누가 갱신할지 — 본 검증은 로드맵/계약을 대상으로 하나, 모순의 근원은 설계 SSOT다. SSOT를 그대로 두면 향후 구현/검증에서 반복 충돌한다.
  • P2를 블로커로 볼지. 로드맵/계약은 PM 확정과 일치해 "구현은 100KB로 진행 가능"이다. SSOT 정정을 구현 착수의 선행조건으로 강제할지, 병행 보정으로 둘지 PM 판단을 요청한다(본 검증은 SSOT 모순을 FAIL 사유로 본다).
