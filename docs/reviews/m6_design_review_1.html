2. 기존 정합·전진호환 — PASS (실측 대조 통과)

2.1 resolveBin 무수정 재사용 주장 — 실측 일치 ✔

설계는 "safeExec.resolveBin이 이미 name이 절대경로면 .exe·실행파일 검증만 수행(safeExec.js:82)" → resolveBin 자체 미변경, resolveTool 래퍼에서 두 번 호출로 충족을 주장한다. 실측 확인 결과 정확하다.

resolveBin 결과 캐시(_binCache)가 있어, 사용자 지정 경로/PATH 결과가 캐시된다. 사용자가 설정에서 경로를 변경했을 때 캐시 무효화가 필요할 수 있으나(opts.force 존재), 설계는 이를 다루지 않는다 — 경미(P3-2 참조).

2.2 openInVsCode toolId 일반화 — 계약 비파괴 ✔

실측 actions.openInVsCode(args, ctx)(line 64)와 preload open:(id)=>{id:String(id)}(preload.js:23). 설계의 open(id, toolId?) 확장은 toolId 미지정 시 'code'로 폴백(§4.1·§6.2)이라 기존 호출 무변경 동작 — 전진호환 성립. api-contract/electron-migration의 기존 응답코드(ID_NOT_FOUND·PATH_NOT_ALLOWED·PATH_GONE·CODE_CLI_NOT_FOUND·OPEN_FAILED) 보존 + TOOL_NOT_FOUND 추가. 기존 코드 깨짐 없음.

주의(반영 권고, P3-1): 설계 §6.2 의사코드의 inflightKey:'open:'+toolId+':'+id는 실측('open:'+id, actions.js:92)과 키 형식이 다르다. 의도된 일반화로 보이나(툴별 분리), api-contract/electron-migration 갱신 시 "inflightKey 형식 변경"을 명시하지 않으면 구현자가 기존 형식과 혼동할 수 있다. 동작 영향은 없다(내부 키).

3. 데이터 모델 — PASS (P2-1 권고)

config(tools) vs ui-state.json(favorites/order/sortMode) 분리 근거(ADR-M6-3)는 타당하다 — CLI 공유성·검증 정책·손상 허용도가 다른 데이터를 분리하는 결정은 합리적이며, 파일 2개로 한정(과분할 안 함)도 적절. 스키마·검증 규칙(toolId 화이트리스트·id 형식·개수 상한)도 기존 config.js의 normalize 패턴(normalizeScanRoots 등)과 일관.

id 안정성 가정(MQ-3)은 §3.3·§14에 명시되고 graceful skip(소멸 id 표시 무시·영속 유지)이 일관되게 서술됨. PM 확정(skip만)과 정합.

P2-1 config.tools 영속 경로(writer)가 미명세

무엇: setToolPath/pickToolExecutable가 config.tools를 어떤 함수로 0600 영속하는지 명세 공백.

위치: §3.1·§4.1·§10(tree). 설계는 "folders.persistScanRoots의 0600 패턴 재사용"이라 하나, 실측 persistScanRoots(folders.js:170-207)는 existing.scanRoots = roots로 scanRoots 키만 쓰는 전용 함수다 — tools를 못 쓴다(line 185 하드코딩).

문제: ui-state.json은 신규 uiStateStore.write로 해결되나, config.tools는 ① persistScanRoots를 tools도 쓰도록 일반화할지 ② tools 전용 writer를 둘지 ③ uiStateStore류 범용 atomic-writer로 통합할지 미결. 구현자가 persistScanRoots를 그대로 호출하면 tools가 저장되지 않거나, scanRoots를 덮어쓸 위험.

권고: §3 또는 §10에 "config.tools 영속 = persistScanRoots를 키-범용(예: persistConfigKeys({tools}))으로 소폭 일반화하여 기존 키 보존 병합 후 0600 쓰기"를 1줄 명시. 또는 ipc/tools.js가 자체 atomic writer를 두되 "기존 키 read-merge 보존"(persistScanRoots line 178-185 패턴)을 반드시 따른다고 명시.

참고(블로커 아님): config.js loadConfig(line 214-227)는 현재 고정 키만 반환하고 미지의 tools 키를 통과시키지 않는다. 설계가 "normalizeTools 추가 + loadConfig 병합"을 §10에 명시했으므로 이 수정은 인지됨. LIMITS.maxTools(32) 추가도 기존 LIMITS(maxScanRoots:64) 패턴과 일관. 정합 OK.

5. sortMode 2모드 상호작용 — PASS

auto(modified/name/size 정렬, 드래그 불가) ↔ manual(order, 드래그 가능) 2모드 토글이 §3.2·§7에 명확. 드래그 시 setOrder가 sortMode='manual' 동반 전환(§4.1 setOrder 응답 {order, sortMode:'manual'}), 정렬 셀렉터 재선택 시 auto 복귀. "둘을 동시 적용하지 않아 모호성 제거"라는 단일 규칙이 명료. 키보드 대체(위로/아래로 버튼, 동일 setOrder 경로)도 N-07 정합으로 명시. table 밀도에서의 동작(manual이면 order 순 행 정렬)까지 다룸. 상호작용 빈틈 없음.

경미 관찰(권고 아님): auto 모드일 때 사용자가 드래그를 시도→manual 자동 전환되는데, 이때 기존 정렬 셀렉터 UI 상태와 sortMode 표시 동기화(셀렉터가 "수동"을 반영)는 renderer 구현 디테일로 남는다. 설계 의도는 명확하므로 블로커 아님.

6. 실행가능성 — PASS (P2-2 권고)

변경이 신규 파일(tray.js·clipboard.js·tools.js·uiState.js·toolRegistry.js·uiStateStore.js)에 격리되고 기존 수정이 5개 최소 지점(생명주기·preload allowlist·register 등록·actions.open 일반화·config/paths 추가)으로 명확. 순수 함수+주입 구조 유지로 헤드리스 테스트 가능. 메인/IPC·renderer 착수 가능. P2-1(config.tools writer) 외 블로커 없음.

P2-2 quit 취소 시 scanController 이미 dispose 가능성

무엇: 트레이 '종료'→Q4 다이얼로그 "취소" 경로의 자원 상태 미정의.

위치: §9.2 close 핸들러 + before-quit 상호작용.

문제: 설계는 onQuit가 isQuitting=true; app.quit() 호출. Electron에서 app.quit()은 before-quit를 먼저 발화하고, 설계의 before-quit는 disposeResources()(scanController.dispose, 실측 main.js:204-205)를 즉시 실행한다. 이후 win close 핸들러에서 스캔 중이라 Q4를 띄우고 사용자가 "취소"하면 isQuitting=false로 되돌리지만, scanController는 이미 dispose된 상태일 수 있다(watchdog 타이머 해제 등). 즉 "취소했는데 진행 중이던 스캔의 자원이 이미 정리됨" 불일치.

권고: ① before-quit의 disposeResources를 "Q4 확정(실제 종료) 시점"으로 늦추거나, ② onQuit를 app.quit() 대신 win.close() 트리거→close 핸들러에서 Q4 통과 후에만 destroy+quit하는 단일 경로로 통일. 설계 §9.2에 "before-quit dispose와 Q4 취소의 순서"를 1단락 명시 권고. (현행 실측은 close에서 destroy만 하고 quit 안 하므로, 트레이 종료 경로 신설 시 이 순서를 새로 정의해야 함 — 설계가 이 신규 상호작용을 아직 명시 안 함.)
