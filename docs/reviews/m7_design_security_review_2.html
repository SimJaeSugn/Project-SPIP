1. 재검증 요지 · 방법

1회차 판정은 조건부 PASS였고 게이트 통과 조건은 높음 2건(SEC-H1 CSP 세션 모델·SEC-H2 UI redressing/클릭통과)을 설계 델타에 명문 반영하는 것이었다. rev.2는 이를 PM 확정 사항과 함께 §4.1·§5·ADR-M7-3·ADR-M7-5로 반영했다고 명시한다. 본 2회차는 ① 두 높음 항목이 리스크 서술이 아니라 통제(MUST)로 확정됐는지, ② 그 확정이 실측 코드의 동작과 정합한지(추측 배제), ③ 보통·낮음 단언 반영, ④ 보존 통제 회귀 무약화, ⑤ rev.2가 새 공격 표면을 만들지 않았는지를 코드 대조로 확인한다.

핵심 결론: SEC-H1·SEC-H2 모두 구조적으로 해소되었고, 그 근거가 실측 코드(main.js:151 CSP 1회 등록·security.js:42 onHeadersReceived 단일 핸들러·register.js:93 공통 guard)와 정합한다. rev.2가 도입한 통제(movable:false·blur 시 floating 해제·focus 게이팅·setIgnoreMouseEvents 미사용)는 모두 표면을 축소하며 새 위협을 만들지 않는다. 높음 0·치명 0·신규 위협 0 → PASS 상향.

해소 SEC-H1 — CSP 세션 모델 (적대적 재확인)

확인: 실측 main.js:151은 applyCspHeaders(win.webContents.session)를 부팅 시 1회 호출하며, 코드 전역에 partition 지정이 전혀 없다(grep 0건). 따라서 신규 위젯 BrowserWindow는 webPreferences에 partition을 주지 않는 한 default session을 공유하고, 이미 등록된 onHeadersReceived(security.js:42, 세션당 1핸들러)가 app://favorites.html 응답에도 CSP를 주입한다. rev.2 §4.1 의사코드가 deps에서 applyCspHeaders를 의도적으로 제거하고 "재호출 금지" 주석을 단 것은 이 메커니즘과 정확히 정합한다(재호출 시 동일 세션 핸들러를 덮어써 무의미하거나, partition 분리 시 누락 — 둘 다 회피).

잔여: 구현 단계에서 ⓐ 위젯 webPreferences에 partition을 실수로 넣지 않을 것, ⓑ favorites.html 메타 CSP가 CSP_POLICY 문자열과 1:1 일치할 것을 스모크로 검증(설계가 이미 구현 게이트 항목으로 명시). 설계 책임 범위 내 해소 완료.

해소 SEC-H2 — UI redressing/클릭통과 (적대적 재확인)

확인: 1회차가 요구한 4개 통제가 전부 확정 명세로 승격됐다 — (1) movable:false 고정[§4.1:248], (2) blur 핸들러가 setAlwaysOnTop(false)로 floating 해제 후 100ms grace hide·생성 시 alwaysOnTop:false[§4.1:249,257-258], (3) focus 게이팅으로 부수효과 액션을 포커스 시에만 활성[§5 focus 게이팅 행], (4) setIgnoreMouseEvents 미사용 = 클릭통과 비허용[§4.1:224·ADR-M7-5]. transparent 부담 폴백을 ADR-M7-5에서 A/B/C 비교로 결정(B 채택, C 폴백)했고 클릭통과 forward를 명시적으로 거부(A)했다.

적대적 재평가: 이 통제들은 위젯의 "보이되 입력은 자기 영역만 소유, 비포커스 시 위로 뜨지 않음, 비포커스 시 부수효과 차단"이라는 닫힌 입력 경계를 만든다. 이는 공격 표면을 늘리지 않고 오직 축소한다. focus 게이팅이 main이 아닌 렌더러 토글이라는 점은 심층방어 보조일 뿐 — 핵심 권위는 여전히 main 재검증(SEC-M4)이므로 게이팅 우회가 일어나도 실행/복사/즐겨찾기는 guard·pathGuard·safeExec로 재검증된다(우회가 권한 상승으로 확대 불가). 해소 완료.

6. 확인 필요 (위험 수용 — 잔여, 비차단)

  • MQ7-3 (메인 UI에서 위젯 트리거): rev.2는 트레이 경유만 필수로 보고 메인 버튼→위젯은 PM 결정으로 남김. 채택 시 신규 invoke spip:showFavoritesWidget 1개 추가 가능 — guard·senderFrame 자동 승계로 저위험이나 "신규 invoke 0" 주장이 변경됨. 도입 시 1회 재검 권고(현 판정에는 비차단).
  • 구현 게이트 스모크(설계가 이미 명시, 추적용): ① 위젯 응답 헤더에 CSP 존재 ② favorites.html 메타 = index.html 메타 = CSP_POLICY 문자 단위 일치 ③ 위젯에서 window.open/외부 location 차단(EC-5) ④ partition 미지정 확인. 설계 책임은 충족, 구현 단계 검증 항목으로 인계.
위 2건은 치명·높음을 만들지 않으며 보안 게이트 통과를 막지 않는다. 구현 보안 검증 단계에서 스모크로 닫는다.

7. 종합 판정

판정: PASS (상향). 1회차 조건부 PASS의 게이트 조건이던 높음 2건(SEC-H1·SEC-H2)이 rev.2에서 구조적으로 해소되었고, 그 근거가 실측 코드(main.js:151 CSP 1회 등록·partition 부재·security.js:42 onHeadersReceived 단일 핸들러·register.js:93 공통 guard)와 정합함을 확인했다. 보통 4(SEC-M1~4)·낮음 3(SEC-L1~3)은 전부 설계 단언/체크리스트로 명문 반영됐다. rev.2 델타는 새 공격 표면을 만들지 않았다(통제 강화·보안 중립만).

발견 건수(2회차): 치명 0 · 높음 0 · 보통 0(미해소) · 낮음 0(미해소) · 신규 위협 0. (1회차 9건 전부 설계 반영으로 클로즈.)

요약: M7의 신규 보안 표면은 여전히 위젯 렌더 컨텍스트 1개 + push 채널 1개로 정확히 한정되며, 메인 창과 동등한 통제(동일 webPreferences·app:// 로드·CSP default session 공유 자동 적용+메타 단일원천·하드닝·senderFrame·textContent·guard 재검증 승계)와 더 좁은 preload(6채널, ipcRenderer/generic invoke 비노출 MUST)로 봉합된다. frameless·transparent·alwaysOnTop의 UI redressing 표면은 비드래그·표시 중에만 floating·focus 게이팅·클릭통과 비허용의 4중 통제로 닫혔다. 보안 게이트 통과 — 구현 단계 진입을 승인한다. 직접 수정하지 않고 검증·확인만 수행했다.
