Project-SPIP — 아키텍처 보안 검증 보고서

로컬 단일 사용자 도구의 아키텍처 설계(architecture.html §11 4중 통제·신뢰경계·pathGuard·safeExec, adr.html ADR-002/004)를 적대적 관점에서 위협 모델링했다. 검증 대상은 설계 청사진이다(구현 코드 없음: lib/ 미존재, package.json은 골격, 외부 의존성 0 — 의존성 취약점 스캔 비해당). 따라서 본 보고는 "설계가 위협을 구조적으로 막도록 명세했는가"를 판정하고, 구현 단계에서 반드시 검증할 시정·확인 항목을 산출한다.

0. 종합 판정

판정: 조건부 PASS (CONDITIONAL PASS)

설계의 보안 골격(127.0.0.1 바인딩 · 동일출처 검증 · id 역참조 화이트리스트 · shell:false 인자배열)은 위협 모델의 핵심 축을 방향성으로는 올바르게 잡았고, 경로 순회·셸 인젝션·정적 서빙 노출의 1차 표면은 적절히 차단한다. 그러나 치명 0 / 높음 3 / 중간 4 / 낮음 3의 잔여 위협이 있으며, 특히 심볼릭 링크/실경로(realpath) 정규화 누락, PATH 하이재킹(실행 파일 절대경로 미고정), 스캔 대상 부분신뢰 입력(package.json/.git) 파싱 방어 부재가 높음 등급이다. 치명은 없으나 높음 항목이 존재하므로 무조건 PASS는 불가하다. 아래 H-등급 시정 항목을 설계서에 반영(명세 보강)하고 구현 단계 수용 기준에 못박는 것을 조건으로 다음 게이트를 통과시킨다.

3. 체크리스트별 위협 분석

3.1 경로 순회 / 화이트리스트 우회

부분 PASS: id 역참조 + path.resolve 정규화 + 정확 일치 + staticHandler public 격리 설계는 ../ 순회·접두사 부분일치·클라이언트 경로 조작을 막는다(PASS-3/5).

H-1. 심볼릭 링크/접합점/하드링크가 화이트리스트를 우회할 수 있다

위치 §5 pathGuard 계약(isAllowed), §11.1, walker
STRIDE 정보노출(I) · 권한상승(E)
심각도 높음 · 악용 가능성: 중
시나리오 설계는 "path.resolve로 정규화 후 등록 경로와 정확 일치"라고만 명세한다. path.resolve는 ../는 접어주지만 심볼릭 링크/Windows 디렉터리 접합점(junction)을 해소하지 않는다. 등록된 화이트리스트 경로(스캔 루트 안) 자체가 심링크이거나 그 하위에 외부를 가리키는 링크가 있으면, "정확히 일치"하는 등록 경로로도 code가 실제로는 스캔 루트 밖 디렉터리를 열게 된다. 또한 스캔 시 walker가 심링크를 따라가면 루트 밖 경로가 화이트리스트에 등록되어, 검증을 "통과"하면서도 경계를 벗어난다(TOCTOU 포함: 스캔 후 경로를 심링크로 교체).
권고 ① pathGuard와 walker 등록 시점 모두 fs.realpathSync(실경로)로 해소한 결과를 기준으로 화이트리스트를 구성·대조한다. ② walker는 기본적으로 심링크를 따라가지 않거나(fs.lstat로 링크 판별 후 정책 적용), 따라갈 경우 실경로가 스캔 루트 실경로의 하위인지 재확인한다. ③ 대소문자 비민감 파일시스템(Windows/macOS)과 유니코드 정규화(NFC/NFD) 차이를 일치 비교에 반영한다. ④ UNC 경로(\\server\share)·드라이브 상대경로(C:foo)·확장 경로(\\?\)를 정규화/거부 규칙에 포함한다. 이 항목들을 §11.1과 pathGuard 계약에 명세로 추가할 것.

3.2 셸 / 명령 인젝션

부분 PASS: shell:false + 인자 배열 + 사용자 입력 셸 비연결 + 실행파일명 고정 상수는 고전적 명령/인자 인젝션을 구조적으로 차단한다(PASS-4). 경로의 공백/한글/특수문자도 인자 배열로 안전 처리(F-2 엣지 정합).

H-2. 실행 파일 절대경로 미고정 → PATH 하이재킹 / 실행파일 위장으로 RCE

위치 §5 safeExec, §11.2(실행파일명을 고정 상수 'code'/'git'로 허용), ADR-002
STRIDE 위장(S) · 권한상승(E) → 임의 코드 실행
심각도 높음 · 악용 가능성: 중(Windows에서 특히)
시나리오 설계는 "실행 파일명을 고정 상수 'code'/'git'만 허용"한다고 하지만, 이는 파일명만 고정일 뿐 경로 해석을 PATH에 위임한다. Windows에서 spawn('code',...,{shell:false})는 현재 작업 디렉터리/PATH 우선순위에 따라 공격자가 심어둔 code.exe·code.bat를 실행할 수 있다(특히 git collector는 cwd: projectPath로 실행 — 스캔 대상 프로젝트 폴더 안의 위장 git.exe/git.bat가 우선될 위험). 스캔 대상은 부분신뢰(TB-B)이므로, 악성 리포를 클론해 둔 사용자는 스캔/열기만으로 임의 코드가 실행될 수 있다.
권고 ① 가능하면 실행 파일의 절대경로를 1회 해석(resolve)해 고정하고(예: where/which 결과를 신뢰 PATH 기준으로 검증), 이후 그 절대경로로 spawn한다. ② spawn 옵션에서 cwd를 신뢰 경로로 두거나, git 호출 시 cwd를 프로젝트로 두더라도 실행 파일은 절대경로로 지정해 cwd의 위장 바이너리가 선택되지 않게 한다. ③ Windows에서 .bat/.cmd 자동 실행 위험을 인지하고 PATHEXT 기반 확장 해석을 막는다. ④ 이를 safeExec 계약과 §11.2에 명세할 것.

3.3 로컬 HTTP 노출 (멀티유저 / 리바인딩 / CSRF)

부분 PASS: 127.0.0.1 바인딩(PASS-1) + 동일출처/CT 검증(PASS-2)은 (a) 외부 네트워크 접근과 (b) 단순 폼 CSRF의 1차 표면을 차단한다.

M-1. 동일출처 검증 우회 여지 (DNS 리바인딩 / Origin 부재 / Host 신뢰)

위치 §9, §11.3 (Origin/Host + Content-Type 검증)
STRIDE 위장(S) · 변조(T)
심각도 중간 · 악용 가능성: 낮~중
시나리오 (b) DNS 리바인딩: 공격 페이지가 자신의 도메인을 127.0.0.1로 재바인딩하면 브라우저는 그 도메인을 "동일 출처"로 보낼 수 있어 Origin 검증만으로는 부족하다 → Host 헤더가 기대 127.0.0.1:<port>/localhost:<port>인지 화이트리스트 검증(allowlist)이 필수. (a) Origin 부재 요청(일부 클라이언트/리바인딩) 처리 정책 미명세 — "Origin이 없거나 불일치면 거부"를 명시해야 함. application/json 요구는 좋으나 fetch는 이를 우회 가능하므로 보조 수단일 뿐.
권고 §11.3에 다음을 명세: Host 헤더를 기대 호스트:포트 allowlist와 정확 비교, Origin 부재/불일치 시 거부(폐쇄적 기본값), 상태변경은 POST+JSON 강제 + (저비용 강화책) 서버 기동 시 1회 생성하는 세션 토큰을 index.html에 주입해 POST에 동봉 검증. 멀티유저 OS(c)에 대해서는 캐시 권한(M-2)으로 보완.

(참고) T-3c 동일 머신 타 사용자/프로세스

판정 127.0.0.1은 동일 머신의 다른 로컬 사용자/프로세스에게는 도달 가능하다(루프백은 사용자 격리를 하지 않음). 단일 사용자 PC 가정상 영향은 낮으나, 멀티유저 환경에서는 타 사용자가 /api/projects로 경로 목록을, /api/open으로 액션을 호출할 수 있다. M-1 토큰 + M-2 파일 권한으로 완화. 위험 수용 결정 후보(아래 §5).

3.4 민감정보 노출

부분 PASS: 스냅샷은 파일 내용 미포함·메타데이터만·외부 전송 0(PASS-6) — 노출 영향 최소화 설계는 적절.

M-2. 캐시/설정 JSON 파일 권한 미명세

위치 §8, §11.4, serializer / common/paths
STRIDE 정보노출(I)
심각도 중간
시나리오 projects.json은 사용자의 전체 프로젝트 절대경로·디렉터리 구조·git 브랜치 등 프라이버시 정보를 담는다. 저장 위치(%LOCALAPPDATA% 등)는 명시됐으나 파일 권한이 미명세다. 기본 umask/상속 권한에 따라 멀티유저 OS에서 타 사용자가 읽을 수 있다. 원자적 쓰기의 임시 파일 권한·정리(누수)도 명세 필요.
권고 serializer가 캐시/설정 파일을 소유자 전용 권한(POSIX 0600, 디렉터리 0700; Windows는 사용자 ACL)으로 생성하도록 §11.4/serializer 계약에 명세. 임시 파일도 동일 권한·동일 디렉터리(rename 원자성 보장)·실패 시 정리.

3.5 부분 신뢰 입력 (package.json / .git / 폴더명)

핵심 갭: 설계는 환경 부재·손상 파일에 대한 graceful 격리(견고성)는 잘 명세했으나(R-13/N-05), 적대적 입력(고의로 만든 악성 파일)에 대한 방어는 명세되지 않았다. 스캔 대상은 신뢰 경계 밖(TB-B)이다.

H-3. 부분신뢰 입력 파싱 방어 부재 (JSON 폭탄 · 크기 상한 · ReDoS)

위치 §6 detector, collectors/language·git·freshness, serializer
STRIDE 서비스거부(D) · 변조(T)
심각도 높음 · 악용 가능성: 중
시나리오 language collector는 package.json을 파싱한다. 공격자(또는 무심코 클론한 악성 리포)가 ① 수백 MB의 거대 package.json → 메모리 고갈, ② 깊게 중첩된 JSON → 파서 부하, ③ deps 키에 제어문자/거대 문자열을 넣어 직렬화/렌더 폭증, ④ 확장자 비율 계산·언어 추정에서 비효율 정규식(ReDoS)을 트리거할 수 있다. 단일 항목 격리(try)는 무한 루프/메모리 폭발을 막지 못한다(프로세스 전체가 죽거나 30초 SLA를 초과).
권고 ① 파싱 전 파일 크기 상한(예: package.json 1MB 등 합리적 한도) 검사 후 초과 시 skip+warning. ② JSON.parse 입력 바이트 한도·깊이 가드. ③ 언어 추정 등 모든 정규식은 선형 시간 보장(역추적 폭발 패턴 금지) 또는 입력 길이 제한. ④ 수집한 문자열 필드(name/description/branch)는 길이 절단·제어문자 제거. 이를 collectors 공통 계약과 §6.1/§12 견고성에 "적대적 입력" 항목으로 명세할 것.

L-1. 대시보드 저장형 XSS 표면 (오염된 메타데이터의 DOM 전파)

위치 §6.3 app.js — view (카드/상세 렌더)
STRIDE 정보노출(I) · 변조(T)
심각도 낮음(로컬·단일 사용자라 영향 제한, 단 토큰·로컬 API 접근 시 상승)
시나리오 악성 리포의 package.json name/description, git 브랜치명에 <img onerror> 등 페이로드가 들어가면, 스캔→스냅샷→/api/projects→DOM 삽입 경로로 저장형 XSS가 가능하다. 실행 시 그 스크립트가 동일 출처에서 /api/open(M-1 통과)·/api/projects를 호출하면 경로 유출/액션 트리거로 상승한다.
권고 view 렌더는 textContent/명시적 이스케이프만 사용하고 innerHTML 직접 결합 금지. CSP 메타(인라인/외부 스크립트 제한)를 index.html에 추가 권고. §6.3 계약에 "모든 동적 텍스트는 이스케이프"를 명시.

3.6 DoS / 자원 (전체 드라이브 스캔 · 심링크 루프)

M-3. 심링크 루프 / 과도한 깊이 → 스캔 무한 순회·자원 고갈

위치 §6 walker, R-03(M4), §8.2 depthLimit(M4)
STRIDE 서비스거부(D)
심각도 중간
시나리오 FeatureSpec F-1은 순환 심링크/과도 깊이를 깊이 한도에서 중단(R-03)한다고 하나, 설계서는 depthLimit와 심링크 처리를 모두 M4로 미룬다(§8.2 "MVP에서도 안전 상한 권고"는 권고에 그침). MVP에서 전체 드라이브는 아니어도 사용자가 큰 루트나 심링크 루프(a→b→a)를 지정하면 무한 순회/30초 SLA 초과가 발생한다. R-05(전체 드라이브)는 M4지만 MVP 루트 지정만으로도 위험이 성립한다.
권고 MVP에서 depthLimit 안전 상한을 강제(기본값)하고, walker가 방문한 실경로(realpath/inode) 집합을 추적해 재방문 시 가지치기. 심링크는 기본 미추적 또는 1회 해소 후 중복 차단. R-03을 "M4 권고"가 아닌 "MVP 필수 안전장치"로 격상해 §6.1/§8.2에 명세.

M-4. spawn 타임아웃 / 동시 실행 상한 부재

위치 §5 safeExec, §6.2 actionHandlers(/api/open), collectors/git
STRIDE 서비스거부(D)
심각도 중간
시나리오 git collector가 행(hang)하거나(손상 리포·거대 리포), '열기' 연타가 서버 측에서 다수 프로세스를 spawn하면 자원이 고갈된다. RA2가 "타임아웃"을 언급하나 safeExec 계약에 타임아웃·kill·동시 실행 상한이 명세되지 않았다(클라 디바운스는 서버 보호가 아님).
권고 safeExec에 기본 타임아웃 + 초과 시 강제 종료(kill), 출력 버퍼 상한을 계약으로 명세. /api/open은 서버 측 레이트리밋/in-flight 상한(id별)으로 중복 spawn 차단.

L-2. 설정 파일(scanRoots/excludes) 입력 검증 약함

위치 §8.2 spip.config.json, common/config
STRIDE 변조(T)
심각도 낮음(사용자 자신이 작성하는 파일이라 신뢰도 높음)
시나리오 config는 "잘못된 값은 경고 후 기본값 폴백"이라 견고성은 있으나, scanRoots 항목의 경로 타입/존재/정규화 검증과 excludes 패턴의 안전성(거대/악성 glob) 명세가 약하다. 화이트리스트의 원천이 스캔 루트이므로 루트 정규화는 H-1과 연결된다.
권고 config 로드 시 scanRoots를 절대경로로 정규화(realpath)·타입/존재 검증, excludes 패턴 길이/개수 상한. H-1 정규화 규칙과 일관 적용.

L-3. 에러 응답의 내부 정보 노출 정책 미명세

위치 §9 오류 응답 계약
STRIDE 정보노출(I)
심각도 낮음
시나리오 표준 에러 코드(PATH_NOT_ALLOWED 등)는 잘 정의됐으나, message 필드에 원시 예외/스택/내부 절대경로가 그대로 실릴 수 있다(특히 NO_SNAPSHOT/PATH_GONE 경로 에코). 로컬 도구라 위험은 낮으나 XSS(L-1)와 결합 시 정보 수집 표면이 된다.
권고 message는 사람이 읽는 고정 안내문만, 원시 예외/스택은 콘솔 로그로만(응답 비노출). 경로 에코 최소화. §9에 "에러 응답에 스택/내부 경로 비노출" 정책 명시.

5. 확인 필요 (위험 수용 결정 후보)

RC-1 (멀티유저 OS / T-3c·M-1·M-2): 제품 가정은 "로컬 단일 사용자 PC"다. 동일 머신에 신뢰할 수 없는 타 로컬 사용자가 존재하는 환경(공용 PC·터미널 서버)을 지원 범위에 포함할지 결정 필요. 포함하지 않는다면 T-3c와 멀티유저 노출은 "가정상 위험 수용"으로 문서화하고, M-1 세션 토큰·M-2 파일 권한은 저비용 심층방어로 여전히 권고한다. 포함한다면 인증 부재(§9.1 Won't) 재검토가 필요하다.

RC-2 (스캔 대상 신뢰 수준): 사용자가 스캔하는 폴더에 신뢰할 수 없는 외부 클론 리포가 섞일 수 있음을 제품이 명시적으로 인정하는가? 인정한다면 H-2/H-3/L-1은 단순 권고가 아니라 필수 방어로 확정된다(본 보고는 인정 전제로 등급화함).

이 두 결정은 제품 위험 수용 범위에 해당하므로 PM/tech-lead 판단이 필요하다. 중대성은 "조건부"이며 게이트 차단 항목(H-1~3)은 결정과 무관하게 반영 필수다.
