Project-SPIP — SH-3 위협 모델 · 구현 가드 요구 명세(DoD)

본 문서는 코드 리뷰가 아니라 착수 전 위협 모델·요구 정의다. SH-3은 본 앱 최초의 사용자 승인 egress 표면(메인 프로세스 → 임의 인터넷 호스트)으로, 렌더러 connect-src 'none'를 유지한 채 main에서만 외부 통신을 허용한다. 아래 위협별 등급과 테스트 가능한 가드 요구 체크리스트는 backend-dev가 SH-3 구현 시 Definition of Done으로 사용한다. 치명/높음 요구가 하나라도 미충족이면 보안 게이트를 통과시키지 않는다.

판정 · 요약

판정: 조건부 진행 (CONDITIONAL — 아래 가드 요구 충족 시에만 착수 승인)

SH-3은 치명(SSRF·DNS rebinding) 위협을 본질적으로 내포한다. 설계(§9.2 IP 분류표·IP 핀·hop 재검증)는 방향이 옳고 충분히 구체적이다. 따라서 설계 자체는 게이트 통과 가능하나, 아래 치명 4 + 높음 7 가드 요구가 구현·테스트로 증명되어야 종료 게이트(post-implementation security-review)를 통과한다. 본 문서는 그 요구를 테스트 가능한 형태로 고정한다.

T-01 SSRF — 사설/루프백/링크로컬/메타데이터 IP로의 크롤 I·E 치명 공격자가 http://169.254.169.254/latest/meta-data/ 또는 http://127.0.0.1:1234/v1(로컬 LLM)·http://192.168.1.1/admin을 북마크로 붙여넣어 add → main이 내부 서비스 응답을 og 메타로 반사. 클라우드/VM이면 IAM 자격 탈취. → D-SSRF-1~6 전부.
T-02 DNS rebinding / TOCTOU — 검증 후 연결 전 DNS가 내부 IP로 바뀜 S·I·E 치명 공격자 도메인이 1차 resolve엔 공인 IP, 연결 시점엔 127.0.0.1 반환(낮은 TTL). 호스트명 화이트리스트나 "검증 후 재해석"은 우회됨. → D-SSRF-3(IP 핀: 검증한 그 IP로 직접 연결, 연결 시 재해석 금지).
T-03 리다이렉트 우회 — 공인 URL이 3xx로 내부 IP·非http 스킴으로 점프 I·E 치명 최초 URL만 검증하면 Location: http://169.254.169.254/ 또는 Location: file:///etc/passwd로 우회. → D-RDR-1~4(각 hop마다 G1+G2 재검증·스킴 재확인·hop 상한).
T-04 og:image 2차 SSRF — 본문은 공인이나 og:image URL이 내부 IP I·E 치명 정상 페이지가 <meta property="og:image" content="http://127.0.0.1:9200/">를 심어 이미지 fetch가 내부 접속. → D-IMG-1(og:image도 본문과 동일 SSRF 게이트·IP 핀·hop 재검증).
T-05 IPv4-mapped IPv6 / 비표준 표기 우회(::ffff:127.0.0.1, 0x7f.1, 017700000001, 2130706433) I·E 높음 분류기가 IPv4-mapped를 풀지 않거나 8진/16진/정수 표기를 정규화하지 않으면 루프백 차단 우회. → D-SSRF-4(매핑 풀어 재판정)·D-SSRF-5(파싱 정규화).
T-06 응답 바디 고갈 — 무한/거대 HTML로 메모리·이벤트루프 점유 D 높음 악성 호스트가 끝없는 스트림 전송 → main OOM/멈춤. → D-RES-1(스트리밍 누적 512KB 상한, 초과 시 소켓 destroy)·D-RES-2(타임아웃 8s).
T-07 Decompression bomb — gzip/br로 작은 응답이 해제 후 폭발 D 높음 1KB gzip → 해제 시 1GB. 압축 바이트만 세면 상한 무력화. → D-RES-3(Accept-Encoding: identity 압축 미요청; 만약 압축 수신 시 해제 후 누적 바이트로 상한 재확인).
T-08 ReDoS — 정규식 og 파서가 악성 HTML에 지수 백트래킹 D 높음 중첩 인용/속성 폭탄으로 main 이벤트루프 정지. → D-PARSE-1(비탐욕·앵커·역참조 회피)·D-PARSE-2(매치 횟수 상한)·입력 512KB 선행 상한.
T-09 비http(s) 스킴 / 자격정보 URL(user:pass@) S·I 높음 javascript:·file:·data: 크롤 시도 또는 자격 누출. → D-URL-1(scheme∈{http,https})·D-URL-2(username/password 비어야). (열기 단계 openExternal 스킴 재검증은 별도 — D-OPEN-1)
T-10 이미지 content-type 위조 / SVG 스크립트 내장 I·E 높음 응답이 image/svg+xml(스크립트 포함) 또는 HTML인데 image로 위장 → data:URI로 렌더 시 위험. → D-IMG-2(content-type image/{png,jpeg,gif,webp}만)·D-IMG-3(SVG 제외)·D-IMG-4(매직바이트 sniff 권장).
T-11 자동 재크롤의 은닉 egress — 토글 off인데도 백그라운드 접속 I·R 높음 6시간 스케줄러가 토글 off에도 tick 동작하면 사용자 모르게 외부 호스트에 주기 접속(프라이버시·열람 신호 누출). → D-SCHED-1(off=완전 무동작)·D-SCHED-2(상승세션 크롤 스킵).
T-12 저장형 XSS — 크롤된 title/desc/og를 innerHTML로 렌더 E·T 중간 L-1 불변식이 있으나 SH-3이 새 표시 필드(title/desc/status)를 도입 → 렌더 경로 회귀 위험. → D-RENDER-1(textContent만)·D-RENDER-2(data:URI MIME 화이트리스트).
T-13 data:URI 크기 폭증 — 64항목×100KB가 IPC payload·메모리 압박 D 중간 list 일괄 응답이 ~8.6MB. → D-IMG-5(개별 100KB 상한)·D-CACHE-2(총 캐시 용량 상한·GC)·지연 로드(권장).
T-14 캐시 경로 탈출 / 파일명 주입 — 해시키 외 값으로 디렉토리 탈출 T·I 중간 bannerKey가 검증 없이 파일경로로 쓰이면 ../ 주입. → D-CACHE-1(키=고정형식 sha256 32hex만, 파일명은 키로만 구성)·확장자 화이트리스트.
T-15 정보 노출 — 에러/로그에 실경로·해소 IP·전체 URL·스택 누출 I 중간 BLOCKED_HOST 응답에 내부 IP를 실으면 내부망 스캐닝 도구화. → D-ERR-1(고정 에러코드만 반환)·D-ERR-2(IP/URL/스택은 로그 전용·host만, llmClient hostOnly 선례).
T-16 동시 크롤 고갈 — 다수 add/refresh 병렬로 소켓·메모리 폭주 D 중간 스케줄러+수동 refresh가 동시에 64건 크롤. → D-RES-4(동시 크롤 상한·in-flight 큐·배치 watchdog).
T-17 쿠키/Authorization 누출 · UA 핑거프린팅 I 중간 세션 쿠키·인증 헤더 전송 또는 식별 가능 UA. → D-PRIV-1(쿠키 저장·전송 0)·D-PRIV-2(Authorization 미전송)·D-PRIV-3(고정 최소 UA·Referer 없음).

4. 구현 가드 요구 체크리스트 (DoD · 테스트 가능)

각 항목은 테스트로 증명되어야 한다(헤드리스 단위 또는 주입 모킹). 치명·높음은 종료 게이트 차단 조건이다 — 하나라도 미충족이면 SH-3 보안 게이트 FAIL.

A. SSRF IP 게이트 ssrfGuard

  • D-SSRF-1 치명 — IP 분류표 전수 차단: 루프백(127/8, ::1) · 사설(10/8, 172.16/12, 192.168/16, fc00::/7) · 링크로컬(169.254/16, fe80::/10) · 메타데이터(169.254.169.254, 100.100.100.200, fd00:ec2::254) · CGNAT/특수(100.64/10, 0.0.0.0/8, 192.0.0/24, 192.0.2/24) · 멀티캐스트/예약(224/4, 240/4, 255.255.255.255, ff00::/8, ::, 0.0.0.0). 테스트: 분류표 각 행 + 경계값(예 10.255.255.255 차단/11.0.0.0 허용) 전수 단위.
  • D-SSRF-2 치명 — BLOCKED_HOST는 DNS resolve 후 결과 IP 기준으로 판정한다(호스트명 화이트리스트 불충분). resolve가 복수 IP 반환 시 하나라도 차단 대상이면 거부. 테스트: DNS lookup 주입 모킹으로 차단 IP 반환 시 BLOCKED_HOST.
  • D-SSRF-3 치명 — IP 핀: 게이트를 통과한 그 IP로 직접 연결한다(options.host=검증IP + servername=원래host + Host 헤더, 또는 lookup 콜백이 검증 IP만 반환). 연결 시점 도메인 재해석 금지(DNS rebinding 차단). 테스트: lookup 콜백이 검증 IP만 내보내는지·연결 시 재해석 호출 0 확인.
  • D-SSRF-4 높음 — IPv4-mapped IPv6(::ffff:a.b.c.d)는 IPv4로 풀어 재판정. 테스트: ::ffff:127.0.0.1·::ffff:169.254.169.254 차단.
  • D-SSRF-5 높음 — 숫자 IP 직접 입력 및 비표준 표기(8진 017700000001·16진 0x7f000001·정수 2130706433·축약 127.1) 정규화 후 분류 — 또는 new URL().hostname이 정규화하지 못하는 형태는 거부. 테스트: 각 표기 입력 시 차단 또는 BAD_INPUT.
  • D-SSRF-6 높음 — 빈 호스트·0.0.0.0·대괄호 IPv6·후행점(127.0.0.1.) 등 엣지 호스트도 게이트 통과. 테스트: 엣지 호스트 집합 차단.

B. URL 1차 검증 ssrfGuard / config.validateHttpUrl

  • D-URL-1 높음 — scheme ∈ {http, https}만. 그 외(file:·ftp:·javascript:·data:) 거부. 테스트: 비http(s) 입력 거부.
  • D-URL-2 높음 — username/password(user:pass@host) 비어야 함, 아니면 거부. 테스트: 자격 URL 거부. (validateBriefingUrl 선례 일반화 — 반환형 {ok,value,external} 불변, briefing 호출처 회귀 0 확인.)
  • D-URL-3 중간 — 길이 ≤ 2048. 파싱 실패는 BAD_INPUT. 테스트: 초과 길이·파싱불가 거부.

C. 리다이렉트 urlMeta

  • D-RDR-1 치명 — 각 hop의 Location을 새 URL로 파싱 후 G1(D-URL-*)+G2(D-SSRF-*) 전부 재실행. 최초 1회 검증은 무효. 테스트: 주입 모킹으로 hop2가 내부 IP면 BLOCKED_HOST.
  • D-RDR-2 높음 — hop 상한 ≤ 3, 초과 시 CRAWL_FAILED. 순환 리다이렉트도 상한으로 종료. 테스트: 4 hop·순환 시 실패.
  • D-RDR-3 높음 — hop의 스킴이 http(s) 아니면 거부(프로토콜 다운그레이드/비http 점프 차단). 테스트: Location: file:/// 거부.
  • D-RDR-4 중간 — 리다이렉트 시 자격·쿠키 비전파(애초에 미전송이므로 자연 충족, 회귀 테스트로 고정).

D. 자원 한도 urlMeta

  • D-RES-1 높음 — 응답 본문 스트리밍 누적 512KB 초과 시 즉시 소켓 destroy(전체 버퍼링 후 검사 금지). 테스트: 모킹 스트림이 상한 초과 시 destroy 호출·부분만 보관.
  • D-RES-2 높음 — 타임아웃 ≤ 8s(연결+응답 전체). 만료 시 abort·CRAWL_FAILED. 테스트: 지연 모킹으로 타임아웃 발동.
  • D-RES-3 높음 — Accept-Encoding: identity로 압축 미요청(decompression bomb 차단). 압축 응답 수신 시 해제 후 누적 바이트로 상한 재확인 또는 거부. 테스트: 요청 헤더에 identity·gzip 응답 거부/상한.
  • D-RES-4 중간 — 동시 크롤 상한(in-flight 큐) + id별 상한 + 배치 watchdog. 테스트: 동시 호출 수 제한 확인.
  • D-RES-5 중간 — GET만, Accept: text/html, content-type이 text/html 계열 아니면 본문 미파싱(폴백 메타). 테스트: 메서드·비HTML content-type 폴백.

E. HTML 파싱 ogParse

  • D-PARSE-1 높음 — 정규식은 비탐욕·앵커 고정·역참조 회피(ReDoS 방어). 입력 512KB 선행 상한 전제. 테스트: 병리적 입력(긴 따옴표 폭탄)에 처리시간 선형 유지.
  • D-PARSE-2 높음 — meta/title 스캔 매치 횟수 상한(예 첫 N개만). 테스트: meta 1만개 입력에서 상한 후 중단.
  • D-PARSE-3 중간 — 엔티티 디코드는 화이트리스트(&amp; &lt; &gt; &quot; &#39; &#NNN;)만, 숫자 엔티티는 범위 검증. 테스트: 알려진 엔티티만 디코드.
  • D-PARSE-4 중간 — 모든 추출 문자열 sanitize(C0/DEL·방향성 제어 제거) + 필드별 길이상한. og:image는 절대 URL로 resolve. 테스트: 제어문자 제거·길이 절단. (config.sanitizeSystemPrompt 정책 재사용 권장.)

F. og:image 다운로드 urlMeta + imageCache

  • D-IMG-1 치명 — og:image URL은 본문과 동일 SSRF 게이트(D-URL-*·D-SSRF-*·IP 핀·hop 재검증)를 다시 거친다. 테스트: og:image가 내부 IP면 fetch 안 함·bannerKey=null.
  • D-IMG-2 높음 — content-type ∈ {image/png, image/jpeg, image/gif, image/webp}만. 그 외 거부. 테스트: text/html·image/svg+xml content-type 거부.
  • D-IMG-3 높음 — SVG 완전 제외(image/svg+xml 거부). 테스트: SVG content-type·.svg 거부.
  • D-IMG-4 중간 — 매직바이트 sniff로 선언 mime과 실제 바이트 일치 검증(content-type 위조 방어, 권장). 테스트: PNG 선언+HTML 바이트 거부.
  • D-IMG-5 높음 — 이미지 크기 ≤ 100KB(스트리밍 누적), 초과 시 거부·bannerKey=null(그라데이션 폴백). 테스트: 100KB 초과 거부.

G. 캐시·전달 imageCache

  • D-CACHE-1 중간 — 파일명은 sha256 32hex 키 + 화이트리스트 확장자로만 구성. bannerKey는 /^[0-9a-f]{32}$/ 외 거부(경로 탈출·주입 차단). 테스트: ../ 포함 키 거부·정규식 검증.
  • D-CACHE-2 중간 — 0600 write(상승세션 보류 정책 정합), 총 캐시 용량 상한 + 미참조 파일 GC. 테스트: 권한 0600·상한 초과 GC.
  • D-CACHE-3 중간 — toDataUri(bannerKey)는 캐시 파일을 읽어 검증된 mime으로만 data:URI 생성(MIME 화이트리스트·base64 검증). 테스트: 알 수 없는 키→null, mime 화이트리스트.

H. 렌더 경계 (L-1) 렌더러 app.js / IPC 응답

  • D-RENDER-1 중간 — name/title/sub/desc/status/cat 등 모든 크롤 문자열은 렌더러에서 textContent만. innerHTML 결합 금지. 배너는 <img src="data:…">(속성 주입). 테스트/검토: 렌더 경로 grep로 innerHTML 부재 확인.
  • D-RENDER-2 중간 — color는 /^#[0-9a-fA-F]{6}$/만, bannerImage는 data:image/(png|jpeg|gif|webp);base64, 접두 검증. 테스트: 비정상 color/data:URI 폐기(정규화).
  • D-RENDER-3 중간 — 렌더러 connect-src 'none' 무변경 확인(외부 fetch 0). CSP는 img-src 'self' data: 그대로. 테스트/검토: index.html CSP diff 0.

I. 프라이버시·스케줄러 urlMeta / scheduler

  • D-PRIV-1 중간 — 쿠키 저장·전송 0, 리다이렉트 간 쿠키 비전파. 테스트: 요청 헤더에 Cookie 부재.
  • D-PRIV-2 중간 — Authorization·인증 헤더 미전송. 테스트: 헤더 집합에 인증 부재.
  • D-PRIV-3 중간 — 고정 최소 User-Agent, Referer 미전송. 테스트: UA 고정·Referer 부재.
  • D-SCHED-1 높음 — 6시간 자동 재크롤 토글 off면 스케줄러 tick 완전 무동작(egress 0). 기본 ON이나 off 시 어떤 외부 접속도 없어야 한다. 테스트: 토글 off에서 tick 호출 시 crawl 0회.
  • D-SCHED-2 중간 — 상승(elevated) 세션이면 크롤 스킵(불필요 egress 회피)·write 보류. 앱 시작 직후 즉시 tick 금지(첫 tick 지연). 테스트: elevated 모킹 시 크롤 스킵.

J. 열기·에러 노출 electron/ipc/shelf.js

  • D-OPEN-1 높음 — url 열기는 shell.openExternal 전에 http/https 재검증(임의 스킴을 OS 핸들러로 넘기는 위험 차단). 테스트: javascript:·file: ref 열기 거부.
  • D-ERR-1 중간 — 응답은 고정 에러코드만(BLOCKED_HOST, CRAWL_FAILED, BAD_INPUT, LIMIT, INTERNAL, UNSUPPORTED_TYPE 등). 실경로·해소 IP·전체 URL·스택 비노출. 테스트: 차단 시 code만, IP/URL 부재.
  • D-ERR-2 중간 — 로그도 host만(전체 URL·IP·키 비노출). llmClient.hostOnly 선례 재사용. 테스트/검토: 로그 인자에 IP/URL 부재.

6. 문서 위생 지적 (구현 전 정정 권장)

H-DOC-1 낮음(문서) — file 열기 도구 서술 불일치. 설계 shelf-widget-design.html §10.3("folder/file → safeExec(resolveTool('code'),[real]…)… 도구 미설정 시 shell.openPath 폴백")와 §13 H-2("folder/file 열기는 safeExec 절대경로")는 file도 safeExec(code)로 여는 것처럼 읽힌다. 그러나 PM #2 확정·§8 IPC 표·api-contract.md는 file = shell.openPath(OS 기본앱) 항상, folder만 VS Code(safeExec)다. → 구현은 확정본(file=shell.openPath, folder=safeExec)을 따르고, §10.3·§13 prose를 그에 맞게 정정할 것. 보안 영향은 낮음이나(둘 다 H-2 규약 내), 구현자가 §10.3을 그대로 따르면 PM 결정과 모순된 동작이 나오므로 착수 전 정정 권장.
참고(상충 아님). §10.3은 url 열기 시 shell.openExternal + http/https 재검증을 명시 — 이는 D-OPEN-1과 정합(유지).
