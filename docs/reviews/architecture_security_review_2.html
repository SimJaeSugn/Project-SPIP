요구 사항 walker 등록 시점·pathGuard 검증 시점 모두 realpath 기준 화이트리스트인가? canonicalize()/isAllowed() 계약이 심링크/접합점/대소문자/유니코드/UNC/드라이브상대/\\?\를 실제로 다루는가?
증거 §5 pathGuard 계약(L400~411)에 canonicalize(rawPath) 신설 — (1) path.resolve로 ../ 접기, (2) fs.realpathSync로 심링크/Windows 접합점(junction) 실경로 해소, (3) 대소문자 비민감 FS 폴드, (4) 유니코드 NFC 정규화, (5) UNC·드라이브상대(C:foo)·확장경로(\\?\) 정규화 또는 거부를 명시. isAllowed()는 "스냅샷 등록 시점 canonicalize 실경로 집합"과 "요청 canonicalize 실경로"의 정확 일치만 허용, 실경로 해소 실패는 false → PATH_GONE/403. §11.1이 등록·검증 동일 규칙을 "단일 진실의 원천"으로 강제하고 TOCTOU 표면 축소를 명시. walker 계약(L386·394~395)·snapshotStore(§6.2)가 등록 경로를 realpath로 정규화하도록 일치. ADR-006 ①과 §1 핵심전제에도 못박힘.
판정 등록·검증 양 시점 동일 규칙, 요청된 정규화 형태(심링크·접합점·대소문자·유니코드·UNC·드라이브상대·확장경로)를 모두 계약에 명세 → 구조적 해소.

ADR-002 rev.2(L107·122·128·137)가 결정·근거·결과로 추적, ADR-006 ②로 게이트 못박음.

2.1 신뢰경계 SVG — TB-B 통제 추가 확인

확인됨: §11 그림 3 SVG가 2경계 + 8중 통제로 갱신됨. TB-A: ① 127.0.0.1 바인딩, ② Host allowlist+Origin+세션토큰, ③ pathGuard(realpath), ④ safeExec(절대경로·타임아웃). TB-B(신규): ⑤ 입력 파싱 가드(크기·깊이·ReDoS-free), ⑥ 실행파일 절대경로(resolveBin·cwd 의존제거), ⑦ realpath 등록·루프차단(방문 실경로 추적), ⑧ 출력 이스케이프+CSP. TB-B 위협(거대/중첩 package.json·위장 git.exe·심링크 루프·오염 name)에 각 통제가 차단 화살표로 매핑됨. 1회차 SVG의 "TB-B 통제 계층 미명세" 갭이 해소됨. §14 추적성 표·§12 NFR 표에도 8중 통제가 요약됨.

4. 구현 단계 검증 조건 (추적 항목 · 게이트 차단 아님)

높음 3건은 설계 계약에 구조적으로 해소되었다. 다만 이들은 구현 디테일에 효과성이 좌우되므로, 코드 보안 검증 게이트에서 다음을 실증해야 한다(설계 PASS를 막지 않는 추적 항목).

5. 확인 필요 (잔여 위험 수용 — 정보 공유)

RC-1 재확인(추가 조치 불필요): 동일머신 타 로컬 사용자/프로세스로부터의 격리는 MVP 비목표로 PM이 수용했고 설계(§11.7)에 명시됐다. 127.0.0.1 루프백은 사용자 격리를 하지 않으므로, 공용 PC·터미널 서버를 지원 범위에 넣게 되면 인증 부재(§9.1 Won't) 재검토가 필요하다. 현 단일 사용자 가정 하에서는 M-1 세션토큰·M-2 파일권한 심층방어로 충분하며 본 재검증은 이 수용을 존중한다.

위 RC-1·RC-2는 1회차에서 PM이 이미 확정했고 설계에 반영되어, 본 2회차에서 새로 올릴 위험 수용 결정은 없다. V-1·V-2는 구현 게이트로 이월되는 추적 항목이다.
