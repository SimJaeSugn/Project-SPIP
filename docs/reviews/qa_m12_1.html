② 항목 1 — 회귀 0 (최우선) PASS

중앙 출처는 lib/common/elevationState.js 모듈 싱글턴이며 기본값 _elevated=false. setElevated는 electron/main.js onReady 1곳에서만 호출되므로 CLI(scan.js)·dev·테스트·portable 실행 경로에는 set이 도달하지 않아 항상 false다. write 3경로와 rescan 가드 모두 if (_elev.isElevated()) 진입 전에는 기존 코드 흐름과 바이트 단위로 동일(분기만 추가, 기존 라인 무변경) — 정규화·원자적 write·rename·chmod 로직 보존을 라인 대조로 확인했다.

무변경 확인(교차 검증): lib/common/paths.js·electron/autoUpdate.js에 elevation|elevated|Elevation 매치 0건. 위치 결정(appDir/APPDATA 종속)·자동업데이트 트리거는 손대지 않아 기존 사용자 데이터 위치·업데이트 동작 불변.

회귀 테스트가 명시적으로 false 경로를 커버한다: writeSnapshot/uiStateStore.write/persistConfigKeys elevated:false → 파일 생성 + 내용 라운드트립 검증, rescan false → SCAN_STARTED + acquire 호출 검증, 그리고 deps 미주입(기본 중앙 모듈 + reset된 false) 케이스도 정상 write/스캔을 확인한다. 전체 768 그린이라 인접 모듈(snapshotStore·folders 등) 동작도 깨지지 않았다.

③ 항목 2 — 상승 판정 PASS

  • SID 권위: hasHighIntegritySid가 S-1-16-12288(High) 또는 S-1-16-16384(SYSTEM) 포함 시에만 true. Medium(S-1-16-8192)·빈문자열·null은 false. 테스트로 직접 검증됨.
  • APPDATA 휴리스틱 = 트리거-only: appdataMismatch가 불일치여도 whoami가 Medium이면 medium-or-below로 false — 단독 차단 안 함. 일치(no-trigger) 시 whoami 호출조차 안 함(비용 0, called=false 검증).
  • 판정 불가 폴백 4종 모두 false: non-win32 / no-trigger / whoami-unresolved(resolveBin null) / whoami-failed(execFileSync throw→null). 가용성 우선 보수 폴백이 코드·테스트 일치.
  • win32 외 false: darwin·linux 모두 non-win32 조기 반환(트리거·whoami 미평가).
  • whoami 고정경로·인자(H-2/M12-S-01): resolveWhoamiBin이 %SystemRoot%\System32\whoami.exe(없으면 C:\Windows)를 resolveBin(abs) 절대경로 분기로 검증 — PATH 순회 없음. execWhoamiGroups가 execFileSync(bin, ['/groups'], {shell:false, windowsHide:true, timeout:5000})로 실행. 모킹 테스트로 args·shell·bin 계약 확정.
관찰(버그 아님): hasHighIntegritySid는 stdout 전체에서 SID 문자열을 indexOf로 탐색한다. 정상 whoami /groups 출력에서 그 SID는 무결성 레이블 행에만 나타나므로 false positive 위험은 실질적으로 없으나, "라인/레이블 한정 파싱"이 아닌 단순 substring이라는 점은 설계 의도대로의 보수적 선택임을 기록한다(차단 방향 오탐이 아니라 가용성 영향 없음).

④ 항목 3 — write 보류 3경로 PASS

세 함수 모두 디렉터리 생성·임시파일 open·rename 이전에 elevated 분기를 둔다 — 부분 파일·잔여 tmp가 관리자 프로필에 남지 않는다.

경로위치elevated:true 동작반환
serializer.writeSnapshot정규화 직후, ensureDirFor 이전디스크 무접촉{path,bytes,written:false}
uiStateStore.write정규화 직후, body/dir 이전디스크 무접촉정규화 메모리 상태 객체
config.persistConfigKeyspatch 검증 직후, read-merge 이전디스크 무접촉(read도 안 함)없음(void)

가장 민감한 경로 검증: persistConfigKeys({mailAccounts:[{...pass:'SECRET-PW'}]}) elevated:true → fs.existsSync(cfgPath)===false. 메일 비밀번호가 관리자 프로필에 새 0600 파일로 떨어지지 않음을 테스트가 직접 확인한다. false 경로는 모두 정상 write + 내용 라운드트립.

경계면 일치: 세 함수의 (opts.deps||ctx.deps).elevationState || require('./elevationState') 주입 패턴이 동일하고, main.js의 단일 setElevated 출처와 require 캐시 싱글턴으로 연결됨 — 출처 1·참조 N 단일화 계약 성립.

⑤ 항목 4 — rescan 가드 PASS

electron/ipc/actions.js rescan에서 elevated 분기가 함수 최선두(controller 유효성 체크·sanitizeRescanOpts·scanRoots 평가·acquire()/start() 모두 이전)에 위치. { ok:false, code:'ELEVATED' } 즉시 반환. 테스트로 controller._acquired===false(락·스캔 미시작) 확인. 비상승·미주입(기본 모듈)은 SCAN_STARTED + acquire 호출로 불변. serializer 보류와 합쳐 이중 안전.

⑥ 항목 5 — 배너·push PASS

  • push 시점·페이로드: electron/main.js win.once('ready-to-show') → win.show() 후 _elevated면 pushElevationWarning(). payload는 고정 {elevated:true}만 — APPDATA 경로·프로필명·whoami 출력 비전송. 창 파괴 레이스 가드(isDestroyed) + try/catch.
  • preload 중계: onElevationWarning(cb)가 ipcRenderer 원본 비노출·콜백만 중계, unsubscribe 반환. 비함수 cb는 no-op.
  • 배너 textContent(L-1): public/app.js el('span',{text: ELEVATION_BANNER_TEXT}) — 고정 문구 상수, 데이터 의존 0. innerHTML 미사용. 경로·프로필 비노출. 1회만(_elevationBannerEl 가드).
  • graceful: subscribeElevationWarning이 hasBridge()·typeof spip.onElevationWarning==='function' 미충족 시 즉시 return — 웹/테스트·비상승(push 미수신)에서 배너 미표시. pagehide teardown에서 구독 해제.
  • 스타일: public/styles.css .elevation-banner sticky 상단 z-index:200. 동일 다크 팔레트.

⑨ 헤드리스 불가 — 수동 스모크 (잔여 검증)

아래는 실 OS·실 권한 상승·실 자동업데이트가 필요해 node --test로 재현 불가하다. 릴리즈 전 1회 수행 권장(블로커 아님 — 코드 경계면은 PASS).

  1. 실 상승 판정: 설치본을 "관리자 권한으로 실행" → 상단에 고정 문구 배너 노출, 변경 저장(즐겨찾기·테마·메일계정) 시도 시 디스크 미반영, 재스캔 버튼이 ELEVATED로 거부되는지. 일반 권한 재실행 시 배너 없음·정상 저장 복귀.
  2. 실 자동업데이트 경로: 이전 버전 설치 → 자동 업데이트 트리거 → 재실행 세션이 상승 컨텍스트가 되는지/되지 않는지(A안 allowToChangeInstallationDirectory:false로 per-machine 상승 자체가 제거되었는지) 실측. 업데이트 직후 기존 데이터(즐겨찾기·설정) 손실·오염 없음 확인.
  3. 관리자 프로필 비오염: 상승 세션 중 어떤 저장 동작을 해도 %USERPROFILE%(Administrator)\AppData\Roaming\…\spip 하위에 신규 config/ui-state/cache 파일이 생성되지 않는지(특히 비밀번호 포함 config) 파일시스템 직접 확인.
  4. whoami 실측: 상승 콘솔에서 whoami /groups가 S-1-16-12288를 실제로 출력하는지(환경별 로캘 무관 SID 기반이라 안전하나 1회 확인).
  5. 설치 디렉터리 페이지 제거: NSIS 인스톨러에서 설치 경로 변경 페이지가 사라졌는지(A안), per-user 기본 설치로 UAC 상승 프롬프트가 뜨지 않는지.
