악용 시나리오
설정된 baseURL 서버가 종료 토큰 없이 무한 스트림을 보내면 메인의 text와 렌더러 streamText가 무제한 증가한다. maxTokens(1024 기본)는 서버가 존중할 때만 유효하고, timeout은 토큰이 계속 흐르면 idle 타임아웃에 안 걸릴 수 있다. baseURL은 사용자 자신의 서버라 위협 표면은 낮으나, 오작동 로컬 서버나 잘못 붙인 외부 endpoint에서 발생 가능.
심각도 근거
단일 사용자·자기 endpoint·opt-in이라 낮음. AbortController·단일 in-flight가 동시 폭주는 이미 차단.
권고 조치
스트림 누적에 절대 바이트/문자 상한(예 maxTokens 기반 또는 고정 상한)을 두고 초과 시 abort + 부분 결과 표시. 렌더러 streamText에도 동일 상한(표시는 어차피 PARSE_REASON_MAX로 잘림). 회귀 테스트 권고.

설정 폼 복원을 위해 baseURL 전체를 렌더러로 회송한다. validateBriefingUrl이 자격증명 포함 URL의 영속을 차단하므로 저장된 baseURL에는 user:pass가 없고, 회송 대상도 신뢰 렌더러(guard 통과)뿐이라 실질 누출은 아니다. 다만 키와 달리 baseURL은 평문 회송되므로, 향후 config 파일을 외부 편집해 자격증명 URL을 직접 심으면(검증 우회) 그 값이 그대로 회송될 수 있다(이론적).
심각도 근거
신뢰 렌더러 한정 + 자격증명 영속 차단 + 설정 폼 표시는 정당 용도. 낮음.

키를 persistConfigKeys로 0600 config에 평문 저장(상승 세션 시 write 보류). OS 키체인 미사용은 단일 사용자 로컬 도구로서 메일 비밀번호와 동일하게 일관된 위험 수용. 로컬 LLM은 통상 키 불필요(apiKey || 'not-needed')이라 실제 키 보관은 드묾.
정보I-2외부 baseURL 허용 — 신뢰 불가 데이터의 외부 전송은 사용자 책임(경고 표기됨)
관찰
비-localhost baseURL을 허용하면 프롬프트(메일 본문·커밋 메시지 포함 가능)가 그 외부 host로 전송된다. 정당 용도(자기 온프레미스/클라우드)이고, external 플래그 + 설정 UI에 "⚠ 외부 서버로 데이터가 전송됩니다" 경고(briefing-warn, textContent)가 표기된다. opt-in(기본 enabled:false)이라 무단 전송은 발생하지 않는다. 위험 수용은 사용자 결정.
정보I-3carry-over 2차 인젝션 표면 — 정규화로 차단되나 모델 출력 재투입 경로 존재
관찰
미처리 항목(carry-over)의 title이 다음 프롬프트에 재투입될 수 있는데(2차 인젝션 표면), 실측상 buildCarryOverData→clamp가 길이상한(≤200)·제어/방향성 문자 제거·JSON 인코딩을 적용해 역할 위조 불가. 다만 현 오케스트레이터는 buildPrompt({items, carryOver: []})로 carry-over를 프롬프트에 싣지 않고 표시 항목만 보낸다 — 2차 인젝션 경로가 사실상 비활성. 향후 carry-over를 프롬프트에 포함하도록 바꾸면 위 정규화가 반드시 유지되어야 함(테스트로 못박기).
