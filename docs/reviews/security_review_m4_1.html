3. 발견 항목

M-1 (중간) — watchdog가 in-flight 락을 강제 해제하지 않음 (C-4 구현 간극)

위치lib/server/scanController.js _forceError(L98-102) / watchdog 콜백(L117-120) / finally(L150-154)
위협(STRIDE)서비스거부(Denial of Service) — 가용성
심각도중간 (악용 가능성: 낮음)
악용 시나리오scanner.scan의 어떤 async await가 무한정 hang되면 watchdog는 phase='error'로만 바꾸고 running은 true로 잔존 → 이후 모든 rescan이 영구 409 SCAN_IN_PROGRESS. 설계 M4-M-3는 "watchdog 초과 시 락 강제 해제로 영구 409 방지"를 명시했으나 구현은 이를 충족하지 않음. 완화 요인: (1) 동기 fs(readdirSync/lstatSync)가 죽은 UNC/네트워크 드라이브에서 블록되면 이벤트 루프가 멈춰 watchdog 타이머 자체가 못 돔 → 어차피 watchdog로 못 잡는 경로. (2) 유일한 진짜 async await인 safeExec(git collector)은 자체 SIGKILL 타임아웃 보유. (3) walker는 maxDirs/timeBudgetMs 내부 상한으로 bounded. → 실제 "watchdog는 도는데 finally는 영원히 안 도는" 무한 hang 경로가 좁음. 적대적 입력으로 신뢰성 있게 트리거되는 권한/기밀 위협 아님 → 중간.
실측 근거직접 주입(watchdogMs=80, scan 2000ms 모의): watchdog 발화 직후 측정 running=true, phase=error가 ~2초 지속, scan 완료 후에야 running=false. 즉 락 해제는 watchdog가 아니라 scan settle의 finally가 수행. 기존 테스트(m4-rescan.test.js:241)는 작은 실트리가 빨리 끝나 finally가 락을 풀어 watchdog의 락 해제를 입증하지 못함(거짓 안심).
권고 조치_forceError에서 this.running=false를 함께 해제하고(또는 watchdog 콜백에서 직접), 늦게 settle되는 scan의 finally가 새 스캔(다른 scanId)을 덮어쓰지 않도록 세대 토큰(generation/scanId 일치 확인)으로 가드. watchdogFired 플래그처럼 "이 run이 이미 폐기됨"을 판별해 finally가 running을 건드리지 않게 한다. 회귀 테스트는 실제로 hang하는 scan 모의(영원히 resolve 안 되는 프로미스)로 watchdog 발화 후 즉시 running===false + 재acquire 성공을 단언해야 한다(현 테스트의 거짓 안심 제거).

L-1 (낮음) — 드라이브 루트가 currentPath로 basename 노출(드라이브 문자)

위치lib/server/scanController.js shortenPath(L35-40)
위협(STRIDE)정보노출(Information Disclosure) — 경미
심각도낮음 (악용 가능성: 매우 낮음)
악용 시나리오스캔 루트가 드라이브 루트(예: all-drives에서 C:\)일 때 shortenPath('C:\\')→"C:" 반환 → scan-status 응답에 드라이브 문자가 잠깐 노출. 사용자 경로/디렉터리명은 노출되지 않고 드라이브 문자(이미 사용자가 스캔 대상으로 지정)뿐이며 전이적. C-3의 절대경로 누설 목표는 유지(사용자 경로 0). 정보 가치 미미.
실측 근거직접 주입: shortenPath('C:\\')→"C:". 일반 경로(C:\Users\bob\dev\myproj→dev/myproj)는 드라이브/상위경로 모두 비노출.
권고 조치(선택) shortenPath가 세그먼트 1개뿐이고 그것이 드라이브 표기(/^[A-Za-z]:$/)면 일반 라벨(예: "드라이브 루트")로 치환하거나 null 반환. 보안 차단 사유 아님 — 표시 품질 개선 수준.
그 외 전 항목 발견 0: C-1·C-2·C-3·C-5·M-1(경계)·V-1·V-2·L-1(프론트)·시크릿·의존성은 코드 실재 + 동작 실측에서 결함 미발견. 특히 라이브 HTTP 프로브 15종이 전부 기대 코드(403/405/200)로 응답했고, 직렬화 scan-status에 절대경로·드라이브·사용자경로 문자열이 전무했다.

PM/확인 필요(위험 수용): C-4는 게이트 차단 사유가 아니나 가용성 잔여 위험이다. (a) M4 릴리스 전 시정 후 재검증, 또는 (b) "watchdog가 즉시 락을 풀지 않는다"를 인지한 위험 수용 — 둘 중 하나를 PM이 결정 권고. 무한 async hang 경로가 좁아 즉시 차단은 아니나, 설계 약속(M4-M-3)과의 정합을 위해 (a)를 권장.
