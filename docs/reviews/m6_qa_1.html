2. preload ↔ renderer 계약 일치 (경계면 교차)

지난 onMenu/open 불일치(BUG-1) 전례를 의식해 9개 신규 채널의 인자·반환 shape·code 집합을 preload(electron/preload.js) ↔ renderer(public/app.js) ↔ main 핸들러(electron/ipc/*) 3면 교차 대조. renderer는 ipc(method, ...args) → spip[method](...args) 어댑터로 호출하므로 인자 순서가 그대로 전달된다.

  • onTray payload {action} 객체 — preload가 cb({action})로 중계, renderer dispatchTrayAction이 msg.action 추출. 일치

7. 발견 항목 (관찰 — 블로커/높음 없음)

런타임을 깨는 경계면 불일치는 발견되지 않음. 아래는 설계 정합·무해한 미사용 경로 수준의 관찰.

O-1 [낮음] 트레이 spip:tray:dashboard는 push되지 않음 — renderer 핸들러는 미사용 경로

  • 무엇이: renderer dispatchTrayAction/onTrayCommand가 dashboard action을 처리하지만, main은 spip:tray:favorites만 send하고 spip:tray:dashboard는 보내지 않음.
  • 어디서: electron/main.js onShowDashboard(L185, win.show()/focus()만) ↔ public/app.js onTrayCommand(L2477 dashboard 분기).
  • 기대 vs 실제: 설계와 일치 — m6-design은 '대시보드 열기'를 win.show()로만 정의(L472), favorites만 push(L473). 즉 의도된 동작이며 renderer 분기는 forward-compatible dead-path.
  • 심각도: 낮음 — 동작 영향 없음. 정리하려면 renderer dashboard 분기 제거 또는 main에서 push 추가(택1). 통합 차단 아님.

O-2 [낮음] setToolPath 빈문자열 경로 → NOT_ABSOLUTE (NOT_FOUND 아님)

  • 무엇이: validateToolPath가 len===0을 NOT_ABSOLUTE로 매핑. renderer는 빈 입력을 null(지정 해제)로 변환해 호출하므로 사용자 경로로는 도달하지 않음.
  • 어디서: electron/ipc/tools.js validateToolPath(L75).
  • 기대 vs 실제: 직접 빈 문자열 전달 시 의미상 "경로 없음"이나 NOT_ABSOLUTE 반환. renderer 경로상 무해.
  • 심각도: 낮음 — 사용자 표면 없음.

O-3 [낮음] NOT_FOUND code는 계약 표·renderer 매핑에 있으나 main이 산출하지 않음

  • 무엇이: 설계 §4.1 setToolPath code 집합에 NOT_FOUND 포함, renderer describeToolError도 매핑하지만, validateToolPath는 미존재 파일을 NOT_EXECUTABLE로 통합(주석에 명시 "미존재·비실행·비.exe 모두 여기로").
  • 어디서: tools.js L81 ↔ m6-design.html §4.1 ↔ app.js describeToolError.
  • 기대 vs 실제: 데드 code(영원히 발생 안 함). 런타임 불일치 없음(renderer는 어떤 code든 graceful). 계약 문서 상 미세 불일치일 뿐.
  • 심각도: 낮음 — 동작 무영향. 정리 시 design code 표에서 NOT_FOUND 제거 권고(코드품질 영역, code-reviewer 위임).
