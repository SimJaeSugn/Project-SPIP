신뢰 경계 · 데이터 흐름

파란 흐름 = 요청(System+DATA). 초록 흐름 = LLM 응답 → 표시전용(textContent). 편집 가능한 것은 메인 단계의 System 텍스트뿐이며, DATA 인코딩·분류 소유·출력 표시전용은 그 우회 경로에 영향받지 않는다.

점검 1 — 구조적 방어 불변 (핵심)

사용자가 systemPrompt를 임의 변경하거나 비워도 다음이 코드 레벨로 그대로 강제됨을 확인:

  • DATA 인코딩: buildSignalData/buildCarryOverData → 항목당 clamp(제어·방향성 문자 제거+길이상한) → JSON.stringify({signals,carryOver}). System 텍스트 분기(override || 시드)와 완전히 독립된 경로.
  • 분류 소유: category는 정책이 결정해 DATA에 박제, 모델에는 설명/가이드만 요청. 프롬프트 편집으로 분류 권한이 모델로 넘어가지 않음.
  • 출력 표시전용: streamText·title·reason·guide 전부 el(...{text})=textContent 렌더(public/app.js), 명령/링크 자동실행 경로 없음. resolveItem은 모델 문자열이 아닌 key(정규식 검증)만 사용.
  • 회귀 테스트: [systemPrompt] 인젝션 방어 불변 테스트가 "악성 System override + 악성 라벨 동시 주입" 시 구조 위조 실패·방향성 제거·분류 정책 소유를 검증. 35/35 통과.

평가: 편집 가능 프롬프트가 구조적 방어를 약화하지 않음 — PASS

점검 2 — systemPrompt 자체 안전 (길이 / 정제)

  • 길이 상한: preload String() 강제 → IPC ≤16000 초과 거부(BAD_ARGS) → normalize clamp 8000. 이중 방어로 거대 문자열 DoS 차단.
  • 정제: sanitizeSystemPrompt가 C0/DEL·방향성 제어문자(U+202A~E, U+2066~9) 제거, \n/\t만 보존. 공백뿐이면 ''(시드 폴백).
  • 신뢰 영역(System)이므로 인젝션 정제는 불필요하나, 비정상 입력(거대/제어/방향성) 방어는 적용됨. 16000자는 토큰·메모리상 무해 수준이라 실질 자원고갈 위협 없음(F-1).

평가: 길이/정제 적정 — PASS

점검 3 — 저장 · 노출

  • 0600 저장: persistConfigKeys가 임시파일→fsync→rename→chmod 0600. 상승 세션이면 디스크 write 보류(메모리 유지).
  • getSettings 노출: systemPrompt/defaultSystemPrompt 회송은 시크릿 누출 아님(프롬프트는 비밀 아님). apiKey는 hasApiKey:boolean만, baseURL은 검증 통과값만 회송 — 자격증명 비노출 유지.
  • 렌더러: textarea는 .value, placeholder는 setAttribute — innerHTML 미사용(L-1 유지).

평가: 저장·노출 안전 — PASS

점검 4 — IPC shape

  • 신규 채널 0: 기존 spip:briefing:getSettings/setSettings 확장만. preload 표면 변화 없음(필드 추가뿐).
  • shape 검증: validateSettingsArgs가 systemPrompt를 문자열/길이(≤16000)로 검증, null→''(시드 복원), 그 외 타입/과길이는 BAD_ARGS 거부.
  • setSettings 응답은 getSettings shape 재사용 — 키 평문 회송 0.

평가: IPC 표면·검증 안전 — PASS
