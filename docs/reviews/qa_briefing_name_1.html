QA 검증 — 브리핑이 해시 ID 대신 프로젝트 이름을 호명하는가

버그: 브리핑이 프로젝트 name 대신 내부 해시 id를 호명. 수정: name을 데이터 경로 5단계로 전파, 프롬프트가 해시 대신 label(name) 전달, 제목 폴백을 name화. 본 검증은 동작·경계면 정합·회귀를 본다 (코드 품질·보안 취약점·문서 정합은 각 검증자 몫).

검증 방식: 경계면 교차 비교(5단계 양쪽 동시 대조) + npm test 실제 실행 + done payload shape 대조 · 날짜 2026-06-25

종합 판정

PASS — 5/5 항목 통과

버그 해소 확인. 프롬프트 DATA의 식별자가 해시 targetId → 사람이 읽는 label(name)로 교체되었고, 해시는 항목 매칭용 key에만 남아 모델에 식별자로 노출되지 않는다. 제목/표시 폴백이 모두 targetLabel(name)을 먼저 사용하고, name이 없으면 신호유형으로 graceful 폴백한다. 데이터 경로 5단계(deriveSnapshot → normProject → signal.targetLabel → item.targetLabel → prompt label)가 끊김 없이 이어진다. 회귀 0 — 기존 신호 판정·carry-over·done payload shape 불변.

1. 버그 해소 (핵심) — PASS

무엇을 봤나: buildSignalData가 모델에 무엇을 식별자로 싣는지, 해시가 프롬프트 DATA에 노출되는지, 제목 폴백이 무엇을 쓰는지를 양쪽 코드와 테스트 어서션으로 대조했다.

  • label = name 탑재. buildSignalData가 항목당 { key, type, category, label, context }를 싣고, label은 clamp(it.targetLabel, TITLE_MAX) — 즉 프로젝트 name이다. 과거의 해시 식별자 필드는 없다.
  • 해시 비노출. DATA JSON에 target(해시) 필드가 존재하지 않는다. 해시는 key(항목 매칭·dedup용)에만 남는다. 어서션 obj.signals[0].target === undefined, obj.signals[0].key === hash로 직접 검증된다.
  • 제목 폴백 name화. _mapViews의 title 폴백이 v.title || it.title || it.targetLabel || it.signalType 순서 — 모델이 title을 비우면 name(targetLabel)을, name마저 없으면 신호유형으로 떨어진다. 해시는 폴백 사슬 어디에도 없다.
  • graceful 폴백. name이 비면 label=''이 되고, 시스템 프롬프트가 "label이 비어 있으면 신호 유형으로 일반적으로 표현한다"고 명시. 제목 폴백 사슬도 빈 label → signalType으로 안전 종료.
  • 시스템 지시 강화. SYSTEM_PROMPT에 "DATA의 label로만 부른다 / 내부 key·해시는 절대 노출하지 말라" 2줄이 추가됨 (테스트가 /label/·/해시/ 정규식으로 존재 확인).
인젝션 방어 회귀도 동시 확인: name에 RTL(U+202E)·제어문자(BEL)·JSON 구조 위조 페이로드를 주입해도 clamp가 제어/방향성 문자를 제거하고 JSON.stringify가 구조 위조를 흡수(단일 항목 유지), 시스템 프롬프트 무오염. name 도입이 기존 N-08/M-3 방어를 약화시키지 않았다.

3. 회귀 0 — PASS

  • 신호 판정 불변. dirty/ahead/behind/attention/deadline/mail/disk/scan 판정 조건식은 손대지 않았다. mk 시그니처에 마지막 인자 targetLabel만 추가 — 기존 호출은 인자가 부족해도 typeof !== 'string' → ''로 안전.
  • signal shape. { type, targetId, targetLabel, category, urgent, meta } — 기존 필드 그대로, targetLabel 추가뿐. targetId는 매칭·dedup·키 산출(itemKey)용으로 그대로 유지되어 안정 키가 바뀌지 않는다(carry-over 키 호환).
  • item shape. targetLabel 필드 추가. itemKey는 여전히 signalType + ':' + targetId 기반 — name이 키에 섞이지 않아 carry-over 매칭 불변.
  • auto-resolve / carry-over / trigger. isResolved·selectCarryOver·mergePersist· evaluate.trigger/urgent는 name을 참조하지 않는다 — 영향 없음.
  • done payload shape 불변(중요). toDonePayload는 여전히 { key, category, title, reason, guide, ref }만 내보낸다. targetLabel·targetId는 페이로드에 포함되지 않으므로 IPC 계약·프런트 무영향. name은 title 폴백을 통해서만 사용자에게 도달(텍스트 표시).
  • 전체 그린. 기존 891 + 신규(policy/items/prompt name 케이스) = 901 모두 통과.

잔여 리스크 / 수동 스모크 권고

자동화 한계 — 실제 LLM 호명은 미검증. 단위 테스트는 프롬프트 DATA에 name이 실리고 해시가 빠진 것까지만 보장한다. "모델이 실제로 그 label로 사용자를 호명하는가"는 시스템 지시 준수에 달려 있어 실 LLM 1회 수동 스모크를 권한다:
  • 브리핑 enable + 실제 baseURL/모델 설정 → dirty 프로젝트 1개 만들어 트리거 → 생성된 title/reason에 프로젝트 이름(예: My-Cool-Project)이 보이고 32자 hex 해시가 보이지 않는지 육안 확인.
  • name이 빈 프로젝트(스캔 메타 누락)로도 트리거해 "미커밋 변경" 류 일반 표현으로 떨어지는지 확인.
이 스모크는 코드 결함 검증이 아니라 모델 프롬프트 준수 확인이므로, 실패 시 원인은 시스템 지시 문구 튜닝(프롬프트 영역)이다.
