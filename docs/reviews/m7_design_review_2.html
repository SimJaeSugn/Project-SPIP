요약

아키텍트가 1회차 PASS(조건부)에서 권고한 P2 2건(R1·R2)·P3 3건(R3·R4·R5)을 rev.2에 반영했다. 본 재검증은 ① 각 권고의 실제 해소 여부 ② 보안 조건부 PASS 반영(SEC-H1/H2)이 요구·기존 기능 회귀를 유발하지 않는지 ③ 최종 창 속성·IPC 계약의 일관·완결을 실측 코드 대조로 확인했다.

결과: 5건 모두 정규 설계로 승격·해소됐고, 해소 과정에서 새로운 모순·회귀를 만들지 않았다. rev.2가 추가로 반영한 SEC-H1(CSP 세션 공유 모델)·SEC-H2(UI redressing 4통제)도 실측 코드(applyCspHeaders(session) 1회 등록·세션당 1핸들러)와 정합하며, "독립 토스트 창"·"우측하단" 요구와 충돌하지 않는다. 구현 착수 가능(PASS). 잔여 미결은 MQ7-3(메인 진입점) 1건으로, 설계가 "트레이 경유만 필수, 메인 버튼은 PM 결정"으로 명시해 블로커가 아니다.

보안 반영(SEC-H1/H2) 회귀 확인

rev.2가 보안 조건부 PASS(높음 2)를 함께 반영했다. 이 통제가 R-22 요구(독립 토스트 창·우측하단)·기존 기능 회귀를 만드는지 추적성·정합 관점에서 점검했다(실제 보안 충분성은 security-reviewer 소관).

반영 항목회귀실측 대조 / 정합
SEC-H1 — CSP 세션 공유(앱 1회 등록·위젯 재호출 안 함)없음 실측: main.js:151 applyCspHeaders(win.webContents.session)(session 인자만·partition 무관)·security.js:40~46 session.webRequest.onHeadersReceived 등록. onHeadersReceived는 세션당 1핸들러가 사실이며, 위젯이 default session 공유 시 앱 1회 등록이 위젯 로드에도 자동 적용된다는 설계 주장은 코드와 정합. ensureWidget deps에서 applyCspHeaders 제거(242행)·재호출 금지 주석(253~254) 일관. 메타 CSP는 CSP_POLICY 단일 원천 동일(SEC-L2). 위젯이 CSP 미적용으로 새는 표면 없음 — 회귀 0.
SEC-H2 — 4통제(movable:false·hide 시 alwaysOnTop 해제·focus 게이팅·클릭통과 비허용)없음 ① movable:false·드래그핸들 미사용(248행)은 MQ7-2 확정과 일치, "우측하단 고정 토스트"는 R-22 요구(우측하단 등장)와 오히려 부합 — 사용자가 위치 이동을 요구하지 않았고 positionBottomRight(§4.2)가 매 show 재계산. ② show 시 floating ON·hide/blur 시 OFF(258·271·275) — 토스트 가시성은 표시 중 유지되므로 "토스트답게 위로 뜸" 요구 충족, 숨김 시만 해제라 UX 회귀 없음. ③ focus 게이팅은 표시는 비포커스 허용·액션 실행만 focus 후(§5·§13)라 토스트 표시 자체를 막지 않음. ④ 클릭통과 비허용(setIgnoreMouseEvents 미사용)은 ADR-M7-5에서 "위젯 클릭=위젯, 밖 클릭=blur"라는 단순 경계로 정리 — 토스트 인터랙션(열기·복사·해제) 정상. 독립 토스트 창 요구·기존 기능 회귀 0.
위젯 비드래그 고정 vs "우측하단 토스트" 요구 부합부합사용자 요구는 "우측 하단 등장"이며 위치 이동(드래그)은 요구 사항이 아니다. movable:false+positionBottomRight 고정은 요구를 정확히 충족하며, 드래그 이동은 MQ7-2에서 "후속·별도 보안 점검"으로 명시 보류 — 요구 충족과 보안 통제가 양립. 충돌 없음.
