요약: M6 설계는 R-18(사용자 지정 외부 툴 실행)이라는 구조적으로 가장 위험한 신규 표면을 도입하면서도, 기존 H-1(pathGuard)·H-2(resolveBin .exe 한정·shell:false)·M-4(in-flight 상한) 체인을 재작성 없이 재사용하는 올바른 방향을 택했다. 실측 결과 safeExec.resolveBin은 설계 주장대로 절대경로·.exe·실행파일 검증 분기를 이미 보유(safeExec.js:82~91)하고, safeExec는 spawn 직전 .exe를 재검증(:146~152)한다 — 사용자 지정 경로에도 동일 적용된다는 설계 주장은 코드 접지된다. 치명 항목은 없다. 다만 높음 2건(M6-H-1 resolveBin 캐시로 인한 TOCTOU·재검증 누락 위험, M6-H-2 args 사용자 지정 인자 주입 표면)은 설계 명세에 방어가 빠져 있어 구현 전 설계 반영이 필수다. 이를 반영하면 PASS로 승격 가능하다.
잘 보존된 부분(PASS 명시): 신규 IPC 8채널 전부 register.js guard 경유(senderFrame·try/catch·고정 code) 명세 / 클립보드 main writeText 텍스트 한정·읽기 미노출 / ui-state id가 실행·경로에 미사용(표시·정렬 메타 한정) / 즐겨찾기 슬라이더 메인창 오버레이(신규 BrowserWindow·preload·CSP 0) / contextIsolation·sandbox·preload allowlist·CSP·textContent(L-1)·0600 원자적 쓰기 보존 / pathGuard H-1 open 일반화 후에도 불변.

1. 자산 · 신뢰 경계 · 데이터 흐름 (M6 델타)

보호 대상: ① 사용자 OS(임의 코드 실행 방지) ② 사용자 파일시스템(허용 경로 외 접근 방지) ③ 설정·UI 상태 무결성. 핵심 신뢰 경계는 renderer→IPC→main이며, M6는 여기에 "사용자가 지정한 실행 파일 경로"라는 신규 부분신뢰 입력을 추가한다.

2. 발견 항목 (STRIDE · 등급화)

2.1 높음 (구현 전 설계 반영 필수)

ID취약점위치 / STRIDE심각도악용 시나리오 · 권고 조치
M6-H-1 resolveBin name-캐시로 인한 사용자 지정 경로 TOCTOU / 재검증 누락 §6.1 resolveTool · safeExec.js:37,75~91,89
Elevation / Tampering		 높음 시나리오: 실측 resolveBin은 _binCache(Map, name→absPath|null)로 결과를 영구 캐시한다(safeExec.js:38,77,89). 설계의 resolveTool은 rb(tool.path)를 실행 시점마다 호출하지만, 절대경로 키가 캐시에 있으면 fs 재검증 없이 캐시값 반환한다. 공격(XSS, RC-2 가정)이 ⓐ 존재하는 양성 .exe 경로를 setToolPath로 등록(검증 통과·캐시됨) → ⓑ 같은 경로에 악성 실행 파일을 덮어쓰기(사용자 쓰기 가능 위치, 예 %LOCALAPPDATA%) → ⓒ open 시 캐시된 절대경로가 재검증 없이 spawn = 임의 코드 실행. 또한 setToolPath 검증 통과 후 실제 spawn까지 사이 파일 교체(고전 TOCTOU)도 동일.
권고: ① 사용자 지정 경로는 setToolPath 시점 검증과 resolveTool/spawn 직전 검증을 둘 다 명세하고, 후자는 캐시 우회(예 rb(path, {force:true}) 또는 isExecutableFile 즉시 재호출)로 spawn 직전 최신 fs 상태를 본다. ② 설계 §3.2가 "검증은 실행 시점에 resolveBin이 수행"이라 했으나 캐시로 그 보장이 깨짐을 명시하고, resolveTool이 사용자 경로에 한해 force 재해석함을 설계에 박는다. ③ 가능하면 사용자 지정 경로의 허용 디렉터리(쓰기 불가 Program Files 등)를 권고 안내(완전 차단은 과함, MQ로).
M6-H-2 사용자 지정 args 인자 주입 — 에디터 플래그 악용 §3.2 tools.args · §6.2 exec(r.bin,[...r.args,real])
Elevation / Tampering		 높음 시나리오: shell:false·인자 배열이라 셸 메타문자 주입(OS 커맨드 인젝션)은 차단된다(PASS). 그러나 args는 대상 프로그램에 그대로 전달되는 신뢰 안 된 플래그다. 다수 에디터/툴이 위험 플래그를 가진다 — 예: VS Code --install-extension <vsix>, Chromium 계열 --no-sandbox, 일부 툴의 --exec/-c/--eval. XSS가 setToolPath로 path=신뢰툴, args=["--위험플래그", ...]을 등록하면, 검증을 통과한 양성 실행 파일을 무기화해 확장 설치·샌드박스 무력화·코드 실행으로 이어질 수 있다(LOLBin 패턴). 설계는 args를 "string·≤16·≤256" 형태 검증만 하고 내용/의미 검증이 없다.
권고: ① args를 기본 비허용/공백으로 두고, 정말 필요한 알려진 안전 플래그만 화이트리스트로 받는 것을 1순위로. ② 최소한 --로 시작하지 않는 위치 인자 금지, 그리고 경로(real)는 항상 args 뒤 마지막 위치 고정(현 설계 [...r.args, real]은 충족 — 명시 유지). ③ args가 임의로 허용되면 "사용자 지정 args는 해당 툴에 대한 신뢰 위임"임을 위협모델에 명시하고, args 편집을 pickToolExecutable(네이티브 dialog) 경로에서는 노출하지 않는 등 setToolPath args 입력 자체를 MVP에서 보류하는 안을 ADR로 검토.

2.2 보통

ID취약점위치 / STRIDE심각도악용 시나리오 · 권고 조치
M6-M-1 toolId 일반화로 인한 PATH 폴백의 RCE 표면 미세 확대 §6.1 resolveTool 폴백 rb(toolId)
Elevation		 보통 시나리오: resolveTool은 사용자 경로 실패 시 rb(toolId)로 PATH 탐색 폴백한다. toolId는 ^[a-z0-9_-]{1,32}$로 제한되나(PASS), XSS가 setOrder류가 아닌 open(id, toolId)에 임의 toolId를 줄 수 있다. toolId가 PATH상 동명 실행 파일(예 사용자가 만든 code.exe가 우선순위 PATH 디렉터리에 존재)을 가리키면 의도 외 바이너리 실행 가능. 단 resolveBin은 cwd(".")·빈 항목을 PATH에서 제외(safeExec.js:96~98)하고 .exe만 허용하므로 표면은 작다.
권고: open의 toolId를 config.tools에 등록된 id로만 제한(레지스트리 멤버십 검증)하고, 미등록 toolId의 PATH 자동 탐색은 예약 id(code) 등 명시 화이트리스트로 한정. 임의 toolId의 PATH 탐색 폴백을 막아 "renderer가 PATH 임의 바이너리 이름을 호출"하는 경로를 차단.
M6-M-2 pickToolExecutable의 exe 필터는 UI 편의일 뿐 — main 재검증 의존 명시 필요 §4.1 pickToolExecutable · §6.3
Spoofing / Elevation		 보통 시나리오: 설계는 pickToolExecutable을 "OS가 준 절대경로(신뢰 출처)"로 본다. 출처 신뢰는 타당하나, dialog의 filters:[exe]는 표시 필터일 뿐 강제가 아니다(사용자가 "모든 파일"로 바꿔 .bat/.lnk 선택 가능, OS에 따라 .lnk는 타겟 해석 위험). 설계가 "setToolPath와 동일 검증 후 영속"이라 했으므로 결과적으로 resolveBin이 .exe만 통과시켜 안전하지만, 이 "dialog 결과도 신뢰하지 않고 main 재검증" 원칙을 설계 표에 명시적으로 못박아야 한다(현재는 "OS 신뢰 출처"로만 기술되어 검증 누락 구현을 유발할 수 있음).
권고: §4.1 pickToolExecutable 행에 "dialog 반환 경로도 setToolPath와 동일한 resolveBin 검증 필수(필터는 신뢰 근거 아님)"를 명문화. .lnk는 .exe 아니므로 자동 거부됨을 확인 항목으로.
M6-M-3 close-to-tray 백그라운드 상주가 만드는 표면 — second-instance·트레이 push 신뢰 경계 §9.2 second-instance / tray push
Spoofing / DoS		 보통 시나리오: 앱이 트레이에 상주하면 ⓐ second-instance 이벤트가 외부(다른 프로세스의 동명 실행)로 트리거되어 창을 표시·복원시킨다 — payload(argv/cwd)를 신뢰해 처리하면 위험하나, 설계는 win.show()/restore()/focus()만 하고 argv를 파싱·실행하지 않음(PASS 방향). ⓑ 트레이 메뉴 클릭→spip:tray:* push는 main이 메인 win.webContents로만 보냄(senderFrame류 아님, 단방향 push라 신뢰 경계 안). 잔여 위험은 second-instance argv를 향후 "열 프로젝트 경로"로 쓰는 확장 시 신뢰 경계 붕괴.
권고: §9.2에 "second-instance payload(argv/cwd)는 절대 신뢰·실행하지 않으며 win.show 트리거로만 사용" 불변식을 명시. 트레이 push action은 dashboard|favorites 고정 화이트리스트임을 명문화(임의 action push 금지).
M6-M-4 ui-state.json 신규 영속 파일의 손상/거대/주입 graceful 명세는 있으나 크기·파싱 상한 부재 §3.2 uiStateStore · §14
DoS / Tampering		 보통 시나리오: id 형식(^[0-9a-f]{1,64}$)·개수 상한(favorites 512)·중복 제거·화이트리스트는 잘 설계됨(PASS 방향, id가 실행/경로에 미사용이라 오염 영향 극소). 그러나 파일 자체의 바이트 크기 상한·JSON 파싱 전 크기 가드가 명세에 없다. 로컬 파일을 임의 조작할 수 있는 공격자(또는 디스크 손상)가 수백 MB JSON·깊은 중첩을 넣으면 JSON.parse 단계에서 메모리/시간 DoS. config.js graceful 패턴 재사용이라 했으나 그 패턴에도 명시적 크기 상한이 있는지는 설계상 불명.
권고: uiStateStore.read에 파일 크기 상한(예 1MB)·읽기 후 길이 가드 → 초과 시 기본값 폴백을 명세. order/favorites 배열 길이 상한을 파싱 직후에도 재적용(슬라이스). schemaVersion 불일치 시 폴백 규칙 명문화.

4. 확인 필요 (위험 수용 결정)

항목질문 / 결정 필요권고 기본값
SQ-1(M6-H-2 관련) tools.args 사용자 자유 입력을 MVP에 포함할지. 자유 args는 "양성 실행 파일의 위험 플래그 무기화" 표면을 연다. 화이트리스트만 받거나, MVP에서 args 편집을 보류할지.MVP는 args 빈 배열 고정(편집 미노출). 추후 알려진 안전 플래그 화이트리스트로 확장.
SQ-2(M6-H-1 관련) 사용자 지정 실행 경로의 허용 위치를 제한할지(쓰기 가능 사용자 디렉터리의 .exe 등록을 경고/차단). 완전 차단은 정상 사용자 워크플로(포터블 에디터)를 막을 수 있음.차단 대신 spawn 직전 강제 재검증(캐시 우회)을 필수화. 위치 제한은 경고 안내까지만.
SQ-3open(id, toolId)의 toolId를 config.tools 등록 멤버로 제한할지(M6-M-1), 아니면 예약 id의 PATH 자동탐색을 유지할지.등록 멤버 + 예약 id(code) 화이트리스트로 제한.
SQ-1·SQ-2는 R-18 핵심 위험과 직결되므로, PM/사용자 결정 전이라도 설계 문서에 방어 명세(spawn 직전 재검증·args 기본 차단)를 선반영한 뒤 정책만 조정하는 것을 권고한다.

5. 결론 · 게이트 처리

판정: 조건부 PASS — 치명 0, 높음 2(M6-H-1·M6-H-2), 보통 4, 낮음 3.

게이트 조건(구현 착수 전 설계 반영 필수):
  1. M6-H-1: 사용자 지정 경로는 setToolPath 검증 + spawn 직전 캐시 우회 재검증(force/즉시 fs 확인)을 둘 다 설계에 명시. resolveBin name-캐시가 "실행 시점 검증"을 무력화함을 §3.2·§6.1에 반영.
  2. M6-H-2: tools.args를 기본 차단/화이트리스트로 좁히고, real 경로 마지막 위치 고정을 명문화(SQ-1 결정 반영).
위 2건을 설계에 반영하면(2회차 재검토 시) PASS 승격 가능. 보통 4건은 구현 시 동시 반영 권고, 낮음 3건은 백로그.

M6 설계의 전반 방향(기존 H-1/H-2/M-4 체인 재사용·신규 표면 격리·오버레이 선택·텍스트 한정 클립보드)은 보안적으로 건전하며, 위협을 새로 만들기보다 기존 통제에 편입하려는 의도가 일관된다. 다만 R-18은 본질적으로 "사용자가 무엇을 실행할지 정하는" 기능이라 캐시·인자 두 지점에서 방어가 명시적으로 박혀야 안전하다. 코드는 직접 수정하지 않으며, 위 시정 항목을 담당 아키텍트/개발자에게 이관한다.
