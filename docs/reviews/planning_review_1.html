판정: FAIL

검증 대상: docs/PRD.html, docs/FeatureSpec.html, docs/UserStories.html · 정합성 비교: package.json, README.md. 전반적으로 구조·수용 기준·범위 정의의 완성도는 높으나, 추적성 누락(비기능 요구 미연결), MVP 경계 문서 간 불일치, 수용 기준 기준값의 상하위 문서 역전, 몇몇 정량 기준의 검증 불가능성이 확인되어 반영이 필요하다.

검증자: 독립 리뷰어 · 검증일: 2026-06-21 · 기준 폴더(docs/Planning/) 부재 → 일반 기획 기준 + 제품 컨텍스트 룰로 검증

1. 요약

확정 요구(인사이트 4종, 스캔 2모드, 순수 HTML/JS, VS Code 열기, git 연동, CLI)는 모두 R-ID로 반영되어 있고 셸 인젝션·경로 순회·로컬 바인딩 등 핵심 보안 리스크도 다뤄졌다. 그러나 다음 4개 영역에서 FAIL 사유가 발견됐다.

  • 추적성 단절: 비기능 요구 N-01(성능)·N-06(데이터)·N-07(접근성/UX)이 UserStories의 어떤 스토리에도 연결되지 않아 백로그에서 누락 → 구현·테스트로 추적 불가.
  • MVP 경계 불일치: PRD 내부(M3 vs "MVP 정의" 노트)와 PRD↔UserStories 간 MVP 구성 요구 집합이 서로 다르다(특히 R-13, N-07).
  • 수용 기준 역전: FeatureSpec의 "2초 이내" 등 정량 기준이 상위 문서 PRD에 없는데 하위 문서가 신설 → 단일 출처 위반.
  • 검증 불가능한 기준: "체감 <200ms", "2초 이내" 등 측정 주체·조건이 불명확하거나 임의값.
결론: 설계 착수 전 아래 P1 항목(추적성·MVP 경계)을 반드시 정리해야 한다. P2/P3는 설계와 병행 정리 가능하나 명시 권고.

권고: US-E4(성능, N-01: "200개 루트 30초 이내" AC), US-E5(데이터/인증, N-06), US-E6(접근성, N-07: 키보드 포커스·색상 비의존 배지)를 추가하고 R-ID 컬럼에 연결.
P1 MVP 경계 3중 불일치 PRD §12 (단계표 + "MVP 정의" 노트), UserStories §MVP 스프린트 문제: MVP 구성 요구 집합이 세 곳에서 다름(N-07 포함 여부, R-13/R-14 단계 귀속).

P1 FeatureSpec 화면 기능 미상세 FeatureSpec.html (현재 F-1, F-2만) 문제: R-10(대시보드 표시)·R-11(필터/정렬/검색)이 명세서에 정상/예외/엣지 흐름으로 상세화되지 않음. 헤더는 "두 개 핵심 기능"이라 의도된 축소이나, MVP Must 기능의 화면 명세 공백은 구현 모호성을 남김.
권고: F-3(대시보드 목록·통계, R-10)·F-4(필터/정렬/검색, R-11)를 추가하거나, 최소한 PRD/US에서 화면 동작(빈 상태, 0건, 필터 AND 동작, 정렬 키)을 충분히 확정.
P2 수용 기준값 상하위 역전 FeatureSpec F-2("2초 이내"), F-1 AC 문제: "버튼 클릭 후 2초 이내 code 실행" 기준이 상위 PRD R-12에 없는데 하위 명세가 신설. 단일 출처 원칙 위반(상위와 충돌 시 추적 혼선).
권고: 시간 기준을 PRD R-12 수용 기준에 먼저 정의하거나, FeatureSpec은 "PRD R-12 기준 준수"로만 참조하고 임의 수치 제거.
P2 검증 불가능한 정량 기준 PRD R-11("체감 <200ms"), §3 탐지정확도 "≥95%" 문제: "체감"은 측정 주체·방법 불명. "탐지 ≥95%"는 분모(전체 실제 프로젝트 모집단) 정의가 없어 테스트 불가.
권고: (i) 200ms는 "필터 조작 입력→리스트 재렌더 완료 N건 기준"으로 측정 조건 명시 또는 정성 기준으로 강등. (ii) 95%는 "사전 정의된 검증용 프로젝트 세트 대비 탐지율"로 모집단·측정 절차 명시.
P2 OoS와 MoSCoW 분류 모순 PRD §9 범위 밖 표 문제: "watch/자동 재스캔", "태그/메모 편집"을 "범위 밖" 섹션에 두면서 Could로 분류 — "범위 밖"인데 이번 버전 후보라는 모순.
권고: 섹션 제목을 "MVP 범위 밖(후속 후보 포함)"으로 바꾸거나, 진짜 제외(Won't)와 후속 후보(Could)를 표에서 시각적으로 분리.
P3 R-16 우선순위 vs 마일스톤 정합 PRD §12 M4, FeatureSpec F-1 트리거 문제: R-16(재스캔)은 Could인데 FeatureSpec F-1 트리거에 "대시보드 재스캔(R-16)"이 동등 기재 → Could 기능이 핵심 기능 트리거로 보일 소지.
권고: F-1 트리거에서 R-16을 "(Could, 후속)"로 표기.
P3 README ↔ PRD 기능 범위 불일치 README.md "주요 기능" 문제: README는 인사이트를 "요약 정보(최종 수정 시각 등)"로만 기술 — git 상태(R-08)·언어 분포(R-06)·규모(R-09)·필터/검색(R-11)·VS Code 열기(R-12)가 빠져 PRD 범위와 어긋남.
권고: README 주요 기능을 PRD R-06~R-12 기준으로 동기화(외부 산출물 정합).
P3 보안 리스크 강화 명시(누락 아님) PRD N-03, FeatureSpec F-2 엣지 문제(경미): 셸 인젝션 방지는 "인자 배열 전달"로 다뤘으나, R-16/재스캔 시 사용자 입력 루트 경로의 검증, 그리고 git CLI 호출 인자(브랜치명 등)에 대한 인젝션 표면은 명시 안 됨.
권고: N-03에 "모든 외부 프로세스 호출(code/git)은 shell:false + 인자 배열"을 일반 규칙으로 승격. (상세 보안 판정은 security-reviewer 영역)

6. 확인 필요 (사용자/PM 판단 쟁점)

  • MVP에 N-07(접근성)을 포함할 것인가? 세 문서가 엇갈리는 핵심 원인. 단일 사용자 로컬 도구 특성상 MVP에서 제외(Should 강등)도 합리적 — 결정 후 통일 필요.
  • FeatureSpec 상세 범위: 의도적으로 2개 기능만 명세한 것인지, R-10/R-11도 명세 대상인지. 설계 모호성 감소 차원에서 추가 권고.
  • PRD §11 미해결 질문 Q1~Q4(기본 루트, stale 90일, 포트, 영속 위치)는 가정으로 남겨 두었으나, 일부(기본 루트·영속 위치)는 설계 착수 전 확정 권고.

7. 통과한 항목 (재확인)

  • 확정 요구의 R-ID 부여 자체는 빠짐없음(인사이트 4종/스캔 2모드/순수 HTML·JS/열기/git/CLI).
  • 핵심 보안 리스크 4종(드라이브 성능·셸 인젝션·경로 순회·로컬 바인딩)이 N-02/N-03/R-02/R-03 및 엣지 케이스로 다뤄짐.
  • package.json(bin: spip→scan.js, scripts: scan/start/dev, engines node>=16, type commonjs)이 PRD N-04 및 R-14와 정합.
  • Given/When/Then 형식의 수용 기준 다수가 측정 가능하게 작성됨(R-01 중첩 1건, R-02 재귀 차단, R-08 git N/A 등).
