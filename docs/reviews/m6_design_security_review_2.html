요약: 게이트 조건 높음 2건이 구조적으로 해소됐고, 설계 명세는 실측 코드와 정합한다. M6-H-2는 PM 확정으로 tools.args 필드를 스키마에서 전면 삭제하고 실행 인자를 [projectPath] 단일 고정으로 못박아 — 사용자가 인자를 주입할 경로 자체가 부재하므로 LOLBin 무기화 표면을 원천 제거했다(완화가 아니라 제거). M6-H-1은 spawn 직전 resolveBin(path,{force:true}) 캐시 우회 강제 재검증을 resolveTool·open 흐름에 명세했고, 실측 safeExec.js:75~91에 opts.force 분기·_clearBinCache(:116)·spawn 직전 .exe 재검증(:146~152)이 모두 실재해 설계 주장이 코드 접지된다 — "등록 후 파일 교체→재검증 없이 spawn"되는 TOCTOU 창이 닫혔다. 보통 4(M6-M-1~4)·낮음 3(L-1~3)도 본문에 반영. rev.2가 만든 신규 위협은 없다(트레이/close-to-tray·persistConfigKeys 일반화·DoS 가드 모두 신뢰 경계 보존). 치명·높음 0 → 보안 게이트 통과(PASS).
잔여 권고(게이트 비차단): 본 PASS는 설계에 대한 것이다. 구현 단계에서 ① setToolPath 성공 시 캐시 무효화(_clearBinCache 또는 해당 키 force 재해석)의 실제 호출 ② resolveTool이 매 open마다 force 호출하는지 ③ normalizeTools가 입력 args 키를 실제로 drop하는지 ④ uiStateStore.read의 1MB 크기 가드 ⑤ second-instance argv 미파싱을 코드 리뷰(m6 security_review 구현 회차)에서 실측 확인할 것. 설계는 충분하나 구현이 명세를 이탈하면 H-1/H-2가 재발할 수 있다.

1. 1회차 게이트 조건 해소 검증 (높음 2건)

1.1 M6-H-1 — resolveBin 캐시 TOCTOU / 재검증 누락

검증 항목1회차 요구rev.2 반영 / 실코드 정합판정
spawn 직전 캐시 우회 강제 재검증 사용자 지정 경로는 resolveTool/spawn 직전 캐시 우회(force) 재검증을 명세 §6.1 resolveTool 의사코드: const abs = rb(tool.path, { force: true })(절대·존재·.exe·realpath). §6.2 open 흐름이 spawn 직전 resolveTool 호출. 실코드 정합: safeExec.js:77 if (!opts.force && _binCache.has(name)) return ... → force=true면 캐시 우회·fs 재확인(:82~90). 설계 주장과 정확히 일치. 해소
setToolPath 시 캐시 무효화 등록 시점 검증 + 캐시 폐기 §4.1 setToolPath: "성공 시 캐시 무효화(_clearBinCache 또는 force) 후 영속". §6.1 warn-box(P3-2) 캐시 무효화 정책 명문. 실코드 정합: safeExec.js:116~118 _clearBinCache() export 실재(:264). 해소

M6-H-1 종결: 설계가 {force:true} 캐시 우회를 resolveTool·setToolPath·getTools resolved 산출 모든 경로에 일관 적용하도록 명세했고, 그 force 의미(캐시 무시 재해석)가 실측 safeExec.resolveBin(:75~91)에 이미 구현돼 있어 설계가 허구 위에 서지 않는다. 1회차의 "name-캐시가 실행 시점 검증을 무력화" 시나리오는 더 이상 성립하지 않는다.

게이트 통과 근거:
  1. M6-H-1 해소: spawn 직전 resolveBin(.,{force:true}) 캐시 우회 강제 재검증을 resolveTool·setToolPath·getTools 전 경로에 명세 — 실측 safeExec의 force 분기(:77)·_clearBinCache(:116)·spawn .exe 재검증(:146)과 정합. 영구 캐시로 인한 무재검증 TOCTOU 창 폐쇄.
  2. M6-H-2 해소: PM 확정으로 tools.args 스키마 전면 삭제·실행 인자 [projectPath] 고정 — 사용자 인자 주입 경로 원천 부재(완화가 아닌 제거).
  3. 보통 4·낮음 3 전부 본문 반영, rev.2 신규 구조에 신규 위협 없음, 보존 통제(guard·CSP·contextIsolation·H-1/H-2/H-3·EC-1~6·오버레이 표면 0) 회귀 없음.
치명·높음이 0이므로 보안 게이트 통과(PASS). M6 구현 착수 가능.
구현 회차로 이관(게이트 비차단, 코드 리뷰 필수 확인 항목): 본 PASS는 설계 명세 기준이다. 구현 시 ① resolveTool이 매 open마다 force 호출 ② setToolPath 성공 시 캐시 무효화 실호출 ③ normalizeTools가 레거시 args 키 drop ④ uiStateStore.read 1MB 가드·_safeParse 경유 ⑤ second-instance argv 미파싱 ⑥ doFinalQuit 멱등·dispose 단일 경로를 실측 코드로 재확인해야 한다. 설계가 정확해도 구현이 이탈하면 H-1/H-2가 재발할 수 있어 구현 보안 검증에서 회귀 테스트로 못박을 것.

M6 설계 델타 rev.2는 1회차 게이트 조건을 정공법으로 해소했다 — 특히 args 전면 제거는 화이트리스트보다 강한 결정이며, force 재검증은 실코드 능력에 정확히 접지된다. 코드는 직접 수정하지 않으며, 위 구현 확인 항목을 담당 아키텍트/개발자에게 이관한다.
