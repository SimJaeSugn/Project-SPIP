종합 판정

CONDITIONAL PASS (조건부 통과)

설계는 추적성·기존 코드 정합·ADR 품질에서 견고하다. 감사가 인용한 코드 사실(전체 replaceChildren 재렌더, FOCUS_SEL/SCROLL_SEL 복원, store._dragging 가드, destroyCardSortable/initCardSortable, 메일 60초 폴링, order 해시 전용 정규식, preload onMenu, todoAdding 플래그)을 실제 코드에서 모두 확인했고 설계 주장과 일치한다. 다만 R-25 게이트의 보류 조건 누락, coalesce 단일 디바운스 타이머의 다중 트리거 정확성, onEnd flush 타이밍, 영문 입력 회귀 가능성 등 P1~P2 반영 항목이 존재한다. 이들을 반영하면 PASS. 미반영 시 R-25 기반 위에 얹는 모든 후속 요구가 미묘한 회귀를 안고 출발한다.

2. 반영 항목 (우선순위순)

우선순위: P1 치명(미반영 시 회귀·정확성 결함) · P2(설계 빈틈, 구현 전 확정 필요) · P3(개선·명확화).

[P1] R1 — RG.deferred() 게이트가 기존 보류 조건 busyMailSummary를 누락

무엇: §3.2의 deferred()는 composing|dragging|overlayOpen|editing 4조건만 검사. editing=store.todoAdding으로만 매핑.
위치: m9-design.html §3.2 (162~167행), 대조 코드 public/app.js:4581 maybeAutoRefreshMail.
문제: 현재 메일 라이브 갱신은 busyMailSummary || showSettings || showHelp || todoAdding일 때 보류한다. 설계의 deferred()로 통합하면 busyMailSummary(메일 요약 fetch 진행 중) 보류가 사라져, P6(R-31)에서 폴링이 in-flight 요청과 겹쳐 재렌더가 일어날 수 있다. "P1은 기존 동작 100% 보존"이라는 게이트(§11)를 정의상 위반.
권고: shouldDeferRender(flags)의 flags에 busy(또는 in-flight 일반화) 조건을 명시 추가하고, P1 이관 시 "기존 보류 조건 5종(overlay×3, dragging, busy, todoAdding) → deferred 입력으로의 1:1 매핑표"를 설계에 박을 것. 누락 조건이 없음을 단위테스트(각 플래그 true→defer true)로 못 박는다.

[P1] R2 — coalesce "정확히 1회"가 단일 전역 큐·단일 디바운스 타이머라 다중 트리거 정합성 미정의

무엇: §3.5 createCoalescer()는 pending boolean 하나, RG.coalesce.request()는 비보류 시 단일 debouncedRender() 호출.
위치: m9-design.html §3.5 (201~219행), §4 흐름도.
문제: ① 비보류 상태에서 메일 push·커밋 폴링·projectsUpdated가 디바운스 창(120ms) 안에 연달아 오면 debouncedRender()가 합쳐지는 것은 맞으나, 그 사이 보류 상태로 전환되면(예: 사용자가 조합 시작) 이미 예약된 debouncedRender 타이머가 보류를 무시하고 발화할 수 있다 — 설계에 debounce 타이머가 deferred()를 재검사한다는 명시가 없다. ② 보류 해제 지점이 여러 개(compositionend·onEnd·closeSettings)인데 둘 이상이 거의 동시에 풀리면 각각 flushIfPending()을 호출 → 첫 flush가 pending을 소비하므로 1회는 보장되나, "어느 해제 지점이 flush 책임을 지는가"가 미정의라 누락(아무도 안 부름) 위험도 존재한다(예: 드래그+모달 동시).
권고: (a) debouncedRender 실제 실행 시점에 deferred()를 재검사해 보류면 mark()로 되돌리는 로직을 계약에 명시. (b) flush 책임 모델을 단일화 — 예: 모든 보류 사유가 해제될 때(deferred()가 false로 전이) 단 한 경로에서 flush하도록 정의하고, 다중 사유 동시 보류/해제 케이스를 단위테스트로 명세.

[P2] R3 — R-26 "영문/숫자 입력 회귀 0"의 검증 경로가 설계에 비어 있음

무엇: R-26 수용기준에 "영문/숫자 입력 시 기존 debounce 검색 동작 유지(회귀 없음)"가 있으나, 설계 §3.2/§3.7은 조합 보류 경로만 기술.
위치: FeatureSpec R-26 수용기준 4번 · m9-design.html §3.2, §3.7(R-26 행).
문제: 검색 핸들러를 RG.coalesce.request() 경유로 바꾸면, 비조합 영문 입력의 디바운스 동작이 기존 debounce(render,120)(코드 3533)와 동일 체감인지가 게이트의 핵심인데 설계가 이를 다루지 않음. coalesce의 디바운스 창이 검색 전용 debounce를 대체하면서 미묘한 지연/이중 디바운스가 생길 수 있다.
권고: 검색 input 경로에서 기존 debounce와 coalesce 디바운스의 관계(대체/중첩 여부)를 §3.2에 명시하고, "영문 빠른 타이핑 시 렌더 횟수·최종 검색어"를 검증하는 경로를 수용기준 매핑에 추가.

[P2] R4 — onEnd의 _dragging=false 즉시 설정 + commitReorder 마이크로태스크 지연과 flush 타이밍 충돌

무엇: 기존 initCardSortable onEnd(코드 4914~4923)는 store._dragging=false를 동기로 설정한 뒤 commitReorder를 Promise.resolve().then(...)로 지연(render가 Sortable 인스턴스를 destroy하지 못하도록).
위치: m9-design.html §3.5/§4(해제 지점 "Sortable onEnd → flushIfPending") · 코드 public/app.js:4914-4923.
문제: 설계는 "onEnd에서 flushIfPending 호출"이라 하지만, onEnd 시점에 이미 _dragging=false이고 commitReorder는 마이크로태스크 뒤에 render를 유발한다. flushIfPending이 onEnd 동기 본문에서 호출되면 commitReorder render와 이중 렌더가 되거나, 반대로 commitReorder가 부르는 render가 RG.widget.destroyAll로 진행 중인 Sortable을 파괴하는 기존에 회피했던 문제가 R-32(홈 섹션 Sortable)로 재현될 수 있다. R-32 위젯 라이프사이클과 onEnd flush의 순서가 미정의.
권고: onEnd → (마이크로태스크) commitReorder → 영속 후 단일 render, 그리고 flushIfPending이 그 render에 흡수되도록 순서를 §3.4/§3.5에 명시. 홈 섹션 Sortable에도 기존 "마이크로태스크 지연으로 destroy 회피" 패턴이 적용됨을 R-32 설계에 박을 것.

[P2] R5 — patchRegion 옵트인이 위젯 라이프사이클·보존과 상호작용 미정의(반쪽 계약)

무엇: §3.3 patchRegion(containerEl, buildFn, keepSelectors)는 "keepSelectors 노드를 adoptNode로 이동 후 재배치"로만 서술, 본문 없음.
위치: m9-design.html §3.3 (179행), §10 차트 깜빡임 완화에서 patchRegion 재언급.
문제: patchRegion이 컨테이너만 부분 교체하면 그 컨테이너 안의 stateful 위젯(차트·Sortable)을 RG.widget.destroyAll/mountAll이 전역 render 기준으로 관리하던 전제가 깨진다 — 부분 갱신 영역의 위젯은 누가 destroy/mount하는가가 미정의. R-26 "상위 해법"·R-33 차트 깜빡임 완화가 모두 이 미정의 함수에 의존한다.
권고: 1차 범위에서 patchRegion을 명시적으로 제외(R-26은 capture/restore + 조합 보류만으로 충족 가능)하거나, 채택한다면 부분 갱신 영역 내 위젯의 destroy/mount 책임·보존 노드 adopt 순서를 계약으로 확정. 현재처럼 "필요 시 옵트인"으로 두면 R-33 깜빡임 리스크의 완화책이 공허해진다.

[P3] R6 — 5분 폴링(R-31, P6) × 드래그/차트(R-32/R-33, P7/P8) 통합 회귀가 순서상 늦게 검증됨

무엇: §11 구현 순서가 R-31(P6) 다음에 R-32(P7)·R-33(P8).
위치: m9-design.html §11.
문제: 명세가 명시한 핵심 위험("실시간이 드래그·차트를 악화")의 통합 검증이 P7/P8에 가서야 가능하다. P6에서 R-31을 켠 채 P7/P8을 진행하므로, 그 사이 단계는 폴링이 활성인 환경에서 작업하게 되어 디버깅 노이즈가 크다.
권고: P6 완료 게이트에 "R-32/R-33 미구현 상태에서도 5분 폴링이 기존 차트(div 막대)·기존 카드 드래그를 파손하지 않음"을 회귀 항목으로 추가하거나, R-31 폴링을 P8 직전(P7.5)으로 늦추는 대안 검토.

[P3] R7 — 동형 구현(ADR-M9-1) 2곳의 드리프트 방지 메커니즘이 "주의"에 그침

무엇: ADR-M9-1·§3.6은 순수 로직을 lib/common/renderGuard.js와 app.js RG에 동형 구현하고 "테스트가 계약"이라 함.
위치: m9-design.html §3.6, ADR-M9-1, 리스크 표.
문제: 두 구현이 같은 입력→출력을 보장하려면 동일 케이스를 양쪽에 적용해야 하나, app.js는 자동 테스트 비대상(F-3). 즉 lib 쪽만 테스트되고 app.js RG의 동형성은 사람 눈에 의존 → 드리프트가 실제로 발생할 수 있는 구조.
권고: 순수 판정 로직(shouldDeferRender/coalescer)을 app.js에 복제하지 말고, preload 노출(안 A)을 재검토하거나, 최소한 동형성을 보장하는 케이스 테이블(JSON fixture)을 단일 출처로 두고 양쪽이 참조하도록 권고. 드리프트 위험을 "낮음"으로만 적지 말 것.

[P3] R8 — Q5(도움말 버튼) 미해결 질문이 P4(R-27) 진행을 블로킹

무엇: §14 Q5(헤더 도움말 버튼 존치/흡수)가 미해결, 기본값은 "흡수+버튼 제거".
위치: m9-design.html §9.1·§14, FeatureSpec §7.3 Q5.
문제: R-27 공통 액션 영역 구성이 Q5에 따라 달라지는데, P4 착수 전 확정되지 않으면 헤더 골격 재작업 가능. Q-A(featureAdd 드래그 포함)·Q-B(masonry 방식)는 화이트리스트/CSS 토글로 흡수 가능해 영향이 작으나 Q5는 구조에 닿는다.
권고: P4 착수 게이트로 Q5를 사용자 확정시킬 것(아래 "확인 필요" 참조). 기본값 진행도 허용되나 명시적 결정을 권고.
