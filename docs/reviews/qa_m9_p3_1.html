2. F5 / 단축키 충돌 · PASS

충돌 시나리오결과 / 근거
입력창 포커스 중 텍스트 삽입 충돌충돌 없음. matchShortcut(app.js:579-587)은 글자 키 단독('o','r' 수식키 없음)을 null 반환(app.js:584-585의 ctrl && 가드). 따라서 입력 중 일반 타이핑은 절대 가로채지 않음. 테스트 shortcuts.test.js:56-59가 이를 고정.
F5 + 수식키 조합(Ctrl+F5 등)안전. key==='F5' && !ctrl && !shiftKey && !altKey(app.js:583)로 수식키 동반 시 비매칭. 테스트 shortcuts.test.js:61-64.
입력창에서 Ctrl+O / Ctrl+R관찰(보통). 전역 keydown(app.js:5491)이 입력 포커스 여부를 검사하지 않아, 텍스트 입력 중에도 폴더추가/재스캔이 발화하고 preventDefault가 걸린다. 텍스트 삽입을 깨지는 않음(Ctrl+O/R은 글자 입력이 아님)이므로 데이터 손상·기능 손실은 없다. 다만 메일/제외/할일 입력 중 의도치 않은 동작 트리거 가능 — 아래 버그 B-1 참조. 수용 기준 위반은 아님(설계는 입력창 가드를 요구하지 않음).
전역 ESC 핸들러와 중복 디스패치분리됨. Esc는 matchShortcut이 null 반환(app.js:586), 별도 ESC 핸들러(app.js:5480-5487)가 단독 처리. SHORTCUTS 안내표엔 'Esc' 표기만(디스패치 비대상). 이중 처리 없음.
두 keydown 리스너 상호작용안전. ESC 리스너는 e.key!=='Escape'면 early-return, 단축키 리스너는 matchShortcut===null이면 early-return. 키 공간이 겹치지 않아 부작용 없음.

3. SHORTCUTS 단일 출처 정합 · PASS

디스패치 ⊆ 안내표. keydown 디스패치가 실제 실행하는 action 집합 {pickFolders, rescan, refresh}(app.js:5495-5500)는 모두 SHORTCUTS 상수(app.js:566-571)에 존재한다. 안내표 렌더 renderShortcutSettings()(app.js:2191-2198)는 동일 SHORTCUTS를 그대로 순회(SHORTCUTS.forEach) → 표시 집합 = 상수 = 디스패치의 상위집합. 불일치 경로 없음. 'Esc'는 안내표엔 있으나 matchShortcut 비대상(전역 ESC 소관)으로, 표 ⊇ 디스패치 관계가 유지됨. 테스트 shortcuts.test.js:75-82가 고정.
L-1(textContent) 준수. 안내표는 helpRow(sc.keys, sc.label)로 렌더되며 el(...,{text:...}) 경로(textContent)만 사용. 정보 섹션(renderInfoSettings)도 text:·helpRow만 사용, innerHTML 미사용 확인. XSS 표면 추가 없음.
