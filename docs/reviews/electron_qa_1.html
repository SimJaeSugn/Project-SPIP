2. preload ↔ renderer 계약 일치 (경계면 교차) — FAIL

renderer(public/app.js)가 호출하는 bridge 메서드와 preload(electron/preload.js)가 window.spip에 노출하는 함수명을 자동 추출·대조했다.

renderer 사용(app.js)preload 노출일치
addRoots · getConfig · getProjects · getScanStatus · getStats · onScanProgress · pickFolders · removeRoot · rescan — 9개 모두 양쪽 일치
open(id) (app.js:1917 ipc('open', id))없음 — preload는 openInVsCode로 노출불일치
— (사용 안 함)openInVsCode, getHealth미사용

설계 계약(electron-migration.html §4.1 표 269행 · §4.3 preload 의사코드 351행)은 채널 spip:openInVsCode를 renderer 표면 spip.open(id)로 노출하도록 못박는다:

// 계약 §4.3 preload 의사코드
open: (id) => ipcRenderer.invoke('spip:openInVsCode', { id: String(id) }),

renderer는 계약대로 open을 호출하지만, 구현된 preload는 함수명을 openInVsCode로 잘못 지었다. → BUG-1(블로커).

PASS 인자/반환 shape는 일치 검증됨: rescan(o) opts는 {withSize,allDrives} Boolean 강제 양쪽 동일; addRoots(paths)는 배열 map(String) 후 {paths} 래핑(handler가 args.paths 추출) 일치; onScanProgress(cb)는 구독 해제 함수를 반환하고 renderer unsubscribeScan()이 그것을 호출·null화 — 누수 없음 일치.

7. 발견 버그

BUG-1 · preload가 open 대신 openInVsCode로 노출 → R-12 전면 불능  블로커

무엇이대시보드에서 프로젝트 "열기"를 눌러도 VS Code가 절대 열리지 않고, 항상 "Electron 환경이 아닙니다(window.spip 없음)" 류 오류 토스트가 뜬다(실제로는 메서드명 부재).
어디서경계면: electron/preload.js:22 ↔ public/app.js:1917(openProject) ↔ 어댑터 ipc() app.js:643-651
기대 vs 실제 기대(계약 §4.1/§4.3): preload가 open: (id) => ipcRenderer.invoke('spip:openInVsCode', {id:String(id)}) 노출.
실제: preload가 openInVsCode: (id) => ...로 노출. renderer는 spip.open을 찾지만 window.spip.open === undefined. 어댑터 ipc('open',id)가 typeof spip['open'] !== 'function' 분기로 bridgeMissing() 반환 → 항상 실패.
재현실 GUI: 프로젝트 카드 "열기" 클릭 → VS Code 미실행 + 오류 토스트.
헤드리스 실증: offscreen 창에서 window.spip.open('x') → {"__missing":true}(함수 부재). Object.keys(window.spip)에 open 없음/openInVsCode 있음.
심각도블로커 — 핵심 요구 R-12가 100% 동작 불능. 핸들러는 정상이라 1줄 수정으로 해결.
수정안backend-dev(devops): preload의 키를 openInVsCode → open으로 변경(채널명 'spip:openInVsCode'는 유지). 또는 별칭 open 추가. 미사용 getHealth는 무해(유지 가능).

NOTE-1 · 테스트가 경계면(메서드명)을 검증하지 않음  낮음(품질 격차)

288개 테스트는 순수 함수만 다룬다. preload 노출 키 ↔ renderer 사용 키의 일치를 자동 검증하는 테스트가 없어 BUG-1이 그린 상태로 통과했다. 재발 방지를 위해 "preload 노출 키 ⊇ renderer 사용 키" 정적 대조 테스트 추가를 권장(QA가 본 검증에서 사용한 스크립트로 충분). — 코드품질 영역이라 code-reviewer와 공유.

NOTE-2 · preload 노출 표면에 미사용 항목  낮음

getHealth·(BUG-1 수정 후) openInVsCode 잔존 가능성. 계약상 getHealth는 가용 채널로 정의되어 있어 노출 자체는 위반 아님. BUG-1 수정 시 openInVsCode 키는 제거(중복 방지). 동작 영향 없음.

8. 사용자 PC 확인 필요 목록 (헤드리스 한계 — 추측 PASS 금지)

아래는 실 GUI·OS·외부 바이너리 의존이라 헤드리스에서 단정할 수 없다. 사용자 PC(Windows 11)에서 npm start 후 수동 확인 필요.

  1. 실제 VS Code 열기(R-12): BUG-1 수정 후 카드 "열기" → 실제 code <path> 실행되는지. code CLI PATH 유무에 따라 CODE_CLI_NOT_FOUND 메시지 분기도 확인.
  2. 네이티브 폴더 대화상자(R-04, pickFolders): "폴더 선택…" → OS 대화상자 표시·다중선택·취소(CANCELLED)·채택 후 루트 목록 갱신.
  3. 실 스캔 진행 push(R-15): "스캔 시작/재스캔" → scanning 뷰에 dirs/found 카운트·currentPath 실시간 갱신 → done 시 대시보드 자동 전환. push 이벤트가 끊김 없이 도착하는지.
  4. 창 생명주기·메뉴(main.js wiring): 단일 인스턴스 락(둘째 실행이 기존 창 포커스), 메뉴 "폴더 선택/재스캔" 항목, ready-to-show 표시.
  5. 스캔 중 종료 다이얼로그(Q4): 스캔 진행 중 창 닫기 → "스캔이 진행 중입니다. 종료하시겠습니까?" 확인 다이얼로그.
  6. CSP/하드닝 실효(EM-M-1/§6.2): 외부 네비게이션·새 창·webview가 실제로 차단되는지, 메타+헤더 CSP 이중주입이 콘솔 위반 없이 동작하는지.
  7. 패키징(electron-builder): npm run build:portable 산출물에서 app:// 자산·preload·RUN_AS_NODE 차단이 packaged 모드에서 동작하는지(app.isPackaged 경로).

9. 종합 판정 · 통합 리스크
