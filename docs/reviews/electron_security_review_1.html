2. 게이트 항목 EC-1~EC-6 — 구현 실재 + 동작 실측

EC-1 — IPC 스키마 거부 실증 PASS

실재: register.js guard()가 전 채널(10개)을 ① senderFrame 검증 → ② try/catch(INTERNAL)로 감싼다. actions.js sanitizeOpenId(id string·0<len≤512·plain object)·sanitizeRescanOpts(plain object만·키 화이트리스트 {withSize,allDrives}·Boolean() 강제), folders.js addRootsResolve(배열 검사·항목 len≤4096·≤64 절단)·removeRootResolve(len 검사).

실측(적대적 프로브): open id에 123/null/len513/""/["a"]/{toString}/{evil}/[]/42 전부 REJECT(유효 문자열만 통과). rescan opts에 prototype 오염키·evilKey·배열·문자열·null 주입 → 화이트리스트 2키만 산출, 그 외 무시. addRoots 비배열 → INVALID_PATH, len 5000 항목 → NOT_FOUND. npm test ipc-* 48건 + 전체 288건 통과.

읽기 채널(getProjects/getStats/getHealth/getConfig/getScanStatus)은 인자를 받지 않고 무시. 쓰기/액션 채널은 타입·길이·개수·키 화이트리스트로 비정상 입력을 main에서 거부. 거부 동작 실측 완료.

EC-2 — 경로 통제 보존 실증 PASS

실재: openInVsCode는 id로 store.getById 역참조 → pathGuard.canonicalize(null→PATH_GONE) → isAllowed(불일치→PATH_NOT_ALLOWED) → resolveBin('code') 절대경로 → safeExec(codeBin,[real],{shell:false,detached:true,maxInflight:2}). 인자로 경로를 받지 않고 id로만 역참조하며 exec에는 검증된 실경로(real)를 전달. addRoots/pickFolders는 canonicalizeDir(realpathSync.native+isDirectory)·isSystemDir·dedup 통과분만 normalizeScanRoots로 0600 영속. safeExec는 absBin 비절대→reject, Windows .exe만 허용(.bat/.cmd 거부), shell:false 고정.

실측: path-like id ../../etc/passwd → store 미매칭 시 ID_NOT_FOUND(exec 미호출), 매칭+화이트리스트 외 → PATH_NOT_ALLOWED, exec에는 항상 canonical real만. addRoots에 C:\Windows/C:\///\\?\C:\Windows → SYSTEM_DIR, /etc·미존재 → NOT_FOUND, 비배열 → INVALID_PATH. removeRoot에 무관 임의경로 → NOT_FOUND(타 루트 제거 불가), 정확 일치만 제거 확인.

EC-3 — CSP 코드 PASS · 실차단 사용자 확인

실재: security.js CSP_POLICY는 default-src 'none'; script-src 'self'; style-src 'self'; img-src 'self' data:; font-src 'self'; connect-src 'none'; object-src 'none'; base-uri 'none'; form-action 'none'; frame-ancestors 'none' — 설계 대비 약화 없음(전 디렉티브 보존·강화). applyCspHeaders가 session.webRequest.onHeadersReceived로 헤더 주입(이중주입). index.html meta는 정확히 동일 문자열(정합 사본). app:// 프로토콜 핸들러는 decodeURIComponent 후 path.normalize+루트 접두 검사로 traversal 차단.

실측: app:// traversal 프로브 — /../../etc/passwd·/..%2f..%2fpackage.json·/%2e%2e/..·/sub/../../../package.json·/..\..\package.json 전부 403 FORBIDDEN(디코드를 normalize 전에 수행해 인코딩 우회 차단), /index.html·/styles.css·/만 ALLOW. CSP 문자열 동일성·전 디렉티브 보존 단위테스트 통과.

사용자 PC 확인 필요: CSP가 실제 Chromium 렌더러에서 인라인/외부/data 스크립트를 실행 차단하는지(오염 픽스처로 XSS 미발화)는 GUI 런타임 검증이므로 1회 스모크 권장. 코드/정책 정합은 실증됨.

EC-4 — Renderer 격리 코드 PASS · 런타임 사용자 확인

실재: main.js webPreferences contextIsolation:true · nodeIntegration:false · sandbox:true · webSecurity:true. preload.js는 window.spip에 9개 함수만 노출 — ipcRenderer 원본·범용 invoke(channel,…) 미노출, 채널명 하드코딩, 이벤트 구독은 콜백만 받고 해제 함수 반환. hardenWebContents가 will-navigate(우리 origin 외 차단)·setWindowOpenHandler:deny·will-attach-webview:preventDefault. shell.openExternal 미사용(코드 전체 검색 0). 원격 URL 미로드(loadURL('app://index.html')만). app.js는 토큰·fetch 전부 제거, IPC 어댑터만 사용.

실측: preload allowlist에 generic invoke 부재 확인. app.js 그렙 — innerHTML 데이터결합 0, 모든 서버 유래 문자열 textContent. RC-2 평가: 저장형 XSS가 renderer에서 성립해도 노출 표면은 9개 allowlist 함수뿐이고 모든 인자는 main에서 재검증되므로 Node 권한 격상·임의경로 접근 불가(sandbox+nodeIntegration:false로 폭발반경 봉쇄).

사용자 PC 확인 필요: sandbox:true 상태에서 전 기능이 동작하는지(preload가 우리 모듈 require 안 함)·will-navigate/팝업 차단의 실 동작은 GUI 스모크 항목.

EC-5 — 발신자 검증 실증 PASS

실재: 모든 ipcMain.handle이 guard()를 거치며 isTrustedSender(event)가 false면 {ok:false,code:'FORBIDDEN'}로 무시(핸들러 미실행). 실측: http://evil.com·https://app.evil.com(접두 혼동)·file:///etc/passwd(allowFileUrl off)·app-evil://(스킴 혼동)·null·빈 객체 전부 거부, app://index.html만 신뢰. register 비신뢰 발신자 FORBIDDEN 테스트 통과.

단, 낮음 1건(EI-L-1): origin 분기 origin.startsWith('app:')는 app://../../etc를 신뢰로 오판(아래 §3).

EC-6 — 패키징 무결성 코드 PASS · 빌드 산출물 사용자 확인

실재: main.js 최상단 if(process.env.ELECTRON_RUN_AS_NODE) process.exit(0) — 앱 로직 전에 무력화. build/afterPack.js가 fuses 5종을 굽는다(실측 grep 전부 확인): RunAsNode:false · EnableNodeCliInspectArguments:false · OnlyLoadAppFromAsar:true · EnableEmbeddedAsarIntegrityValidation:true · EnableNodeOptionsEnvironmentVariable:false. electron-builder.yml asar:true·test/docs 제외·쓰기 데이터 %APPDATA%\spip 분리·설치 디렉터리 읽기전용·미서명 위험 수용 명시(주석 "PM 확정 Q5", SmartScreen 안내). @electron/fuses 미설치 시 경고만(빌드 비차단).

사용자 PC 확인 필요: fuses가 실제 빌드 산출물 바이너리에 구워졌는지·RUN_AS_NODE 기동 차단은 패키징 후 1회 실증 항목(헤드리스 빌드 불가).

6. 사용자 PC 확인 필요 (헤드리스 한계 · 추측 PASS 금지)

아래는 GUI/패키징 런타임이라 헤드리스에서 실증 불가 — 코드·정책 정합은 PASS이나 실 동작은 사용자 PC 1회 스모크로 확정해야 한다. 게이트 비차단이나 운영 전 필수.
  • CSP 실차단(EC-3): 오염 메타/외부·인라인·data 스크립트 픽스처로 실제 Chromium에서 XSS 미발화 확인.
  • renderer 격리·탐색 차단(EC-4): sandbox:true 전 기능 동작 + will-navigate/setWindowOpenHandler/webview 차단 실 동작.
  • 패키징 fuses(EC-6): electron-builder 빌드 후 산출 바이너리에 fuses 5종 실제 적용 + ELECTRON_RUN_AS_NODE=1 기동 차단(즉시 종료) 확인.
  • senderFrame 실 동작: 실제 app:// 프레임의 senderFrame.url/origin 값으로 신뢰 분기 정상 + 비신뢰 프레임 FORBIDDEN.
위 항목은 위험 수용을 요하지 않는 실증 미완 항목이며 차단 사유 아님. 미서명 배포(Q5)·RC-1/RC-2는 설계 단계에서 PM 위험 수용 확정됨(추가 결정 불요).
