Project-SPIP — M4 설계 델타 보안 검증 보고서

MVP(M1~M3) 보안 베이스라인(아키텍처 2회차 재검증: 8중 통제·TB-A/TB-B·치명/높음 0)을 기준으로, M4 델타(R-16 재스캔·R-15 진행표시·R-09 size·R-05 전체드라이브·R-03 depthLimit·N-07 접근성)가 새로 추가하는 공격 표면을 적대적 관점에서 위협 모델링했다. 핵심 질문: 기존 8중 통제가 신규 상태변경·신규 GET·디스크 전체 도달 범위 확대에 실제로 승계되는가, 그리고 신규 표면이 베이스라인 위험 수용(RC-1 타사용자 비격리·RC-2 부분신뢰)을 확대하지 않는가. 검증 대상은 설계 청사진이다(구현 코드 없음 — npm audit/시크릿 스캔 비해당, 외부 의존성 0).

0. 종합 판정

판정: 조건부 PASS (치명 0 · 높음 2 · 중간 3 · 낮음 2 · PASS 6)

M4 델타의 보안 골격은 견고하다 — M-1(Host allowlist+Origin+세션토큰+CT)이 신규 상태변경 /api/rescan에 그대로 선통과로 승계되고, all-drives는 config opt-in 게이트 + 5중 가드로 기본 차단이며, size·depthLimit은 H-3/M-3의 상한 가드를 확장한다. 경로를 본문으로 받지 않는(config-only) 설계는 H-1을 유지한다. 그러나 두 가지 높음 위협이 설계 반영을 요구한다: ① 신규 무인증 GET /api/scan-status가 절대 경로(currentPath)·scanId를 노출하여 베이스라인의 RC-1(동일머신 타사용자 비격리) 위험 수용 범위를 정보노출 방향으로 확대하고 L-3(내부경로 응답 비노출) 정책과 상충한다(M4-H-1). ② all-drives 시스템 디렉터리 prefix 제외가 H-1 canonicalize 규칙과 동일 시점·동일 정규화로 적용된다는 보증이 설계에 없어, 심링크/접합점·Windows 8.3 단축명·대소문자·UNC로 C:\Windows 등 제외 대상을 우회 순회할 잔여(M4-H-2)가 있다. 치명/높음 잔여가 있으므로 보안 게이트는 통과시키지 않는다 — 아래 §4 H급 2건의 설계 반영 후 재검증(2회차)을 조건으로 한다.

1. M4 신규 공격 표면 · 신뢰경계 (변화 지점)

M4는 새 신뢰 경계를 만들지 않는다(설계 주장 — 대체로 타당). 그러나 기존 경계에서 데이터가 흐르는 방향·범위가 바뀐다: (a) TB-A에 신규 상태변경 1건(rescan)·신규 읽기 1건(scan-status)이 추가되고, (b) TB-B(스캔 대상=부분신뢰)의 도달 범위가 지정 루트 → 디스크 전체로 확대되며(all-drives), (c) 부분신뢰 입력에서 신뢰 영역으로 흐르는 데이터에 currentPath(진행 경로)가 추가되어 무인증 GET으로 TB-A 너머로 다시 흘러나간다. 아래 SVG는 베이스라인 그림 위에 M4 신규 흐름(굵은 주황)만 덧그린 것이다.

1.1 보호 자산 (M4 관점)

  • 호스트 파일시스템 구조(절대 경로) — all-drives 시 디스크 전체 디렉터리 레이아웃·사용자명·프로젝트 위치. currentPath·프로젝트 path로 노출.
  • 스캔 자원(CPU·I/O·시간) — N-01 30초 예산. 재스캔 트리거·거대 트리·size 측정·폴링이 소모.
  • scanId · 진행 state — 추측/노출 시 상태 추론·잠재적 상관관계.
  • 캐시 무결성 — 동시 재스캔이 캐시 동시 쓰기를 일으키면 손상.

3. 높음 위협 — 상세 (게이트 차단)

M4-H-1. 무인증 GET /api/scan-status의 절대경로·scanId 정보 노출 높음

위치 §3 R-15, §8.3 ScanProgress (apiHandlers.js GET /api/scan-status)
STRIDE 정보노출(I)
관찰 ScanProgress는 currentPath(순회 중 절대 경로), scanId, counts, startedAt을 담는다. 이 라우트는 GET이므로 M-1(상태변경 POST 전용)이 적용되지 않는다 — 설계도 §3.1에서 "무상태 GET 1건"으로 명시. 즉 토큰 없이 누구나 폴링 가능하다.
악용 시나리오 베이스라인은 RC-1(동일머신 타 로컬 사용자/프로세스 비격리)을 위험 수용했고 127.0.0.1은 사용자 격리를 하지 않는다(검증2 §5). all-drives 재스캔이 도는 동안 같은 머신의 다른 로컬 프로세스가 /api/scan-status를 1초 폴링하면, 인증 없이 디스크 전체의 절대 경로 트리(사용자명·프로젝트 위치·디렉터리 구조)를 점진 수집할 수 있다. counts로 프로젝트 규모도 추론. 베이스라인 L-3은 "내부 절대경로는 응답 비노출, 콘솔 로그 한정"을 못박았는데, 이 신규 GET은 의도적으로 절대경로를 응답하므로 L-3 정책과 정면 상충한다. MVP에는 없던 신규 정보노출 채널이다.
권고 조치
  1. 정보 최소화(우선): 응답의 currentPath를 전체 절대경로 대신 마지막 1~2 세그먼트(basename) 또는 드라이브 루트 기준 상대 경로로 축약. 진행 표시 UX에는 basename으로 충분하다. startedAt·scanId 등 상관 가능한 식별자는 최소화.
  2. 대안/병행 — 동일출처 강제: GET에도 Origin/Referer 또는 세션토큰 약식 검증을 적용해 같은 머신의 비브라우저 폴러를 배제(M-1을 읽기에도 부분 승계). 최소한 currentPath를 담는 응답은 토큰 검증 게이트 뒤에 둔다.
  3. 설계에 L-3 예외 사유를 명문화하거나 L-3 준수로 정렬할 것(현재 상충 미해소).
  4. 위 1)을 채택하면 §8.3 계약과 R-15 그림 1의 currentPath 정의를 "축약 표시명"으로 갱신.

M4-H-2. all-drives 시스템 디렉터리 prefix 제외의 정규화·시점 불일치로 인한 우회 순회 높음

위치 §5.2 (시스템 디렉터리 제외 — "경로 prefix 기반"), excludeRules.js · driveEnum.js
STRIDE 정보노출(I) · 서비스거부(D)
관찰 §5.2는 시스템 디렉터리 제외를 "경로 prefix 기반"(C:\Windows, C:\Program Files*, /proc 등)으로 추가한다고만 명세한다. 그러나 베이스라인 H-1은 보안 결정에 반드시 canonicalize()(realpath·심링크/접합점 해소·대소문자 폴드·NFC·UNC/드라이브상대/\\?\ 정규화)를 등록·검증 동일 시점·동일 규칙으로 적용하라고 못박았다. 제외가 raw 문자열 prefix 비교로 수행되면 canonicalize와 어긋난다.
악용 시나리오 all-drives가 활성(opt-in)된 상태에서: (1) Windows 8.3 단축명 — C:\PROGRA~1은 C:\Program Files의 실경로이나 prefix 문자열 비교를 통과해 제외를 우회. (2) 접합점/심링크 — 다른 위치의 junction이 C:\Windows를 가리키면 raw prefix는 다르나 realpath는 시스템 폴더. (3) 대소문자/UNC — c:\windows, \\?\C:\Windows, \\127.0.0.1\C$\Windows 변형. 이로써 시스템 디렉터리가 순회되어 (a) 시스템 경로/구조 정보노출(I) 확대, (b) 거대/특수 시스템 트리 순회로 N-01 예산 초과·자원고갈(D)이 발생한다. all-drives가 도달 범위를 디스크 전체로 넓혔기 때문에 영향이 베이스라인보다 크다.
권고 조치
  1. 시스템 제외 판정을 canonicalize() 실경로 기준으로 수행하도록 §5.2에 명시 — raw prefix 금지. 제외 비교는 walker가 디렉터리를 canonicalize한 직후·진입 전에 적용(가지치기 시점 통일).
  2. prefix가 아닌 경로 세그먼트(canonical 컴포넌트) 단위 비교로 부분 일치 우회 차단(예: C:\WindowsApps가 C:\Windows 제외에 잘못 걸리거나 그 반대를 방지).
  3. 8.3 단축명·UNC·확장경로(\\?\)는 canonicalize 단계에서 정규화 또는 거부(H-1 규칙 재사용)됨을 §5.2에 명문화.
  4. 심링크/접합점은 walker M-3(심링크 미추적)로 진입 자체가 차단되나, 제외 판정이 미추적 가지치기보다 앞단임을 보장(H-3 "격리/처리보다 가드 앞단" 원칙 승계).

4. 중간 · 낮음 위협 — 상세

M4-M-1. rescan withSize body 옵션의 config 게이트 비대칭 중간

위치 §2.2 옵션 본문, §8.1
STRIDE 서비스거부(D) · 변조(T)
관찰 allDrives:true는 config.allowAllDrives===true일 때만 수락(강등+note) — 안전. 그러나 withSize:true는 동일한 config 게이트 명세가 없어, 요청 본문만으로 비싼 size 측정(node_modules stat 수만 회)을 강제할 수 있다. M-1을 통과한 브라우저(또는 RC-1상 동일머신 폴러가 토큰 탈취 시)가 size 측정을 반복 유발.
완화 사실 전역 단일 락(M4-P-2)이 동시 1개로 제한하고, size 자체에 budgetMs/maxDepth/maxEntries 상한(M4-P-5)이 있어 폭주는 제한적. 그래서 치명/높음이 아닌 중간.
권고 조치 withSize도 서버 config 상한(예: config.size.allowOnDemand)에 종속시키거나, 최소한 size 활성 여부의 최종 결정권을 서버 config가 갖도록 명세(allDrives와 대칭). 미해결 질문 Q1과 함께 PM이 "본문 옵션을 config 허용 범위 내로만"이라는 원칙을 withSize에도 명시 적용.

M4-M-2. ABS_MAX_DEPTH=64 상향 + all-drives 결합 자원 예산 중간

위치 §6 (ABS_MAX_DEPTH=64), §5.2 all-drives
STRIDE 서비스거부(D)
관찰 MVP의 안전상한은 24였다(M-3에서 "항상 강제"). M4는 이를 기본값으로 강등하고 절대 상한을 64로 신설한다. 즉 사용자가 config로 깊이를 24→64까지 늘릴 수 있고, all-drives 디스크 전체와 결합하면 순회량이 MVP 대비 크게 증가할 수 있다. 64는 유한하므로 무한루프는 아니나, "MVP 안전상한을 약화하지 않는다"는 R-03 원칙과 미묘하게 어긋난다(절대상한이 24→64로 완화됨).
권고 조치 (1) all-drives 모드 기본 깊이를 낮게(설계 §6의 12 권고) 강제하고, all-drives에서는 ABS 상한도 더 낮게(예: 24) 별도 적용 검토. (2) ABS_MAX_DEPTH=64 + all-drives + maxEntries 상한 조합이 N-01/N-05를 깨지 않음을 구현 게이트(V-2 연장)에서 벤치로 실증. (3) entry/시간 상한이 깊이와 독립적으로 전체 순회에도 적용되는지(현재 size에만 명시) walker 레벨 전역 상한 추가 검토.

M4-M-3. 백그라운드 스캔 실패 시 락 영구 점유(데드락) 회복 경로 미명세 중간

위치 §2.1 동시성 모델, §2.5 ScanController
STRIDE 서비스거부(D)
관찰 acquire()가 running=true로 락을 잡고 백그라운드 start()가 비동기 실행된다. 설계는 완료 시 phase=done, 예외 시 phase=error를 명시하나, 예외/크래시 시 running 락이 반드시 해제된다는 보증(finally·watchdog)이 시그니처에 없다. 백그라운드 스캔이 처리되지 않은 예외로 죽으면 running이 영구 true가 되어 이후 모든 재스캔이 409로 영구 거부(가용성 데드락)될 수 있다.
악용/유발 적대적 트리(거대/특수 파일)나 권한 오류가 격리되지 않고 상위로 전파되면 트리거. all-drives에서 발생 확률↑.
권고 조치 (1) start()는 try/finally로 항상 락 해제(성공·실패·예외 무관)함을 계약에 명시. (2) watchdog 타임아웃(예: 절대 상한 시간 초과 시 강제 해제+phase=error)을 ScanController에 추가. (3) phase=error도 락 해제 후 idle로 복귀함을 §2.3 완료/실패 행에 명문화. (N-05 "재스캔 중 0 다운"과 정합 필수.)

M4-L-1. scanId 엔트로피·용도 미명세 낮음

위치 §2.3, §8.3 (scanId:"<hex>")
STRIDE 정보노출(I)
관찰 scanId가 crypto 기반 충분 엔트로피인지, 인가에 쓰이는지(쓰인다면 추측 위험), 단순 표시용인지 미명세. 현재 설계상 인가에 쓰이지 않고 단일 락이라 영향은 낮음.
권고 조치 scanId는 (1) crypto.randomBytes 기반 충분 엔트로피, (2) 인가 토큰이 아님(표시·상관용)을 명시. 인가에 쓸 계획이면 추측 불가성 보증 추가.

M4-L-2. 진행 응답 전 필드의 textContent 렌더 강제 범위 낮음

위치 §3.2, §8.3, §7.1 aria-live
STRIDE 변조(T) → 저장형 XSS(L-1)
관찰 설계는 currentPath에 대해 textContent 렌더를 명시하나, note(all-drives 강등 안내 등)는 서버 생성이라 비교적 안전해도, 진행 패널 전 필드를 일괄 textContent로 렌더한다는 포괄 규칙이 명문화되어 있지 않다. aria-live 영역에 부분신뢰 문자열을 갱신할 때 누락 위험.
권고 조치 §7.1/§3.2에 "ScanProgress의 모든 문자열 필드(currentPath·note 포함)는 textContent로만 렌더, aria-live 갱신도 textContent"를 L-1 포괄 규칙으로 명시. (currentPath를 basename 축약하면 M4-H-1과 동시 완화.)
