M12-S-01 whoami 고정 절대경로 실행 — 충족

위치: lib/common/elevationGuard.js (resolveWhoamiBin · execWhoamiGroups), lib/common/safeExec.js (resolveBin 절대경로 분기)

  • PATH 검색 0: resolveWhoamiBin 은 %SystemRoot%(부재 시 C:\Windows 폴백) + System32\whoami.exe 로 절대경로를 조립한 뒤 resolveBin(abs) 절대경로 분기에 넘긴다. 이 분기는 path.isAbsolute 확인 후 .exe 확장자 화이트리스트·실존 파일 검사만 하며 PATH 순회를 일절 하지 않는다(PATH 하이재킹 0).
  • 인터폴레이션 0: execWhoamiGroups 는 childProcess.execFileSync(bin, ['/groups'], {shell:false, windowsHide:true, timeout:5000, maxBuffer:1MB}) 로 인자를 배열 고정 실행. 셸 해석 경로 없음(H-2 준수). 테스트가 shell:false·인자 ['/groups']·고정 bin 을 모킹으로 검증.
  • %SystemRoot% 변조/부재 안전: SystemRoot 가 비거나 변조돼 잘못된 경로가 되면 resolveBin 이 실존 파일 검사에서 null 반환 → detectElevation 이 whoami-unresolved 로 비상승 폴백. 공격자가 SystemRoot 를 위장 경로로 바꿔도 임의 바이너리가 실행될 여지는 없다(.exe·실존·절대경로 3중). 단, 신뢰 환경(main 프로세스 env)이라 이 변수 자체의 변조 위협은 로컬 단일사용자 모델 밖.

M12-S-02 write 보류 완전성 (3경로) — 충족

위치: serializer.writeSnapshot · uiStateStore.write · config.persistConfigKeys · actions.rescan

  • 3경로 전부 디스크 미진입: 세 함수 모두 정규화(JSON 직렬화는 메모리상) 직후, ensureDirFor(디렉터리 생성)·임시파일 openSync('wx')·rename 이전에 isElevated() 검사로 조기 반환한다. 따라서 상승 세션에서는 디렉터리·임시(.tmp)파일·최종파일 어느 것도 생성되지 않는다(테스트 fs.existsSync === false 로 확인).
  • 설정(메일 비밀번호) 포함: 가장 민감한 persistConfigKeys 가 보류 대상에 포함돼, 메일 계정 pass 가 관리자 프로필에 새 0600 파일로 떨어지는 정보노출이 차단된다(전용 테스트 존재).
  • rescan 이중 안전: actions.rescan 이 acquire()/start() 이전에 ELEVATED 조기거부 → 스캔 자체가 시작되지 않아 스냅샷 write 트리거가 원천 차단(write no-op 와 이중 방어).
  • 기존 데이터 무손상(읽기 허용): loadConfig·uiStateStore.read 는 read-only 이며 가드 대상이 아니다. 보류는 write 만 막아 기존 사용자 데이터를 읽어 정상 표시할 수 있고, 덮어쓰기/삭제가 없어 손상 위험 0. writeDefaultConfig(raw fs write·미가드)는 CLI(scan.js) 전용으로 Electron 상승 세션 경로에서 호출되지 않음(확인 완료).

M12-S-03 감지 보수성 (false positive 미차단) — 충족

위치: elevationGuard.detectElevation · appdataMismatch

  • SID 단독 차단: APPDATA 불일치(로밍 프로필·APPDATA 리다이렉트 등 false positive 원인)는 "whoami 검사 트리거"로만 쓰이고, 유일한 차단 근거는 whoami /groups 의 High/SYSTEM ML SID(S-1-16-12288/S-1-16-16384)다. 불일치만으로는 절대 차단하지 않는다(전용 테스트로 Medium SID + 불일치 → 비상승 확인).
  • 판정 실패 = 비상승 폴백: win32 외, 트리거 없음, whoami 미해석, 실행/파싱 실패의 모든 경로가 elevated:false 로 폴백. 오탐으로 정상 사용자의 저장을 막는 쪽을 더 큰 피해로 보고 가용성을 우선하는 보수적 설계가 일관되게 구현됨.

M12-S-04 배너 L-1 · 정보노출 · preload 단방향 — 충족

위치: public/app.js(showElevationBanner) · public/styles.css · electron/preload.js(onElevationWarning) · electron/main.js(pushElevationWarning)

  • 고정 문구·L-1: 배너는 상수 ELEVATION_BANNER_TEXT 만 el(..,{text}) = textContent 로 렌더한다. 동적값 결합 0, innerHTML 미사용. 콜백 function(){ showElevationBanner(); } 가 payload 를 아예 참조하지 않아, payload 가 오염돼도 화면에 들어갈 경로가 없다.
  • 정보노출 0: push payload 는 { elevated:true } 고정 신호. main 이 실제 APPDATA 경로·프로필명·whoami 출력·SID 를 보내지 않는다. 진단 reason 토큰(high-integrity 등)은 로그(main 로컬)에만 남고 렌더러로 전달되지 않으며 민감값을 담지 않는다.
  • 단방향·ipcRenderer 비노출: preload 는 콜백 중계 + unsubscribe 만 노출하고 ipcRenderer 원본을 노출하지 않는다. main 의 webContents.send ↔ preload 의 ipcRenderer.on 단방향이며, 이 채널에 대한 ipcMain.handle/on 핸들러가 없어 렌더러→main 인바운드 표면이 0.

M12-S-05 표면 · 격리 — 충족

위치: electron-builder.yml · preload.js · 기존 paths 정책

  • 표면 최소: 신규 IPC 는 단방향 push spip:elevation:warning 1개뿐. 새 invoke/handle 없음 → 신규 공격 표면 사실상 0.
  • per-user·위치 불변: nsis.perMachine:false(per-user) + allowToChangeInstallationDirectory:false(설치 위치 고정)로 설치 표면·경로 변동을 줄인다. 데이터는 기존대로 OS 앱데이터 폴더 0700·파일 0600 정책 유지(write 보류는 정책을 바꾸지 않고 write 시점만 건너뜀).
