P2 — 품질 개선 항목 (6)

#심각도무엇 / 위치왜 문제 · 권고
1P2 /api/open 응답이 code 프로세스 종료까지 대기
lib/server/actionHandlers.js:122 open()
lib/common/safeExec.js:223 (close에서만 resolve)		 safeExec는 child 'close' 이벤트에서만 Promise를 resolve한다. open()은 이를 await하므로, 응답은 "spawn 시작"이 아니라 code 프로세스가 종료된 뒤 전송된다. code 런처가 빠르게 분리·종료되는 환경에서는 문제없지만, 첫 실행/런처가 부모에 붙는 경우 응답이 지연되고 in-flight 카운터도 그동안 점유된다. 계약 §9 P2-3 "성공 = spawn 시작 성공", R-12 "2초 내 피드백"과 의미가 어긋날 수 있다.
권고: open 액션은 spawn 직후(또는 child 'spawn' 이벤트) 즉시 OPENING 응답을 반환하고, detach(unref()) 처리. 또는 safeExec에 detached/fireAndForget 모드를 추가해 액션 경로에서 사용. 현재 테스트는 code 미설치 환경이라 이 지연을 드러내지 못한다.
2P2 스캔 child 프로세스 정리: stdout 상한 초과 시 stderr는 무한 누적 가능
lib/common/safeExec.js:208-220		 stdout는 maxBuffer 초과 시 즉시 SIGKILL하지만, stderr 핸들러(216-219)는 상한 초과 시 누적만 멈추고 kill하지 않는다. stderr만 폭주하는 악성/이상 프로세스는 메모리 누적은 막히나 프로세스가 타임아웃까지 살아남아 자원을 점유한다. 비대칭 처리라 의도를 읽기 어렵다.
권고: stderr도 상한 초과 시 stdout과 동일하게 kill하거나, 두 경로를 공통 헬퍼로 묶어 대칭화. 의도가 "stderr는 살려둔다"면 주석으로 명시.

6P2 HEAD 요청을 GET처럼 처리 → 본문 전송 가능성
lib/server/router.js:77-80
lib/server/staticHandler.js:181-187 handle()		 router는 GET과 HEAD를 함께 staticHandler로 보내지만(77행), staticHandler는 메서드를 구분하지 않고 항상 res.end(buf)로 본문까지 전송한다. HEAD 응답에 본문을 실으면 HTTP 규약 위반이며 일부 클라이언트가 혼란을 겪는다. (보안 위험은 낮아 P2.)

확인 필요 (설계 의도 확인 요청)

  • open 응답 시멘틱(P2-1): "spawn 시작 성공 즉시 응답"(계약 P2-3 문구)으로 갈지, "프로세스 종료까지 대기 후 응답"(현 구현)으로 갈지 PM/tech-lead 확정 요청. 전자가 계약·R-12 정합.
