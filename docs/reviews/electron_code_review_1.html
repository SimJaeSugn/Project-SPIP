차단 사유(P1-1): public/app.js의 전송 어댑터가 ipc('open', id)를 호출하지만, preload.js는 해당 채널을 openInVsCode로만 노출한다. spip.open은 존재하지 않으므로 어댑터의 typeof spip[method] !== 'function' 가드가 항상 참 → bridgeMissing() 반환. 결과적으로 카드·표·드로어의 "열기" 버튼이 항상 "내부 오류"를 띄운다. (qa의 GUI 스모크에서 잡혀야 하나 코드 레벨에서 확정 가능.)

1. 모듈 경계 흐름(전송 vs 도메인)

전송/도메인 분리는 설계대로 지켜졌다. 다만 renderer→preload 채널명 계약이 한 곳(open vs openInVsCode)에서 어긋났고, 과도기 HTTP 핸들러(apiHandlers/actionHandlers)가 동일 로직을 이중 보유한다.

5. 잘한 점(유지)

  • 도메인/전송 분리: ipc/*는 Electron API를 import하지 않고 (args,ctx)→result 순수 함수로, register.js만 wiring·발신자 검증을 담당. 헤드리스 단위테스트 61건 전부 PASS.
  • scanController onProgress(F-1): 기존 _merge 무변경 + 세대 가드(_isActive)로 버려진 run의 늦은 콜백 차단, try/catch로 구독자 예외 격리. 회귀(미지정 호출) 테스트까지 포함 — 모범적.
  • 입력 재검증: actions.sanitizeOpenId/sanitizeRescanOpts가 renderer 입력을 main에서 재검증(키 화이트리스트·Boolean 강제·길이 상한). preload 1차 고정과 이중 방어.
  • folders 영속: 임시파일→fsync→rename, 0600, 기존 키 보존 병합, 실패 시 tmp/fd 정리 — 리소스 누수·부분쓰기 방어가 견고.
  • makeProgressSender: isDestroyed() 가드 + try/catch로 창 닫힘 레이스에서 안전. preload onScanProgress는 unsubscribe 함수를 정확히 반환.
  • L-1 일관: 프론트 전 경로/서버 문자열을 textContent로만 렌더, CSP 메타·헤더 이중주입 정합.
