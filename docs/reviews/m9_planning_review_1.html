M9 백로그 기능명세 — 기획 검증 보고서

검증 대상: docs/FeatureSpec-backlog-m9.html(신규 요구 R-25~R-33) · 기준 문서: docs/ref.md(원본 백로그 8) · docs/pm/codebase-audit.html(코드베이스 감사) · docs/traceability.html(요구 ID 체계).

종합 판정: CONDITIONAL PASS (조건부 통과)

치명적 결함 없음. 다만 추적성 역추적표 부재(P2)와 R-25 수용기준의 검증경계 모호(P2) 등 보완 권고 4건. 권고 반영 후 설계 착수 가능.

1. 종합 판정 및 요약

판정: CONDITIONAL PASS — 명세는 원본 백로그 8개를 빠짐없이 9개 요구로 분해했고, 감사가 지목한 공통 근원(전체 재렌더)을 기반 요구 R-25로 분리한 결정은 타당하다. ID 체계도 충돌이 없다. 그러나 아래 보완 권고(P2 3건·P3 1건)를 반영하면 설계 착수의 안정성이 높아진다. FAIL을 부를 치명적 결함(P1)은 발견되지 않았다.

회의적 검토 결과 한 줄 요약: "8→9 분해·의존성·보안 정합은 견고. 약점은 (1) ref↔요구 1:1 역추적표가 산문에만 있고 표로 없음, (2) R-25 기반 요구가 후속 전체를 막는 단일 병목인데 그 범위·완료 정의가 후속 요구들의 가정보다 느슨함."

2. 체크리스트별 판정

검증 항목판정근거
1. 완결성 (8항목 추적·누락·중복) PASS ref.md 8개 항목이 모두 요구로 매핑됨: 헤더통일→R-27, 실시간→R-31, 드래그→R-32, 차트→R-33, 메뉴제거→R-28, 단축키정보→R-29, 설정분할→R-30, 한글버그→R-26. 8→9 증가는 감사 교차이슈의 "공통 근원(전체 재렌더)"을 기반 R-25로 분리한 결과로 명시적 근거가 있음(누락·근거없는 신설 아님). 중복 없음.
2. 수용기준 검증가능성 PASS (보완) 대부분 검증가능(단위테스트·사용자 PC 1회 확인 명시, 헤드리스 한계 솔직). 다만 R-25의 "보존 영역 API"·"표준 훅"은 구현 형태 선택지를 또는으로 나열해 무엇이 충족인지 경계가 모호(P2-2). R-32 "워터폴 빈틈 없이"는 정량 기준 부재(P3).
3. 일관성 (의존성·MoSCoW 충돌) PASS R-25 기반 → R-26·R-31·R-32·R-33 의존 관계가 다이어그램·각 요구에 일관 기재. 우선순위 역전 없음(기반/Must가 Should/Could보다 선행). 구현순서 권고가 의존성 준수. R-28↔R-29 단일출처 공유, R-28·R-29→R-30 수용 관계 모순 없음.
4. 범위 명확성 (in/out) PASS §5 비범위 7항목 명시(가상DOM·단축키편집·설정검색·섹션추가삭제·신규데이터수집·프로젝트카드재배치·신규 npm 의존성). 범위 과대/과소 징후 없음. R-32가 기존 카드 reorder(R-19/R-23)와 명확히 구분됨.
5. 제약 정합 (보안·의존성) PASS L-1(textContent)·CSP(self-host·인라인 style 블록 금지)·런타임 의존성 최소(self-host 자산 별개 취급)가 §1 불변식 + 각 요구 수용기준에 반영. R-33이 감사의 "Chart.js canvas 안전 vs SVG 라이브러리 style-src 위험" 판단을 정확히 승계. 신규 IPC(R-32)의 guard·고정 에러코드·0600 영속 명시.
6. 사용자 스토리 품질 (역할·목적·이유) PASS (보완) 모든 요구에 "~로서 / ~하고 싶다 / 왜냐하면" 3요소 구비. 단 R-25 스토리는 역할이 "개발자/사용자"로 혼합되어 수혜자가 모호 — 기반 요구의 성격상 허용 가능하나, 사용자 가치 진술이 약함(P3).
추가. ID 체계 무결성 (R-25~R-33 충돌) PASS traceability.html 확인 결과 기능요구 최대값은 R-24, 비기능 N-07, 보안 V-2. R-25~R-33은 R-24 다음으로 충돌 없이 이어짐. 명세 헤더의 "R-24 / N-07까지 사용중" 기재가 정확. N-07(접근성) 승계 주장도 traceability N-07 행과 정합.

3. 반영 항목 (생산자 시정 요청)

치명(P1) 0건. 아래는 권고(P2·P3)로, 설계 착수를 막지는 않으나 반영을 권한다.

F-1  P2  ref↔요구 1:1 역추적표 부재

  • 무엇: 원본 백로그 8개 항목이 어느 요구로 가는지 명시적 매핑표(역추적 행렬)가 없다.
  • 위치: 명세 §1·§2(개요·인덱스) — 매핑이 §1 산문("UX 일관성… R-27" 식)에만 흩어져 있음.
  • 문제: 추적성은 이 저장소의 1급 산출물(traceability.html) 원칙이다. 산문 매핑은 빠뜨린 항목을 자동으로 드러내지 못해, 검증자/구현자가 8→9 분해의 완결성을 매번 수작업 대조해야 한다. 특히 ref 2번("모든 섹션 실시간")이 R-31 하나로 가는지, R-25에도 걸치는지 표가 없으면 모호.
  • 권고: ref.md 항목번호 → 요구ID(들) 8행 매핑표를 §2 근처에 추가하고, R-25는 "감사 교차이슈 #1 파생(원본 항목 아님)"으로 명시. 후속 traceability.html에 R-25~R-33 행을 추가할 때 그대로 재사용 가능하게.

F-2  P2  R-25 기반 요구의 완료 정의(DoD)가 후속 요구의 가정보다 느슨함

  • 무엇: R-25 수용기준이 "보존 영역 API: withPreservedNode 또는 destroy/recreate 표준화", "갱신 가드 공통화"처럼 구현 선택지를 OR로 나열하고, 무엇이 갖춰져야 R-25 완료인지(=후속이 의존할 수 있는 계약)가 확정되지 않았다.
  • 위치: R-25 "수용 기준" 4·5번 항목, "상세 기능 명세" 보존 영역 API 항목.
  • 문제: R-25는 R-26·R-31·R-32·R-33 4개가 동시에 의존하는 단일 병목이다. 그런데 R-32는 "SortableJS init/destroy 표준 훅", R-33은 "차트 destroy/recreate 훅", R-26은 "composition 가드"를 각각 R-25에서 제공받는다고 가정한다. R-25 완료 정의가 "또는"으로 열려 있으면, R-25를 만든 뒤에도 후속이 기대한 훅이 실제로 존재하는지 보장되지 않아 통합 단계에서 재작업 위험이 있다.
  • 권고: R-25 수용기준에 후속이 의존하는 공개 계약(API surface)을 확정 목록으로 박는다 — 예: ① shouldDeferRender(state) 순수함수, ② composition 단일 진입점(store._composing 세팅 보장), ③ 인스턴스 destroy/recreate 표준 훅(차트·Sortable 공용 시그니처), ④ coalesce 1회 보장. "또는" 표현을 줄이고, R-26/R-32/R-33의 수용기준이 참조하는 이름과 R-25가 제공하는 이름을 일치시킨다.

F-3  P2  R-31 ↔ R-26 의존이 다이어그램과 본문에서 어긋남

  • 무엇: 의존성 다이어그램(§3)은 R-31→R-26으로 점선("실시간이 IME 깨짐 악화")을 그리지만, 구현순서 권고와 R-31 본문은 "R-31은 R-26 이후 활성화"라고 한다. 또 R-26 수용기준 마지막 항목은 "R-31이 켜진 상태에서도…"를 검증 대상으로 둔다.
  • 위치: §3 다이어그램 점선 레이블 + §3 note 구현순서 + R-26 수용기준 5번 + R-31 의존성 칸.
  • 문제: 셋의 방향이 미묘하게 다르다. "악화 관계"(R-31이 R-26 문제를 키움)와 "활성화 순서"(R-26 먼저)와 "R-26이 R-31 켜진 상태를 테스트"가 섞여, 구현자가 어느 것을 먼저·무엇을 누구 수용기준에서 검증하는지 혼동할 수 있다. 특히 R-26 수용기준 5번은 R-31에 의존하는데 R-26은 R-25에만 depends로 표기됨(R-31 의존 누락 또는 순서 역전 가능).
  • 권고: 관계를 한 문장으로 단일화: "R-26은 R-25 위에서 단독 완결(R-31 불요). R-31 활성화의 선결 조건으로 R-26 완료를 둔다(R-31 depends: R-25, R-26)." R-26 수용기준 5번의 'R-31 켜진 상태' 검증은 R-31 수용기준으로 옮기거나, R-26에 'R-31 의존' 표기를 추가해 일관시킨다.

F-4  P3  정량 기준·스토리 수혜자 모호 (경미)

  • 무엇: (a) R-32 "워터폴/그리드로 빈틈 없이 재배치"에 정량/판정 기준 없음. (b) R-25 스토리 역할이 "개발자/사용자" 혼합.
  • 위치: R-32 수용기준 2번, R-25 스토리.
  • 문제: (a) "빈틈 없이"는 검증자가 통과/실패를 가르기 어려운 모호어 — 사용자 PC 확인으로 빠지면 회귀 검출 불가. (b) 기반 요구라 불가피하나 사용자 가치 진술이 약해 우선순위 방어 근거가 얇다.
  • 권고: (a) "재배치 후 컬럼 간 높이 편차가 가장 큰 섹션 1개 높이 이내" 같은 관찰 가능한 판정 기준, 또는 "masonry 레이아웃 적용 + 시각 확인(사용자 PC)"으로 한계를 명시. (b) R-25 스토리를 "사용자로서 … 입력/드래그/차트가 깨지지 않기를 원한다"로 사용자 가치 중심 1문장 + "개발 관점 근거"로 분리 서술.

4. 가장 위험한 약점 (Top 2)

① R-25 단일 병목 + 느슨한 완료 정의 (F-2). 9개 중 4개가 R-25에 의존하는데 R-25의 제공 계약이 "또는"으로 열려 있다. R-25를 잘못 좁게 만들면 R-26~R-33 통합 시 일괄 재작업이 발생하고, 너무 넓게 잡으면 비범위(가상 DOM) 경계를 넘는다. 설계 단계에서 R-25의 API surface를 먼저 확정하지 않으면 이 배치 전체의 일정 리스크가 한 요구에 집중된다.
② R-31×R-26 관계 진술의 비일관 (F-3). 악화/순서/교차검증 세 진술이 미묘하게 다른 방향을 가리켜, 구현자가 "R-26을 먼저 끝냈는데 R-31 켜니 또 깨진다"는 상황을 만들 여지가 있다. 의존 방향과 검증 소속을 한 문장으로 못 박아야 한다.
참고(검증 범위 밖): 두 약점 모두 설계 단계에서 해소 가능한 명세 수준 모호함이지, 기획의 완결성·추적성을 깨는 결함은 아니다. 따라서 종합 판정은 FAIL이 아닌 CONDITIONAL PASS다.

5. 사용자 판단 필요 (명세가 이미 식별)

명세 §6이 Q1~Q5(헤더 통일 방향·설정 카테고리·차트 폴링 주기·차트 구현 수단·도움말 버튼 존치)를 미결정으로 올리고 각각 기본 진행값을 제시했다 — 이는 적절한 처리다. 검증자는 이 5건을 추가 결함으로 보지 않으며, 기본 진행값이 모두 보안·디자인 근거를 동반해 미응답 시에도 진행 가능하다. PM이 Q4(Chart.js vs SVG 자작)만은 R-33 착수 전 확정 권고 — self-host/CSP 검증 부담이 수단에 따라 크게 갈리기 때문.
