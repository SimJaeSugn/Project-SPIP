요약

M7 설계는 두 사용자 요청(R-22 즐겨찾기 독립 토스트 창 · R-23 카드 reorder FLIP 애니메이션)을 추적 가능·실행 가능 수준으로 설계했고, 실측 코드(파일·함수·라인 인용)와의 정합이 대부분 정확하다. ADR-M6-5(오버레이) 번복 근거가 실측 모듈화(security.js 헬퍼·registerAppProtocol·register.js 창무관 guard)로 타당하게 뒷받침된다. PM 확정 4건(MQ7-1~5)도 본문·기본값에 반영됐다.

구현 착수를 막는 블로커는 없다(PASS). 다만 ① PM이 확정한 "blur+grace" 결정이 §4.1 의사코드에 미반영(grace 없는 즉시 hide) ② setFavorite broadcast가 setOrder/setSortMode와 비대칭(위젯-메인 동기화 범위 한정 근거 부족) — P2 2건. 그리고 ③ FLIP 키 속성이 실측 data-card-id와 설계 data-id 불일치 등 P3 3건은 구현 전 정정 권고.

반영 항목 (구현 전 정정 권고 — 블로커 아님)

ID우선무엇 / 위치 / 문제 / 권고
R1P2 무엇: PM 확정 "blur+grace"가 §4.1 핵심 의사코드에 미반영.
위치: m7-design.html §4.1 ensureWidget 의사코드(widget.on('blur', () => widget.hide())) vs §13 MQ7-1(기본 권고 "100ms grace + 명시 ×")·PM 확정(blur+grace+×/Esc).
문제: 설계 본문 의사코드는 grace 없는 즉시 hide다. §13 리스크 표·MQ7-1에서만 grace를 "옵션"으로 언급해, PM이 확정한 grace가 정규 설계로 승격되지 않았다. 구현자가 의사코드를 따르면 PM 결정과 어긋나고, 위젯 내 포커스 경합 시 의도치 않은 닫힘(리스크 표 자인) 그대로 노출.
권고: §4.1 의사코드를 "blur → setTimeout(hide, 100ms) + 위젯 재포커스 시 타이머 취소" 형태로 갱신하고, Esc·헤더 ×의 명시 hide 경로를 §4.3 창 계약(hide())에 함께 못박는다. MQ7-1을 "확정"으로 상태 갱신.
R2P2 무엇: broadcast 동기화가 setFavorite에만 적용, setOrder/setSortMode 비대칭.
위치: §6.3·§7. 신규 push는 spip:favorites-changed 1개뿐.
문제: 위젯은 favoriteViewModels를 쓰는데 이는 order/sortMode에도 영향받는다(실측 favoriteViewModels app.js:789, slider가 order 정렬 사용). 메인창에서 카드 순서를 바꾸면(setOrder) 위젯의 즐겨찾기 표시 순서가 stale될 수 있으나, 설계는 favorites 변경만 broadcast한다. "단일 원천 일관"이라는 §7 주장과 부분 충돌. MVP에서 위젯이 1장 슬라이딩(MQ7-4)이면 영향 작으나, 근거가 명시되지 않음.
권고: ① order 변경은 위젯 표시에 영향 없음(즐겨찾기는 집합이며 위젯은 자체 순서 사용)을 §7에 명문화하거나 ② 위젯 show마다 getUiState 재조회(§4.3 show 시퀀스 ④에 이미 언급된 경로)를 "stale 해소의 정규 수단"으로 확정. 비대칭을 의도된 범위 축소로 근거화.
R3P3 무엇: FLIP 키 속성 불일치 — 설계 data-id vs 실측 data-card-id.
위치: §9.2 의사코드(querySelectorAll(sel+' [data-id]')·elx.dataset.id) / §10 트리(line 568 "+ 카드 data-id") vs 실측 app.js:1870 card.dataset.cardId = vm.id(=data-card-id).
문제: 카드는 이미 data-card-id를 갖는데 설계는 신규 data-id 부여를 가정한다. 둘을 혼용하면 중복 속성·셀렉터 오류 가능. 설계가 기존 속성 존재를 인지·재사용하는지 불명.
권고: §9.2를 기존 data-card-id 재사용([data-card-id]·dataset.cardId)으로 통일하거나, data-id 신설 시 data-card-id를 대체함을 §10에 명시(중복 금지).
R4P3 무엇: preload.js onTray 축소가 실측 양면(preload 노출 + app.js 소비) 중 한쪽만 기술.
위치: §8.1 표(preload.js onTray를 dashboard만으로 축소).
문제: 실측 preload.js onTray(80~90행)는 ['dashboard','favorites'] 양쪽 구독이고, app.js의 onTrayCommand(2471)도 favorites 분기를 갖는다. 설계는 preload 축소만 언급하고 app.js onTrayCommand favorites 분기 제거(§8.2에서 별도 언급은 있음)와의 연결을 §8.1 표에서 누락. tray.js TRAY_ACTIONS는 favorites 유지라 3층(tray/preload/app)이 일시 불일치 상태가 됨.
권고: §8.1에 "preload onTray·app.js onTrayCommand favorites 분기를 함께 제거(또는 무시)하되 tray.js TRAY_ACTIONS는 호환 유지"를 한 줄로 명시해 3층 정합을 단일 표에 모은다.
R5P3 무엇: table 밀도 FLIP 미적용이 R-23 "카드 reorder" 요구 범위와 정합하나 명시 약함.
위치: §9.2 적용 범위·ADR-M7-4 결과.
문제: 실측 reorder(reorderByDrop/Move)는 cards·table 모두에서 트리거 가능한지 불명(renderCards만 확인, table 행 reorder 경로 미확인). table에 reorder가 있다면 애니메이션 누락이 UX 비대칭.
권고: 실측에서 table 밀도에 드래그 reorder가 존재하는지 확인 후, 존재하면 "MVP 미적용·후속"을 사용자 영향과 함께 명시(현 §9.2 cards만 표기는 R-19 §7 정합 주장에 의존하나 근거 링크 보강).

정합 대조 — 실측 확인 결과(통과 근거)

  • webPreferences 동등성: main.js:140~146 {contextIsolation:true,nodeIntegration:false,sandbox:true,webSecurity:true} — 위젯 §4.1과 1:1 일치. 정확.
  • 보안 헬퍼 재사용: security.js에 CSP_POLICY·applyCspHeaders·hardenWebContents·TRUSTED_ORIGIN='app://' 모두 실재 — 창 무관 재사용 가능(번복 근거 타당).
  • guard 창 무관성: register.js isTrustedSender가 app:// origin 정확 일치(접두 매칭 아님, EI-L-1) — "위젯도 분기 없이 통과" 주장 정확. guard 표면 불변.
  • app:// 매핑 이탈 차단: registerAppProtocol(main.js:98~124, normalize+rootWithSep 검사 108~113) — "새 페이지 추가만으로 동일 보호" 정확. 인용 라인(108~113) 정확.
  • doFinalQuit 단일 경로: main.js:260~264(disposeResources→tray.destroy→app.exit). 두 종료 분기(235·238) 모두 doFinalQuit 경유 → 위젯 dispose 1줄 추가 위치가 옳음(P2-2 단일 경로 불변).
  • setFavorite 반환 계약: uiState.js:47~65가 {ok:true,favorites} 반환 — §6.3 broadcast 래핑이 res.favorites 재사용하는 전제 정확.
  • 제거 대상·인용 라인: 필터바 슬라이더 버튼 1611~1618·onTrayCommand favorites 2473~2476·commitReorder render 2588·onShowFavorites 186~193 — 모두 실측과 정확히 일치.
  • ADR-M6-5 번복 정당성: m6-design.html:694~702 오버레이 결정·"보안 표면 2배" 우려가 실재 → 번복 맥락·근거 정확.
보안 상세는 security-reviewer 소관. 본 검증은 추적성 관점만 확인했다 — 위젯 6종 통제(webPreferences·app://·CSP·하드닝·senderFrame·textContent)와 broadcast 단방향이 추적표(§11)·§5에 연결돼 있고 실측 헬퍼가 존재함을 확인. 위젯 컨텍스트의 실제 보안 충분성·preload 6채널 노출면 적정성은 security 검증으로 넘긴다.

확인 필요 (PM/생산자 판단)

  • R2 관련: MVP 위젯이 "1장 슬라이딩"(MQ7-4 확정)이면 order broadcast 비대칭이 무해한지, 아니면 위젯도 메인 순서를 반영해야 하는지 — 사용자 기대 확인.
  • R5 관련: 실측 table 밀도에 드래그 reorder 경로가 있는지(코드 확인 결과 renderCards 외 table reorder 트리거는 미발견) — 있으면 FLIP 범위 결정 필요.

두 항목 모두 중대 쟁점은 아니며 P2/P3 반영으로 흡수 가능. 별도 AskUserQuestion 상신 없이 생산자 보완으로 처리 권고.
