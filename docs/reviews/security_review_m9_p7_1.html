setHomeLayout IPC · homeLayout 스키마 보안 검증 (R-32)

검증 대상: lib/common/uiStateStore.js(normalizeHomeLayout·normalizeState·defaultState) · electron/ipc/uiState.js(setHomeLayout·toResponse) · electron/ipc/register.js(guard) · electron/preload.js(setHomeLayout 노출) · test/uiStateStore.test.js·test/ipc-uistate.test.js. 기준: 보안 불변식(L-1·H-1·H-2·CSP·0600·guard senderFrame·단일 신뢰 경계) + 설계 보고서 M-1(m9_design_security_review_1.html). 적대적 관점 · 코드 미수정.

종합 판정: PASS — 치명·높음 0건 · 중간·낮음 0건 · 설계 M-1 시정 완료 확인

1. 요약

P7 데이터 계층 구현은 M9 설계 보안 검증에서 유일한 구현 게이트였던 M-1(homeLayout 영속 일관성)을 완전히 해소했다. normalizeState·defaultState·toResponse 3곳 모두 homeLayout을 다루며, write가 단일 정규화 게이트로서 키를 보존함이 라운드트립 테스트로 못 박혔다. 신규 IPC spip:setHomeLayout은 경로·실행·해시가 아닌 고정 enum(7개 섹션 id) 배열만 받고, normalizeHomeLayout이 유일 신뢰 경계로 항상 적용되어 적대적 입력(미지값·중복·비배열·비문자열·프로토타입 키)을 전부 흡수한다. guard senderFrame·0600 원자적 write·preload 최소 노출 모두 기존 패턴과 동형이며 새 위험 표면을 넓히지 않는다.

전체 테스트 668 pass / 0 fail(실행 확인). M-1 라운드트립·적대적 IPC 입력 케이스 포함 전부 그린. 치명·높음·중간·낮음 발견 0건.

2. 신뢰 경계 · 데이터 흐름 (P7 구현 검증)

신뢰 불가 입력 preload 타입 고정 메인 검증·영속

STRIDE: 신규 표면의 유일하게 의미 있는 위협은 변조(Tampering) — 임의 IPC 페이로드/손상 상태 파일. 권한상승(EoP)은 구조적 차단(섹션 id에 경로·실행 의미 0). 정보노출은 응답에 민감정보 0(homeLayout 배열만)으로 해당 없음. 변조는 normalizeHomeLayout이 화이트리스트 순열만 반환(외부 값 유입 0)해 흡수하며, 이중으로 write→normalizeState가 재적용한다.

3. 점검 결과

완료 3.1 — M-1 (normalizeState·defaultState·toResponse 3곳 일관) 검증 통과

점검 항목
설계 M-1이 지적한 "한 경로에만 화이트리스트 적용 시 부분 적용·검증 우회·데이터 소실" 패턴이 구현에서 제거됐는가.
확인 결과
① defaultState() — homeLayout: HOME_SECTION_IDS.slice() 포함(원본 배열 노출 방지로 slice 복제, 양호).
② normalizeState() — homeLayout: normalizeHomeLayout(obj.homeLayout)를 반환 객체에 포함. write가 저장 직전 무조건 normalizeState를 재적용하므로, 여기 누락 시 모든 쓰기가 키를 조용히 버렸을 것 — 실제 포함됨(C-M-1 주석 마커 병기).
③ toResponse() — homeLayout: state.homeLayout 포함. getUiState/setHomeLayout 응답 모두 노출.
④ 라운드트립 — write({homeLayout: custom}) → read() 후 동일 순서 보존을 test/uiStateStore.test.js "write/read — homeLayout 라운드트립 보존 (C-M-1)"가 실측. 키 침묵 폐기 = 부분적용 우회 패턴 없음 확정.
⑤ reconcileState() — normalizeState 거친 뒤 favorites/order만 손대고 나머지 키(homeLayout 포함)는 Object.assign 베이스로 보존. getUiState 재정리 경로에서도 homeLayout 소실 없음.
판정
3곳 모두 일관 처리 + 라운드트립 게이트 그린. 설계 M-1 시정 완료. 잔여 위험 없음.

완료 3.2 — 신규 IPC 입력 견고성 (적대적 페이로드 · 단일 신뢰 경계) 검증 통과

점검 항목
임의/악의적 입력(외부 enum 값·중복·비배열·거대 배열·비문자열·프로토타입 오염)에 견고하며, normalizeHomeLayout이 렌더러 검증에 의존하지 않고 항상 적용되는가.
확인 결과
· 미지/외부 enum 값 — HOME_SECTION_SET.has(id) 화이트리스트 통과만 push. 'bogus' 제거 확인(테스트).
· 중복 — seen Set으로 1회만. ['mail','mail'] → 단일 확인.
· 비배열/누락 — Array.isArray(input) false면 루프 미진입, 누락 보충 단계가 전체 기본 순서 복원. 'nope'·{}·undefined args 모두 기본 순서 반환(graceful, 테스트).
· 비문자열 혼입 — typeof id !== 'string' 거부. 숫자 7·42 제거 확인.
· 거대 배열(DoS) — 출력은 항상 HOME_SECTION_IDS 7개의 순열(고정 enum이 길이 상한 역할). 입력이 10만 개여도 화이트리스트 7개만 통과 → 결과 크기 고정. 단 입력 순회 비용은 입력 길이 비례(아래 I-2 선택 강화).
· 프로토타입 오염 — '__proto__'·'constructor' 등은 HOME_SECTION_SET에 없어 거부. 출력은 새 배열에 push만, Object.assign({}, state, {homeLayout})는 평범한 키 병합으로 오염 벡터 없음. 또한 read 경로는 _safeParse(깊이 32)·isPlainObject 가드 통과.
· 단일 신뢰 경계 — setHomeLayout 핸들러가 normalizeHomeLayout(args.ids)를 핸들러 진입 즉시 호출(렌더러/preload 검증과 무관하게 항상). preload의 ids.map(String)은 UX 편의 1차 고정일 뿐 보안 의존 아님. write 내부 normalizeState가 2차 재적용.
· guard — register.js가 guard('spip:setHomeLayout', …)로 등록 → senderFrame 정규 origin 정확 일치(EI-L-1) + try/catch 고정 {ok:false,code:'INTERNAL'}(스택 미노출). 기존 패턴과 동형.
판정
모든 적대적 벡터를 화이트리스트가 흡수. 외부 값 유입 0.

완료 3.3 — 신뢰 경계·표면 (preload 최소 노출 · 응답 · 0600) 검증 통과

확인 결과
· preload 노출 — setHomeLayout: (ids) => invoke('spip:setHomeLayout', {ids: Array.isArray(ids)?ids.map(String):[]}). 채널명 하드코딩, ipcRenderer 원본·범용 invoke 미노출, 단일 인자(섹션 id 배열). 새 위험 표면 안 넓힘 — 기존 setOrder와 동형.
· 응답 — {ok:true, homeLayout:[...]}. 경로·비밀번호·내부 상태·절대경로 0. 민감정보 노출 없음.
· 0600 원자적 write — setHomeLayout이 store.write 경유. write는 임시파일(wx,0600)→fsync→chmod→rename→chmod 유지. 변경 없음.
판정
최소 노출 유지 · 민감정보 0 · 0600 불변.

완료 3.4 — enum 안전 (인젝션 무관 · 화이트리스트 순열) 검증 통과

확인 결과
· HOME_SECTION_IDS = ['attention','productivity','activity','todos','mail','disk','featureAdd'] — 고정 식별자. 경로(pathGuard)·실행(safeExec)·해시·SQL·셸과 무관한 순수 표시 메타. 인젝션 무관(H-1/H-2 적용 대상 아님).
· normalizeHomeLayout 반환값은 항상 HOME_SECTION_IDS의 순열(7개, 누락 보충). "완전 재정렬 입력 보존" 테스트가 사용자 순서 보존을, "화이트리스트 외 제거 + 누락 보충" 테스트가 순열 불변식(r.sort() === HOME_SECTION_IDS.sort())을 검증. 외부 값이 저장/응답으로 새는 경로 0.
판정
고정 enum이라 인젝션 표면 없음. 화이트리스트 순열만 산출.

정보 I-1 — 비정상 페이로드의 침묵 흡수(탐지 공백) — 선택적 로깅

위협(STRIDE)
부인/탐지 공백 — 손상된 렌더러가 미지/대량 페이로드를 반복 전송해도 로그 신호 없음
평가
기존 setSortMode/setTheme의 graceful 폴백과 동형이라 일관적이고 보안 위험은 낮다(enum 화이트리스트라 결과는 항상 안전, guard가 1차 방어). 위험 등급 아님 — 정보.
권고(선택)
graceful 유지하되, normalizeHomeLayout 결과가 입력과 달라진 경우(미지/중복 제거 발생) logger.warn 1회 기록을 두면 침해 분석에 도움. 필수 아님.

정보 I-2 — preload 입력 길이 절단으로 순회 비용 0화 — 선택적 강화

위협(STRIDE)
서비스거부(DoS) — 렌더러가 거대 배열을 전달하면 메인이 전수 순회(결과 크기는 7로 고정이나 순회 O(n))
평가
로컬 단일 사용자·신뢰 origin guard 환경이라 실질 위협 거의 없음. 결과 크기는 항상 7개로 고정되어 메모리/저장 증폭 없음. 정보 등급.
권고(선택)
preload의 ids.map(String)에 .slice(0, HOME_SECTION_IDS.length) 같은 상한 절단을 더하면 순회 비용을 enum 길이로 고정. 또는 normalizeHomeLayout 입력 루프에 길이 가드. 필수 아님.
